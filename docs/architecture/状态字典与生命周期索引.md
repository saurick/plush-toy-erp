# 状态字典与生命周期索引 / Status Dictionary And Lifecycle Index

本文是 plush-toy-erp 状态体系的顶层设计与统一查找入口。它先定义状态应按哪些所有者分层、目标态如何演进，再回答当前实现中“canonical key 在哪里、由谁校验、怎样流转、页面去哪里取中文文案”。它不替代代码、Schema、migration、测试或目标环境运行证据。

判断 Workflow task、Process 与 Fact 的责任边界，继续阅读 [`状态工作流事实边界.md`](状态工作流事实边界.md)。判断角色、菜单和流程投影，阅读 [`../product/多甲方角色能力与流程编排.md`](../product/多甲方角色能力与流程编排.md)。

## 使用规则 / Reader Path

1. 先用顶层树确认对象属于 Source Document、Workflow、Process、Fact / Ledger、MasterData lifecycle 还是派生投影。
2. 用成熟度标记区分目标设计与当前证据：`[C] Current` 已有本地代码 / Schema 真源，`[P] Planned` 已正式评审但尚未完整实现，`[D] Deferred` 只保留长期边界，不预先登记 canonical key。
3. `Planned / Deferred` 可以进入顶层设计，但不能混入当前 API、数据库约束、页面选项或发布声明；进入实现前必须重新核对正式评审和当前仓库。
4. `Current` 的 canonical key、允许集合和流转判断以代码 / Schema 为准；本文负责架构、导航和证据索引。
5. 数据库 CHECK 只能证明落库形状；涉及权限、来源、并发、幂等和跨表事实的规则仍由 usecase / repo 负责。
6. 前端中文标签只是显示投影，不能反向决定允许状态、状态迁移或事实含义。
7. 判断目标环境当前状态时，还必须核对已发布 commit、migration、配置和运行证据；本地代码与本文不能单独证明已经部署。

## 状态字典树 / Status Dictionary Tree

以下两棵树结构一一对应：中文树用于业务和项目阅读，英文树用于代码检索与跨工具引用。`Current` 节点展示当前主要生命周期，`Planned / Deferred` 节点展示已评审演进或长期边界；两者都不替代下方索引指向的代码、Schema、migration 和测试。

### 中文状态字典树

```text
plush-toy-erp 状态体系
├─ [C] 源单据（业务承诺）
│  ├─ 销售订单：草稿 → 已提交 → 生效 → 已关闭 / 已取消
│  ├─ 采购订单：草稿 → 已提交 → 已批准 → 已关闭 / 已取消
│  ├─ 加工合同：草稿 → 已提交 → 已确认 → 已关闭 / 已取消
│  ├─ 生产订单：草稿 → 已发布 → 已关闭；草稿 / 已发布 → 已取消（受事实门禁约束）
│  └─ 物料清单生命周期：草稿 → 生效 → 已归档 / 已停用
├─ [C] 工作流（岗位协同）
│  ├─ 协同任务：待开始 / 可执行 / 处理中 / 阻塞 → 已完成 / 已退回 / 已取消 / 已关闭
│  └─ 业务进度投影：立项 → 资料 / 齐套 → 生产 → 质检 → 入库 → 出货 → 对账 / 结算
├─ [C] 流程运行时（流程编排）
│  ├─ 流程实例：运行中 → 已完成 / 已取消 / 已阻塞
│  ├─ 流程节点：等待中 → 运行中 → 已完成 / 已跳过 / 已失败 / 已阻塞
│  └─ 处理结果：节点或白名单领域命令的处理结果，不是通用单据状态
├─ [C] 业务事实与台账
│  ├─ 入库 / 退货 / 调整：草稿 → 已过账 → 已取消
│  ├─ 质检：草稿 → 已提交 → 通过 / 不通过 / 已取消
│  ├─ 出货：草稿 → 已出货 → 已取消
│  ├─ 生产 / 委外事实：草稿 → 已过账 → 已取消
│  ├─ 库存预留：生效 → 已释放 / 已消耗 / 已取消
│  ├─ 业务财务：草稿 → 已过账 → 已结清 / 已取消
│  ├─ 库存批次：可用 / 冻结 / 拒收 / 停用
│  └─ 订单行：开放 → 已关闭 / 已取消
├─ [C] 主数据生命周期（主数据可用性）
│  └─ 启用 / 停用 / 归档等对象专属状态；停用不等于删除历史引用
├─ [P] 已评审演进
│  └─ 当前没有已登记但尚未实现的新增 canonical 状态；能力接入计划不等于新增状态
└─ [D] 长期演进边界
   ├─ 质检明细、缺陷字典与抽检方案：待独立领域评审，不预设状态 key
   ├─ 生产在制、领料、返工与工艺路线：待生产 / MES 边界评审，不复用协同状态
   └─ 完整应收、应付、收付款、核销与会计台账：待财务领域评审，不由当前业务财务状态外推
```

### English Status Dictionary Tree

```text
plush-toy-erp status system
├─ [C] Source Document (business commitment)
│  ├─ sales_order: draft → submitted → active → closed / canceled
│  ├─ purchase_order: draft → submitted → approved → closed / canceled
│  ├─ outsourcing_order: draft → submitted → confirmed → closed / canceled
│  ├─ production_order: DRAFT → RELEASED → CLOSED; DRAFT / RELEASED → CANCELLED (fact-gated)
│  └─ BOM lifecycle: DRAFT → ACTIVE → ARCHIVED / DISABLED
├─ [C] Workflow (role collaboration)
│  ├─ task: pending / ready / processing / blocked → done / rejected / cancelled / closed
│  └─ business projection: project → engineering / material readiness → production → quality → inbound → shipment → reconciliation / settlement
├─ [C] Process Runtime (orchestration)
│  ├─ instance: active → completed / cancelled / blocked
│  ├─ node: waiting → active → completed / skipped / failed / blocked
│  └─ outcome: result of a node or allowlisted domain command, not a universal document status
├─ [C] Fact / Ledger (business fact)
│  ├─ receipt / return / adjustment: DRAFT → POSTED → CANCELLED
│  ├─ quality: DRAFT → SUBMITTED → PASSED / REJECTED / CANCELLED
│  ├─ shipment: DRAFT → SHIPPED → CANCELLED
│  ├─ production / outsourcing fact: DRAFT → POSTED → CANCELLED
│  ├─ stock_reservation: ACTIVE → RELEASED / CONSUMED / CANCELLED
│  ├─ finance_fact: DRAFT → POSTED → SETTLED / CANCELLED
│  ├─ inventory_lot: ACTIVE / HOLD / REJECTED / DISABLED
│  └─ source-document item: open → closed / canceled
├─ [C] MasterData lifecycle (master-data availability)
│  └─ object-specific active / inactive / archived states; deactivation does not delete historical references
├─ [P] Reviewed evolution
│  └─ no reviewed-but-unimplemented canonical status is currently registered; capability rollout does not imply a new status
└─ [D] Long-term evolution boundary
   ├─ quality items, defect catalog and sampling plans: separate domain review required; no status key reserved
   ├─ WIP, material issue, rework and routing: production / MES boundary review required; workflow states are not reused
   └─ full AR, AP, collection, payment, settlement and accounting ledger: finance review required; not inferred from current finance facts
```

读树时必须守住三条边界：Workflow task `done` 不等于 Fact `POSTED`；业务投影 `shipping_released` 不等于 shipment `SHIPPED`；Process node `completed` 只表示该节点按既定合同结算，领域事实仍以对应 usecase 和事实表为准。

## 目标态与演进登记 / Target Evolution Register

| 成熟度 | 方向 | 顶层设计边界 | 进入 Current 的条件 |
| --- | --- | --- | --- |
| Planned | 当前无新增 canonical 状态 | 已评审的页面、API、seed 或部署接入继续沿用现有状态，不为交付阶段另造业务状态 | 若正式评审新增状态，再登记 owning layer、key、迁移与测试 |
| Deferred | 质检明细、缺陷与抽检 | 属于 Quality Fact 扩展，不把 Workflow task outcome 当质检事实 | 完成 schema / usecase / API / UI 与状态迁移评审 |

## 分层语义概览 / Layer Semantics

| 层级 | 状态表达什么 | 不表达什么 |
| --- | --- | --- |
| Source Document | 订单、合同或计划从草稿到生效、关闭、取消的业务承诺生命周期 | 不证明库存、质检、出货或财务事实已经发生 |
| Workflow task | 某个责任角色的协同任务是否可做、处理中、阻塞、完成或终止 | `done` 不等于任何 Fact 已过账 |
| Workflow business projection | 跨岗位主链当前走到哪里，供看板和协同查询 | 不是库存、出货、应收应付的事实真源 |
| Process instance / node | 编排实例和节点 attempt 的运行、阻塞、结算与 outcome | 节点完成不自动代替领域命令的事实结果 |
| Fact / Ledger | 已发生并可审计、取消、冲正或调整的业务事实 | 不由页面显示状态或 Workflow payload 补造 |
| MasterData lifecycle | 主数据是否可继续用于新业务、是否停用或归档 | 停用不等于删除历史引用 |

同名 key 必须结合对象解释。例如 Workflow task 的 `rejected` 是协同任务终态，质检 `REJECTED` 是质检事实结论；Workflow business projection 的 `shipped` 只表示出货协同链路完成，实际出货仍以 shipment `SHIPPED` 及其库存事务为准。

## 前端显示合同 / UI Projection Contract

- 页面优先消费共享状态映射或模块页面配置，不在单页重复硬编码同一组 key。
- 未知 key 必须使用业务可理解的保守 fallback，不直接向业务用户暴露 schema key、RBAC、API 或原始异常。
- 客户配置可以调整低风险显示标签，但不能改变 canonical key、允许流转、权限或事实含义。
- Workflow business projection 的后端登记表与前端 `BUSINESS_STATUS_OPTIONS` 必须保持 key 集合一致；相关测试负责阻断遗漏和多余 key。
- 其它领域只有在确实存在跨页面重复或漂移风险时才抽取共享映射；不要为单个只读页面先造万能状态平台。

## 新增或修改状态 / Change Checklist

1. 明确 owning layer 和业务含义，避免复用同名状态掩盖 Workflow / Fact 或 Source Document / Fact 边界。
2. 在唯一代码常量、Schema enum / CHECK 或 catalog 中登记 canonical key；若已有真源可承接，不新增平行字典。
3. 在后端 usecase / repo 中锁住合法迁移、RBAC、owner / assignee、来源、事务、幂等、并发和终态规则。
4. 覆盖创建默认值、编辑或来源切换、取消/冲正、历史数据、列表/详情/搜索/打印/导出的缺值与残值路径。
5. 同步共享中文显示映射；只有用户可见行为受影响时才升级浏览器回归。
6. 补 unit、repo、service、PostgreSQL concurrency 或 browser tests 中真正对应风险的形态。
7. 在本文同步成熟度：设计评审通过后可进入 `Planned`；只有代码 / Schema、流转、测试和必要页面映射就绪后才进入 `Current`；长期方向保留为 `Deferred`，不预占 key。
8. 若对象类别、真源路径或长期职责发生变化，同步本文、架构 README、文档清单和相关正式产品文档；普通新增 key 不在多份 Markdown 复制整份枚举。

## 明确不做 / Non-goals

- 不建立跨所有领域的万能 `business_status` 表或可由客户上传任意状态机的配置系统。
- 不让前端标签、Workflow payload、seed、fixture 或历史文档成为状态真源。
- 不把本地测试绿色写成目标环境已 migration、已发布或客户已验收。
- 顶层设计允许表达经正式评审的目标态和长期方向；未实现内容必须标记 `Planned / Deferred`，不得混入 `Current` 证据或写成已交付能力。
