# 状态字典与生命周期索引 / Status Dictionary And Lifecycle Index

本文是 plush-toy-erp 状态体系的目标设计与统一入口。状态树只描述系统最终认可的对象、canonical key 和合法流转；旧 AI 草稿、本地实验、未发布字段及过渡方案不进入状态树，也不成为运行时合同。

判断 Workflow、Process 与 Fact 的责任边界，继续阅读 [`状态工作流事实边界.md`](状态工作流事实边界.md)。判断角色、菜单和流程投影，阅读 [`../product/多甲方角色能力与流程编排.md`](../product/多甲方角色能力与流程编排.md)。

## 设计规则 / Design Rules

1. 先确认状态属于 Source Document、Workflow task、Process Runtime、Fact / Ledger、MasterData lifecycle 还是派生投影；不同层即使同名也不得共用状态机。
2. canonical key、允许集合和流转判断必须有唯一代码或 Schema 真源；前端标签、Workflow payload、seed、fixture 和文档不能反向定义状态。
3. 状态树不表达实现成熟度。代码、versioned migration、目标环境发布和客户验收分别取证，不能混写成一个“已完成”。
4. 数据库 CHECK 只约束落库形状；权限、来源、并发、幂等、跨表事实和终态规则仍由 usecase / repo 守住。
5. 未知状态一律 fail closed：不映射成其他正式状态、不提供动作、不建立别名或双读分支。

## 状态字典树 / Status Dictionary Tree

以下两棵树结构一一对应。箭头表示合法转换，斜杠表示并列终点或分类；所有节点都是目标设计的一部分，不附带历史或计划标签。

│  ├─ 出货：草稿 → 已出货 → 已取消
│  ├─ 生产 / 委外事实：草稿 → 已过账 → 已取消
│  ├─ 库存预留：生效 → 已释放 / 已消耗 / 已取消
│  ├─ 业务财务：草稿 → 已过账 → 已结清 / 已取消
│  ├─ 库存批次：可用 / 冻结 / 拒收 / 停用
│  └─ 订单行：开放 → 已关闭 / 已取消

   └─ 对象专属的启用 / 停用 / 归档；停用不删除正式业务引用

### English Status Dictionary Tree

```text
plush-toy-erp status system
├─ Source Document (business commitment)
│  ├─ sales_order: draft → submitted → active → closed / canceled
│  ├─ purchase_order: draft → submitted → approved → closed / canceled
│  ├─ outsourcing_order: draft → submitted → confirmed → closed / canceled
│  ├─ production_order: DRAFT → RELEASED → CLOSED; DRAFT / RELEASED → CANCELLED (fact-gated)
│  └─ BOM: DRAFT → ACTIVE; DRAFT / ACTIVE → ARCHIVED; ARCHIVED → ACTIVE
├─ Workflow (role collaboration)
│  ├─ task: create → ready; ready → blocked / done / rejected; blocked → ready
│  └─ business projection: source-specific phase keys, not a universal linear lifecycle
├─ Process Runtime (orchestration)
│  ├─ instance: active → completed / blocked
│  ├─ node: waiting → active → completed / blocked
│  └─ outcome: result of a node or allowlisted domain command, not a universal document status
├─ Fact / Ledger (business fact)
│  ├─ receipt / return / adjustment: DRAFT → POSTED → CANCELLED
│  ├─ quality: DRAFT → SUBMITTED → PASSED / REJECTED / CANCELLED
│  ├─ shipment: DRAFT → SHIPPED → CANCELLED
│  ├─ production / outsourcing fact: DRAFT → POSTED → CANCELLED
│  ├─ stock_reservation: ACTIVE → RELEASED / CONSUMED / CANCELLED
│  ├─ finance_fact: DRAFT → POSTED → SETTLED / CANCELLED
│  ├─ inventory_lot: ACTIVE / HOLD / REJECTED / DISABLED
│  └─ source-document item: open → closed / canceled
└─ MasterData lifecycle
   └─ object-specific active / inactive / archived states; deactivation preserves formal references
```

读树时必须守住三条边界：Workflow task `done` 不等于 Fact `POSTED`；业务投影 `shipping_released` 不等于 shipment `SHIPPED`；Process node `completed` 只表示节点按合同结算，领域事实仍以对应 usecase 和事实表为准。

## 精确合同索引 / Canonical Contract Index

| 层级与对象 | 目标状态集合 | 创建与转换真源 | 约束与显示入口 |
| --- | --- | --- | --- |
| Workflow task | `ready / blocked / done / rejected` | `server/internal/biz/workflow_metadata.go`；公共创建固定为 `ready`；`ready → blocked / done / rejected`；`blocked → ready` | Workflow usecase / repo / JSON-RPC action、task / event schema、共享页面映射及直接 tests |
| Workflow business projection | 有受控生产者的来源专属阶段 key | 受控任务动作、Process Runtime 或领域 usecase 内部副作用；各来源不共享统一转换图 | `workflow_metadata.go` taxonomy、`workflow_business_states` 与前后端 key 合同测试 |
| Process instance / node | instance：`active / completed / blocked`；node：`waiting / active / completed / blocked` | Process Runtime 的创建、推进、claim、outcome 与幂等合同 | process schema、repo 事务 / CAS 与直接 tests |
| BOM | `DRAFT / ACTIVE / ARCHIVED` | 创建固定为 `DRAFT`；activate / archive 的显式转换 | `server/internal/biz/inventory.go`、BOM repo、schema 与直接 tests |
| 销售 / 采购订单与订单行 | 对应 lifecycle helper 的精确集合 | 对应 usecase 与 repo 的正式动作 | `server/internal/core/status/`、order / item schema CHECK 与直接 tests |
| 加工合同与合同明细 | 对应 outsourcing lifecycle 精确集合 | outsourcing order usecase / repo 的正式动作 | `server/internal/biz/outsourcing_order.go`、schema CHECK 与直接 tests |

| Fact / Ledger | 各事实对象的精确集合 | 对应领域 usecase 的 post / cancel / reversal / settle / ship | 各事实 schema、库存事务、页面共享映射与直接 tests |

## Workflow 任务合同 / Workflow Task Contract

公共创建只产生 `ready`。任务状态与合法转换只有：

```text
ready   → blocked / done / rejected
blocked → ready
```

- `done / rejected` 是终态；终态不重开。返工或重新协同时创建新任务或新节点 attempt。
- 同状态写入不是幂等；网络重试必须重放同一个 `idempotency_key` 对应的 receipt。
- `blocked / rejected` 必须提供非空原因；恢复要求新的解除说明并清理任务上的旧阻塞原因。
- `resume` 固定执行 `blocked → ready`。任务的 `business_status_key` 与 `workflow_business_states` 投影继续保持 `blocked`，后续业务阶段只能由来源 usecase 的受控动作显式推进。
- 任务看板只接受 `all / ready / blocked / rejected / done / overdue / dueSoon`。`overdue / dueSoon` 是到期派生筛选，不是 canonical 状态；不提供聚合状态别名。
- break-glass 只能绕过受审计的人员范围，不能绕过状态转换、版本、幂等、事务、Process Runtime 结算或 Fact 边界。

## 实现、迁移与发布证据 / Implementation, Migration And Release Evidence

| 证据层 | 结论 |
| --- | --- |
| 中央状态合同 | `workflow_metadata.go` 的 registry / transition 已按四个 Workflow task key 收口；它只证明中央允许集合与转换图。 |
| 跨层调用方与生成物 | **尚未收口**。Workflow repo / service / API / UI、seed / fixture / tests 与 Ent generated 仍需逐项同步，不能由中央合同外推。 |
| versioned migration | **尚未收口，未迁移**。手写 Schema 或 generated 代码不能证明 fresh / upgrade 数据库已获得目标 CHECK、字段或索引。 |
| 目标环境 | **未发布**。本地代码与定向测试不能证明目标环境已执行 migration、加载对应制品或完成业务验收。 |

### 一次性数据边界 / One-time Data Boundary

- 正式运行代码只识别目标集合；遇到集合外 key 必须明确失败，不读取、映射、双写或静默降级。
- 迁移前先做数据审计。非生产数据优先重置或重新 seed；若发现必须保留的正式业务记录，单独评审可审计的一次性转换，明确映射、人工确认、回滚点和失败条件。
- 一次性转换完成后删除临时工具或关闭入口，不把转换逻辑留在 usecase、repo、API、UI 或查询筛选中。
- 当前只能说“中央 registry / transition 合同已收口；跨层调用方、Ent generated 与 versioned migration 尚未收口；目标环境未发布”。

## 分层语义 / Layer Semantics

| 层级 | 状态表达什么 | 不表达什么 |
| --- | --- | --- |
| Source Document | 订单、合同或计划从草稿到生效、关闭、取消的业务承诺生命周期 | 不证明库存、质检、出货或财务事实已经发生 |
| Workflow task | 某个责任角色的协同任务是否可执行、阻塞、完成或退回 | `done` 不等于任何 Fact 已过账 |
| Workflow business projection | 某个来源对象投影到哪个协同阶段 | 不是统一状态机，也不是库存、出货或财务真源 |
| Process instance / node | 编排实例和节点 attempt 的运行、阻塞、结算与 outcome | 节点完成不替代领域命令的事实结果 |
| Fact / Ledger | 已发生并可审计、取消、冲正或调整的业务事实 | 不由页面显示状态或 Workflow payload 补造 |
| MasterData lifecycle | 主数据是否可继续用于新业务、是否停用或归档 | 停用不删除正式业务引用 |

## 前端显示合同 / UI Projection Contract

- 页面消费共享状态映射或模块配置，不在单页复制另一套允许集合。
- 未知 key 显示业务可理解的“状态不可用”，同时隐藏动作；不得映射成最近似状态。
- 客户配置可以调整低风险显示标签，不能改变 canonical key、流转、权限或事实含义。
- Workflow task 动作按精确合同投影：`ready` 可完成、阻塞、退回或催办，`blocked` 只可恢复或催办，`done / rejected` 只读。

## 新增或修改状态 / Change Checklist

1. 明确 owning layer 与业务含义，避免复用同名状态掩盖 Workflow / Fact 或 Source Document / Fact 边界。
2. 在唯一代码常量、Schema enum / CHECK 或 catalog 中登记 canonical key，不新增平行字典。
3. 在后端 usecase / repo 中锁住合法迁移、RBAC、owner / assignee、来源、事务、幂等、并发和终态。
4. 同步 versioned migration、共享显示映射和真正对应风险的 unit / repo / service / PostgreSQL / browser tests。
5. 分别记录代码、迁移、发布和验收证据；任一层缺失都不得外推。

## 明确不做 / Non-goals

- 不建立跨所有领域的万能 `business_status` 表或允许客户上传任意状态机的配置系统。
- 不让前端标签、Workflow payload、seed、fixture 或旧文档成为状态真源。
- 不在状态树中登记旧 AI 方案、实现成熟度或临时技术草案。
- 不为目标集合外的 key 建立别名、双读、双写、聚合筛选或永久 fallback。
- 不把本地测试绿色写成已迁移、已发布或客户已验收。
- Source Document 与 Fact 中正式定义的关闭、取消、冲正语义属于其目标生命周期，不得与 Workflow task 状态混用。
