# 产品 / SKU / BOM 边界评审 / Product / SKU / BOM Boundary Review

## 结论

现有 `products / product_skus / materials / units / warehouses / bom_headers / bom_items` 已经是当前主数据与工程资料真源。BOM version 和 SKU 粒度只能扩展这些真源，不得重复设计 `products / product_skus / materials / units / bom_*`，也不得把 永绅 yoyoosun 客户 Excel 字段直接写成 Product Core 必填字段。

```text
BOM 是工程资料 / 物料需求来源。
BOM 不是库存事实。
BOM 不能直接改库存余额。
采购需求可以从 BOM 派生，但采购事实必须由采购单 / 入库单产生。
```

## 现有真源

| 对象 | 当前状态 | 口径 |
| --- | --- | --- |
| `units` | 已有 Ent schema / migration / relation | 单位主数据，库存、BOM、采购事实共用 |
| `materials` | 已有 Ent schema / migration / relation | 物料主数据，包含 code、name、category、spec、color、default_unit |
| `products` | 已有 Ent schema / migration / relation | 成品主数据，包含 code、name、style_no、customer_style_no、default_unit 和可选产品单重（净重） |
| `product_skus` | 已有 Ent schema / migration / relation | 成品规格主数据，可选维护 SKU 单重；一旦维护单重必须同时有显式 SKU 默认单位 |
| `warehouses` | 已有 Ent schema / migration / relation | 仓库主数据，不属于本文件新增范围 |
| `bom_headers` | 已有 Ent schema / migration / relation | BOM 头，按 product + version，单产品最多一个 ACTIVE |
| `bom_items` | 已有 Ent schema / migration / relation | BOM 明细，关联 material / unit，有 quantity 和 loss_rate |
| `inventory_lots` | 已有事实/追溯表 | 批次追溯，不是 SKU 主数据 |

## 是否需要 product_skus

`product_skus` 已进入 V1 schema 主路径，作为产品可售 / 可库存规格主数据真源。显式 SKU 已贯通销售订单行、库存批次 / 流水 / 余额、生产 / 委外事实、出货和预留；这不代表所有产品必须有 SKU，也不代表 BOM 已强制切到 SKU。

SKU 可能承接：

- 颜色。
- 尺寸。
- 包装版本。
- 按具体规格维护的可选净重。
- 客户款号。
- 客户 SKU。
- 条码 / 外部编码。
- 图片或样图引用。

SKU 不应承接：

- 库存事实。
- 出货事实。
- 客户专属报表临时列。
- BOM 明细事实。
- 永绅 yoyoosun 样本中未通用化的任意 Excel 列。

## SKU 与产品编号、款式、颜色、尺寸关系

| 概念 | 推荐层级 | 说明 |
| --- | --- | --- |
| product.code | Product Core | 产品资料编号 / 内部产品编号，现有 `products.code` |
| product.style_no | Product Core 可选 | 款式编号，现有 `products.style_no` |
| product.customer_style_no | Product Core 可选 | 客户款号，现有 `products.customer_style_no` |
| product.unit_net_weight_kg | Product Core 可选 | 按产品默认单位记录的单件净重，统一以 kg 保存；未知为 NULL，非空必须大于 0 |
| product_sku.sku_code | Product Core | 具体可销售 / 可生产规格编码，现有 `product_skus.sku_code` |
| product_sku.unit_net_weight_kg | Product Core 可选 | 按 SKU 显式默认单位记录的规格净重，统一以 kg 保存；填写单重时 `default_unit_id` 必须同时非空，未填单重才允许下游回退产品单重 |
| color / color_no | SKU / lot 分层 | 标准颜色可进 SKU；染批、批次色号属于 lot |
| size | Product Core | 如果同一 product 有多尺寸规格，放 SKU |
| version | BOM / SKU 分层 | 工程版本优先放 BOM version；销售包装版本可放 SKU |
| image / attachment | 文档 / media 引用 | 不进入库存事实；V1 可先做引用字段草案 |

## 单重与出货快照

单重真源和历史出货事实分层处理：

- `products.unit_net_weight_kg` 以产品 `default_unit_id` 为计量基准。
- `product_skus.unit_net_weight_kg` 是显式 SKU 覆盖；填写时必须同时维护 SKU `default_unit_id`，不从产品默认单位暗中补齐 SKU 单重基准。SKU 未填单重时，下游才回退产品单重和产品默认单位。
- 出货单从 `DRAFT` 确认为 `SHIPPED` 时，后端在同一事务内解析每条有效行的单重和计量基准，冻结到 `shipment_items.unit_net_weight_kg_snapshot`。
- 全部有效行都能解析时，后端使用 decimal 累加 `quantity × unit_net_weight_kg_snapshot`，将最终结果写入 `shipments.total_net_weight_kg`。任一有效行缺值、基准单位不匹配或不可解析时，不生成部分合计。
- `shipments.total_net_weight_kg` 也允许在无法完整自动计算时保留可空人工实际总净重；第一版不因单重或总净重缺失阻断出货。
- 已确认出货的行快照和总净重不随产品 / SKU 主档后续修改，不对历史记录回填；取消出货时保留原快照和总净重，由库存 `REVERSAL` 表达冲正。
- 毛重、单位换算、装箱 / 物流 / 打印、报关和提单仍不在这一版范围。

## BOM 按 product 还是 sku

当前已实现的 BOM 是按 `product_id`。早期 V1 推荐：

- V1 保持现有 product-level BOM 作为主路径。
- 如果同一产品不同颜色 / 尺寸的材料用量、颜色、包装差异会影响采购需求，则评审 BOM header 可选 `product_sku_id`。
- 在未证明 SKU 影响 BOM 前，不把 BOM 强制改为 SKU 粒度。
- 不用 BOM 改版直接修改历史采购或库存事实。

## BOM 改版

现有 `bom_headers.version / status / effective_from / effective_to` 已能表达基础版本，当前已接入 `bom` JSON-RPC、`bom.*` RBAC 和 `/erp/purchase/material-bom` V1 页面。BOM Version 当前主路径是：

- 草稿 `DRAFT` 可维护头信息和明细。
- `ACTIVE` 版本唯一性保持，同一产品激活新版本时将旧 `ACTIVE` 设为历史版本（底层状态 `ARCHIVED`）。
- 已激活 BOM 不允许直接改头或明细；改版应复制新 `DRAFT` 后再激活。
- 已被订单 / 采购需求引用的 BOM 后续应保留版本快照。
- BOM 改版影响未来需求，不回写已过账库存、采购入库或质检事实。
- 当前不改变 schema，不加入 `product_sku_id` 粒度，不生成采购需求 / MRP，不处理替代料或成本核算。

## 材料替代

材料替代不建议 V1 必做。原因：

- 会影响采购需求、可用库存、质检和成本。
- 需要替代规则、优先级、有效期、客户限制和审计。
- 如果替代会影响库存和采购事实，必须进入 Product Core 架构评审。

## BOM 到采购需求

BOM 可以作为采购需求来源：

```text
sales_order_item
-> product / product_sku
-> active BOM version
-> material demand
-> purchase demand / purchase order candidate
```

但采购事实必须由采购单或入库单产生：

```text
purchase_order = 采购承诺。
purchase_receipt = 采购入库事实。
inventory_txns = 库存落账事实。
```

BOM 不写 `inventory_txns`，不更新 `inventory_balances`，不决定 `purchase_receipts` 是否 posted。

## product_skus 当前落地边界

结论：`product_skus` 已作为 V1 Product Core 主数据落地，并完成显式 SKU 的订单来源和库存事实 grain 闭环。

当前已完成：

- 新增 `product_skus` Ent schema / Atlas migration / generated code。
- `product_skus` 关联 `products`，可选关联 `units` 作为 SKU 默认单位。
- MasterData JSON-RPC / RBAC / UI 已支持 SKU 创建、编辑、查询、启停；已被业务事实引用的 SKU 不能改挂产品。
- SKU 可选维护 `unit_net_weight_kg`；填写时必须同时有显式 `default_unit_id`，不以产品单位暗中补齐。
- `sales_order_items / inventory_lots / inventory_txns / inventory_balances / production_facts / outsourcing_facts / shipment_items / stock_reservations` 使用 nullable `product_sku_id`；显式值参与来源校验、库存 grain、幂等、锁、扣减和冲正。
- 保留 `product_id` 主路径；`product_sku_id=NULL` 是独立的未分规格产品库存，不自动回填、不与任一 SKU 共池，也不按颜色 / 尺寸快照猜测归类。
- 库存台账可按 SKU 筛选并区分具体规格与未分规格库存；真实 PostgreSQL 已验证四组 SKU / lot partial unique index、规格池隔离和并发预留。

当前未完成：

- BOM header 是否支持 `product_sku_id`。
- 从真实客户导入或订单颜色自动创建 SKU。
- 旧未分规格库存的人工重分类 / 调账工具。

## 字段分类

| 字段 / 资料 | 分类 | 推荐落点 |
| --- | --- | --- |
| product code、name、style_no、customer_style_no、default_unit、unit_net_weight_kg | Product Core | 现有 `products`；单重按产品默认单位记录，默认单位变化后必须重新确认 |
| material code、name、category、spec、color、default_unit | Product Core | 现有 `materials` |
| BOM version、status、effective dates、quantity、loss_rate | Product Core | 现有 `bom_headers / bom_items`，按需扩展 |
| sku_code、barcode、sku_name、color、size、packaging_version、default_unit、unit_net_weight_kg | Product Core | `product_skus`；SKU 单重只以显式 SKU 默认单位为基准 |
| 色卡、作业指导书、图片 | Industry Template Candidate / Attachment | 先做引用和模板，不塞进库存事实 |
| yoyoosun Excel 中的设计师、特殊颜色描述、客户内部列名 | Customer Material | 作为线索，不直接必填 |
| 字段显示、必填、编号规则 | Customer Config | 配置草案 |
| 染批、供应商批号、生产批号 | Fact / Lot | `inventory_lots`，不放 SKU 主数据 |

## 明确不落地

- 不重复设计 `products / materials / units / warehouses / bom_headers / bom_items`。
- 不让 BOM 直接写库存事实。
- 不把 `product_sku` 写成所有产品必填的库存维度；`product_id` 仍是必填主体，`product_sku_id` 可空，但显式提供时属于 exact inventory grain；`NULL` 是独立未分规格库存，不自动回填、不与任一 SKU 共池。
- 不把 永绅 yoyoosun 客户 Excel 列直接写成 Product Core schema。
- 不做材料替代 V1 强制能力。
