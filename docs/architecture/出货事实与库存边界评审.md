# 出货事实与库存边界评审 / Shipment / Inventory Boundary Review

> 结论：当前真实出货事实已由 `shipments / shipment_items`、`OperationalFactUsecase` 和 `operational_fact` JSON-RPC 承接，`ship_shipment` 会把出货单推进为 `SHIPPED` 并写库存 `OUT`，`cancel_shipment` 会对已出货单写 `REVERSAL` 冲正，重复发货和重复取消保持幂等。第七条后端 workflow usecase 仍只把 `shipment_release done` 收口为“允许出货 / 出货放行”即 `shipping_released`，不直接等同 `shipped`。

## 1. 本轮范围

| 项 | 结论 |
| --- | --- |
| 是否改 Go / JS 运行时代码 | 当前已存在出货事实运行时代码；本文仅同步边界口径。 |
| 是否实现 `shipment_release` 后端 rule | 已实现最小协同状态规则：`done -> shipping_released`，`blocked / rejected -> blocked`。 |
| 是否写 `inventory_txns / inventory_balances / inventory_lots` | `shipment_release` 不写；真实 `ship_shipment` 写库存 `OUT` 并更新余额，`cancel_shipment` 写 `REVERSAL`。 |
| 是否新增 reservation 表 | 已存在 `stock_reservations` 最小表，但出货单发货主路径仍按实际 `OUT` 防负库存，不把放行任务当预留事实。 |
| 是否改 Ent schema / 生成 migration | 本次文档同步不改、不生成。 |
| 是否迁应收 / 开票 / 应付 / 对账 | 当前最小门禁是 `RECEIVABLE / INVOICE` 必须引用 `source_type = SHIPMENT` 且对应 shipment 已 `SHIPPED`；完整财务闭环仍待后续评审。 |

当前出货事实不是由 workflow payload 补造，而是由出货单主路径写入。`server/internal/data/operational_fact_repo_test.go` 覆盖出货单、出货明细、发货、取消冲正、重复发货和重复取消；库存相关测试集中在 `server/internal/data/inventory_repo_test.go` 和 `server/internal/data/inventory_postgres_test.go`。

## 2. InventoryUsecase 当前能力

| 能力 | 当前事实 | 评审 |
| --- | --- | --- |
| 流水语义 | `server/internal/biz/inventory.go` 定义 `IN / OUT / ADJUST_IN / ADJUST_OUT / TRANSFER_IN / TRANSFER_OUT / REVERSAL`。 | 已有出入库、调整、转移和冲正语义；没有单一 `ADJUST`，而是拆成入 / 出方向调整。 |
| 库存事实不可变 | `inventory_txns` 字段 immutable，hook 拒绝 update / delete。 | 符合库存事实追加模型，错误应走 `REVERSAL`。 |
| 余额并发更新 | `ApplyInventoryTxnAndUpdateBalance` 在 DB transaction 内新增流水并更新余额；入库走 upsert 累加，出库走条件更新。 | 可以防止并发出库把账面余额扣成负数。 |
| 防负库存 | 出库类 delta 使用 `UPDATE ... WHERE quantity + delta >= 0`，失败返回 `ErrInventoryInsufficientStock`。 | 已有账面余额防负库存。 |
| decimal 精度 | 数量字段使用 `decimal.Decimal`，PostgreSQL schema 为 `numeric(20,6)`。 | 不使用 float；后续 shipment 行仍需定义单位换算、舍入和小数位策略。 |
| 批次 | `inventory_lots` 支持 `subject_type + subject_id + lot_no`，`inventory_txns / inventory_balances` 均有 nullable `lot_id`。 | 已支持材料和成品批次维度；非批次库存和批次库存分开扣减。 |
| 冲正 / 取消基础 | `REVERSAL` 要求原流水、主体、仓库、单位、数量匹配且方向相反；同一原流水只能冲正一次。 | 有通用冲正原语；采购入库 / 采购退货 / 调整和 shipment 取消均已走专用用例。 |
| 幂等 | `idempotency_key` 唯一；重复提交返回已有流水。`source_type/source_id/source_line_id` 有索引。 | 出货发货 / 取消库存流水使用稳定幂等键：`SHIPMENT:<shipment_id>:<shipment_item_id>:POST / REVERSAL`；重复发货和重复取消不会重复写库存流水。 |
| 仓库 | `warehouse_id` 是流水和余额必填维度。 | 支持仓库维度。 |
| 库位 | 未落 `warehouse_locations`，流水和余额无 `location_id`。 | 不能做精细库位扣减。 |
| 主体区分 | `subject_type` 支持 `MATERIAL / PRODUCT`；业务层校验 MATERIAL 指向 `materials.id`，PRODUCT 指向 `products.id`。 | 材料库存和成品库存可以共用同一套流水 / 余额 / 批次模型。 |
| 可用 / 预留 / 冻结 | 当前已存在 `stock_reservations` 最小表，库存可用量按账面余额扣 active reservation 计算；冻结仍主要由批次状态和后续 hold 规则承接。 | 出货执行可以防负库存；严格的放行即预留、冻结释放和多订单承诺策略仍需单独评审。 |

结论：`InventoryUsecase` 可以作为出货扣减的底层库存事实能力，真实扣减已由出货单主路径调用；仍不能直接从 `shipment_release` payload 写真实出货扣减，因为放行任务不是出货行真源。

| 字段 | 当前情况 | 结论 |

| `sku_id` | `shipment_items` 已有 nullable `product_sku_id` 追溯位，当前出货主路径仍保留 `product_id`。 | SKU 粒度出货校验和自动带值仍待后续评审。 |

| `quantity` | `shipment_items.quantity` 是最终出货行数量。 | 已具备 decimal 数量；部分出货、短溢装和更细规则仍需后续评审。 |
| `unit_id` | `shipment_items.unit_id` 写入库存流水。 | 已具备。 |

| `customer_id` | `shipments.customer_id` 可选追溯客户，`customer_snapshot` 可做展示快照。 | 已具备 nullable 追溯位；财务完整对账仍需后续评审。 |

## 4. `shipment_release done` 与 `shipped`

`shipment_release done` 不应默认等于 `shipped`。

| 语义 | 必要事实 | 当前是否具备 |
| --- | --- | --- |
| 允许出货 / 已放行 | 放行人、放行时间、放行原因、财务或业务限制、可用量检查结果。 | 只具备部分展示和任务完成信息。 |
| 已出货 / 已扣库存 | 出货单、shipment line、产品 ID、单位 ID、仓库 / 批次、出货数量、发生时间、库存 `OUT` 流水。 | 已由 `shipments / shipment_items` + `ship_shipment` 最小具备；经手人等审计字段仍待后续。 |

旧前端 v1 曾在 `MobileRoleTasksPage.jsx` 中把 `shipment_release done` 本地推进为 `shipped`，并继续创建应收任务；这个语义过重。本轮已移除真实移动端运行时的这条 follow-up。更合理且当前已采用的拆分是：

1. `shipment_release done -> shipping_released`：表示允许出货，不写库存。
2. `ship_shipment -> SHIPPED + inventory OUT`：表示实际出库完成，写库存流水和余额。
3. `shipped -> receivable / invoice`：实际出货后再进入应收 / 开票。

当前已迁的 `shipment_release` 后端 rule 只做协同状态：`done -> shipping_released`，`blocked / rejected -> blocked`，强制 reason，不创建 DerivedTask，不写库存，不派生财务。

## 5. 可用量、预留和冻结

`available_qty = on_hand - reserved - frozen` 作为查询口径成立，但前提是三类量都有明确真源。

| 量 | 建议真源 | 当前状态 |
| --- | --- | --- |
| `on_hand` | `inventory_balances.quantity`，可由 `inventory_txns` 聚合校验。 | 已有。 |
| `reserved` | `stock_reservations` 或等价预留事实表，按来源行占用。 | 已有最小基础。 |
| `frozen` | `stock_holds / stock_reservations` 或明确冻结事实表，区分质检、财务、客户、异常冻结。 | 未落独立冻结事实。 |
| `available` | 查询时计算，或余额表冗余并用一致性检查守护。 | 已有最小纯计算口径；完整页面和业务承诺策略仍待后续。 |

出货放行如果要承诺“可发”，应先 reserve，而不是仅检查当前账面余额。否则两个订单同时通过放行检查，最后只有先扣库存的一单成功，另一单会在执行时失败。

财务未放行是否 frozen，要看业务定义：

- 如果财务未放行是“订单不可出货”，应冻结订单对应库存或拒绝创建 reservation。
- 如果财务未放行只是任务状态，不应污染库存冻结量。
- 冻结必须有来源、释放条件和审计，不应藏在 workflow payload 里。

多订单抢同一批库存时，推荐由 reservation 表在同一 DB transaction 中原子占用，并用 active reservation 聚合防超占。出货取消释放 reservation；出货完成把 reservation 消耗为 `OUT`，并关闭 reservation。没有 reservation 表时，只能在实际 `OUT` 时防负库存，不能做严格可用量承诺。

## 6. 三种方案评审

| 方案 | 评审 | 结论 |
| --- | --- | --- |
| A：`WorkflowUsecase` 不写库存，只触发 `InventoryUsecase.Outbound` | 技术上可以调用库存扣减，但 workflow event、业务状态和库存流水必须在一个清晰 transaction 边界内一致。当前 `WorkflowUsecase` 与 `InventoryUsecase` 的 repo transaction 没有统一外层业务事务接口，且 shipment payload 缺出货行事实。 | 不作为首选；容易把协同任务变成库存事实入口。 |
| B：独立出货事实 usecase / `InventoryUsecase` 先完成出货事实，再由 workflow 只读结果 | 当前由 `OperationalFactUsecase` 承接 shipment document、出货行、执行、取消和 `OUT / REVERSAL`；`InventoryUsecase` 负责库存事实原语；`WorkflowUsecase` 只推进协同状态或读事实结果。 | 已按该边界落最小实现；后续可再评审是否拆出独立 `ShipmentUsecase`。 |
| C：前端继续直接创建财务 / 出货任务 | 当前前端 v1 能跑展示链路，但会绕过后端事实边界，且已与新项目后端 usecase 迁移方向冲突。 | 不推荐。后续迁入后应移除真实运行时 follow-up，避免双写。 |

当前已按方案 B 的边界落最小实现：出货事实由 `OperationalFactUsecase` 承接，库存扣减由库存事实原语执行，`WorkflowUsecase` 只承担 `shipment_release` 的协同状态推进，不直接承担库存扣减和财务派生。

## 7. 应收 / 开票前置条件

应收建议在 `shipped` 后生成，而不是 `shipment_release` 后生成。`shipment_release` 只是放行时，客户债权通常还未成立；只有实际出货完成后，才具备应收基础。

开票不应只依赖 `shipment_release done` 或 `receivable_registration done` 任务状态。至少需要：

- 客户 ID、销售订单 / 合同、shipment / outbound 行。
- 出货数量、单价、金额、税率、含税 / 不含税金额。
- 开票申请、发票类型、发票号、红冲 / 作废边界。
- `sales_order / shipment / ar_receivable / ar_invoice` 或等价专表真源。

当前 `shipmentFinanceFlow.mjs` 是前端 v1 任务流：`shipped -> receivable_registration -> invoice_registration`。它可以继续作为展示、调试或临时手动入口辅助，但不应作为财务事实真源。

## 9. 下一轮最小落地建议

推荐按下面顺序推进：

1. 继续保留 `shipment_release` 与 `ship_shipment` 的边界：放行任务只写 `shipping_released`，真实出货只从出货单主路径执行。
2. 评审是否把当前 `OperationalFactUsecase` 中 shipment 子域拆成独立 `ShipmentUsecase`：只有在 reservation、经手人、装箱物流、签收、退货或完整财务联动复杂度上来后再拆。
3. 补 reservation / frozen 的业务策略：至少明确 `on_hand / reserved / frozen / available` 的来源、释放条件和并发策略；不要让 `shipment_release` payload 充当预留或冻结真源。
4. `shipment_release` 最小协同状态已迁：只做 `done -> shipping_released`、`blocked / rejected -> blocked`，并已移除移动端真实运行时本地 `shipped` 和应收派生 follow-up。
5. 已出货和取消冲正继续通过 `ship_shipment / cancel_shipment` 写 `OUT / REVERSAL`；后续若接装箱、物流、签收或退货，需要单独定义状态和事实表边界。
6. 最后再扩完整应收 / 开票：基于 `SHIPPED` 和正式 shipment / receivable / invoice 事实推进，不从 `shipment_release` 直接派生。
