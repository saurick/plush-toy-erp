# 业务事实扩展总评审 / Operational Fact Expansion Review

- 文档类型：架构评审 / Architecture Review
- 状态：业务事实扩展 统一 review、本地最小实现闭环；2026-06-08 目标环境发布、登录态只读 API smoke 和内部模拟事实写入仅作为历史基线 evidence，当前未做 live refresh
- Runtime Source of Truth / 运行时真源：No
- Schema Source of Truth / Schema 真源：No
- Current Implementation Source of Truth / 当前实现真源：No；当前实现以 Ent schema、migration、biz/data usecase、JSON-RPC、前端页面和测试为准

## 1. 结论

业务事实扩展 不拆任何字母子阶段，也不再把生产、委外、出货、库存预留和财务拆成后续字母半阶段。本文先作为 业务事实扩展 的统一 review，一次性覆盖五条事实链的真源、边界、状态、冲正、幂等、RBAC、API、UI、测试和停止条件；当前同一 业务事实扩展 内已落本地最小实现闭环。

本文最初作为 docs-only review，随后同一 业务事实扩展 内已落本地最小实现闭环：schema / migration、repo / usecase、API / RBAC、最小 UI 和测试。归档 evidence 证明 2026-06-08 对应 commit / image 曾在目标环境执行 migration 到 `20260608134530`，并通过当时的健康检查、前端业务事实处理路由、未登录鉴权 smoke、目标试用账号 RBAC 核对、登录态只读 operational_fact API smoke 和内部受控模拟事实写入闭环；本轮并发、幂等、客户配置 fail-closed、预留消费和页面加固尚未重新发布，且未 live refresh 当前目标运行态。客户使用确认属于交付后的业务确认，不再作为本地最小实现阻塞；该结论仍不代表真实客户数据导入、完整打印、完整报表、发票和收付款核销已经交付。

## 2. 范围项

| 范围项 | 目标 | 本文评审结果 | 不做 |
| --- | --- | --- | --- |
| 生产事实 / Production Facts | 明确生产领料、成品入库、返工事实边界 | 已落最小 `production_facts`，过账写 `inventory_txns` IN / OUT，取消写 REVERSAL | 不把生产任务 done 当成入库或成本事实 |
| 委外事实 / Outsourcing Facts | 明确委外发料、委外回货边界 | 已落最小 `outsourcing_facts`，发料写 OUT，回料写 IN，取消写 REVERSAL | 委外结算不写成委外库存事实，进入财务事实 |
| 出货事实 / Shipment Facts | 明确出货计划、实际发货、取消冲正边界 | 已落 `shipments` / `shipment_items`，发货写库存 OUT，取消写 REVERSAL | 不把 `shipping_released` 当成 `shipped` |
| 库存预留 / Stock Reservation | 明确可用量、预留、释放、随出货消费顺序 | 已落 `stock_reservations`，预留本身只影响可用量；消费只能随真实出货库存 `OUT` 同事务发生 | 不从销售订单直接扣库存，不开放独立消费 |
| 财务事实 / Finance Facts | 明确 AR/AP、发票、收付款、对账状态事实 | 已落 `finance_facts` draft / posted / settled / cancelled | 不从放行或任务状态直接生成财务事实 |

## 3. 总边界

- Workflow task done 不等于 Fact posted。
- Source Document 只代表业务承诺，不代表事实落账。
- `shipping_released` 只表示出货放行，不表示 shipped、库存扣减、应收或开票。
- 成品入库任务完成不等于库存入账；真实库存变化必须由事实 usecase 写 `inventory_txns` 并维护余额。
- 财务事实必须晚于真实可审计业务事实，且必须支持冲正、审计和幂等。
- 客户合同、Excel、截图和模拟数据只能作为线索或样本，不直接进入 Product Core。

## 4. 事实链评审结果

### 4.1 生产事实 / Production Facts

| 项目 | 结论 |
| --- | --- |
| 唯一真源 | `production_facts` + `OperationalFactUsecase`；workflow task、`workflow_business_states` 和旧 `business_records` 删除前 evidence 都不是生产事实真源 |
| 已落事实类型 | `MATERIAL_ISSUE`、`FINISHED_GOODS_RECEIPT`、`REWORK` |
| 状态边界 | `DRAFT` / `POSTED` / `CANCELLED`；受保护字段不可直接更新，已过账只能取消冲正 |
| 库存影响 | 生产领料和返工写 `inventory_txns.OUT`；成品入库写 `inventory_txns.IN` 并维护余额 |
| 冲正 | 取消已过账生产事实时写 `inventory_txns.REVERSAL`，不直接改历史流水 |
| RBAC | 复用 `pmc.plan.*` 与 `warehouse.adjustment.create` 守卫创建、过账和取消 |
| API / UI | `operational_fact` JSON-RPC 暴露 create / post / cancel / list；正式 UI 已拆到生产进度、出货单 / 出库、财务等收窄 V1 页面提交 usecase 动作 |
| 剩余项 | 生产订单 Source Document 已完成本地 schema / migration / repo / usecase、成品入库 fact linkage、Close 领域闭环、API/RBAC runtime、分页 readable option API、销售来源 eligibility 同事务复核，以及 Product Core UI / 通用菜单；永绅客户菜单投影、seed 与部署未接入，且未接成本归集或完整报工 |

### 4.2 委外事实 / Outsourcing Facts

| 项目 | 结论 |
| --- | --- |
| 唯一真源 | `outsourcing_facts` + `OperationalFactUsecase`；加工合同打印样式不是委外事实真源 |
| 已落事实类型 | `MATERIAL_ISSUE`、`RETURN_RECEIPT` |
| 状态边界 | `DRAFT` / `POSTED` / `CANCELLED`；委外库存事实和财务结算事实分开 |
| 库存影响 | 委外发料写 `inventory_txns.OUT`；委外回料写 `inventory_txns.IN` |
| 冲正 | 取消已过账委外事实时写 `inventory_txns.REVERSAL`，不覆盖原始外发和回货事实 |
| RBAC | 复用 `purchase.order.*` 与 `warehouse.adjustment.create` 守卫创建、过账和取消 |
| API / UI | `operational_fact` JSON-RPC 暴露 create / post / cancel / list；正式 UI 已拆到生产进度、委外边界、出货 / 财务等收窄 V1 页面提交 usecase 动作 |
| 剩余项 | 委外结算进入 `finance_facts`，未做委外订单专表、质检对接或应付自动生成 |

### 4.3 出货事实 / Shipment Facts

| 项目 | 结论 |
| --- | --- |
| 唯一真源 | `shipments` / `shipment_items` + `OperationalFactUsecase`；`shipping_released` 只是 workflow 协同状态 |
| 已落事实类型 | 出货单草稿、出货行、发货、取消发货 |
| 状态边界 | `DRAFT` / `SHIPPED` / `CANCELLED`；`SHIPPED` 才代表真实出货 |
| 库存影响 | 发货按出货行写 `inventory_txns.OUT`；取消已发货出货单写 `inventory_txns.REVERSAL`；与来源订单行、产品 / SKU、单位和库存 grain 一致的 ACTIVE 预留在同一出货事务内消费 |
| 冲正 | 已 shipped 的错误走取消发货冲正；不直接改原出货流水 |
| RBAC | 复用 `sales_order.update` 创建草稿 / 加行，`warehouse.outbound.confirm` 确认发货和取消 |
| API / UI | `operational_fact` JSON-RPC 暴露 create / add item / ship / cancel / list；UI 继续区分放行与已发货 |
| 来源与并发 | `SHIPPED` 前在事务内锁定销售订单和来源行，校验客户、产品 / SKU、单位、订单状态与累计已出货量；并发出货不得超过订单行数量 |
| 剩余项 | 草稿不占用订单剩余量，最终阻断在确认出货时执行；未接拣货、装箱、物流、打印、退货或自动应收派生 |

### 4.4 库存预留 / Stock Reservation

| 项目 | 结论 |
| --- | --- |
| 唯一真源 | `stock_reservations` + `OperationalFactUsecase`；销售订单行不是预留真源 |
| 已落事实类型 | 创建预留、释放预留、列表；`CONSUMED` 只由真实出货事务内部推进，正式出库管理页当前不提供页面级无来源新建入口 |
| 状态边界 | `ACTIVE` / `RELEASED` / `CONSUMED`；普通业务入口只能主动释放，独立消费入口已移除 |
| 库存影响 | 创建预留时锁定对应库存 grain，再检查 `inventory_balances - active reservations`，不写 `inventory_txns`；任何产品出库都不得把余量降到 ACTIVE 预留合计以下 |
| 冲正 | 释放只变更预留状态；已随出货消费后的库存冲正仍跟随出货事实冲正处理，不把预留恢复成可重复消费的 active 记录 |
| RBAC | 复用 `sales_order.update`、`warehouse.inventory.read` 和 `warehouse.outbound.confirm` |
| API / UI | `operational_fact` JSON-RPC 暴露 create / release / list；UI 不把菜单显隐当权限边界，正式出库管理页只暴露释放和查询，不允许脱离出货单独消费预留 |
| 出货消费 | 出货会锁定来源销售订单行，原子消费同销售订单行、产品 / SKU、仓库、单位和批次的完整预留记录；并汇总该来源行全部库存 grain，保证历史已出货量、本次出货量和剩余 ACTIVE 预留不超过订单量。单条预留不做部分消费，匹配预留超过本次出货量时拒绝，由业务先释放并按需要重新预留 |
| SKU grain | 显式 SKU 已进入批次、库存流水 / 余额、生产 / 委外、出货和预留的同一库存 grain；历史 `NULL` 规格库存独立保留，不自动归类或跨 SKU 共池 |
| 剩余项 | 未做按订单自动创建预留、单条预留部分消费、预留明细专项页、冻结量 read model、BOM SKU 粒度、旧库存重分类或客户导入自动建 SKU |

### 4.5 财务事实 / Finance Facts

| 项目 | 结论 |
| --- | --- |
| 唯一真源 | `finance_facts` + `OperationalFactUsecase` |
| 已落事实类型 | `RECEIVABLE`、`PAYABLE`、`INVOICE`、`PAYMENT`、`RECONCILIATION` |
| 状态边界 | `DRAFT` / `POSTED` / `SETTLED` / `CANCELLED`；posted 后只能结清或取消 |
| 金额与币种 | `amount` 和 `fee_amount` 共用 `currency`；当前币种限制为美金 `USD`、人民币 `CNY`、港币 `HKD`，手续费不另设币种 |
| 前置事实 | 应收 / 发票仍应晚于真实 shipped，且 counterparty customer 必须等于 shipment customer 真源；应付仍应晚于采购入库 / 委外结算等可审计事实；当前不自动派生 |
| 冲正 | `POSTED -> CANCELLED` 保留原 `posted_at`，同一事务记录认证操作人、取消时间和业务原因，并把关联 ProcessRuntime effect 标记为 compensated；未写总账、核销、红冲或反向金额事实 |
| RBAC | 复用 `finance.receivable.*` 与 `finance.payable.*` 守卫创建、过账、结清和取消 |
| API / UI | `operational_fact` JSON-RPC 暴露 create / post / settle / cancel / list；UI 不从放行状态派生应收或开票 |
| 剩余项 | 未做发票明细、收付款核销、对账单、总账、自动派生或报表 |

取消审计采用单一事实束 `cancelled_at + cancelled_by + cancel_reason`。本项目迁移前已经处于 `CANCELLED` 且无法还原原始操作者、时间或原因的行保留 `cancel_audit_version=0`，只读投影固定显示“历史记录，取消审计信息缺失”，不从 `updated_at`、`posted_at` 或其他账号猜值。迁移后的正式取消写入 `cancel_audit_version=1`，三字段必须成组完整；新增记录默认进入版本 1，版本 0 只为本项目迁移事实保留，待历史数据完成可审计处置后删除该分支与约束放行。

## 5. 本地实现摘要

本地实现已完成以下最小闭环：

1. Ent schema 和 Atlas migration：`production_facts`、`outsourcing_facts`、`shipments`、`shipment_items`、`stock_reservations`、`finance_facts`。
2. 后端 usecase / repo：`OperationalFactUsecase` 和 `NewOperationalFactRepo`，库存影响统一复用 `InventoryUsecase` 的事务内流水与余额更新能力。
3. JSON-RPC / RBAC：新增 `operational_fact` handler，复用既有生产、采购、销售、仓库和财务权限码。
4. 前端 UI：旧内部直达页面 `/erp/operations/facts` 已退出正式 ERP 运行时入口并重定向到业务看板；五条事实链的登记、过账 / 发货 / 释放 / 结清 / 取消动作由生产进度、出货单 / 出库、财务等收窄 V1 页面承接；预留消费只随出货事务内部发生，不再提供独立页面动作，也不再提供客户可直达的内部事实汇总新建页。
5. 测试：新增 repo 层生产过账与冲正、库存预留可用量检查、出货发货与取消冲正测试；同步验证客户菜单不展示内部 业务事实扩展 入口。
6. 创建幂等：库存流水、生产事实、委外事实、财务事实和库存预留持久化“业务时间是否由调用方显式提供”的 marker；都省略时间或都显式提供相同时间才可按同 key 重放，显式时间不同或显式 / 省略混用会返回幂等冲突。服务端默认时间不被误当成每次重试的新业务意图，历史 marker 默认 `false`。
7. ProcessRuntime 领域命令先做无副作用 preflight 再绑定 intent fingerprint；应收客户从已出货 shipment 带出，成品质检执行人从认证 actor 带出，未实际落账的 release / ship / due-date 字段不进入命令 payload。

后续扩展仍必须至少写清：

1. 当前唯一真源和既有真源是否可复用。
2. 是否需要扩展 Ent schema、字段或 migration。
3. 状态机、终态、取消、作废和冲正口径。
4. 幂等键、并发保护、重复提交和 source tracing。
5. RBAC 动作权限、角色职责和业务守卫。
6. API 是否只暴露 usecase 能力，不补造事实逻辑。
7. UI 是否只展示和提交事实 usecase，不在前端本地派生事实。
8. 测试层级、验收命令、停止条件和剩余风险。

允许的实现闭环顺序不是历史实施子阶段，只是工程依赖顺序：

1. 先落会被其他链路引用的事实真源和库存影响边界。
2. 再落 API / RBAC 和 UI 操作入口。
3. 最后接客户试用、培训、交付台账和目标环境验收。

## 6. 剩余停止条件

当前业务事实最小实现已完成，但仍不允许：

- 修改 `WorkflowUsecase` 去写库存、出货或财务事实。
- 把 试用模拟数据或客户样本转写成真实导入、出货、库存或财务事实。
- 把内部模拟闭环写成真实客户数据导入、客户已签收或完整业务交付。

业务事实扩展的本地最小实现已闭环；`客户目标发布验收手册` 和 `客户目标发布 evidence` 只记录 2026-06-08 对应版本的历史目标环境模拟事实基线，不证明当前运行态或本轮代码已发布。显式 SKU 库存 grain 已通过本地 PostgreSQL migration、partial unique index、隔离和并发预留验证，但自动派生、报表、打印、核销、物流退货、BOM SKU 粒度、旧库存重分类和客户导入自动建 SKU 仍是后续增强；若启动这些增强，仍必须按 `docs/product/模块实施治理.md` 明确允许路径、禁止路径、验收命令和停止条件。
