# 生产工艺路线与在制品边界评审 / Production Route and WIP Boundary Review

## 结论与适用范围

毛绒生产的正式路线必须由稳定路线代码 / 版本、生产订单路线快照、逐步骤执行和可追溯的在制品批次共同表达，不能再从工序档案排序、委外合同先后、Workflow 任务顺序或页面备注反向推导。

本评审确认 Product Core 的通用边界。永绅已经确认的高层顺序是业务输入，不会形成客户专属 schema 或 usecase。本文也不单独证明代码已经部署、客户配置已经激活、目标环境已经迁移或客户已经验收；本地实现层级仍以当前 schema、migration、usecase、API、页面和实际测试为准。

当前 Product Core 本地源码已落固定 `PLUSH_SEW_HAND_V1` v1、订单工序快照、WIP 批次 / 事件、分段 Quality Fact、包材业务确认、完工入库数量门禁、JSON-RPC / RBAC 与页面入口。这是一条固定产品路线，不是客户可自由编辑的通用路线设计器。对应 `20260717035245_migrate.sql`（路线、WIP、分段质检）和 `20260717043625_migrate.sql`（首道材料归属、委外分配）只已生成并纳入本地迁移链，未 apply 到共享开发库或目标客户数据库，因此尚不能表述为已有目标环境 runtime、已发布或已客户验收。

## 领域归属与唯一真源

| 能力 | 责任层 | 唯一真源 | 不能替代它的内容 |
| --- | --- | --- | --- |
| 工序定义 | MasterData | 启用的工序主数据 | 工序列表显示顺序 |
| 路线与版本 | Product Core / Engineering | 当前固定 `PLUSH_SEW_HAND_V1` v1 步骤合同 | 客户流程图、BOM 行顺序、委外合同顺序 |
| 生产订单路线 | Source Document snapshot | 订单发布时冻结的路线版本和步骤快照 | 后续路线模板修改、Workflow payload |
| 工序执行与 WIP | Production Fact | 订单、步骤、批次、数量和执行方式绑定的领域事实 | 任务 done、自由文本进度、通用库存备注 |
| 外发加工 | Outsourcing Source / Fact | 路线 WIP 绑定的既有加工合同明细，以及独立发料 / 回货事实 | 把合同确认或 WIP 状态当作已经发料 / 回货库存事实 |
| 分段质检 | Quality Fact | 绑定订单、路线步骤、执行批次和关口类型的质量检验 | 工序 `quality_required` 标记、质检任务完成 |
| 成品入仓 | Production / Inventory Fact | 所有必需关口通过后的正式成品入库与库存流水 | 包装完成、仓库任务完成、客户验货备注 |
| 包装材料确认 | Sales / Purchase / Quality 分层 | 业务确认记录、采购入库 / IQC 和包装领用事实分别负责 | 用业务对版替代 IQC，或用 IQC 替代客户版面确认 |

`processes.sort_order` 只控制工序档案的列表显示顺序，不是路线顺序、前置关系或执行状态真源。

## 当前固定路线实现

| 步骤 | 产出 / 关口 | 当前执行规则 |
| --- | --- | --- |
| `FABRIC_PROCESSING` 布料加工 | `CUT_PIECE` / 裁片检验 | 只允许外发，必须整单数量办理，不允许在首道拆 WIP 批次。BOM 行必须显式标记 `production_operation_code=FABRIC_PROCESSING`，发布时至少冻结一条这类材料需求；不按“面料”等名称或类别文本猜测。必须绑定且仅绑定覆盖这些冻结材料的 MATERIAL 加工合同明细，开始前还要有足量已过账委外发料。 |
| `SEWING` 车缝 | `SHELL` / 皮套检验 | 裁片检验只有 `PASS` 后才能转入；从这一步开始才能按产品数量拆分 WIP 子批，各子批独立选择本厂或外发。 |
| `HANDWORK` 手工 | `FINISHED_GOODS` / 成品、针检、抽检及条件客户验货 | 皮套检验通过后才能转入，每个 WIP 批次再独立选择本厂或外发。客户验货只在订单行快照明确要求时加入。 |
| `PACKAGING` 包装 | `PACKED_GOODS` | 只允许本厂。启动前必须完成独立的包材版面 / 版本业务确认；包装 WIP 已 `ACCEPTED` 仍不等于库存已入仓，只是正式完工入库数量的上限。 |

生产阶段关口当前只允许 `PASSED + PASS` 推进。项目尚未建立关口级让步审批策略和审计，所以通用质检的 `CONCESSION` 对生产 WIP 关口 fail closed；这不改变采购、委外回货或出货侧链各自现有的让步边界。

## 正式生产主线

当前 Product Core 只允许生产订单行显式选择固定 `PLUSH_SEW_HAND_V1` v1，其执行合同保持同一条可验证主线：

`布料加工 → 裁片检验 → 车缝 → 皮套检验 → 手工 → 成品检验 → 针检 → 抽检 →（订单要求时）客户验货 → 最终包装 → 成品入仓`

采购到货与主辅料 IQC 是这条生产主线的上游门禁；出货放行和真实 `SHIPPED` 是成品入仓后的下游事实，不合并到生产路线步骤中。

路线必须显式保存步骤顺序和前置关系。`车缝 → 手工` 是前后步骤，不是两个可互换选项；车缝和手工各自再选择本厂或外发，前一步的执行方式不得自动复制为后一步决定。

## 路线版本与订单快照

1. 当前只有固定 `PLUSH_SEW_HAND_V1` v1；页面在生产订单行显式选择，后端不从工序排序或客户名称推导。
2. 发布前必须解析唯一、启用且具备对应内 / 外发能力的布料加工、车缝、手工和包装工序；歧义、缺失或能力不符都 fail closed。
3. 显式路线行还必须从锁定 BOM 冻结至少一条 `production_operation_code=FABRIC_PROCESSING` 的材料需求；没有该机器可读标记时拒绝发布，不用材料名称、类别或备注猜测。
4. 生产订单发布时冻结路线代码 / 版本、步骤顺序、工序快照、必需质量关口、客户验货条件和首道材料需求快照；后续修改 BOM 或工序档案不改写已发布订单。
5. 历史 `route_code=NULL` 订单行继续使用旧完工边界，不按工序排序、委外记录或材料文本猜测回填路线。
6. 后续若引入第二条正式路线或可配置版本，必须另做 schema、发布、快照、变更和审计评审；不在当前固定 v1 上加自由文本分支。

## WIP 与内外发语义

### 本厂生产

本厂步骤完成后的在制品流转统一称为“车间移交”或“WIP 转移”。当前事件保存来源 / 目标工序、产品 / 规格、父子批次沿袭、数量、单位、操作者和时间，但不冒充采购回仓、委外回仓或成品入库。

裁片和皮套在路线执行中作为正式 WIP 工序产出处理；它们不是原材料库存，也不是可销售成品。若后续业务确实需要把某类半成品作为独立可盘点产品，必须另做主数据和库存粒度评审，不能在本路线中隐式升级。

### 外发生产

只有物料或在制品交给外部加工方后返回，才使用“外发回仓”语义。外发源单、发料、回货和质检分别保留 Source Document / Fact 边界；合同确认不等于已经发料，回货不等于已经合格，质检合格也不自动生成应付或成品入仓。

每一笔外发都必须绑定生产订单、路线步骤和精确 WIP 批次（首道根批或后续子批）。通用委外合同仍可复用，但不能形成脱离路线执行的第二套生产进度真源。

## 拆量、数量守恒与返工

- 首道布料加工的正常流必须按生产订单行整单数量外发，不允许拆 WIP 批次。只有裁片检验 `PASS` 并转入 `SEWING` 后，才允许按产品数量一次拆成至少两个子批，并分别选择本厂或外发；本次全部有效子批数量之和必须精确等于父批数量。
- 布料加工的外发绑定只能覆盖已冻结且显式标记 `FABRIC_PROCESSING` 的材料需求；不得把 BOM 其他辅料一并当成首道外发材料，也不得少绑、多绑或用 PRODUCT 委外明细代替。
- FABRIC 正常流返工回到布料加工时不复用原 MATERIAL 合同行；返工批次已经是产品 WIP，若再次外发必须新建并绑定数量相等的 PRODUCT 委外合同行，原材料需求和历史发料不被重复消费。
- 已分配、进行中、待检、合格、返工、报废和剩余数量必须在同一数量粒度上可核对；任何状态变化都不能让数量凭空增加或消失。
- 生产订单仍有 `PLANNED / IN_PROGRESS / OUTSOURCED / WAITING_QUALITY` 在制批次时不得关闭，即使填写短关闭原因也不能绕过。`SPLIT` 父批本身已经由子批承接，不单独阻断；其有效子批仍按各自状态判断。`ACCEPTED / REJECTED / CANCELLED` 作为终态可进入短关闭核对，但短缺原因、已过账事实和委外源单仍分别保留，系统不会替用户自动取消或冲正。
- 并发拆量、重复提交、回货重试和网络结果未知必须使用稳定 intent 和幂等结果重放，不能重复生成 WIP、回货或质检事实。
- 子批后续合并必须是显式、可审计的领域动作；仅因进入同一步骤或同一仓库不得自动合并。
- 不合格返工必须保存原检验、原步骤、返工目标步骤、数量和原因。返工创建新的执行记录，不覆盖或重新打开已经结算的原执行。
- 报废、让步接收和短缺必须有明确权限、原因与数量影响；不得用 Workflow 阻塞 / 完成状态代替。

## 分段质量关口

以下关口是不同业务语义，必须分别记录，不能用一张通用质检单或一个“需质检”布尔值表示全部完成：

| 关口 | 主要来源 | 通过后允许 | 不通过时 |
| --- | --- | --- | --- |
| 主辅料 IQC | 采购到货 / 入库草稿 | 采购入库或布料加工准备 | 阻止对应入库或进入生产 |
| 裁片检验 | 首道布料加工外发回仓形成的裁片 WIP | 车缝分流 | 阻止车缝，进入明确处置 |
| 皮套检验 | 车缝子批完成 | 手工分流 | 阻止手工，退回指定步骤返工 |
| 成品检验 | 手工子批完成 | 针检 | 阻止后续质量关口和包装 |
| 针检 | 成品检验通过的子批 | 抽检 | 阻止抽检或包装，按规则返工 / 报废 |
| 抽检 | 针检通过的子批 | 条件性客户验货或包装 | 阻止包装，进入复检或返工 |
| 客户验货 | 订单明确要求的成品批次 | 包装或出货准备 | 阻止后续；不要求时记录条件不适用，不伪造“通过” |

每个关口必须绑定精确来源步骤和批次，记录抽检数量、判定、原因、证据和操作者。Quality Fact 只负责质量结论；它本身不增减库存，也不以任务完成代替判定。当前生产阶段关口没有关口级让步审批策略和审计，所以只有 `PASS` 放行，`CONCESSION` fail closed；后续若允许让步，必须先增加明确权限、原因、关口策略和持久审计。

## 包装材料确认与品质检验

包装材料同时存在业务确认和品质检验两类责任，必须分开：

- 业务对版 / 外观确认：由业务岗位确认客户版面、文字、颜色、包装版本和订单要求是否一致，保存订单与版本来源。
- 品质正式检验：由品质岗位按材料、批次和检验要求记录规格、外观或抽检结论；需要 IQC 时继续走正式 Quality Fact。

两类独立记录互不代替。业务确认通过不能把不合格包材改成合格，品质检验通过也不能证明版面已经获得客户或业务确认。包装领用、消耗、装箱和成品入仓仍分别走对应生产 / 库存事实。

## 条件性客户验货

客户验货是订单级条件关口，不是所有生产订单的全局强制步骤：

1. 销售订单或生产订单路线快照必须保存“是否要求客户验货”及其来源。
2. 不要求时只记录 `not_applicable` 及规则来源，不生成虚假的客户验货合格记录。
3. 要求时，未检、进行中或不合格均阻止后续包装 / 放行；合格后才允许继续。
4. 任何豁免必须由正式权限、原因和审计记录承接，不能靠备注跳过。
5. 客户验货结果是条件质量关口，不等于客户已经完成整套系统 UAT 或交付签收。

## Workflow、角色与事实边界

- 生产经理负责逐步骤内制 / 外发决定和允许范围内的拆量；实际写入必须经过后端 usecase。
- 生产岗位办理本厂步骤执行和车间 WIP 移交；外发办理仍复用委外源单与事实。
- 品质岗位登记各分段质量事实；任务处理只能推动协同，不直接伪造检验结论。
- 业务岗位负责包材业务确认和订单是否要求客户验货的业务来源；不能写品质结论。
- 仓库只办理真实收货、外发回仓、包装领用和成品入仓事实；车间内部移交不冒充仓库入库。
- PMC 负责计划和风险协同，不把排期任务完成写成路线步骤完成。

后端 RBAC、owner / assignee、状态、版本和数量约束是正式安全边界；前端按钮隐藏、客户菜单显隐或 Workflow 责任池不能提升权限。

## 本地实现与客户交付边界

当前本地源码层已具备：

- 固定 `PLUSH_SEW_HAND_V1` v1、订单路线快照、显式首道材料需求快照、步骤执行和 WIP 数量真源。
- 对应 Ent schema、Atlas migration、repo / usecase、事务、幂等、JSON-RPC、RBAC 和审计。
- 生产订单、路线执行、质量检验和关联记录的正式页面入口。
- 正常、首道整单外发、车缝后拆量、拒绝、返工、条件客户验货、包材确认分离、未批准让步 fail closed、并发和幂等定向测试。
- 生产订单工序办理与质量检验生产阶段读模型的前端入口和定向页面合同。

尚缺的运行 / 交付证据是：将当前 WIP migration 应用到归属明确的隔离 PostgreSQL，完成真实事务回归、目标环境 health / smoke、真实角色权限读回和客户人工验收。

客户配置当前只能收窄模块、页面、岗位动作和责任投影，不能重新定义固定路线步骤、数量守恒、质检事实、库存事实或后端 RBAC 底线。永绅客户包的生产、品质、业务和 PMC 岗位必须在 Product Core 权限上限内显式投影 WIP 能力；该投影存在也不代表 active revision、目标数据库 migration、目标角色 smoke、发布回滚或客户人工签收已完成。

## 验收门禁

1. 调整工序列表排序不会改变已发布生产订单的路线。
2. 车缝未完成或皮套检验未通过时不能开始手工。
3. 车缝与手工可以分别选择不同执行方式。
4. 内制转移只显示 WIP / 车间移交，外发返回才显示回仓。
5. 布料加工按整单数量外发且不可拆批；其绑定的 MATERIAL 合同明细恰好覆盖显式 `FABRIC_PROCESSING` 材料需求，开始前已过账委外发料数量足够。
6. 裁片 `PASS` 转入车缝后才允许产品数量拆批；后续内制、外发拆量及返工仍满足数量守恒，并发操作不能超分配。
7. 裁片、皮套、成品、针检、抽检和客户验货分别可追溯，生产阶段关口只有 `PASS` 放行，未审批的 `CONCESSION` 和其他未通过状态都不能越级。
8. 不要求客户验货的订单记录条件不适用；要求客户验货的订单在通过前保持阻断。
9. 包材业务确认和品质正式检验互不替代。
10. 包装完成不等于成品已经入仓；只有正式成品入库事实写库存。
11. Workflow 任务完成不直接修改路线、质量或库存事实。

## 明确不由本评审证明

- 自动 MRP、产能优化、工时工资、成本归集和完整 MES。
- 客户专属 schema、SaaS tenant 或客户代码分支。
- 目标环境已经发布、migration 已 apply、备份恢复已演练或客户已经签收。
- 未经业务来源确认的客户验货要求、让步权限、报废规则或客户真实数据导入。
