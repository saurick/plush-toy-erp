# 业务附件证据边界评审 / Business Attachment Evidence Boundary Review

## 结论

业务附件主体仍是 Reporting / Audit / Integration 层的证据能力，不是 MasterData、Source Document、Fact 或 Workflow 的状态真源。产品图片复用同一文件存储与权限链路，但按可替换的产品媒体管理，不按不可无痕删除的业务证据解释。

当前实现新增统一 `business_attachments` 表和 `attachment` JSON-RPC 域，把文件挂到既有业务对象：

| owner_type | 对应业务对象 | 当前接入页面 |
| --- | --- | --- |
| `sales_order` | 销售订单 | 销售订单编辑弹窗 |
| `purchase_order` | 采购订单 | 采购订单编辑弹窗 |
| `outsourcing_order` | 委外 / 加工合同源单 | 加工合同编辑弹窗 |
| `purchase_receipt` | 采购入库单 | 入库草稿 / 加行弹窗 |
| `quality_inspection` | 来料质检单 | 质检创建 / 判定弹窗 |
| `shipment` | 出货单 | 出货单新建 / 明细查看弹窗、收窄出货事实页 |
| `finance_fact` | 财务事实 | 应收、应付、发票、对账收窄页面 |
| `production_fact` | 生产事实 | 生产进度收窄页面 |
| `outsourcing_fact` | 委外事实 | 委外事实收窄页面 |
| `product` | 产品主数据的产品图 | 产品基础信息表单；打印草稿只读带值 |
| `product_sku` | SKU 主数据 | 产品 / SKU 页面 |
| `bom_header` | BOM 版本 | 物料清单（BOM）页面 |
| `workflow_task` | Workflow 协同任务 | Workflow V1 页面、岗位任务端详情 |

## 页面布局口径

- 销售订单、采购订单、委外订单、BOM、SKU 等带主表字段和明细 items 的编辑弹窗，以及出货单聚合新建 / 明细查看弹窗中，附件作为主对象证据字段处理，默认放在备注、交付、合同资料或凭证语义附近，并位于订单行、BOM 明细、出货明细等 item 区之前。
- 新建未保存记录允许先选择附件并暂存在当前前端表单内，保存业务记录成功拿到 owner_id 后再自动上传绑定；取消或保存失败不得生成悬空附件。已有记录无附件时显示紧凑空态，不在弹窗末尾撑出独立大区块，避免明细增多后用户必须滚到最后才能上传。
- 页面级选中记录附件，例如收窄事实页、Workflow 桌面页和岗位任务端详情，可保留独立区块，但仍必须绑定当前已保存 owner，并保持“证据能力、不改业务事实”的语义。
- 产品基础信息表单提供两个固定产品图片位：`产品图 1（主图）` 和 `产品图 2（辅图）`。新建 / 编辑时先在当前表单会话内暂存选择、替换或清空动作，产品保存成功后再绑定；取消或产品保存失败不得写入图片。

## 真源边界

- 附件只能证明“该业务记录存在一份外部资料或现场证据”。
- 附件上传、下载和轻量预览不改变订单生命周期、BOM 状态、质检判定、出货状态、库存流水、财务事实、Workflow 任务状态或业务状态投影。
- 出货单 `SHIPPED`、库存 `inventory_txns`、财务 `finance_facts`、质检判定、Workflow task status 等仍由各自 usecase 负责。
- `workflow_task` 附件不等于任务完成；岗位任务端仍必须通过 Workflow API 执行处理、阻塞、退回或完成动作。
- `finance_fact` 附件不替代税控、发票查验、纳税申报、总账凭证、付款审批、收付款核销或多账簿。

### 产品图片媒体例外

- 产品图片只允许 `owner_type=product`、`attachment_type=product_image` 和 `slot_key=primary|secondary`，每个产品每个槽位当前只保留一张。
- 同槽重新上传会在锁定产品记录的同一事务内替换旧图，数据库 partial unique index 再约束每个产品 / 槽位最多一行；`clear_product_image` 只允许清空上述产品图片槽。这个窄例外不恢复普通附件的物理删除能力。
- 产品图片只接受 PNG、JPEG、WEBP，单张仍受 5MB 上限约束；服务端在 base64 解码后先读取尺寸，再按声明格式完整解码，宽高单边不得超过 8192px、总像素不得超过 2000 万。扩展名、声明 MIME、完整图片内容或尺寸不一致都会拒绝；GIF、HEIC / HEIF、PDF、Office 和资料包不能作为产品打印图上传。
- 产品图不进入 `products` 字段，不改变产品、SKU、BOM、订单、库存、生产、委外或质检事实。打印入口读取后冻结到当前打印草稿，后续替换主档图片不会反向改写已经打开的草稿。

## API 与权限

`attachment` JSON-RPC 当前提供：

- `list_attachments`
- `upload_attachment`
- `download_attachment`
- `clear_product_image`：仅清空产品 `primary / secondary` 图片槽，不是普通附件删除接口

权限不另造一套页面旁路；服务端按 `owner_type` 复用所属业务对象的读写权限。Workflow 任务还必须复用 active revision 的行级责任范围：读取要求 `workflow.task.read` 和任务可见，上传只接受 `workflow.task.update`，并要求当前账号是有效 owner scope 或指定处理人；终态任务拒绝追加证据。催办特权不等于附件写权。

前端预览只复用 `download_attachment` 读取内容：PNG、JPG / JPEG、WEBP、GIF 和 PDF 可在当前页面轻量预览；HEIC / HEIF、Word、Excel、CSV、文本、ZIP、邮件证据和 WPS 文件继续下载后查看。预览不新增 `/files` 静态外链、不绕过 owner 权限，也不把附件内容写回业务对象。

当前允许上传的证据格式：

| 分组 | 扩展名 / 类型 | 预览策略 |
| --- | --- | --- |
| 图片证据 | `.png`、`.jpg`、`.jpeg`、`.webp`、`.gif` | 当前页面轻量预览 |
| 手机图片证据 | `.heic`、`.heif` | 允许上传 / 下载；不承诺浏览器内预览或在线转换 |
| PDF | `.pdf` | 当前页面轻量预览 |
| Office 文档 | `.doc`、`.docx`、`.xls`、`.xlsx` | 下载后查看 |
| 数据 / 文本 | `.csv`、`.txt` | 下载后查看 |
| 资料包 | `.zip` | 下载后查看 |
| 邮件证据 | `.eml`、`.msg` | 下载后查看 |
| WPS 文件 | `.wps`、`.et`、`.dps` | 下载后查看；前端按扩展名归一到受控 MIME，不放开任意 `application/octet-stream` |

服务端还做这些守卫：

- 必须是后台管理员登录态。
- `owner_type` 必须在白名单内。
- `owner_id` 必须对应真实业务记录。
- 单个附件最大 5MB；HTTP 路由先把编码请求体限制在 7MB，业务层再在 base64 解码前按编码长度拒绝超限输入，并在解码后复核 5MB。当前下载仍通过 JSON-RPC base64 返回，因此该上限是低配宿主的内存预算门禁，不代表流式大文件能力。
- 普通物理删除接口已退出。除产品图片固定槽位的受控替换 / 清空外，已上传证据的纠错 / 撤销需要先设计主体状态、操作者、原因、持久审计和不可删除事实证据边界，不能恢复宽泛 delete。
- 上传 repo 在同一事务内锁定 owner 行并插入附件；读取内容会复核 owner 仍存在。targeted debug cleanup 与全量 debug clear 都先处理 `business_attachments`，避免清理 owner 后留下可凭附件 ID 读取的孤儿。
- 仅允许上表列出的证据格式，且文件扩展名必须与受控 MIME 合同匹配；不接受任意二进制、脚本、HTML、SVG 或可执行文件。
- JSON-RPC 日志会脱敏 `content_base64`、`base64`、`content`、`file_content` 等大字段或敏感字段。

## 当前不包含

- 不实现对象存储、CDN、缩略图、OCR、病毒扫描、电子签章或全文检索。
- 不实现 Office 在线转换、CSV / 文本在线渲染、客户外链共享或跨权限临时下载链接；当前只允许图片 / PDF 轻量预览。
- 不实现版本化附件、附件审批流、附件归档保管期限、附件水印或客户外链共享。
- 不把附件拆成每个业务表的字段，也不把附件 ID 写回业务事实表。
- 不做客户真实文件导入或历史 backfill。

## 验收口径

代码层至少核对：

1. `business_attachments` migration 和 Ent generated code 存在。
2. `attachment` JSON-RPC 的 canonical list/upload/download 可通过服务端测试或真实 RPC 调用验证；产品图片另验证同槽替换和窄版 `clear_product_image`，camelCase、content alias 和宽泛 delete 均 fail closed。
3. P0/P1 页面只把附件挂到已保存业务记录；新建未保存时只能保留前端待上传附件，保存失败或取消不得生成悬空附件。
4. 上传、下载或预览附件后，所属业务对象状态、事实状态和 Workflow 任务状态不发生变化。
5. 超过 5MB、编码请求体超限、非法 MIME、无权限、不可见 Workflow owner、终态 Workflow owner 或不存在 owner 的请求被拒绝。
