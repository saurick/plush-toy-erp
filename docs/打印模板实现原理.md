# 打印模板实现原理 / ERP Print Template Implementation

> 目的：把当前毛绒 ERP 打印模板的共享编辑壳层、窗口级状态和正文真源收口到一份正式文档，避免后续继续把“普通路由页 + 模板共用一份 localStorage 草稿”的旧认知当成主实现。

## 1. 当前结论

- 当前正式模板包括 `采购合同`、`加工合同`、`物料分析明细表`、`色卡`、`作业指导书`。
- 五套正式模板当前通过“打印中心 -> 独立弹窗 -> 右侧纸面 DOM 输出 PDF / 打印”的主链打开；旧业务页选中 `business_records` 带值入口已退出。后续若从其他领域页带值打印，必须由对应领域模型显式生成打印草稿输入。
- 这条主链已收口为：直接打开模板工作台 URL、窗口级 `state` 隔离结构化草稿、工作台按当前草稿重新渲染纸面；`/print-window-shell.html` 和整窗 HTML 快照只保留为已有窗口状态或极端恢复场景的兼容兜底，不作为可编辑模板的长期真源。
- 五套正式模板共用顶层 `appendixImages[]` 作为模板末尾附图的唯一真源：管理区支持多选、排序、移除和逐张选择 `自动 / 半宽 / 整行`。普通图片自动两张一行，长图自动整行，超长图按打印页安全高度分段；不设业务或应用层张数上限，但仍受浏览器处理 / 存储能力和服务端整份请求安全预算约束。
- 当前正文继续由 React 工作台渲染，不是 builder + 独立静态 runtime，但分页运行时已经统一收口到共享 `printPageMargin` 主链。

## 3. 当前打开链路

### 3.1 从领域页带值打开

旧 `BusinessModulePage -> businessRecordPrintDraft.mjs` 链路已随 `business_records` 删除退出运行时。当前采购订单带值打开采购合同，委外订单带值打开加工合同和外发作业指导书，物料清单（BOM）页面带值打开物料分析明细表、色卡和内部 / 工程资料作业指导书。BOM 来源的作业指导书 mapper 单独把产品业务款号和产品名称映射到纸面表头；委外来源沿用源单冻结的产品编号和名称快照，不拿当前产品主档覆盖历史源单。两条路径都不把编号拼进产品名称；缺少独立打印版次或工序日期真源时对应值槽保持空白。委外单号进入订单号，回货日期、加工厂、来源订单、加工项目、工序和委外数量进入正文上下文，不占用 `版本/版次` 或工序日期值槽，也不把委外明细伪造成生产、质检、库存或 Workflow 事实。后续如果采购、委外、工程资料、生产、品质或出货领域页需要新增带值打印，必须先定义对应领域模型到打印草稿的字段映射，再调用 `openPrintWorkspaceWindow(templateKey, { entrySource: 'business', initialDraft })` 生成窗口级 `stateID`，不能恢复旧通用业务记录作为打印草稿真源。

工程资料业务入口在生成草稿前读取产品媒体：产品图 1 / 2 分别使用 `business_attachments` 的 `product + product_image + primary / secondary`，前端通过受控 list / download API 转成 data URL 后再交给 mapper。物料明细映射为 `header_left / header_right`，作业指导书映射为 `header / header_right`。没有图片是合法空态；读取到已登记图片但下载失败时阻止本次打开并提示错误，不静默打印残缺版本。

产品页选择大源图时会先在浏览器内生成受控打印快照，因此业务入口携带的是压缩后的 data URL，不把原始大图塞进窗口草稿。若 localStorage 因容量或策略无法写入，`openPrintWorkspaceWindow` 会把初始草稿放进当前弹窗的 `window.name` 一次性通道；打印页首次读取后清空 `window.name`，并在当前窗口内按模板和 `stateID` 暂存该初始草稿，仅用于桥接 React 初始化与挂载 effect 的重复读取。当前草稿一旦成功写入 localStorage，该初始化缓存立即失效，后续重挂载不会用旧草稿遮挡新内容；写入仍失败时才继续保留该桥接。

BOM 本身只有一个 `product_id`，可直接读取该产品图片。委外订单只在全部有效产品行都有同一个 `product_id` 时自动读取；多产品或关联不完整时保持图片槽为空并提示人工核对。图片在打开时已经冻结进当前窗口结构化草稿，之后产品页替换图片不会改写已打开窗口；但关闭后重新打开同一历史 BOM 会重新读取产品主档当前图片。这不是 BOM 激活时的图片版本快照，当前也不保存附件 ID、hash 或图片历史版本。

采购订单打开采购合同时，单位显示由采购合同打印草稿映射层收口：优先显示中文单位。中文来源单位保留原文；`米（M）` 这类中文加英文括号的来源取中文前缀；已知英文 code 会映射回中文，例如 `PCS / PC / EA` -> `件`，`PAIR / PR` -> `对`，`SET` -> `套`，`M / CM / MM / YD / FT / IN` -> `米 / 厘米 / 毫米 / 码 / 尺 / 英寸`，`KG / G / T / LB / OZ` -> `千克 / 克 / 吨 / 磅 / 盎司`，`CTN / BOX / PKG / BAG / ROLL / BTL / DRUM` -> `箱 / 盒 / 包 / 袋 / 卷 / 瓶 / 桶`，`DOZ / GRS / REAM / BDL` -> `打 / 罗 / 令 / 扎`。无法确认的来源单位保留原文，避免把未知单位伪造成错误中文。

### 3.2 从打印中心打开

`PrintCenterPage` 调用 `openPrintWorkspaceWindow(templateKey, options)`：

1. 生成窗口级 `stateID`
2. 生成实际模板工作台 URL：`/erp/print-workspace/:templateKey?...&state=:stateID`
3. 先把 `{ templateKey, workspaceURL }` 落到 localStorage 的窗口状态 key，供兼容恢复和登录守卫识别
4. 直接打开工作台 URL，而不是把可编辑主路径压到 `/print-window-shell.html`
5. 工作台根据 `stateID` 读取 / 写入当前窗口专属结构化草稿

这样做的目标有两个：

- 保持浏览器刷新时仍落在真实 React 工作台，由结构化草稿恢复可编辑纸面
- 保留窗口状态作为兼容兜底和入口识别，不让静态 HTML 快照覆盖结构化草稿

### 3.3 工作台加载后

各打印工作台都会：

- 根据 `state` 计算当前窗口专属草稿 key
- 首次加载时先读窗口级草稿；若没有，再兼容回读旧模板级草稿
- 保持当前弹窗地址为 `/erp/print-workspace/:templateKey?...&state=...`，让刷新继续走 React 工作台主路径
- 监听当前整窗 DOM 的变更并回写最新 `windowHTML` 快照作为兼容兜底

这意味着：

- 弹窗当前显示的仍是 React 渲染后的工作台 DOM
- 浏览器地址栏保留真实工作台 URL
- 用户刷新当前弹窗时，工作台会重新读取当前窗口级结构化草稿；壳页只服务旧窗口状态和极端恢复场景

## 4. 草稿与输出真源

### 4.1 草稿真源

- 当前窗口级草稿 key：`__plush_erp_print_workspace_draft__:<templateKey>:<stateID>`
- 旧模板级草稿 key 仅保留兼容回读：
  - 采购合同：`__plush_erp_material_purchase_contract_print_draft__`
  - 加工合同：`__plush_erp_processing_contract_print_draft__`

当前规则：

- 新打开的弹窗默认写窗口级草稿
- 只有当前窗口级草稿不存在时，才回读旧 key
- 不再默认把同模板的多个弹窗强行绑到同一份 localStorage 草稿上
- 五套模板都把顶层 `appendixImages[]` 写入当前窗口级草稿；上传、调序、移除和逐张版式切换会同步更新当前窗口草稿，不回写业务附件或其他窗口
- 单张逻辑图片保存 `layoutMode`、快照宽高，以及一张 `dataURL` 或多段 `segments[]`。超长图只保存分段，不再重复保存一份完整长图；旧草稿只有 `dataURL` 时会归一为 `auto`，尺寸未知时继续按半宽输出，保持旧草稿可读
- 老加工合同的 `attachments['attachment-1', 'attachment-2']` 与老物料明细的 `images.footer_left / footer_right` 仅在读入草稿缺少 `appendixImages` 字段时按原顺序迁移；一旦草稿显式存在 `appendixImages`，包括空数组，它就是唯一真源，旧槽位不得再次复活图片
- 右侧 `contenteditable` 输入通过 `input` 和当前 editable DOM 文本变化写入窗口级草稿；`setDraft` 每次解析出新草稿时都会同步落当前窗口草稿 key，失焦再提交 React 渲染态。输入期草稿以 `draftRef` 为准，避免旧 React state 在 effect 或页面退出时反向覆盖最新输入；页面退出前会先让当前活动 `contenteditable` 失焦，再 flush 当前 draft。`恢复样例 / 空白模板` 这类外部重置必须同步更新右侧纸面 DOM 和结构化草稿，避免 DOM、state 与 localStorage 三者分叉。若现有浏览器窗口仍表现为旧恢复行为，应先排除旧 bundle / 浏览器缓存，再继续定位当前代码。

### 4.2 PDF / 打印真源

- 浏览器打印：直接打印当前弹窗里的右侧纸面 DOM
- 在线预览 PDF / 下载 PDF：`printPdf.mjs` 会先 blur 当前焦点元素，再 clone 当前窗口文档、移除运行时脚本和高亮类，并只保留右侧纸面主区域后交给后端 `/templates/render-pdf`
- PDF 快照只允许静态内容：前端移除脚本、外部 stylesheet 和无打印意义的交互属性，把相关 CSS 规则直接内联且不携带源地址；外链图片由当前已登录页面先读取并栅格化，SVG 在所有输出模式下先转成 PNG / JPEG 类位图。转换失败时直接给出业务提示，不让后端 Chromium 访问外部地址兜底。
- 五套模板的末尾附图都会先在前端转成可持久化图片快照，再按 `appendixImages[]` 顺序并入右侧纸面 DOM 末尾，保证刷新恢复、PDF 预览 / 下载和浏览器打印读取的是同一份内容。纸面按显式行模型顺序扫描：连续半宽图片每两张组成一行，整行图片到来前先结束尚未配对的半宽行并独占下一行，奇数半宽图片保持在左侧
- 半宽行两图顶部对齐，各自使用 `width: 100%; height: auto` 保持原比例；Grid 行高自然取两图中的最高值，下一行不会按较矮图片提前开始。整行图片使用纸面可用宽度，手动 `half / full` 优先于自动判定
- 上传快照按宽度处理：不放大原图，宽度超过 `1600px` 时才缩到 `1600px`，统一生成 JPEG 质量 `0.9` 的安全内嵌位图。自动版式在纵横比超过 `1.35` 或横纵比超过 `1.8` 时使用整行；尺寸未知的旧草稿仍回退半宽
- 目标图高宽比超过 `1.45` 时会连续分段，每段高宽比不超过 `1.25`；分段覆盖原图首尾且不重复保存完整长图。整行分段图片的后续段在打印态从新页开始，分段自身避免被页内裁切；用户手动改为半宽时仍保持分段与原顺序
- 在线 PDF 快照对末尾附图保留上传阶段生成的栅格像素，不再套用普通预览图的二次降分辨率压缩；SVG 和外链安全边界、整份请求预算及服务端 HTML / CSS allowlist 保持不变
- 从产品主档带入的产品图同样先冻结到工程资料草稿，再由物料明细或作业指导书右上角图片槽渲染；PDF、浏览器打印与屏幕预览继续读取同一份右侧纸面 DOM
- 正式模板必须在 `printTemplates.mjs` 声明 `fieldRequirements`、`moduleKeys` 和 `factBoundary: read_snapshot_only`；五套模板的字段合同都声明 `appendix_image_snapshots`。字段合同只描述打印草稿需要哪些业务快照，不代表模板可以生成采购、委外、生产、库存、出货、质检或财务事实。
- `/templates/render-pdf` 的请求体只接受 `title`、`file_name`、`template_key` 和 `html`。客户 key 由部署环境决定，客户端不能提交 `customer_key` 或 `base_url` 选择其他客户或资源基址。
- 服务端先走统一管理员认证 middleware，使用实时 session / RBAC，再从同一次 active revision effective session 同时检查打印 action 和模板依赖 module，避免跨 revision 拼接权限。未知模板、无 active revision、无打印 action 或 module 非 `enabled` 都拒绝渲染。前端新增正式模板时，如果没有同步服务端 PDF 模块门禁和字段合同测试，不能视为已接入打印闭环。
- 服务端再次解析 HTML / CSS allowlist，只接受文档内锚点和受限内嵌位图；脚本执行、缓存和 Chromium 出网均关闭，每次渲染使用独立 browser context。生产镜像以非 root 用户运行并保留 Chromium sandbox；这些静态合同不替代容器 sandbox、受控出网探针和目标环境真实 PDF smoke。
- 末尾附图不设业务或应用层张数上限，但不是无限资源承诺：手机照片和电脑截图会由打印窗口先生成适合纸面输出的图片快照，仍受浏览器内存、当前站点存储配额和整份 PDF 安全预算约束。服务端按整份打印内容控制资源预算：内嵌图片解码后合计最多 64 MiB，HTML 最多 96 MiB，请求体最多 128 MiB。代码默认并发为 4，通用 Compose 使用 2 GiB 容器预算和并发 4；确认资源充足的客户实例可使用 4 GiB 和并发 8。请求、并发和内存仍保留硬上限，避免异常客户端或突发渲染耗尽服务内存。

因此当前正式口径是：

- 右侧纸面 DOM 才是打印输出真源
- 左侧字段面板只是同窗编辑入口，不是另一套打印模板
- 左侧末尾附图管理区负责维护 `appendixImages[]`，但真正输出仍以右侧纸面末尾附图 DOM 为准
- PDF 和浏览器打印都不应该再维护第二份独立 HTML
- 两套合同在当前纸面高度超过单页 A4 时，会动态把整次浏览器打印与在线 PDF / 下载 PDF 的 `@page` 切到 `margin: 5mm 0 5mm`；未跨页时保持 `0` 页边距，避免本应单页的合同被额外挤出空白尾页。

## 5. 五套模板末尾附图覆盖

| 模板 | 当前末尾附图真源 | 老草稿迁移 | 纸面 / PDF / 打印行为 |
| --- | --- | --- | --- |
| 采购合同 | `appendixImages[]` | 无专用老槽位 | 合同正文与签字区之后，按顺序自适应半宽 / 整行混排，超长图连续分段 |
| 加工合同 | `appendixImages[]` | 仅当该字段缺失时，按 `attachment-1` -> `attachment-2` 迁移 | 合同正文与签字区之后，按顺序自适应半宽 / 整行混排，超长图连续分段 |
| 物料分析明细表 | `appendixImages[]` | 仅当该字段缺失时，按 `footer_left` -> `footer_right` 迁移 | 保留右上角业务图片槽；补充图在模板末尾自适应混排并按需分段 |
| 色卡 | `appendixImages[]` | 无专用老槽位 | 色卡正文与页脚之后，按顺序自适应混排并按需分段 |
| 作业指导书 | `appendixImages[]` | 无专用老槽位 | 保留右上角产品图和编号行图片；补充图在全部正文续页之后自适应混排并按需分段 |

表中“无专用老槽位”表示不需要从模板专属图片字段搬运，不表示绕过通用的窗口级草稿兼容回读。两类老槽位的迁移都必须使用“字段是否存在”判断，不能用数组是否为空判断，否则用户显式移除全部附图后会被老槽位反向恢复。

## 6. 当前实现边界

已经收口的部分：

- 独立弹窗编辑，不在打印中心页内直接改纸面
- 直接打开真实模板工作台 URL
- 窗口级 `state` 和窗口级草稿隔离
- 首屏准备态
- 壳页保留兼容恢复，不作为可编辑模板主路径
- PDF / 打印都从当前弹窗里的同一份纸面 DOM 输出
- 五套模板共用 `appendixImages[]`、统一末尾附图管理、自适应半宽 / 整行混排与超长图分段输出
- 两套合同继续共用同一条 A4 续页判定与动态页边距主链

仍未完成的部分：

- 当前正文仍是 React 路由页，不是 builder + 共享 runtime 的纯静态 HTML 输出
- 当前覆盖采购合同、加工合同和三套工程资料模板；正文仍按模板组件渲染，还没有收口成纯 builder + 多模板统一 runtime

## 7. 后续改造守则

- 若继续扩模板，优先复用 `printWorkspace.js + PrintWorkspaceShell.jsx + usePersistentPrintWorkspaceDraft` 这条主链，不要再为新模板单独发明新的弹窗入口和新的草稿 key。
- 若后续继续扩展打印能力，优先补更强的共享编辑 runtime，不要重新退回“每个模板页自己兜刷新 / 自己猜分页”的分叉写法。
- 若修改打印输出逻辑，必须同时检查：
  - 左侧字段面板是否仍和右侧纸面同步
  - 当前窗口级草稿是否仍独立
  - `appendixImages[]` 是否仍是末尾附图的唯一真源，显式空数组是否仍能阻止旧槽位复活
  - 多选、调序、移除和版式切换后，逻辑图片顺序、半宽配对、整行独占和超长图分段是否仍正确
  - PDF 和浏览器打印是否仍读取同一份右侧 DOM
  - 跨页时是否仍会统一切到续页页边距，确保第 `2` 页起正文不会贴页顶开始排
