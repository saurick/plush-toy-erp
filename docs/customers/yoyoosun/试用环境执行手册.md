# 永绅 yoyoosun 试用环境执行手册 / Yoyoosun Trial Environment Runbook

本手册用于在个人本地验收库和 133 试用服务器的独立验收库重放同一版模拟数据，使甲方可以逐页看、查、筛选和打印。当前唯一数据合同是：

| 项目             | 固定值                                                                                |
| ---------------- | ------------------------------------------------------------------------------------- |
| 数据集           | `yoyoosun-manual-acceptance`                                                          |
| 版本             | `2026.07.16-v5`                                                                       |
| 运行批次         | `20260716-V5`                                                                         |
| 页面验收目标     | 50 项：2 个登录与入口、29 个电脑业务页、9 个岗位任务页、5 个打印预览和 5 个打印工作台 |
| 来源单可见前缀   | `YS5`                                                                                 |
| 岗位任务前缀     | `YS-V5`                                                                               |
| 本地验收库       | `plush_erp_acceptance_20260716_v5_dev`                                                |
| 133 验收库       | `plush_erp_uat_20260716_v5`                                                           |
| 133 Compose 栈   | `plush-toy-erp-v5`（与旧 `plush-toy-erp-prod` 并存）                                  |
| 133 V5 隔离端口  | PostgreSQL `55435` / 后端 `8315` / 前端 `5185`                                        |
| 133 数据目录     | `/home/simon/plush-toy-erp-v5/data/postgres`                                          |
| 133 migration 锁 | `/home/simon/plush-toy-erp-v5/run/atlas-migrate.lock`                                 |

这是脱敏模拟数据，不是永绅真实交易、库存、出货或财务事实，也不代表客户已签收。两端使用同一业务语义和数量合同，但数据库 ID 必须各自生成；禁止复制数据库行或把“编号相同”写成“两端已验证一致”。

## 1. 固定边界

- 正式生产部署不自动造数。只有明确的 `local-dev` 和 `customer-trial-133` 验收目标允许执行 V5。
- 本地必须使用非 `8300` 的隔离后端和 `plush_erp_acceptance_*_dev` 库。共享开发库的稳定超级管理员始终是 `admin/adminadmin`；验收脚本不得轮换、重置或归档该账号。
- 133 只使用独立库 `plush_erp_uat_20260716_v5`，不改写旧库。稳态 Compose 必须是 `BOOTSTRAP_ADMIN_ONCE=false`，且不得持久保存 `APP_ADMIN_PASSWORD`。
- 数据生成只走正式 JSON-RPC / usecase 入口。不直接复制表行，不用 Workflow payload 冒充 Fact，不因试用放宽权限、状态、幂等或数据库目标门禁。
- 所有用于页面验收的账号都是 `demo_*` 岗位账号。超级管理员只用于安全前置和必要的配置操作，不用来冒充普通岗位验收。

## 2. 为什么页面会空

页面只读当前连接的后端和数据库。如果 V5 写在隔离后端 `8310`，而浏览器仍打开共享本地 `8300`，或 133 仍指向旧 UAT 库，“出货放行”等页面就会空。这不能用另外补一份页面 fixture 解决，必须在该页实际连接的同一数据库完整重放 V5。

出货有两条不同真源：

| 页面     | 数据真源                               | V5 数量                              |
| -------- | -------------------------------------- | ------------------------------------ |
| 出货放行 | Workflow `task_group=shipment_release` | 4 条，分别是可处理、阻塞、完成和退回 |
| 出货管理 | Shipment Fact                          | 47 张，含 1 张 25 行明细长单         |

Workflow 任务完成不等于出货 Fact 已过账，两者不可混用或相互凑数。

## 3. 一次性执行顺序

### 3.1 冻结代码和数据库身份

1. 记录完整 40 位 Git commit、镜像 digest 和最新 Atlas migration。
2. 为本地与 133 分别创建全新隔离库，执行 migration `status -> apply -> status`；不删除旧库。133 的每条 Compose 命令必须显式带 `-p plush-toy-erp-v5`，并同时带 `compose.yml` 与 `compose.customer-trial-133.yml`，后者只允许声明 `name: plush-toy-erp-v5`。
3. 133 新库只通过受控的一次性 bootstrap 创建正式管理员，成功后读回 admin、`admin_bootstrap.completed` marker、audit 和 RBAC，再以无密码的稳态环境重建 app-server。
4. 核对 `/healthz`、`/readyz` 和 `/readyz/runtime-identity`，确认数据库、commit 与 migration 都是本批次。

133 的 migration 只能使用固定 release 内脚本与 release 外的精确 runtime env。先确认 env 归当前用户所有、权限为 `0600`，文件和父路径都不是符号链接，然后按下列顺序执行：

```bash
cd /home/simon/plush-toy-erp-v5/current/server/deploy/compose/prod
export COMPOSE_OVERRIDE_FILE=/home/simon/plush-toy-erp-v5/current/server/deploy/compose/prod/compose.customer-trial-133.yml
export COMPOSE_ENV_FILE=/home/simon/plush-toy-erp-v5/runtime/.env.customer-trial-133

sh ./migrate_online.sh --status-only
sh ./migrate_online.sh

docker compose \
  -p plush-toy-erp-v5 \
  --env-file "$COMPOSE_ENV_FILE" \
  -f /home/simon/plush-toy-erp-v5/current/server/deploy/compose/prod/compose.yml \
  -f "$COMPOSE_OVERRIDE_FILE" \
  stop app-server

MIGRATION_MAINTENANCE_CONFIRMED=1 sh ./migrate_online.sh --apply
sh ./migrate_online.sh --status-only
```

停写窗口只停 V5 `app-server`，V5 PostgreSQL 继续运行供宿主 Atlas 连接；旧 `plush-toy-erp-prod` 栈及其数据不受影响。V5 迁移不接受宿主或 env 文件覆盖 `MIG_DIR / ATLAS_BIN / PSQL_BIN / POPULATED_UPGRADE_PREFLIGHT`。

133 启动前先跑受控 preflight；它会对 runtime env 生成私有快照，结束前复核原文件的 owner、mode 和 SHA-256，并拒绝 project name、数据目录、migration 锁、数据库或任一宿主端口与旧栈重叠。固定 Jaeger 端口组是 `45775 / 46831 / 46832 / 45778 / 46687 / 54268 / 54250 / 49411 / 44317 / 44318`。preflight 与后续 Compose 必须在同一个干净 shell 执行；宿主不得预先定义 env-file 同名键、`COMPOSE_PROJECT_NAME / COMPOSE_FILE / COMPOSE_PROFILES / COMPOSE_ENV_FILES / COMPOSE_PATH_SEPARATOR` 或 `DOCKER_HOST / DOCKER_CONTEXT / DOCKER_TLS_VERIFY / DOCKER_CERT_PATH`：

```bash
cd /home/simon/plush-toy-erp-v5/current
bash scripts/deploy/production-preflight.sh \
  --env-file /home/simon/plush-toy-erp-v5/runtime/.env.customer-trial-133 \
  --compose-dir /home/simon/plush-toy-erp-v5/current/server/deploy/compose/prod \
  --compose-override /home/simon/plush-toy-erp-v5/current/server/deploy/compose/prod/compose.customer-trial-133.yml
```

preflight 成功后，所有 `docker compose` 命令都必须继续带同样的绝对 `--env-file`、两个绝对 `-f` 路径和 `-p plush-toy-erp-v5`。稳态服务启动后再运行同一命令并追加 `--runtime --expected-release '<40-character-lowercase-git-sha>'`；它会读回四服务的唯一容器、env image ref、image / container content ID、app / web `GIT_SHA`、精确容器名、project label、宿主端口、PostgreSQL 挂载源、app 试用变量和不含凭据的 DSN 身份。

### 3.3 生成岗位账号和全页数据

完整 CLI 、确认串、attestation 和报告路径以 [scripts/qa/README.md](../../../scripts/qa/README.md) 的“双环境全页面手工验收数据”为准。操作顺序固定为：

```text
core 稳定码读回
  -> baseline 空业务库回执
  -> role 岗位账号
  -> source 主数据与来源单
  -> task 九岗位 Workflow 任务
  -> facts 来源驱动业务事实
  -> attachments 附件引用
  -> readiness 只读数量与身份读回
```

## 5. 页面验收标准

1. readiness 必须读回 50/50 页面归属和同批 role/source/task/facts 身份，但 readiness 不能冒充浏览器通过。
2. browser 必须用真实 `demo_*` 账号登录，逐页读 DOM，不只检查 URL 或 HTTP 200。
3. 通用列表的 DOM 数量只能在 runner 的 `baseline` fresh 空库基线已记录且通过、并且空库后只重放当前 V5 时作为同批证据；有历史业务数据的库会在写入前直接阻断，不得用旧数据凑够最低数量。
4. 业务看板不仅要显示数量，还要按当前 RBAC 和 effective session 点开至少一个真实可达来源页。
5. “出货放行”必须找到精确 `YS-V5-CK-02/13/16/19`，分别覆盖可处理、阻塞、完成、退回，且同时有临期和逾期；“出货管理”必须找到 47 张 Fact 并打开精确 25 行长单。
6. 每个打印预览必须至少证明 1 份可见数据；采购、委外和 BOM 每个打印工作台必须先证明本批来源单至少 5 条，再选中实际 25 行源单，验证字段带值、刷新恢复、页面行数与真实 PDF；不得只检查 `%PDF` 文件头。
7. 本地和 133 必须分别生成 readiness 和 browser 报告。一端绿色不能代替另一端。

### 5.1 账号、入口和岗位动作辅助核对

V5 的 50 项浏览器报告是最终页面证据；下面的辅助检查用于在正式逐页验收前定位账号、入口或岗位动作问题，不能替代 V5 readiness、目标环境浏览器报告或 release evidence。

再执行试用入口的无写入浏览器预检：

该预检不读密码、不登录、不调用 JSON-RPC、不启动浏览器、不启动 Vite、不读取客户配置脚本、不写数据库。需要留下真实入口回归的脱敏结果时执行：

无凭据的 effective session 探针只用于记录未登录边界：

```bash
node scripts/qa/customer-config-effective-session-probe.mjs --json
```

探针预期记录 `40302 未登录`，不读取 token、不登录、不发布 / 激活 / 回滚客户配置、不写数据库、不证明 active revision、普通账号菜单投影或后端 RBAC。真实浏览器回归中的 `window.__PLUSH_ERP_EFFECTIVE_SESSION_DIAGNOSTIC__` 只保存脱敏摘要，不保存 token、Authorization header、config hash、raw id、action 列表或客户包 payload。

岗位任务动作另有无写入预检和真实脱敏报告：

## 6. 重放、退出和回滚

- 首次 apply 前，该目标的规范 `dataset/apply-report.json` 必须不存在。阶段失败或完整成功后的同批重放，必须保留原回执，并在原命令追加 `--resume-report output/qa/manual-acceptance/datasets/2026.07.16-v5/<target>/dataset/apply-report.json`；不得删除回执后伪装 fresh apply。
- fresh apply 会捕获一次岗位任务时间锚点并写进总回执；同批 resume 只校验并复用，不按重跑时间改写到期日。本地与 133 分别绑定自己的 fresh 锚点，但语义 digest 和时间策略相同。浏览器若已超过回执中的“即将到期”有效时间，不得继续使用旧批次报告，应换新数据版本并从 fresh 空库重放。
- fresh 和 resume 共用 `dataset/.apply.lock` 原子排他锁，第二个进程会在任何写入前停止。异常退出留下的锁不得直接删除；确认锁内 PID 已退出后，只能按错误提示将原锁重命名为 .stale-\* 归档，再用原参数重跑。
- 相同 `target + dataVersion + runId` 的 resume 必须幂等：各类数量不增长，已有引用指向同一业务对象，报告明确记录 replay；目标、数据库、客户配置、release / migration、连续阶段或组件 digest 任一漂移都要停止。
- 退出验收批次使用 `manual-acceptance-source-retire.mjs`：来源单按正式生命周期取消/归档，主数据停用，不物理删除已生效 Fact 和审计记录。
- 本地回滚是停止隔离服务并切回原本开发库，不动共享 dev admin。133 回滚必须成对恢复上一个 commit/image/database 组合，不得只切镜像或只切数据库。
- 脚本失败时保留已生成的脱敏报告和状态；不自动删表、清库、提权或反复 bootstrap admin。

## 7. 停止条件

以下任一情况必须停止，不得宣称“每页已有数据”：

- commit、image、database、migration 或 customer-config revision 任一不符。
- 133 尚未激活当前 V5 `customer-trial-133` 配置，或仍指向旧库。
- 数据集仅写入隔离后端，但用户实际打开的前端连接了另一后端。
- readiness 不是 50/50，browser 未逐页执行，或 5 份 PDF 未生成。
- 出货放行 4 条任务、出货 47 张 Fact、25 行长单或任一其他类别读回不符。
- 重放导致数量增长、重复编号或引用漂移。
- 普通试用账号必须获得 super admin / debug 权限才能看页面。
- 只有自动化绿色，没有目标环境真实页面和读回证据。
