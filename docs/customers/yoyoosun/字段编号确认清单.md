# 永绅 yoyoosun 字段编号确认清单 / Yoyoosun Field Numbering Confirmation Checklist

本文用于和永绅 yoyoosun 确认字段显示、字段必填和编号规则。本文不是 runtime、schema、migration、API、RBAC、导入或真实账号真源；当前配置草案在 `config/customers/yoyoosun/fieldNumberingConfig.mjs`，且 `runtimeEnabled=false`。

确认前禁止把本文中的客户习惯字段升级为 Product Core、Ent schema 必填字段、真实导入规则或行业默认模板。

## 1. 确认边界

| 事项 | 当前边界 |
| --- | --- |
| 客户 key | `yoyoosun` 是稳定客户 key，不是 SaaS runtime tenant |
| 运行时状态 | 字段 / 编号配置草案尚未接入运行时 |
| Schema | 本清单不新增、不修改 Ent schema 或 Atlas migration |
| RBAC | 本清单不改变后端权限真源 |
| 导入 | 本清单不批准真实 import / backfill |
| 事实层 | 本清单不生成库存、出货、财务、采购入库或生产事实 |

## 2. 客户档案字段

| 确认项 | 当前建议 | 需客户回答 | 不做事项 |
| --- | --- | --- | --- |
| 客户编码 | 保留为可选字段或导入候选；重复时进入 unresolved queue | 编码由系统生成、人工维护，还是沿用历史资料？ | 不因缺失编码自动伪造 |
| 客户简称 | 作为显示偏好候选 | 列表和搜索是否优先显示简称？简称为空时显示全称是否可接受？ | 不改变 `customers` 真实名称字段 |
| 税号 | 财务资料候选，当前不强制必填 | 试用阶段是否必须录入税号？哪些角色维护？ | 不生成发票、应收或付款事实 |
| 地址 / 国家地区 | 客户资料快照候选 | 地址是否需要拆字段，还是先放备注 / 后续资料？ | 不把单客户地址格式写成 Product Core 必填 |

## 3. 供应商字段

| 确认项 | 当前建议 | 需客户回答 | 不做事项 |
| --- | --- | --- | --- |
| 供应商编码 | 保留为可选字段或导入候选 | 供应商和加工厂是否共用编号规则？ | 不因缺失编码自动伪造 |
| 供应商分类 | 进入 Customer Config / 行业模板候选评审 | 材料供应商、加工厂、客户、临时文本是否需要明确分类？ | 不直接改 Product Core 枚举 |
| 结算资料 | 作为客户材料或 future finance 线索 | 试用阶段是否只记录文本，还是需要后续财务模块？ | 不生成应付、发票或付款事实 |
| 厂商料号 | 作为材料 / 采购线索 | 厂商料号归供应商、材料，还是采购源单据？ | 不自动创建采购订单或物料编码 |

## 4. 销售订单字段

| 确认项 | 当前建议 | 需客户回答 | 不做事项 |
| --- | --- | --- | --- |
| 销售订单编号 | 必须唯一；先确认生成方式 | 系统自动生成、人工录入，还是导入保留？ | 不恢复旧 `project-orders` 双写真源 |
| 客户订单号 | 作为外部来源单号候选 | 是否每张销售订单都必须有客户订单号？谁维护？ | 不把客户订单号当内部销售订单主键 |
| 订单日期 | 作为源单据日期 | 录入时默认当天是否可接受？ | 不根据订单日期生成库存或财务事实 |
| 交期 | 表示订单承诺 | 是否按整单交期、明细交期，还是两者都要？ | 不把交期或出货日期当作已出货 |
| 业务员 / 跟单人 | 已进入销售订单录入字段；先作为订单责任文本和搜索 / 列表字段 | 是否需要和登录账号、角色或后续任务绑定？ | 不在前端本地派生后端 workflow fact |
| 联系人 | 已进入销售订单联系人快照；优先从客户主联系人带出，也允许本单手工修正 | 客户订单是否必须指定联系人？电话 / 邮箱是否必填？ | 不把订单联系人快照反写成客户联系人主数据 |

## 5. 销售订单明细字段

| 确认项 | 当前建议 | 需客户回答 | 不做事项 |
| --- | --- | --- | --- |
| 产品 | 必须唯一匹配 existing `products` | 产品由谁维护？订单录入时是否允许临时产品？ | 不自动创建 `product_skus` |
| 款式编号 | SKU / product / industry template 评审线索 | 款式编号与产品编号是否同一概念？ | 不落成 sales_order_items runtime 字段 |
| 颜色 / 尺寸 | SKU 评审线索 | 是否每个产品都有颜色 / 尺寸？是否影响库存和出货？ | 不自动生成 SKU、库存批次或库存余额 |
| 数量 / 单位 | 使用现有订单明细数量和单位 | 单位是否允许只、件、箱、套等混用？ | 不用单位文本绕过现有 unit 真源 |
| 单价 / 金额 | 源单据金额候选 | 试用阶段是否需要显示金额？是否含税？ | 不生成应收、发票或收款 |

## 6. 产品和未来 SKU 编号

| 确认项 | 当前建议 | 需客户回答 | 不做事项 |
| --- | --- | --- | --- |
| 产品编号 | existing products 编码候选 | 产品编号、款式编号和 SKU 编号是否分层？ | 不因颜色 / 尺寸直接创建 SKU |
| 产品名称 / 规格 | existing products 字段候选 | 规格是产品属性、SKU 属性，还是订单备注？ | 不把订单快照当产品唯一真源 |
| 包装版本 | future SKU / BOM 版本线索 | 包装版本是否影响 BOM、库存或出货？ | 不在本阶段落版本 schema |

## 7. 采购和合同编号

| 确认项 | 当前建议 | 需客户回答 | 不做事项 |
| --- | --- | --- | --- |
| 采购订单号 | V2 purchase order 候选 | 当前试用是否必须先出现采购订单源单据？ | 不自动生成采购入库事实 |
| 加工合同号 | 委外源单据线索 | 加工合同是否独立于供应商 / 采购单？ | 不生成委外发料、回货或结算事实 |
| 采购合同样式 | 客户打印样本 | 是否只作为 yoyoosun 样本，还是多个客户共用？ | 不实现通用打印模板内核 |

## 8. 确认结论记录方式

客户确认后只记录非敏感结论：

| 结论类型 | 记录位置 |
| --- | --- |
| 已确认字段显示 / 必填 | 更新 `客户配置草案.md` 和 `fieldNumberingConfig.mjs` |
| 已确认编号规则 | 更新 `客户配置草案.md` 和 `fieldNumberingConfig.mjs` |
| 仍未确认的问题 | 更新 `问题待办.md` 或 `导入待确认队列.md` |
| 属于 Product Core 的候选 | 单独进入产品评审，不直接落 schema |
| 属于客户差异的候选 | 更新 `差异登记.md` 或 Customer Config |

只有明确确认且通过产品 / 架构边界评审的字段，才能进入后续 runtime、schema、导入 loader 或行业模板任务。
