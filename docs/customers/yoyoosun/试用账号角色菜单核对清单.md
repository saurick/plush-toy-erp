# 永绅 yoyoosun 试用账号角色菜单核对清单 / Yoyoosun Trial Account Role Menu Checklist

本文用于试用前核对永绅 yoyoosun 的测试账号、预设角色、桌面菜单和岗位任务端入口。本文不是 runtime、schema、migration、API、RBAC 或真实账号真源；权限真源仍是 `server/internal/biz/rbac.go`，菜单客户配置源仍是 `config/customers/yoyoosun/menuConfig.mjs`，实际前端运行时由部署侧 `customer-config.js` 注入。

本文禁止记录真实密码、显式覆盖密码、短信验证码、token、私钥、数据库连接串或客户真实敏感账号密钥。公开测试默认值一旦 seed 就是可登录凭据，只允许登记的隔离开发库使用；真实账号只在运行环境和授权管理流程中维护。

## 1. 试用账号准备原则

- 普通业务试用账号按角色分配，不使用 `is_super_admin=true` 作为业务试用账号。
- `debug_operator` 只允许 local / dev / test 使用，客户试用账号不分配 debug 权限。
- 一个真实人员可以绑定多个角色，但必须先确认是否确实需要跨岗位试用；不要为了省事把所有角色都给同一个普通账号。
- 岗位任务端入口只由 `mobile.<role>.access` 权限决定；设备只影响默认入口，不决定权限。
- 前端菜单隐藏不是安全边界；后端 JSON-RPC action permission、Workflow / Fact usecase 和任务归属校验才是最终保护层。
- 当前不新增 `tenant_id`，不创建 SaaS runtime tenant，也不把客户 key 当租户隔离字段。

标准角色演示账号为 `demo_boss`、`demo_sales`、`demo_purchase`、`demo_production`、`demo_warehouse`、`demo_quality`、`demo_finance`、`demo_pmc`、`demo_engineering` 和 `demo_admin`。无显式密码时，公开测试默认值 `12345678` 只允许登记的 `plush_erp_*_dev` 隔离开发库，并只生成九个普通业务角色账号；`demo_admin`、调试账号、人工验收账号和客户试用环境必须显式提供独立的非默认密码。

## 2. 账号模板

| 试用人员类型 | 建议角色 | 桌面重点入口 | 岗位任务端 | 核对重点 |
| --- | --- | --- | --- | --- |
| 管理层 / 老板 | `boss` | `工作台`、`任务看板`、主数据和业务只读入口 | `/m/boss/tasks` | 可看全局和审批，不代表能替各岗位写事实 |
| 业务 / 销售 | `sales` | `客户档案`、`销售订单`、相关出货协同入口 | `/m/sales/tasks` | 可维护客户和销售订单；销售订单不写出货、库存或财务事实 |
| 采购 | `purchase` | `供应商档案`、采购相关入口、入库相关入口 | `/m/purchase/tasks` | 采购任务和采购收货协同要区分采购事实入口 |
| 仓库 | `warehouse` | `库存台账`、`入库管理`、`出库管理`、`出货单` | `/m/warehouse/tasks` | 出货放行任务从任务看板 / 岗位任务端处理，不等于已出库；库存事实仍由专门 usecase 控制 |
| 品质 | `quality` | `来料质检`、任务看板里的异常协同任务 | `/m/quality/tasks` | 任务完成不等于质检事实通过；质检事实按 QualityUsecase 口径 |
| 财务 | `finance` | `对账/结算`、`待付款/应付提醒`、`应收提醒`、`发票/开票异常`、`入库管理`（来源只读核对） | `/m/finance/tasks` | 可核对应付关联的采购入库来源，但不能创建、调整、确认入库或办理采购退货；财务能力仍未进入完整客户试用承诺，不从出货放行自动生成 |
| PMC / 计划 | `pmc` | `生产进度`、任务看板里的计划 / 异常协同任务 | `/m/pmc/tasks` | 可看计划和风险，不等于替生产、仓库、品质、财务完成事实 |
| 生产 | `production` | `生产进度`、任务看板里的生产协同任务 | `/m/production/tasks` | 生产任务仍是岗位协同方向，生产事实未完整落地 |
| 工程 | `engineering` | `产品档案`、`工序档案`、`物料清单（BOM）`、任务看板里的工程资料任务 | `/m/engineering/tasks` | 可接住老板审批后派生的工程资料任务；BOM / 工序不写库存、采购、生产或成本事实 |
| 系统管理员 | `admin` | `权限管理` | 无固定业务岗位入口 | 只用于账号、角色、权限维护，不天然拥有业务事实处理权 |

## 3. 桌面菜单核对

| 角色 | 精确桌面菜单 |
| --- | --- |
| `sales` | 工作台、客户档案、销售订单、产品档案、生产订单、库存台账、出货单、任务看板、模板打印中心 |
| `boss` | 工作台、业务看板、任务看板、销售订单、采购订单、加工合同、入库管理、质量检验、库存台账、生产订单、生产排程、生产进度、生产异常、出货单、对账管理、应付管理、应收管理、发票管理、模板打印中心 |
| `engineering` | 工作台、产品档案、材料档案、工序档案、物料清单（BOM）、任务看板、模板打印中心 |
| `pmc` | 工作台、业务看板、产品档案、材料档案、工序档案、物料清单（BOM）、生产订单、生产排程、生产进度、生产异常、任务看板 |
| `purchase` | 工作台、供应商档案、材料档案、产品档案、工序档案、物料清单（BOM）、库存台账、采购订单、加工合同、入库管理、任务看板、模板打印中心 |
| `warehouse` | 工作台、材料档案、产品档案、库存台账、入库管理、出货放行、出库管理、出货单、任务看板 |
| `quality` | 工作台、材料档案、产品档案、工序档案、质量检验、生产订单、生产异常、库存台账、出货单、入库管理、加工合同、任务看板 |
| `finance` | 工作台、客户档案、供应商档案、对账管理、应付管理、应收管理、发票管理、加工合同、销售订单、质量检验、库存台账、入库管理、出货单、任务看板、模板打印中心 |
| `production` | 工作台、供应商档案、材料档案、产品档案、工序档案、生产订单、生产排程、库存台账、加工合同、生产异常、生产进度、任务看板、模板打印中心 |

试用前还要核对以下通用入口边界：

| 核对项 | 预期结果 |
| --- | --- |
| 登录后左侧菜单 | 显示当前 yoyoosun 菜单分组、排序和文案 |
| `客户档案` | 使用正式 customers 入口，不使用旧 `partners` |
| `供应商档案` | 使用正式 suppliers 入口，不使用旧 `partners` |
| `销售订单` | 使用正式 sales_orders 入口，不使用旧 `project-orders` |
| `任务看板` | 作为 Workflow 任务处理主入口保留 |
| `业务看板` | 仅老板和 PMC 显示；其他岗位从工作台、任务看板或岗位任务端查看与本岗有关的阻塞 / 逾期风险 |
| `生产排程`、`生产异常`、`出货放行` | 当前客户菜单显示并准备本批任务；它们仍不是领域事实入口，出货放行任务不等于 shipped fact |
| 旧 `/erp/master/partners` | 不作为产品内页面、菜单、重定向或权限别名保留 |
| 旧 `/erp/sales/project-orders` | 不作为产品内页面、菜单、重定向或权限别名保留 |
| 无权限角色访问业务入口 | 以前端隐藏和后端权限拒绝为准，不能只看菜单 |
| disabled 管理员 | 应被拒绝进入，不因已有角色而放行 |
| 普通非管理员 | 应按 `AdminRequired` 边界拒绝 |

## 4. 岗位任务端核对

| 核对项 | 预期结果 |
| --- | --- |
| `/entry` | 只选择“后台管理 / 岗位任务端”，不在登录前选择岗位 |
| 手机默认入口 | 可默认岗位任务端，但最终仍由账号权限决定 |
| 电脑默认入口 | 可默认后台管理，但最终仍由账号权限决定 |
| `/m/<role>/tasks` | 只有拥有对应 `mobile.<role>.access` 权限的账号可进入 |
| 多角色账号 | 自动进入第一个可用岗位；如后续确有高频切换需求，再单独设计岗位内切换 |
| 岗位任务处理 | 必须同时满足权限码、任务 `owner_role_key`、`assignee_id` 和 `task_status_key` 边界 |

## 5. 试用前人工核对步骤

1. 准备每类试用人员对应账号；文档只记录角色、用途和隔离开发默认值的适用边界，不记录真实或显式覆盖密码。
2. 给普通试用账号绑定业务角色，不给 `debug_operator`，不默认给 `is_super_admin=true`。
3. 用 `admin` 账号核对角色和权限码是否来自当前 RBAC 预设，不使用旧 `admin_users.menu_permissions` 或 `admin_users.mobile_role_permissions`。
4. 分别登录业务、采购、仓库、品质、财务、PMC、生产、工程和管理层账号，核对桌面菜单可见性和岗位任务端可达性。
5. 明确向试用人员说明：`客户档案`、`供应商档案`、`销售订单` 是当前正式试用入口；旧 `partners / project-orders` 不再使用。
6. 明确向试用人员说明：销售订单只是源单据 / 客户订单承诺，不代表出货、库存扣减、应收、开票或收款完成。
7. 记录试用反馈时按业务问题归类，不把单个客户习惯直接写进 Product Core、schema、fact 或 workflow。

## 6. 不在本清单内完成

- 不创建真实账号或真实密码。
- 不执行真实数据导入。
- 不新增 schema、migration、API、RBAC 权限码或角色 runtime。
- 不恢复旧入口、旧重定向、旧权限别名或旧 `business_records` 写入。
- 不新增 SaaS 多租户、`tenant_id`、license server、套餐计费或客户工单系统。
- 不把 yoyoosun 菜单配置当成后端权限或事实规则。

## 7. 通过标准

| 标准 | 通过口径 |
| --- | --- |
| 正式入口 | 试用账号能按角色看到应试用的正式入口 |
| 旧入口退出 | 旧 `partners / project-orders` 不再作为产品入口被使用 |
| 权限边界 | 菜单可见性、后端权限和岗位任务归属边界没有互相替代 |
| 移动端入口 | 有 `mobile.<role>.access` 才能进入对应岗位任务端 |
| 敏感信息 | 公开测试值只用于登记的隔离开发库；文档不记录真实或显式覆盖密码、token、验证码或密钥 |
| 交付边界 | 试用人员知道销售订单、出货、库存和财务事实仍是不同边界 |
