# 永绅流程编排闭环矩阵

本文是内部实施 / 研发使用的“逐流程、逐节点当前系统合同”。它回答每个节点由谁办理、调用什么业务动作、通过 / 退回 / 阻塞后发生什么，以及最终改变的是任务、来源单据还是业务事实。

- 与甲方确认目标职责和流程：看[甲方角色职责与业务流转确认表](甲方角色职责与业务流转确认表.md)。
- 查 9 岗位精确权限、页面和责任池：看[角色能力与流程矩阵](角色能力与流程矩阵.md)。
- 查可演示、可试用、已交付和 UAT：看[客户交付矩阵](客户交付矩阵.md)。

本文不把 `preview_only` 配置写成 runtime，不把 Workflow task done 写成 Fact posted，也不缓存单一“系统已完成”结论。

## 3. 结果分层

| 层 | 典型状态 | 完成后能证明 | 不能证明 |
| --- | --- | --- | --- |
| Workflow task | ready / blocked / done / rejected | 某人已办理、退回或阻塞 | 来源单据已生效、库存 / 出货 / 财务已过账 |
| Process instance / node | active / blocked / completed | 注册流程推进到指定节点 | 节点外的领域状态已自动改变 |
| Source Document | 草稿 / 已提交 / 已批准或生效 / 已关闭 / 已取消 | 销售、采购、生产等业务承诺状态 | 质检、库存、出货、财务事实 |
| Quality Fact | 待检 / 合格 / 让步 / 不合格 / 更正 | 正式质量判定 | 自动入库、退货、返工或出货 |
| Inventory / Shipment Fact | DRAFT / POSTED / SHIPPED / CANCELLED / REVERSAL | 正式库存或出货结果 | 财务已经收付 |
| FinanceFact | DRAFT / POSTED / SETTLED / CANCELLED | 应收、应付、发票等财务来源事实 | 真实收付款、核销、红冲或银行处理完成 |
| FinancePayment | DRAFT / POSTED / REVERSED | 独立收款 / 付款已登记、过账或冲正 | 发票是核销来源，或银行直连 / 总账已完成 |
| FinanceAllocation | POSTED / REVERSED | 收付款对同方向应收 / 应付的核销分配或反向审计 | 原应收 / 应付被删除，或发票已参与收付款分配 |
| FinanceCreditNote | POSTED / REVERSED | 正式页面和服务端都只允许从应收 / 应付发起红冲或反向红冲；来源类型守卫已闭环 | 原事实被改写、删除，或税控处理完成 |

详细生命周期只链接[状态字典与生命周期索引](../../architecture/状态字典与生命周期索引.md)，本文不复制第二份状态机。

## 4. ProcessRuntime 注册流程

### 4.1 `sales_order_acceptance`

| 顺序 | 节点 | 办理 / 能力 | 通过后 | 退回 / 阻塞 | 来源单据 / Fact 结果 | 当前缺口 |
| ---: | --- | --- | --- | --- | --- | --- |
| 1 | `submit_sales_order` | 销售订单提交领域命令 | SalesOrder 进入已提交，自动创建老板审批任务 | 领域命令失败则实例不前进 | 只改变 SalesOrder；不写库存 / 出货 / 财务 | 无 |
| 2 | `order_approval` | 老板责任池 approval，要求 `workflow.task.approve` | 通过后自动执行唯一 `activate_sales_order` 领域命令 | 退回 / 阻塞记录原因并失败关闭，不继续创建下游任务 | 任务结论与领域命令结果分别审计 | 甲方 PMC 前置节点和老板生产资料二审未映射 |
| 3 | `activate_sales_order` | 自动领域命令 | 以同一事务写 `SUBMITTED → ACTIVE` 与 durable result，再进入工程 / PMC 协作 | 失败按 durable command evidence 恢复，实例不伪完成 | 只改变 SalesOrder；不写库存 / 出货 / 财务 | 无公开 submit / activate 旁路 |
| 4 | `engineering_data` | 工程人工任务；永绅选择的变体包含该节点 | 完成后自动创建 PMC 评审任务 | 可退回 / 阻塞 | 工程资料仍由产品 / BOM / 工序 usecase 维护 | 甲方要求的 PMC 前置节点和老板生产资料二审未映射 |
| 5 | `order_review` | PMC 人工任务 | 自动进入 end | 可退回 / 阻塞 | 只完成评审协作 | 不自动发布生产单或生成采购单 |
| 6 | `end` | 流程结束 | 实例 completed | 不适用 | 不额外写 Fact | 无 |

结论：当前已实现“销售提交 → 老板审批 → 唯一激活命令 → 工程 → PMC”的自动交接；审批通过后 SalesOrder 以同一事务进入 `ACTIVE` 并保存 durable result，公开 submit / activate 旁路已关闭。甲方 PMC 前置评审和老板生产资料二审仍待确认，本流程为 `runtime_enabled_local_target_evidence_required`。

### 4.2 `material_supply`

| 顺序 | 节点 | 办理 / 领域动作 | 通过后 | 拒绝 / 失败 | 来源单据 / Fact 结果 | 当前缺口 |
| ---: | --- | --- | --- | --- | --- | --- |
| 1 | `purchase_order_approval` | 老板责任池 approval，要求 `workflow.task.approve` | 通过后自动进入采购批准命令 | 退回 / 阻塞不创建收货或库存事实 | 只形成审批任务结论 | 目标账号 smoke / UAT 待补 |
| 2 | `approve_purchase_order` | 自动领域命令 | `PurchaseOrderUsecase.ApprovePurchaseOrderForProcessCommand` 写唯一批准状态与 durable result | 失败按领域命令恢复合同处理 | 只改变 PurchaseOrder；直批 API 已关闭 | 无双轨 |
| 3 | `purchase_receipt_source` | 采购从已批准采购订单创建收货草稿 | 进入来料质量关口 | 来源、数量、状态或幂等不合法则停在当前节点 | 生成 Receipt DRAFT 和待检来源，不写可用库存 | 无 |
| 4 | `incoming_qc` | 品质评估逐行正式质检 | 全部满足通过 / 让步条件后进入入库 | 任一行拒绝则不能过账 | 只读取正式 Quality Fact 决定资格 | 首次拒绝后的处置不属于本 Runtime 节点，需要独立来源动作 |
| 5 | `warehouse_inbound` | 仓库确认正式入库 | 进入 end | 状态、质检、数量、版本或库存校验失败则不前进 | Receipt POSTED 后才写 inventory IN | 无 |
| 6 | `end` | 流程结束 | 实例 completed | 不适用 | 不额外写 Fact | 无 |

首次来料拒绝已有独立正式处置：品质可按精确收货行和部分数量创建“退回供应商 / 供应商补换”，所有未取消处置累计不得超过来源行。补换确认会生成新的 DRAFT 收货、单行待收、零余额 HOLD 批次和 SUBMITTED IQC，原收货不因部分处置被整单取消；取消只在补换新待收仍可安全撤回时终止该链。当前仍缺目标 migration apply / readback、真实岗位 smoke 与 UAT。

### 4.3 `finished_goods_delivery`

| 顺序 | 节点 | 办理 / 领域动作 | 通过后 | 拒绝 / 失败 | 来源单据 / Fact 结果 | 当前缺口 |
| ---: | --- | --- | --- | --- | --- | --- |
| 1 | `finished_goods_quality` | 品质通过正式质检用例判定 | 进入财务审批 | 不合格则实例不前进 | 写正式 Quality Fact；不等于出货 | 与直连出货前质检 side flow 的入口关系需按实际来源核对 |
| 2 | `shipment_finance_approval` | 财务责任池 approval，要求 `workflow.task.approve` | 通过后自动执行 Shipment 放行领域命令 | 退回 / 阻塞不允许进入正式出货 | 记录审批意见、来源和轨迹；不写收付款 | 目标财务账号 smoke / UAT 待补 |
| 3 | `shipment_finance_release` | 自动领域命令 | CAS 写 Shipment `APPROVED`、版本、actor、时间和流程锚点 | 失败按 durable command evidence 恢复 | 形成不可绕过的 Shipment 放行门禁；不写应收、发票或收付款 | 无公开财务放行 execute 旁路 |
| 4 | `shipment_execution` | 仓库调用正式出货 usecase | 进入应收线索 | 状态、来源、财务门禁、质检、预留或库存失败则不前进 | 成功后 Shipment SHIPPED + inventory OUT | `shipping_released` 或审批任务完成都不能单独当成 shipped |
| 5 | `receivable_lead` | 财务从 SHIPPED 来源创建应收草稿 | 进入 end | 非 SHIPPED 或重复有效来源时拒绝 | 只创建 FinanceFact DRAFT | 仍需独立 POST；不等于收款 |
| 6 | `end` | 流程结束 | 实例 completed | 不适用 | 不额外写 Fact | 无 |

结论：Product Core 已把成品质检、财务 approval、Shipment 版本化放行、正式出货和应收草稿分层接入；普通与流程出货均要求财务门禁为 `APPROVED`。本流程为 `runtime_enabled_local_target_evidence_required`，目标 migration、真实岗位 smoke 和客户 UAT 仍未完成。

## 5. 固定生产路线与 ProcessRuntime 的关系

客户已确认高层主线为：布料加工 → 裁片检验 → 车缝内外发决策 → 皮套检验 → 手工内外发决策 → 成品检验 → 针检 / 抽检 / 条件性客户验货 → 包装入库 → 放行发货。

Product Core 使用固定 `PLUSH_SEW_HAND_V1` 承接生产执行：发布时冻结路线和工序快照，WIP 记录本厂移交或外发回仓，裁片合格后才允许守恒拆批，车缝与手工逐道独立决定本厂或外发，各质量关口写正式 Quality Fact。客户包的 `production_to_inventory` 继续保持 `preview_only`，只解释跨模块意图，不执行或覆盖该领域路线。

版本状态必须分开写：较早固定 WIP migrations 已 apply 到登记的本地验收库和 133 V5 技术试用库，并已有页面 / 角色技术 smoke；当前仓库后续异常处置 migration 尚未整体发布到 133，客户岗位 UAT 也未签收。因此不能笼统写成“全部未 apply”，也不能把当前 HEAD 写成已交付。

## 6. `preview_only` 客户工作流

| workflow | 节点顺序 | 作用 | 当前不能宣称 |
| --- | --- | --- | --- |
| `sales_order_approval` | sales_submit → boss_approval → engineering_data → pmc_review → end | 客户审批目标预览 | 甲方 PMC 前置、老板二审、完整退回链或 SalesOrder 激活已闭环 |
| `purchase_order_approval` | purchase_submit → boss_review → warehouse_prepare → quality_prepare → end | 采购评审与到货准备预览 | 任务完成等于采购批准、入库或应付 |
| `payment_approval` | finance_submit → boss_approval → end | 付款申请 / 审批候选 | 它不会创建、过账或冲正 FinancePayment；当前也没有正式付款审批链 |
| `finished_goods_delivery` | finished_goods_qc → finance_release → shipment_execution → receivable_lead → end | 成品交付目标预览 | 与同名 Product Core ProcessRuntime 相同，或自动写质检、库存、出货、应收 |

所有对象仍是 `workflow_only + preview_only`，`runtimeEnabled=false`。Product Core 当前真实收付款、多来源分配、冲正和红冲走独立 FinancePayment / Allocation / CreditNote 合同；永绅 finance 角色尚未获得收付款页面和权限，133 较早 V5 也未包含该切片。

## 8. 通过、退回、阻塞、取消的统一解释

| 动作 | Workflow / Runtime 结果 | 领域结果 | 恢复方式 |
| --- | --- | --- | --- |
| 通过 / 完成 | 当前任务或节点完成 | 只有白名单领域命令成功时才改变对应单据 / Fact | 进入下一节点 |
| 退回 / rejected | 记录原因并回到允许办理边界 | 不自动撤销已经生效的领域事实 | 补充资料后按新 revision / version 重提 |
| 阻塞 / blocked | 保留当前节点和原因 | 不改来源 / Fact | 解除原因后恢复 ready / active |
| 取消来源单据 | 走对象自己的取消 / 冲正 usecase | 是否写反向事实由对象生命周期决定 | 不通过改任务状态取消 |
| Runtime 补偿 | 只撤回可证明无领域副作用的下游任务 / 节点 | 已有 effect ref 或已完成领域动作时 fail closed | 进入人工纠错或对象专项冲正 |

## 11. 当前结论

1. Workflow、ProcessRuntime、固定 WIP 路线和 Fact 是四个不同层；任何一层完成都不能代替另一层。
2. 三条注册 ProcessRuntime 的销售激活、采购批准和 Shipment 财务门禁已在 Product Core 本地闭环；目标 migration、active revision、真实岗位 smoke 与 UAT 仍须独立取证。
3. 首次来料退回供应商 / 供应商补换、真实收付款和多来源核销已不是 Product Core “完全缺口”；必须继续分列永绅授权、133 当前版本和 UAT。
4. `payment_approval` 和四条客户 workflow 仍为 `preview_only`；它们不能代写 FinancePayment、库存、出货或应收应付事实。
5. 133 较早 V5 技术证据可以证明其固定版本，不证明当前 HEAD；客户 UAT / 签收仍是独立关口。

审批产品层的当前精确边界：所有 ProcessRuntime `approval` 节点必须声明 `workflow.task.approve`，服务端统一审批集合及桌面、协同 Drawer、Workflow 页面和移动岗位端复用同一任务动作、意见、来源和事件轨迹合同；通过 / 退回 / 阻塞继续服从登录态、disabled、RBAC、owner / assignee、active revision、任务 status / version、幂等 receipt 和审计事件。销售、采购和出货财务放行已各自接入唯一领域写入，Workflow task done 本身仍不等于领域写入成功。现有生产异常决定与执行继续以生产异常 usecase 为真源，本轮不新增第二套 WIP approval；PAYMENT 没有正式申请 / 审批 ProcessRuntime，任何审批任务都不得伪造付款完成。
