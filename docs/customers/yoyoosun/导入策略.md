# 永绅客户导入策略 / Yoyoosun Import Strategy

## 结论

当前永绅资料只用于来源核对、产品设计参考、打印模板输入和模拟试用。Product Core 只保留通用 manifest 校验、extract、freeze、dry-run 工具与合成 fixture；私密 manifest 和真实原件由客户专属 Private 仓库管理。当前不提供真实导入执行入口。

客户配置包和客户业务数据必须分开：配置包定义品牌、菜单、字段显示、编号、打印和映射规则；客户、供应商、订单、库存等业务数据进入独立客户数据库，不能嵌入前端配置或 Product Core。

## 分类边界

| 数据 | 当前处理 | 不允许 |
| --- | --- | --- |
| 客户、供应商、联系人 | dry-run 候选；主体无法唯一识别时进入未决队列 | 依据名称猜测合并 |
| 产品、材料、单位、仓库、BOM | 匹配已有正式主数据；缺失时单独确认 | 重复造表或自动伪造主数据 |
| 销售订单及明细 | 只作为 Source Document 候选 | 自动形成出货、库存或应收事实 |
| 采购 / 加工合同 | 先核工程/BOM需求与供应商、材料/产品主体 | 从来源表直接形成入库、应付事实 |
| 库存、出货、质检、财务 | 不导入事实；通过正式领域 usecase 产生 | 由脚本直写或从历史表推测 |
| PDF、图片、非结构化 Excel | 人工复核来源和字段口径 | 默认 OCR 后自动写入 |

## 准备流程

1. 客户私有仓库的 manifest 登记 sourceId、相对路径、hash、用途和是否允许结构化提取。
2. 从客户私有仓库的 tracked `sources/` 使用 Product Core 通用工具做只读校验与提取；不改原件。
3. 冻结 snapshot 与现有 V1 预览，记录校验和与人工复核项。
4. dry-run 输出候选、重复、冲突、未决项和禁止自动导入项。
5. 由业务人员确认映射；当前在此停止，不执行真实写入。

Product Core 普通构建 / CI 不 checkout 客户私有仓库，也不因缺少真实原件失败。客户私有验证使用兄弟目录或 CI multi-checkout，不使用 submodule。

Product Core 当前提交树与索引不再承载私密原件或真实 manifest。永绅 Private 仓库的当前来源 inventory、远端回读和完整性结果以其 manifest、README 与正式验证记录为准；可变产品版本锁定只以私有仓 `product.lock.json` 和两边清洁时的 `FORMAL_PRODUCT_PIN=1` validate 为准，Product Core `HEAD` 推进后必须重新验证。当前树隔离和私有仓验证仍不等于真实导入批准、Git 历史清理或客户签收。

## 未来真实导入

真实数据到位后，需要另行实现与评审通用导入批次：只能调用正式 usecase/API，按批次幂等，保留逐行来源、结果、操作人和错误，支持失败恢复与导入后对账。实现必须与客户 key 解耦，不能恢复 `business_records`，也不能给 dry-run 增加隐藏 `--execute`。
