# 永绅 Excel 字段产品核心映射评审 / Excel Field Product Core Mapping Review

本文评审受控客户来源中 Excel 字段应落到 Product Core、客户配置、导入审核还是业务事实。具体文件名、人员、账号、hash、大小和原始值由客户私有仓库管理，不复制进 Product Core。本文不是导入批准、客户签收或目标环境运行证据，也不替代代码、migration、测试和私有 manifest。

## 当前结论

本次对加工汇总和加工厂商资料两张表逐列复核。原实现有四类实质问题：

1. “加工项目”曾与“工序类别”混用，缺少独立的委外行字段。
2. 加工厂地址没有供应商主数据落点。
3. “加工工序”曾被误当成供应商类型，缺少供应商与工序的能力关系。
4. 厂商资料的“序号”曾可能被当作供应商编码，非日期回货说明也可能进入日期候选。

当前代码已经分别用 `outsourcing_order_items.processing_item`、`suppliers.address` 和 `supplier_process_capabilities` 修正前三项，并修正提取规则：序号不再生成供应商编码，非日期回货说明只进入人工审核文本。产品 / 材料主体、工序、供应商、单位仍必须匹配正式主数据；源表不能自动生成回货、库存、应付、发票或付款事实。

“有对应实现”不等于“每个 Excel 表头都增加一列数据库字段”。状态含义如下：

| 状态 | 含义 |
| --- | --- |
| 已实现 | 有明确的主数据、源单据、快照、页面和 API 落点。 |
| 审核后映射 | 系统有正确落点，但真实来源必须先唯一匹配或确认字段含义，不能按文本自动写入。 |
| 业务事实派生 | 只能从已过账事实计算，不允许从汇总表直接录入或覆盖。 |
| 延后 | 来源为空或语义不足，当前不制造不可靠字段。 |
| 排除 / 私密 | 版式空列、噪声或敏感资料不进入 Product Core 业务字段。 |

## 加工汇总字段逐列映射

| 列 | 来源字段 | 当前状态 | 产品核心落点与逻辑 |
| --- | --- | --- | --- |
| A | 委外加工订单号 | 已实现 | `outsourcing_orders.outsourcing_order_no`。来源为空时不能据此自动创建委外单，由正式新建 / 导入审核生成唯一单号。 |
| B | 产品订单编号 | 已实现 | `outsourcing_order_items.product_order_no_snapshot`，作为当前委外行的来源订单文本快照；产品和材料加工都可保留，不伪造成销售订单外键。 |
| C | 产品编号 | 审核后映射 | 产品加工唯一匹配 `products` 后保存 `product_id + product_no_snapshot`；不能只按名称猜产品，也不能自动建产品。 |
| D | 产品名称 | 审核后映射 | 与产品编号共同匹配产品，订单行保存 `product_name_snapshot`；主数据仍以 `products` 为真源。 |
| E | 加工项目 | 已实现 | 新增 `outsourcing_order_items.processing_item`，记录“脸×1、耳×2”等当前行加工部位 / 内容；它不是工序主数据。 |
| F | 厂家名称 | 审核后映射 | 唯一匹配 `suppliers`，委外单保存 `supplier_id + supplier_snapshot`；简称 / 全称不能直接猜 ID。 |
| G | 工序类别 | 审核后映射 | 来源值先作为正式工序名称候选唯一匹配 `processes`；委外行保存 `process_id`，工序名称 / 类别快照由匹配后的工序主数据冻结，不把来源文本直接写成主数据类别；与 E 列“加工项目”保持独立。 |
| H | 单位 | 审核后映射 | 唯一匹配 `units` 后保存 `unit_id + unit_name_snapshot`；不按 Excel 文本静默创建单位。 |
| I | 单价 | 已实现 | `outsourcing_order_items.unit_price`，允许空值，填写时必须为非负金额。 |
| J | 数量 | 已实现 | `outsourcing_order_items.outsourcing_quantity`，必须为正数。 |
| K | 加工金额 | 已实现 | `outsourcing_order_items.amount`，服务端以“数量 × 单价”计算并校验；源表金额只用于一致性核对，不能覆盖出不同结果。 |
| L | 备注 | 已实现 | `outsourcing_order_items.note`，保存当前委外行备注。 |
| M | 下单人 | 审核后映射 | 确认是委托方代表后进入订单 `contract_party_snapshot.buyerContact`；不写成加工厂联系人。 |
| N | 联系电话 | 审核后映射 | 与下单人一起确认后进入 `contract_party_snapshot.buyerPhone`；不得与供应商联系人电话混用。 |
| O | 回货日期 | 已实现 / 审核 | 合法日期进入订单或行 `expected_return_date`。非日期说明不会进入日期字段，只保留 `return_date_source_text + review_required` 供人工处理。 |
| P | 空白列 / 版式说明 | 排除 | 没有业务表头，不建 schema，不参与导入。 |
| Q | 用料数量 | 延后 | 来源数据为空，且可能是一对多材料需求，不能用委外行单一数值假装完整模型；后续需结合 BOM / 委外发料设计单独评审。 |
| R | 材料单位 | 延后 | 与 Q 列同属尚未确认的材料需求明细；不能脱离材料和数量单独保存单位文本。 |
| S | 加工入库数量 | 业务事实派生 | 只能按委外行汇总已过账 `RETURN_RECEIPT` 回货事实；不能从源单据表格直接写入库存或回货事实。 |
| T | 数量差异（采购-入库） | 业务事实派生 / 待确认 | 来源数据为空且“采购”措辞与委外语义不一致。若确认是委托数量差异，应由“委托数量 - 已过账回货数量”动态计算，不新增可编辑快照。 |

| --- | --- | --- | --- |
| A | 序号 | 排除 | 只作为来源行定位，不是稳定供应商编码。提取器不再把序号写入 `supplier.code / document_no`。 |
| B | 厂家简称 | 审核后映射 | `suppliers.short_name`；与全称共同用于人工去重和匹配。 |
| C | 厂家全称 | 审核后映射 | `suppliers.name`；供应商正式编码仍由既定编码规则或人工确认。 |
| D | 加工工序 | 已实现 / 审核后映射 | 新增 `supplier_process_capabilities`，关联 `suppliers` 与可承接的 `processes`。它是资料查询能力，不是供应商类型，也不自动限制委外订单选择。 |
| E | 联系人 | 审核后映射 | `contacts`，必须在供应商唯一匹配后写入 `owner_type=SUPPLIER + owner_id`。 |
| F | 联系电话 | 审核后映射 | 对应同一供应商联系人。明确的大陆手机号进入 `contacts.mobile`，座机或其他号码进入 `contacts.phone`；不能与公司对接人电话混用。 |
| G | 开票类型 | 延后 | 来源数据为空，属于 future finance / 结算配置；不从样本反推通用枚举。 |
| H | 开票点数 | 延后 | 来源数据为空，属于 future finance；不自动生成税率、发票或应付事实。 |
| I | 加工商地址 | 已实现 | 新增 `suppliers.address`，供应商表单、列表、导出和 JSON-RPC 共用同一真源。 |
| J | 银行卡号 | 私密隔离 | 不进入通用供应商字符串字段、日志或公开配置。提取证据只保留“来源存在 / 已脱敏”布尔标志，真实值留在客户私密资料并等待付款账户模型评审。 |
| K | 公司对接人 | 审核后分流 | 不是供应商联系人。提取为独立 `customer_material` 审核项；确认后可进入某张委外单的委托方快照，或以后进入内部责任关系模型。 |
| L | 对接人电话 | 审核后分流 | 与 K 列成对处理，不写入供应商联系人。 |
| M | 备注 | 审核后映射 | 确认是有效业务说明后可进入 `suppliers.note`；纯数字、占位或异常值只留人工审核，不自动写主数据。 |

## 读写、残值与打印边界

| 路径 | 当前规则 |
| --- | --- |
| 新建 / 编辑委外行 | `processing_item` 与 `process_id` 分开填写；前者是行内容，后者是可复用工序。空字符串会清除旧值。 |
| 产品 / 材料切换 | `product_id` 与 `material_id` exactly-one；切换时清除另一主体的 ID 和编号 / 名称快照，但保留独立的来源产品订单编号。 |
| 金额 | 服务端统一按数量和单价计算；来源金额只做校验，避免页面、导入和打印各算一套。 |
| 供应商能力 | `process_ids` 为当前资料关系；更新时非空数组替换、空数组清空、缺省保持。新关联工序必须启用且允许委外。 |
| 联系人 | 外部联系人归供应商；公司侧下单人 / 对接人归委托方快照或内部责任关系，两类电话不能互相兜底。 |
| 加工合同打印 | 表头由统一列定义生成；“加工项目”和“工序类别”分别显示。工序列优先显示委外行冻结的正式工序名称，旧数据缺名称时才回退类别快照；产品 / 材料编号与名称使用同一主体感知表头。 |
| 历史数据 | 新字段均为 nullable；旧委外行、旧供应商没有值时显示空态，不伪造或从其他字段回填。 |
| 导入 | 当前仍是 review-first 候选提取，不执行真实客户数据自动写入，不生成 Workflow / Fact。 |

## 不进入本轮 Product Core 的内容

- Q / R 两列没有真实数据，且需要一对多材料需求模型，不能为了“表头齐全”增加两个无真源标量。
- S / T 是回货事实和差异计算，不是委外源单据可编辑字段。
- 开票类型、开票点数和银行卡号属于 future finance / 付款账户边界；真实银行资料必须继续留在客户私密仓库。
- 公司对接人身份尚未确认，不能强塞进供应商联系人。
- 本次 schema migration 已生成；在归属明确的数据库执行 migration 前，运行中的旧数据库仍不具备新增列和关系表。

## 与其他 Excel 类别的既有边界

- 采购订单行的 `product_order_no_snapshot / product_no_snapshot / product_name_snapshot` 继续作为来源展示快照，不反写产品主数据，也不新增销售订单外键。
- 材料、单位、产品、工序和供应商继续先走主数据唯一匹配；BOM、采购、委外来源不能自动创建库存、质检、应付或付款事实。
- 色卡、图片和作业指导书长文本继续按工程附件 / 打印资料评审，不写进采购或委外行。
- 布料加工回货如何关联质检、材料入库和应付，仍必须作为 Fact 链路单独评审，不能由加工合同确认自动过账。
