# 永绅客户闭环交付验收清单

> 本清单用于把产品核心能力、永绅客户配置、角色流程、打印字段、来源材料、测试数据和发布证据对齐到同一套验收口径。当前仍以试用 / dry-run / preview 为主，未获得 authenticated active revision readback 前不得标记客户签收。

## 1. 产品核心与永绅客户边界

| 项目 | 当前证据 | 验收判断 | 边界 |
| --- | --- | --- | --- |
| Product Core | 当前代码、Ent schema、Atlas migration、QA 测试 | 可继续本地拉测 | 不因客户配置改变 schema / migration |
| yoyoosun 客户配置 | `customerPackage.mjs` / runtime manifest / projection matrix | 可试用配置资产 | `businessFlows / stateMachines / processPolicies` 仍是 preview-only |
| 角色矩阵 | `roleFlowMatrix.mjs` / `角色能力与流程矩阵.md` | 可编译为 active revision 的角色 / 能力 / 责任池收窄投影 | 不创建或提升 Product Core RBAC，不写任务或事实 |
| 客户来源映射 | `rawSourceFormMap.mjs` / `raw-source-form-map.md` | Product Core 只保留脱敏类别映射；真实来源覆盖由私有 manifest 验证 | 不直接写 runtime，不做 OCR 自动导入，不复制私有 inventory |
| 试用数据 | `trialDataFixture.mjs` | 可用于本地 / 试用 UI 和打印验证 | 不污染真实客户数据 |
| 配置投影 | `projectionMatrix.mjs` / `配置投影覆盖矩阵.md` | runtime visibility / 正式字段合同 / preview 已分层 | 正式字段合同不当作 active field policy；当前只消费三个 surface 的列表 / CSV 列可见性 |

## 2. 角色验收

| 角色 | 必须具备 | 不允许 |
| --- | --- | --- |
| 销售 | 客户、销售源单、销售协同任务 | 直接写库存、质检、出货、应收、发票 |
| 老板 | 审批、退回、经营查看 | 绕过业务 usecase 状态和权限校验 |
| 工程 | 产品、SKU、材料、工序、BOM | 直接生成生产、库存、采购、质检、出货、财务事实 |
| PMC | 生产评审、计划、风险 | 把计划写成生产完工或成品入库 |
| 采购 | 供应商、采购订单、采购合同打印 | 把采购承诺写成入库、库存或应付事实 |
| 仓库 | 入库准备、库存查看、出货执行 | 用任务完成直接增减库存 |
| 品质 | 来料质检、成品质检、异常处理 | 把质检任务完成直接写库存或出货事实 |
| 财务 | 财务放行、应收 / 应付草稿线索 | 冒充税控、总账、收付款核销或完整财务系统 |
| 生产 / 委外 | 委外订单、加工合同、生产进度线索 | 直接写成品入库、质检通过、结算或付款事实 |

## 3. 打印字段验收

### 采购订单 → 采购合同

必须能带出：

- 采购订单号、下单日期、预计回货日期。
- 供应商名称、联系人、电话、地址。
- 买方公司、联系人、电话、地址、签字人。
- 产品订单编号、产品编号、产品名称、材料品名、厂商料号、规格、单位、数量、单价、金额、备注。

边界：

- 客户配置默认值只覆盖买方 / 订货方字段。
- 供应商字段必须来自业务快照。
- 明细缺失时不能用模板样例补造。
- 空字段不能 fallback 成内部 ID。

### 委外订单 → 加工合同

必须能带出：

- 委外单号、下单日期、预计回货日期。
- 加工方名称、联系人、电话、地址。
- 委托单位、委托人、电话、地址、签字人。
- 产品订单编号、产品编号、产品名称、工序名称、加工厂商、工序类别、单位、数量、单价、金额、备注。

边界：

- 客户配置默认值只覆盖委托方字段。
- 加工方字段必须来自业务快照。
- 无来源明细不能补造样例行。

## 4. 私密来源资料验收

| 检查项 | 当前处理 | 不允许 |
| --- | --- | --- |
| Product Core 当前树 | 当前提交树与索引不承载真实原件、私密 manifest 和未脱敏输出；普通 CI 只用合成 fixture | 因缺少客户私有仓库让普通构建 / CI 失败 |
| 客户私有仓库 | 保存真实原件、私密 manifest、评审和验证入口；与 Product Core 采用兄弟目录 / CI multi-checkout | submodule / subtree、公开仓库、凭据或长期 URL |
| Excel 类来源 | 私有 manifest 校验、只读提取、dry-run、人工 review | 自动真实导入、直接写库存 / 财务 / 出货 |
| PDF / 图片 / 截图 | Private 仓库原件 + 人工核对线索 | OCR 自动入库、作为唯一业务事实真源 |
| Private 仓库存储 | 每客户隔离；当前 inventory、完整性与远端回读以私有仓 manifest、README 和正式验证记录为准 | 误设 public、共享高权限凭据、未回读就宣称迁移完成 |
| Product Core 版本锁 | 签收候选提交必须与私有仓 `product.lock.json` 一致，并在两边清洁时通过 `FORMAL_PRODUCT_PIN=1` validate | 在 Product Core `HEAD` 推进后沿用旧 lock、旧验证或在本文固定易漂移 SHA |
| Git 历史 | 单独维护窗口评审清理 | 在普通提交中自动 force push 或把当前树删除写成历史已清理 |

## 7. 当前结论

Product Core 当前提交树与索引已完成客户私密原件隔离。客户私有仓的当前存储、远端完整性和 Product Core 版本锁定状态，只以私有仓 manifest、README、`product.lock.json` 与 formal validate 证据为准，本清单不缓存提交号或完成状态。既有 Git 历史治理、authenticated runtime readback、目标角色 smoke 和客户签收仍需独立验收；当前树隔离不能替代客户正式交付。
