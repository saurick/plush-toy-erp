# 永绅甲方角色职责与业务流转确认表

本文用于甲乙双方面谈确认永绅的目标岗位职责、审批关系、跨岗位交接、异常处理和本期范围。它是会议工作底稿，不是系统已实现、已发布、已迁移或已验收的证明。

真实姓名、签字、联系方式、录音和会议原件只保存在永绅客户专属受控资料库；产品仓只保留脱敏模板、岗位结论、流程结论和差异编号。

### 1.1 本次确认什么

1. 每个岗位为什么存在、接收什么、办理什么、交出什么。
2. 每个审批由谁发起、审核什么、通过后去哪、退回给谁、阻塞后怎样恢复。
3. 任务、单据、质检、入库、出货和财务结果分别在什么条件下才算完成。

### 1.2 本次不直接确认什么

- 不在本表确认个人账号、真实数据、密码、电话号码或银行资料。
- 不把“业务上希望这样走”写成“系统已经这样运行”。
- 不把本地测试、目标技术试用或页面可见写成甲方已完成用户验收（UAT）。
- 不在会议现场用模糊措辞新增开发承诺；争议项登记决策人和截止日期。

每个准备确认的流程必须独立记录六个维度，禁止只写“已完成”。甲方现场只确认目标业务和交付范围；产品、配置、环境与用户验收状态由乙方会前预填并当场解释，不要求甲方判断或签认代码、配置、部署证据。

当前状态快照：目标技术试用环境已有较早固定版本的部署、配置和账号 / 页面技术检查记录；后续增加的审批办理、异常处置及收付款能力尚未整体更新到该环境。甲方用户验收（UAT）仍未完成。面谈和签认只使用文档控制中冻结的状态基线与证据编号；后续[客户交付矩阵](客户交付矩阵.md)更新不能反向改写本次会议看到的版本。

## 3. 一页业务总链

```mermaid
flowchart LR
  customer["客户需求"] --> sales["销售建单"] --> boss["老板审核"] --> engineering["工程资料"] --> pmc["PMC 齐套与排程"]
  pmc --> purchase["采购与到货"] --> iqc["品质 IQC"] --> material["仓库正式入库"]
  material --> production["生产 / 委外 WIP"] --> processQc["分段质检与返工"] --> packing["包装与完工入库"]
  packing --> delivery["出货准备与放行"] --> shipped["仓库实际发货"] --> finance["应收 / 应付 / 发票 / 收付款"]
```

全链共同规则：审批或任务完成只说明协同结论；只有对应业务单据正式生效、质检正式判定、库存正式过账、出货正式确认或财务正式过账，才形成相应业务结果。

## 9. 状态含义确认

## 11. 当面对接方法

1. 会前按 R01–R09、A01–A07 生成 v0.3 受控工作副本，只请甲方修改“目标职责和流程”，不要让甲方判断代码状态。
2. 现场选一张代表性订单，从客户需求走到收付款；按 P01–P09 顺序逐段确认，不逐页念权限。
3. 每个节点先判断是审批、业务评审、办理任务还是放行门禁，再填写发起人、办理人、审批人、输入、通过、退回、阻塞恢复和最终业务结果。
4. 有争议的事项登记到 C01–C09，不在现场用“差不多”“以后再说”合并不同规则。
5. 会后 24 小时内发会议纪要；已确认事项进入[决策日志](决策日志.md)，未决事项进入[问题待办](问题待办.md)，仍是推断的内容进入[假设登记](假设登记.md)。
6. 已确认且需要改变系统的事项进入[客户交付矩阵](客户交付矩阵.md)的“客户差异与决策”；签认后再次改变才进入[变更请求流程](变更请求流程.md)。
