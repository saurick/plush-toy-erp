# plush-toy-erp 自动化测试体系与验收策略

## 0. 文档目的

本文用于定义 `plush-toy-erp` 从当前 V1 最小闭环到未来完整 ERP 产品化过程中的自动化测试体系。

本文重点回答：

* 每一类改动应该跑哪些测试。
* 每个业务层应该如何测试。
* 哪些测试必须自动化。
* 哪些测试适合 Codex 每轮执行。
* 哪些测试适合 CI 执行。
* 哪些测试适合阶段验收前执行。
* 如何防止 Workflow / Fact 混淆。
* 如何防止 customer-specific 数据污染 Product Core。
* 如何防止旧 `business_records` 与 V1 正式模型形成双真源。
* 如何保证未来多个客户私有化部署时，产品内核稳定、配置差异受控。

---

## 1. 自动化测试总原则

### 1.1 测试目标

自动化测试不是为了追求数量，而是为了保证：

* 产品内核稳定。
* 业务事实可信。
* Workflow 和 Fact 不混淆。
* 状态机不乱跳。
* 权限不只靠前端隐藏按钮。
* 导入不会污染正式数据。
* 客户配置不会破坏核心规则。
* 多客户交付时不需要复制代码。
* 每轮 Codex 改动都能被快速审查。

### 1.2 测试分层原则

测试必须分层：

| 层级              | 目标                             | 典型工具 / 命令                   | 适用场景                      |
| --------------- | ------------------------------ | --------------------------- | ------------------------- |
| 静态检查            | 格式、边界词、禁止路径、diff 检查            | git diff --check、grep、lint  | 所有改动                      |
| 单元测试            | 验证函数、状态机、usecase guard         | go test、pnpm test           | biz / helper / API client |
| Repo / DB 测试    | 验证 Ent repo、DB 约束、事务           | go test + enttest / test DB | data repo                 |
| API / RBAC 测试   | 验证接口、鉴权、动作权限                   | handler tests               | JSON-RPC / RBAC           |
| UI 组件测试         | 验证页面、表单、按钮、API client contract | pnpm test                   | 前端页面                      |
| 路由 / 冒烟测试       | 验证页面能打开，基础渲染不炸                 | Playwright mocked route     | 新页面 / 路由                  |
| 样式回归            | 验证布局、样式、L1 场景                  | pnpm css、pnpm style:l1      | UI / 样式                   |
| E2E 测试          | 验证真实用户路径                       | Playwright + 后端 + DB        | 阶段验收                      |
| 数据导入 dry-run 测试 | 验证导入预演、unresolved queue、不写库    | fixture tests               | import tooling            |
| 部署冒烟测试          | 验证部署、迁移、备份、恢复                  | scripts / smoke             | 私有化交付                     |

### 1.3 禁止用测试掩盖边界问题

测试不能允许以下行为通过：

* Workflow task done 写库存、出货、财务事实。
* `shipping_released` 被当成 `shipped`。
* `sales_order` 被当成 shipment fact。
* `business_records` 被当成正式事实真源。
* current 客户样本字段直接进入 Product Core。
* API 绕过 usecase 直接写 Ent。
* UI 绕过 API 直接写本地假数据。
* 只隐藏前端按钮，不做后端权限校验。
* 导入工具自动生成 shipment / inventory / finance facts。
* 未经人工确认的数据被静默导入。

## 2. 测试层级定义

## 2.1 T0：静态检查

### 目标

所有改动都必须至少经过静态检查。

### 检查内容

* 当前 Git 状态。
* diff 统计。
* 空白 / 格式错误。
* 未跟踪文件。
* 禁止字段。
* 禁止动作。
* 禁止路径。
* 实现状态措辞。

### 基础命令

```
git status --short
git branch --show-current
git log --oneline -3
git diff --stat
git diff --check
git ls-files --others --exclude-standard
```

### 常用边界 grep

```
grep -R "tenant_id" docs/customers docs/product docs/architecture docs/reference config deployments server web || true
grep -R "shipping_released" docs/customers docs/product docs/architecture docs/reference web server || true
grep -R "shipped_quantity\|shipment_id\|inventory_txn_id\|invoice_id\|payment_id\|ar_id\|ap_id\|product_sku_id" server web docs || true
grep -R "markAsShipped\|shipSalesOrder\|reserveStock\|deductInventory\|generateInvoice\|generateReceivable\|receivePayment" server web docs || true
grep -R "ChangeUsecase\|change_records" server web docs || true
```

---

## 2.2 T1：文档测试

### 目标

文档改动不破坏项目真源，不制造信息差。

### 适用场景

* roadmap。
* cutline。
* transition audit。
* import draft。
* risk register。
* current-source-of-truth。
* progress。
* Codex goal。

### 必须检查

| 检查项                              | 目的           |
| -------------------------------- | ------------ |
| 是否把 draft 写成 implemented         | 防止文档提前承诺     |
| 是否新增 tenant_id                   | 防止 SaaS 提前污染 |
| 是否把 current 字段写成 Product Core    | 防止单客户污染产品    |
| 是否把 shipping_released 写成 shipped | 防止出货事实混淆     |
| 是否建议自动生成库存/财务事实                  | 防止事实层污染      |
| 是否引用了正确真源                        | 防止历史聊天或截图当真源 |
| 是否更新 current-source/progress     | 防止进度信息差      |

### 文档-only 命令

```
git status --short
git diff --stat
git diff --check
git ls-files --others --exclude-standard
grep -R "Runtime Implemented: Yes\|Ent Schema Implemented: Yes\|Migration Implemented: Yes" docs/product docs/architecture || true
```

### 文档-only 不需要跑

一般不需要跑：

```
cd server && go test ./internal/biz ./internal/data
cd web && pnpm test
```

除非本轮意外改了代码、前端配置或文档 registry。

## 2.3 T2：Schema / Migration 测试

### 目标

保证 Ent schema、generated code、Atlas migration 正确，并且没有越界表或字段。

### 适用场景

* 新增 Ent schema。
* 修改 Ent schema。
* 生成 Ent code。
* 生成 Atlas migration。

### 必须检查

| 检查项                   | 目的                                       |
| --------------------- | ---------------------------------------- |
| 是否只新增允许的表             | 防止 scope creep                           |
| 是否生成 tenant_id        | 禁止当前阶段 SaaS                              |
| 是否生成 deferred 表       | 防止提前做 product_skus / shipments / finance |
| migration 是否只包含本轮允许对象 | 防止误迁移                                    |
| generated code 是否编译   | 防止 schema 语法错误                           |
| DB status 是否清楚        | 防止 migration pending 被误当已应用              |

### 命令

```
cd server && make data
cd server && make migrate_status
cd server && go test ./internal/data/model/schema
cd server && go test ./internal/biz ./internal/data
```

### migration 检查

```
grep -R "tenant_id" server/internal/data/model server/internal/data/model/migrate || true
grep -R "product_skus\|purchase_orders\|shipments\|stock_reservations\|invoice\|payment\|reconciliation\|change_records" server/internal/data/model server/internal/data/model/migrate || true
```

### 注意

`make migrate_status` 显示 pending 不等于错误。

但必须明确：

* migration 是否只生成。
* migration 是否已 apply。
* 目标库是否 pending。
* 后续是否需要 apply / smoke。

---

## 2.4 T3：Repo / Usecase 测试

### 目标

保证后端 usecase 规则、状态机、guard、事务边界正确。

### 适用场景

* 新增 repo。
* 新增 usecase。
* 修改业务 guard。
* 修改状态机。
* 修改 MasterData / Source Document / Fact 层。

### 必须覆盖

| 测试对象             | 必须测试                                           |
| ---------------- | ---------------------------------------------- |
| MasterData       | create / update / get / list / active guard    |
| Contacts         | owner_type 合法性、owner_id 存在性、primary 唯一         |
| Sales Order      | lifecycle guard、customer active guard          |
| Sales Order Item | product/unit active guard、quantity/money guard |
| Inventory Fact   | idempotency、posted/reversed、余额一致性              |
| Shipment Fact    | release != shipped、库存扣减、预留消耗                   |
| Finance Fact     | invoice/payment/reversal/reconciliation        |

### 命令

```
cd server && go test ./internal/biz ./internal/data
cd server && go test -count=1 ./internal/biz ./internal/data
```

### 测试原则

* usecase 测业务规则。
* repo 测持久化和事务。
* API 不复制业务规则。
* UI 不复制业务规则。
* 状态跳转必须测试非法路径。
* 终态保护必须测试。
* 幂等必须测试。

---

## 2.5 T4：API / RBAC 测试

### 目标

保证 API 走 usecase，后端权限真实生效。

### 适用场景

* 新增 JSON-RPC / HTTP API。
* 修改 API handler。
* 新增权限码。
* 修改 RBAC 角色权限。
* 修改 auth / permission 逻辑。

### 必须覆盖

| 场景             | 目的                         |
| -------------- | -------------------------- |
| 未登录            | 不能访问                       |
| 非管理员           | 不能执行管理动作                   |
| disabled admin | 不能绕过                       |
| 无权限            | 返回权限错误                     |
| 有权限            | 正常执行                       |
| super admin    | 按现有规则通过                    |
| API 走 usecase  | 不绕过业务 guard                |
| 禁止方法不存在        | 不暴露 shipment/finance 等越界动作 |

### 命令示例

```
cd server && go test ./internal/data -run 'TestJsonrpcData'
cd server && go test ./internal/biz -run 'Test.*Permission'
cd server && go test ./internal/biz ./internal/data ./internal/service ./internal/server
```

### API / RBAC 禁止项

API 不得：

* 直接写 Ent 绕过 usecase。
* 在 API 层复制状态机。
* 在 API 层复制 contacts owner guard。
* 暴露 `shipSalesOrder`。
* 暴露 `reserveStock`。
* 暴露 `generateInvoice`。
* 暴露 `receivePayment`。
* 只靠前端按钮隐藏做权限。

---

## 2.6 T5：前端 UI 测试

### 目标

保证页面可用、路由可打开、API client contract 稳定、按钮权限不误导业务边界。

### 适用场景

* 新增页面。
* 新增路由。
* 新增 API client。
* 修改表单。
* 修改按钮权限。
* 修改样式 / 布局。
* 修改菜单入口。

### 必须覆盖

| 测试项                    | 目的                                     |
| ---------------------- | -------------------------------------- |
| API client contract    | 方法名、参数、返回值稳定                           |
| helper tests           | 状态文案、权限按钮、字段格式                         |
| page render            | 页面默认态不崩                                |
| route render           | 路由可打开                                  |
| mocked RPC             | 不依赖真实后端也能检查页面                          |
| lint/css/test/style:l1 | 保持前端质量                                 |
| forbidden grep         | 不出现 shipped_quantity / shipment action |

### 命令

```
cd web && pnpm lint
cd web && pnpm css
cd web && pnpm test
cd web && pnpm style:l1
```

### 可选浏览器冒烟

如果项目已有 Playwright：

```
使用 mocked RPC route render 覆盖新增页面默认态
```

### UI 禁止项

UI 不得：

* 把 `shipping_released` 显示成已出库。
* 显示 `sales_order` 为 shipped。
* 展示或编辑 `shipped_quantity`。
* 展示 shipment / inventory / finance facts，除非对应事实层已完成。
* 用 local fake state 当真源。
* 绕过 API 写数据。
* 删除旧 business_records 入口，除非已有 cutover goal。
* 改 seedData / docs registry，除非本轮明确允许。

## 2.7 T6：数据导入 dry-run 测试

### 目标

保证数据导入在写库前能发现字段冲突、重复、缺失、歧义和 forbidden facts。

### 适用场景

* current 客户数据导入。
* business_records 旧记录迁移。
* Excel / PDF / seedData / source snapshot 解析。
* import loader。
* backfill tooling。

### 必须覆盖

| 测试项                  | 目的                                    |
| -------------------- | ------------------------------------- |
| source parsing       | 解析来源数据                                |
| field classification | 字段分类正确                                |
| target mapping       | 映射到正确模型                               |
| duplicate detection  | 发现重复客户/供应商/产品                         |
| unresolved queue     | 未解决项进入队列                              |
| forbidden facts      | 禁止自动生成 shipment / inventory / finance |
| no-write mode        | dry-run 不写正式 DB                       |
| preview output       | 输出可审阅预览                               |
| acceptance checklist | 导入前必须确认                               |

### 禁止项

导入不得自动生成：

* shipments。
* stock_reservations。
* inventory_txns。
* AR/AP。
* invoice。
* payment。
* product_skus。
* purchase_orders。

### 未来命令示例

当 import tooling 存在后：

```
cd server && go test ./internal/importer
cd server && go test ./internal/importer -run DryRun
cd server && go test ./internal/importer -run Unresolved
cd server && go test ./internal/importer -run NoWrite
```

当前阶段如果没有 import tooling，不得伪造测试。

## 2.8 T7：业务事实测试

### 目标

保证库存、出货、财务、生产、委外等事实层可信。

### 适用场景

* shipment usecase。
* stock reservation。
* inventory outbound。
* AR/AP。
* invoice。
* payment。
* reconciliation。
* production issue。
* finished goods receipt。
* outsourcing issue/receipt。

### 必须覆盖

| 事实层               | 必须测试                               |
| ----------------- | ---------------------------------- |
| Inventory         | posted / reversed、余额变化、批次关系        |
| Reservation       | reserved / released / consumed、可用量 |
| Shipment          | draft / picked / shipped、出库事实      |
| Sales fulfillment | partial / full 从 facts 推导          |
| Finance           | AR/AP/invoice/payment/reversal     |
| Production        | 领料扣减、成品入库                          |
| Outsourcing       | 委外发料、回货、质检、结算                      |
| Idempotency       | 同一动作不能重复生成事实                       |
| Audit             | 谁在何时做了什么                           |

### 高风险测试

必须测试：

* 重复提交。
* 幂等键重复。
* 终态保护。
* 失败回滚。
* 并发请求。
* 负库存保护。
* 已冲正事实不能再冲正。
* 已结算不能继续改事实。

## 4. 按模块的测试重点

## 4.1 MasterData 测试重点

| 对象        | 测试重点                                            |
| --------- | ----------------------------------------------- |
| customers | code unique、create/update/list/get、active guard |
| suppliers | supplier_type、active guard、code unique          |
| contacts  | owner_type、owner_id、primary contact、disable     |

必须测试：

* 不存在 owner。
* 非法 owner_type。
* 同一 owner 设置多个 primary。
* 禁用联系人。
* 主数据 inactive 后引用行为。

---

## 4.2 Sales Order 测试重点

必须测试：

* 创建订单时 customer 必须存在且 active。
* 创建订单行时 product 必须存在且 active。
* 创建订单行时 unit 必须存在且 active。
* ordered_quantity 必须大于 0。
* amount / unit_price 不能为负。
* lifecycle 只能合法跳转。
* 不能进入 shipped。
* 不能写 shipment / inventory / finance。

## 13. 最终原则

自动化测试体系必须服务于这几个核心目标：

```text
不让 Workflow 写事实。
不让 Source Document 冒充 Fact。
不让 derived status 冒充原始事实。
不让 current 客户样本污染 Product Core。
不让 business_records 形成双真源。
不让前端菜单误导产品成熟度。
不让导入工具自动生成出货、库存、财务事实。
不让权限只停留在前端按钮。
不让部署和迁移没有回滚。
```

最终产品要做到：

```text
代码有测试。
文档有边界。
事实有来源。
状态可追溯。
导入可预演。
发布可回滚。
客户可验收。
多客户可复制。
```
