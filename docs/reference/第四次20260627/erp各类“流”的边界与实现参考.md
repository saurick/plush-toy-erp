# ERP 各类“流”的边界与实现参考

> 文件定位：用于 ERP 项目中统一理解“业务流、工作流、审批流、状态流、任务流、异常流、通知流、自动流转”等概念，避免设计时把所有流程混成一团。
>
> 核心原则：**业务准确度优先，效率优先，可维护，可扩展，边界清晰，不把流程做成负担。**

---

## 1. 为什么要区分这些“流”

ERP 不是简单的 CRUD，也不是单纯的审批系统。

一个成熟 ERP 通常同时包含：

- 业务流：业务从开始到结束的主过程。
- 状态流：业务对象生命周期中的状态变化。
- 工作流：人、角色、岗位、部门之间的任务流转。
- 审批流：工作流中专门处理审核、同意、驳回的部分。
- 任务流：某个环节需要谁去处理、确认、执行。
- 异常流：驳回、返工、取消、挂起、补料、改单等异常闭环。
- 通知流：提醒、催办、超期、结果通知。
- 自动流转：系统根据规则自动推进后续动作。

如果不区分这些概念，后果通常是：

1. 审批流被误认为是全部工作流。
2. 工作流和业务状态耦合在一起，后续很难改。
3. 不同客户的差异都写死在业务代码里。
4. 流程越做越复杂，反而降低 ERP 效率。
5. 页面、权限、状态、任务、审批混在一起，后期维护困难。

---

## 2. 总体关系图

```text
业务流 Business Flow
  ├─ 状态流 / 状态机 State Flow
  ├─ 工作流 Workflow
  │    ├─ 审批流 Approval Flow
  │    ├─ 任务流 Task Flow
  │    ├─ 协同流 Collaboration Flow
  │    └─ 异常流 Exception Flow
  ├─ 通知流 Notification Flow
  ├─ 自动流转 Auto Flow
  └─ 规则 / 策略 / 扩展点
```

一句话理解：

> **业务流决定业务怎么走；状态流决定对象当前处于什么阶段；工作流决定谁来处理；审批流决定谁来同意；规则策略决定什么条件下走哪条路；扩展点承接客户差异。**

## 3. 各类“流”的定义与边界

| 类型 | 解决的问题 | 典型例子 | 是否建议独立建模 |
|---|---|---|---|
| 业务流 | 一项业务从开始到结束怎么完成 | 销售订单 -> 采购 -> 入库 -> 生产 -> 出库 -> 对账 -> 结算 | 是 |
| 状态流 / 状态机 | 一个业务对象当前处于什么状态，允许如何迁移 | 草稿、待审、已审、执行中、已完成、已归档 | 是 |
| 工作流 | 谁在什么节点处理什么任务，处理完流向哪里 | 销售提交、主管确认、采购处理、仓库入库 | 是 |
| 审批流 | 谁审核，是否通过，驳回到哪里 | 采购审批、付款审批、订单审批 | 是，作为工作流子能力 |
| 任务流 | 谁要执行什么任务 | 打样、报价、采购、质检、发货 | 是 |
| 协同流 | 多角色、多部门之间如何配合 | 销售、采购、生产、仓库、财务协同 | 可合入工作流 |
| 异常流 | 业务不按正常路径走时如何闭环 | 驳回、返工、取消、补料、改单、挂起 | 是 |
| 通知流 | 什么情况提醒谁 | 审批提醒、超期提醒、到货提醒 | 是，但不要侵入业务核心 |
| 自动流转 | 不需要人工操作时系统自动推进 | 审批通过后自动生成采购需求 | 是 |
| 规则 / 策略 | 什么条件触发什么流程或节点 | 金额超过 1 万走老板审批 | 是 |
| 扩展点 | 客户差异逻辑挂在哪里 | 客户 A 多一级审批，客户 B 自动跳过某环节 | 是 |

## 4. 业务流 Business Flow

### 4.1 定义

业务流描述的是 ERP 中真实业务的主过程。

例如毛绒玩具 ERP：

```text
客户需求
  -> 报价
  -> 销售订单
  -> 物料需求
  -> 采购
  -> 入库
  -> 生产
  -> 质检
  -> 出库
  -> 对账
  -> 收款
  -> 归档
```

业务流回答的是：

> 这件业务从头到尾要经过哪些业务环节？

### 4.2 业务流不应该直接等于审批流

错误设计：

```text
销售订单 = 审批流程
```

更合理的设计：

```text
销售订单是业务对象
销售订单有生命周期状态
销售订单在某些节点可能需要审批
审批只是业务推进中的一个控制点
```

### 4.3 设计原则

业务流应该表达稳定的业务主干，不应该塞太多客户差异。

推荐：

```text
产品核心业务流：保持稳定
行业模板流程：承接行业通用差异
客户配置包：承接客户级差异
扩展点：承接代码级特殊差异
```

## 5. 状态流 / 状态机 State Flow

### 5.1 定义

状态流描述业务对象的生命周期。

例如销售订单：

```text
草稿
  -> 待审批
  -> 已确认
  -> 执行中
  -> 已完成
  -> 已归档
```

也可能存在异常状态：

```text
已驳回
已取消
已挂起
已关闭
```

状态流回答的是：

> 这张单现在处于什么阶段？还能不能继续推进？允许执行哪些动作？

### 5.2 状态机是 ERP 的底盘能力

状态机应该负责：

- 当前状态是什么。
- 当前状态允许哪些动作。
- 某个动作会迁移到什么状态。
- 谁有权执行这个动作。
- 迁移时需要记录什么审计证据。
- 迁移失败时如何回滚或提示。

### 5.3 状态流不要被审批流绑死

错误设计：

```text
待主管审批
待财务审批
待老板审批
```

这些更像审批节点，不一定应该全部变成业务状态。

更推荐：

```text
业务状态：待审批
审批节点：主管审批、财务审批、老板审批
```

否则状态会爆炸：

```text
待主管审批
主管已审待财务审批
财务已审待老板审批
老板驳回待销售修改
销售修改后待主管复审
...
```

这种设计后期非常难维护。

### 5.4 推荐做法

业务状态保持粗粒度：

```text
草稿
待审批
已确认
执行中
已完成
已归档
已取消
已挂起
```

审批节点放到工作流实例里：

```text
workflow_instance.current_node = 财务审批
approval_task.status = 待处理
business_object.status = 待审批
```

## 6. 工作流 Workflow

### 6.1 定义

工作流描述的是：

> 一项业务在不同角色、岗位、部门之间如何流转处理。

工作流不只审批，还包括：

- 分派任务。
- 人工处理。
- 部门协作。
- 节点流转。
- 并行处理。
- 异常回退。
- 自动节点。
- 通知与催办。

### 6.2 工作流的核心对象

建议至少抽象这些对象：

| 对象 | 说明 |
|---|---|
| workflow_template | 流程模板 |
| workflow_node | 流程节点 |
| workflow_transition | 节点流转规则 |
| workflow_instance | 某次业务流程实例 |
| workflow_task | 分配给人的任务 |
| workflow_action | 用户在节点上执行的动作 |
| workflow_audit_log | 流程审计记录 |

### 6.3 工作流和业务对象的关系

业务对象不要直接依赖某个写死流程。

推荐关系：

```text
sales_order
  -> workflow_instance
      -> workflow_task
      -> workflow_audit_log
```

业务对象只需要知道：

```text
当前是否有流程实例
当前业务状态是什么
当前可执行动作是什么
```

流程细节交给工作流层。

## 7. 审批流 Approval Flow

### 7.1 定义

审批流是工作流中的一个子类型，专门处理：

- 提交审批。
- 审批通过。
- 审批驳回。
- 加签。
- 转交。
- 撤回。
- 重新提交。
- 多级审批。
- 并行审批。
- 会签。
- 或签。

所以：

> **审批流是工作流的一环，但不是工作流的全部。**

### 7.2 审批流应该解决的问题

| 问题 | 示例 |
|---|---|
| 谁审批 | 销售主管、财务、老板 |
| 审几级 | 金额越大审批层级越高 |
| 如何通过 | 单人通过、全部通过、任一通过 |
| 如何驳回 | 驳回到发起人、驳回到上一节点 |
| 是否允许撤回 | 未审批前允许撤回 |
| 是否允许加签 | 特殊客户需要老板加签 |
| 审批结果如何影响业务状态 | 通过后订单已确认，驳回后回草稿 |

### 7.3 审批流不要写死在业务代码里

错误做法：

```go
if order.Amount > 10000 {
    approveByBoss()
} else {
    approveByManager()
}
```

推荐做法：

```text
业务提交
  -> 规则引擎判断审批路径
  -> 创建工作流实例
  -> 创建审批任务
  -> 审批完成后回调状态机
```

审批规则放到配置、规则、策略中：

```text
if amount > 10000:
  route = manager -> boss
else:
  route = manager
```

## 8. 任务流 Task Flow

### 8.1 定义

任务流描述的是：

> 某个业务节点需要谁去执行什么任务。

任务流不一定是审批。

例如：

```text
销售创建订单
  -> 采购确认物料
  -> 仓库确认库存
  -> 生产确认排期
  -> 质检执行检验
  -> 仓库安排出库
```

这些都属于任务流，但不是审批流。

### 8.2 任务流适合承接操作性环节

例如：

| 业务环节 | 任务 |
|---|---|
| 打样 | 设计师上传样品图 |
| 采购 | 采购员确认供应商 |
| 入库 | 仓库员确认数量 |
| 质检 | 质检员录入结果 |
| 出库 | 仓库员确认发货 |
| 财务 | 财务确认收款 |

### 8.3 任务流的设计重点

任务流要关注：

- 任务分配给谁。
- 谁可以领取。
- 谁可以转交。
- 是否允许多人协作。
- 是否有截止时间。
- 是否支持催办。
- 完成任务后如何流转。
- 任务是否影响业务状态。

## 9. 协同流 Collaboration Flow

### 9.1 定义

协同流强调多角色、多部门之间的配合。

例如：

```text
销售接单
  -> 技术确认工艺
  -> 采购确认材料
  -> 生产确认产能
  -> 财务确认账期
  -> 销售回复客户
```

这里不一定每一步都是审批，但每一步都需要不同角色参与。

### 9.2 协同流适合放在工作流层

协同流不建议单独做成一套完全独立的系统。

更推荐：

```text
工作流节点类型：
  - 审批节点
  - 人工任务节点
  - 自动节点
  - 并行节点
  - 条件节点
  - 协同确认节点
```

协同流只是工作流模板中的一种编排方式。

## 10. 异常流 Exception Flow

### 10.1 定义

异常流描述业务不按正常路径走时如何处理。

常见异常：

- 驳回。
- 撤回。
- 取消。
- 关闭。
- 挂起。
- 返工。
- 补料。
- 改单。
- 重开。
- 作废。
- 冲销。
- 归档。
- 反审核。

### 10.2 异常流一定要闭环

错误设计：

```text
审批驳回 -> 状态变成已驳回
```

但后续没有说明：

- 谁修改？
- 修改后能不能重新提交？
- 历史审批记录是否保留？
- 驳回后库存、采购、生产是否回滚？
- 已生成的下游单据怎么办？

正确设计应该明确：

```text
审批驳回
  -> 业务状态回到草稿或待修改
  -> 当前审批任务关闭
  -> 保留审批记录
  -> 发起人收到待修改任务
  -> 修改后可以重新提交
  -> 新审批实例关联旧审批历史
```

### 10.3 异常流设计原则

异常流必须回答：

| 问题 | 要求 |
|---|---|
| 异常动作谁能执行 | 角色与权限明确 |
| 异常后状态是什么 | 状态机明确 |
| 相关任务如何处理 | 关闭、重开、转交、取消 |
| 下游数据如何处理 | 回滚、冻结、作废、生成冲销记录 |
| 审计如何记录 | 必须保留 |
| 是否允许重新进入主流程 | 要明确 |

## 11. 通知流 Notification Flow

### 11.1 定义

通知流负责把变化告诉相关人。

例如：

- 有新的审批任务。
- 订单被驳回。
- 采购超期。
- 物料入库完成。
- 生产任务延期。
- 财务已确认收款。

### 11.2 通知流不要侵入业务核心

错误设计：

```go
approveOrder()
sendEmail()
sendSms()
sendDingTalk()
```

推荐设计：

```text
业务事件产生
  -> event_bus
  -> notification_service
  -> 根据订阅配置发送通知
```

### 11.3 通知流建议事件驱动

例如：

```text
OrderSubmitted
ApprovalTaskCreated
ApprovalPassed
ApprovalRejected
PurchaseDelayed
InventoryReceived
PaymentConfirmed
```

通知模块订阅事件，不直接影响业务主流程。

## 12. 自动流转 Auto Flow

### 12.1 定义

自动流转是系统根据规则自动推进业务。

例如：

```text
审批通过后自动确认订单
库存充足时自动跳过采购
采购入库后自动更新生产可用物料
质检通过后自动允许出库
收款完成后自动关闭应收
```

### 12.2 自动流转适合放在规则和事件之后

推荐：

```text
业务事件
  -> 规则判断
  -> 自动动作
  -> 状态迁移
  -> 记录审计
```

例如：

```text
ApprovalPassed
  -> 判断订单是否需要财务复核
  -> 不需要则自动变成已确认
  -> 生成订单确认日志
```

### 12.3 自动流转必须可追溯

所有自动流转都要记录：

- 触发事件。
- 命中的规则。
- 执行动作。
- 执行结果。
- 失败原因。
- 关联业务对象。
- 执行时间。

不能出现系统偷偷改状态，但查不到原因。

## 13. 规则、策略和扩展点

### 13.1 规则 Rule

规则适合处理可配置、可表达的条件。

例如：

```text
订单金额 > 10000，需要老板审批
客户信用等级 = C，需要财务审批
物料类型 = 特殊材料，需要采购经理审批
```

规则特点：

- 通常可配置。
- 可视化程度较高。
- 不应该写复杂代码。
- 适合业务人员或实施人员维护。

### 13.2 策略 Strategy

策略适合处理同一能力的不同实现。

例如：

```text
审批人选择策略
  - 按角色找人
  - 按部门主管找人
  - 按金额区间找人
  - 按客户配置找人
```

策略特点：

- 接口固定。
- 实现可替换。
- 可按客户、租户、行业模板选择。
- 适合开发人员实现，实施人员选择。

### 13.3 扩展点 Extension Point

扩展点适合处理配置和策略都解决不了的客户差异。

例如：

```text
客户 A 在订单提交前要校验外部系统额度
客户 B 审批通过后要调用第三方财务系统
客户 C 生成采购需求时要套特殊物料映射规则
```

扩展点特点：

- 是受控 hook 点。
- 不能到处乱开口。
- 要有输入输出契约。
- 要有超时、错误处理、审计。
- 不允许破坏核心业务模型。

## 14. 推荐分层设计

```text
API / UI 层
  -> 只负责展示和触发动作，不写复杂流程逻辑

Application Usecase 层
  -> 编排一次业务动作
  -> 调用状态机、工作流、规则、策略、扩展点

Domain 层
  -> 业务对象、业务规则、状态机核心约束

Workflow 层
  -> 流程模板、流程实例、任务、审批、流转

Rule / Strategy 层
  -> 条件判断、路径选择、客户策略

Extension 层
  -> 客户代码级扩展点

Event / Notification 层
  -> 事件发布、通知、异步动作

Audit 层
  -> 操作日志、流程日志、状态迁移日志、业务事实记录
```

## 15. 页面层应该如何处理这些流

页面不应该直接写死流程。

页面应该做的是：

1. 展示业务对象当前状态。
2. 展示当前用户可执行动作。
3. 展示待办任务。
4. 展示审批记录。
5. 展示流程进度。
6. 展示异常原因。
7. 触发动作，例如提交、通过、驳回、撤回、确认、完成。

页面不应该关心：

- 下一个审批人是谁。
- 具体审批路径怎么走。
- 金额超过多少走几级审批。
- 哪个客户要插入哪个特殊节点。
- 某个节点后要不要自动生成下游单据。

这些应该由后端的规则、策略、流程模板和扩展点决定。

## 16. 菜单和角色的关系建议

ERP 中菜单不要只围绕审批设计，而要围绕角色任务和业务对象设计。

### 16.1 常见菜单

```text
我的待办
我的发起
我的审批
我的已办
```

## 18. 一个完整例子：销售订单提交审批

### 18.1 正常流程

```text
销售创建订单
  -> 保存草稿
  -> 提交
  -> 状态机校验：草稿是否允许提交
  -> 规则判断：是否需要审批
  -> 创建工作流实例
  -> 创建主管审批任务
  -> 主管审批通过
  -> 判断是否需要财务审批
  -> 财务审批通过
  -> 状态变成已确认
  -> 发布 OrderConfirmed 事件
  -> 自动生成采购需求或生产需求
```

### 18.2 驳回流程

```text
销售提交订单
  -> 主管审批驳回
  -> 审批任务关闭
  -> 业务状态回到草稿或待修改
  -> 记录驳回原因
  -> 销售收到待修改任务
  -> 销售修改后重新提交
  -> 创建新审批轮次
```

### 18.3 自动跳过审批

```text
销售提交订单
  -> 规则判断：金额小、客户信用正常、无特殊物料
  -> 不需要审批
  -> 状态直接变成已确认
  -> 记录自动确认原因
  -> 发布 OrderConfirmed 事件
```

## 19. 客户差异如何承接

### 19.1 客户 A 多一级老板审批

不要改核心业务代码。

推荐：

```text
客户 A 配置审批模板：
销售主管 -> 财务 -> 老板
```

### 19.2 客户 B 不需要财务审批

推荐：

```text
客户 B 配置审批模板：
销售主管 -> 老板
```

或者：

```text
规则命中后跳过财务节点
```

### 19.3 客户 C 订单确认后要调用外部系统

推荐使用扩展点：

```text
ExtensionPoint: AfterOrderConfirmed
  -> customer_c_after_order_confirmed_handler
```

### 19.4 客户 D 一个角色身兼多职

不要强行增加多个用户。

推荐：

```text
同一个用户绑定多个角色
任务分派时按角色匹配
任务去重展示
审计仍记录实际操作人
```

## 20. 设计边界和禁止事项

### 20.1 不建议做的事

不要：

- 把审批流写死在业务代码里。
- 把每个审批节点都做成业务状态。
- 把通知逻辑写进审批函数里。
- 把客户差异散落在 if customer_id == xxx 中。
- 为每个客户复制一套业务代码。
- 让页面决定流程走向。
- 让流程配置绕过状态机。
- 允许扩展点直接修改核心状态。
- 只设计正常流程，不设计异常闭环。
- 只做审批，不做任务和协同。

### 20.2 必须坚持的边界

必须：

- 业务对象有清晰状态机。
- 工作流实例和业务对象解耦。
- 审批流是工作流的一部分。
- 规则决定是否需要审批和走哪条路径。
- 策略决定某类能力的不同实现。
- 扩展点承接受控客户差异。
- 状态迁移必须可审计。
- 自动流转必须可追溯。
- 异常流程必须闭环。
- 页面只触发动作，不决定核心流程。

## 21. 推荐落地优先级

### 21.1 第一优先级：状态机

先把业务对象状态定义清楚：

```text

哪些动作

从哪到哪

是否记录审计
```

再实现：

```text

任务

```

### 21.3 第三优先级：审批能力


## 22. 一句话结论

ERP 里这些“流”应该区分清楚：

```text
业务流是主线
状态流是底盘
工作流是协同机制
审批流是工作流的一个子能力
任务流承接人工处理
异常流保证闭环
通知流负责提醒
自动流转负责提效
规则策略负责差异选择
扩展点负责受控客户特殊逻辑
```

最终目标不是把流程做复杂，而是：

> **让 ERP 在不同客户、不同角色、不同业务差异下，仍然保持清晰、高效、可维护、可扩展。**
