# ERP 业务数据生命周期、状态流转与删除策略规范

## 1. 文档目的

本文档用于规范 ERP 系统中各类业务数据的生命周期、状态流转、删除限制、作废更正、归档策略、历史追溯和审计要求。

本文档不是只约束主数据，而是约束 ERP 中所有会影响业务链路、历史追溯、库存账、财务账、审批责任和客户交付证据的数据。

主数据是本文档的重要章节，但不是全部范围。

核心目标：

1. 避免业务数据被误删导致历史断链。
2. 避免使用“删除 / 回收站”心智处理 ERP 核心业务数据。
3. 保证订单、采购、库存、生产、财务、审批、对账等链路可追溯。
4. 建立统一的数据状态流转规则。
5. 支持通用产品核心，而不是只适配单一客户。
6. 为前端按钮、后端接口、数据库约束、权限审计、测试用例提供统一依据。

---

## 2. 核心原则

ERP 系统不应该优先思考“怎么删除数据”，而应该优先思考：

> 业务是否发生过？
> 当前处于什么状态？
> 是否还能继续用于新业务？
> 历史是否还能查账、追责、统计、结算？
> 修改、取消、作废、更正是否可追溯？

统一原则：

1. 草稿数据可以删除。
2. 未被引用的配置或主数据可以受控删除。
3. 已被业务引用的数据不允许物理删除。
4. 已提交、已审核、已生效的业务单据不应删除。
5. 已经影响库存、财务、生产、对账的业务事实不应删除。
6. 业务事实错误时，应使用作废、冲销、更正、反向流水，而不是删除。
7. 主数据不用了，应停用、冻结、归档、合并或替代，而不是删除。
8. 历史业务单据必须保存关键快照，不能只依赖当前主数据展示。
9. 所有关键状态变更必须记录审计日志。
10. 数据库层面禁止使用会破坏业务链路的级联删除。

---

## 3. ERP 数据类型分层

ERP 数据不能全部套用同一种删除规则，应按数据类型分层处理。

| 数据类型   | 示例                 | 主要风险                  | 推荐处理方式            |
| ------ | ------------------ | --------------------- | ----------------- |
| 主数据    | 客户、供应商、产品、材料、仓库、工序 | 被订单、BOM、库存、财务引用后删除会断链 | 启用、停用、冻结、归档、合并、替代 |
| 业务过程数据 | 报价单、销售订单、采购订单、生产单  | 删除后业务过程断裂             | 草稿可删，提交后取消、关闭、归档  |
| 业务事实数据 | 入库单、出库单、库存流水、收付款记录 | 删除后库存账、财务账不可信         | 作废、冲销、更正、反向流水     |
| 配置数据   | 模板、字段配置、审批规则、打印配置  | 删除后历史单据展示或流程解释异常      | 启用、停用、版本化、归档      |
| 临时数据   | Excel 导入草稿、未确认解析结果 | 长期堆积、脏数据              | 可删除、可清理           |
| 派生数据   | 报表缓存、统计表、搜索索引      | 与真实业务数据不一致            | 可重建、可清理           |
| 审计数据   | 操作日志、审批日志、状态变更日志   | 删除后无法追责               | 不删除，只归档或按保留策略处理   |

## 4. 总体删除策略

### 4.1 不推荐默认提供删除按钮

ERP 核心业务页面不应默认提供明显的“删除 / 回收站”按钮。

不推荐：

```text
新增 / 编辑 / 删除 / 回收站
```

推荐：

```text
新增 / 编辑 / 提交 / 审核 / 取消 / 关闭 / 作废 / 冲销 / 更正 / 停用 / 冻结 / 归档 / 查看日志
```

删除只适合以下场景：

1. 草稿数据。
2. 临时导入数据。
3. 未被任何业务引用的主数据。
4. 未被使用的配置数据。
5. 可重建的缓存或派生数据。

---

### 4.2 删除前必须做引用检查

任何可能被业务链路引用的数据，在删除前必须检查引用关系。

引用检查至少包括：

1. 是否被订单引用。
2. 是否被 BOM 引用。
3. 是否被采购、生产、库存、财务记录引用。
4. 是否被审批流或操作日志引用。
5. 是否被打印、导出、对账、结算使用过。
6. 是否是系统预置数据。
7. 是否是客户配置保护数据。

只要存在有效引用，就禁止删除。

提示文案示例：

```text
该数据已被业务引用，不能删除。你可以停用、冻结、归档，或查看引用明细。
```

## 5. 主数据生命周期规则

### 5.1 适用对象

主数据包括但不限于：

1. 客户。
2. 供应商。
3. 产品。
4. 材料。
5. 仓库。
6. 工序。
7. 颜色。
8. 尺码。
9. 单位。
10. 联系人。
11. 收货地址。
12. 结算方式。

### 5.2 主数据处理原则

主数据本身不一定是业务事实，但一旦被业务单据引用，就成为历史业务证据的一部分。

规则：

1. 未被引用的主数据可以受控删除。
2. 已被引用的主数据禁止物理删除。
3. 不再使用的主数据应停用。
4. 有风险的主数据应冻结。
5. 长期不用的主数据应归档。
6. 重复主数据应合并。
7. 被新数据取代的主数据应设置替代关系。
8. 历史单据必须展示当时快照。

### 5.3 主数据推荐状态

| 状态  | 含义           | 是否可用于新业务 | 是否保留历史 |
| --- | ------------ | -------: | -----: |
| 草稿  | 尚未正式启用       |        否 |      是 |
| 启用  | 正常使用         |        是 |      是 |
| 停用  | 不再用于新业务      |        否 |      是 |
| 冻结  | 因风险或异常暂时禁止使用 |        否 |      是 |
| 归档  | 长期不用，默认隐藏    |        否 |      是 |
| 已合并 | 重复数据已合并到另一条  |        否 |      是 |
| 已替代 | 被新的数据替代      |      通常否 |      是 |

### 5.4 客户主数据

客户推荐状态：

```text
草稿 / 启用 / 停用 / 冻结 / 归档 / 已合并
```

客户被以下数据引用后禁止删除：

1. 报价单。
2. 销售订单。
3. 发货单。
4. 出库单。
5. 对账单。
6. 收款记录。
7. 客户合同。
8. 售后记录。

客户不用了，应停用。

客户有信用风险、欠款风险、纠纷风险时，应冻结。

重复客户应合并，而不是删除。

---

### 5.5 产品主数据

产品推荐状态：

```text
开发中 / 启用 / 停售 / 停产 / 冻结 / 归档 / 已替代
```

产品不建议只用一个 `enabled` 字段，建议拆成多个业务能力开关：

| 字段             | 含义         |
| -------------- | ---------- |
| can_quote      | 是否可报价      |
| can_sell       | 是否可销售      |
| can_produce    | 是否可生产      |
| can_purchase   | 是否可采购      |
| can_stock      | 是否参与库存     |
| can_after_sale | 是否支持售后     |
| can_use_in_bom | 是否可加入新 BOM |

产品被以下数据引用后禁止删除：

1. 报价单。
2. 销售订单。
3. 样品单。
4. BOM。
5. 生产单。
6. 入库单。
7. 出库单。
8. 库存流水。
9. 成本记录。
10. 售后记录。

产品不卖了，应停售。

产品不生产了，应停产。

产品被新产品取代，应设置 `replaced_by_id`，而不是删除老产品。

---

### 5.6 材料主数据

材料推荐状态：

```text
启用 / 停采 / 停用 / 冻结 / 归档 / 已替代
```

材料建议支持以下能力开关：

| 字段               | 含义         |
| ---------------- | ---------- |
| can_purchase     | 是否可采购      |
| can_receive      | 是否可入库      |
| can_issue        | 是否可领料      |
| can_stock        | 是否参与库存     |
| can_use_in_bom   | 是否可加入新 BOM |
| inventory_frozen | 是否冻结库存使用   |

材料被以下数据引用后禁止删除：

1. BOM。
2. 采购订单。
3. 供应商报价。
4. 入库单。
5. 出库单。
6. 领料单。
7. 退料单。
8. 库存批次。
9. 库存流水。
10. 成本核算。

材料不再采购，应停采。

材料被新材料替代，应设置替代关系。

已有库存是否允许继续消耗，应由业务规则控制。

## 6. 业务过程数据生命周期规则

### 6.1 适用对象

业务过程数据包括：

1. 报价单。
2. 销售订单。
3. 采购申请。
4. 采购订单。
5. 生产计划。
6. 生产单。
7. 外协单。
8. 发货计划。
9. 售后单。

这类数据表达的是：

> 打算做、正在办、已经提交、等待审核、正在执行、已经完成。

### 6.2 推荐状态

业务过程数据推荐状态：

```text
草稿 / 已提交 / 审核中 / 已审核 / 执行中 / 已取消 / 已关闭 / 已完成 / 已归档
```

### 6.3 删除规则

| 状态  | 是否允许删除 | 推荐操作  |
| --- | -----: | ----- |
| 草稿  |     可以 | 删除    |
| 已提交 |    不建议 | 撤回、取消 |
| 审核中 |    不建议 | 撤回、驳回 |
| 已审核 |    不允许 | 取消、关闭 |
| 执行中 |    不允许 | 关闭、终止 |
| 已完成 |    不允许 | 归档    |
| 已取消 |    不删除 | 归档    |
| 已关闭 |    不删除 | 归档    |

核心规则：

> 业务过程一旦进入提交、审核、执行、完成阶段，就不应再通过删除来消除痕迹。

---

## 7. 业务事实数据处理规则

### 7.1 适用对象

业务事实数据包括：

1. 入库单。
2. 出库单。
3. 库存流水。
4. 调拨记录。
5. 盘点记录。
6. 收款记录。
7. 付款记录。
8. 对账记录。
9. 发票记录。
10. 成本记录。

这类数据表达的是：

> 已经发生，并且以后查账、追责、统计、结算都要认它。

### 7.2 禁止删除

业务事实数据原则上不允许删除。

错误处理：

```text
删除入库单
删除出库单
删除库存流水
删除收款记录
删除付款记录
```

正确处理：

```text
作废
冲销
更正
生成反向流水
生成调整记录
保留原始记录
```

### 7.3 库存相关事实

库存相关事实尤其不能删除。

如果入库录错：

```text
不要删除原入库单
应作废或生成反向库存流水
记录原因、操作人、时间
```

如果出库录错：

```text
不要删除原出库单
应作废或生成反向库存流水
必要时重新生成正确出库单
```

如果库存数量错：

```text
不要直接改库存结果
应通过盘点单、调整单、更正流水修正
```

### 7.4 财务相关事实

收付款、对账、发票相关数据不应删除。

错误处理：

```text
删除收款记录
删除付款记录
删除对账记录
```

正确处理：

```text
冲销
作废
调整
重新对账
保留原始记录
```

## 8. 配置数据生命周期规则

### 8.1 适用对象

配置数据包括：

1. 打印模板。
2. 字段配置。
3. 审批规则。
4. 编号规则。
5. 客户专属配置。
6. 表单配置。
7. 权限角色配置。
8. 价格规则。
9. 通知规则。

### 8.2 配置数据不应随意删除

配置数据可能影响历史单据展示、打印、审批解释和客户交付。

推荐处理：

1. 启用。
2. 停用。
3. 版本化。
4. 复制新版本。
5. 归档。

### 8.3 配置版本化

对于模板、审批规则、字段配置，建议使用版本化。

示例：

```text
报价单打印模板 v1
报价单打印模板 v2
报价单打印模板 v3
```

历史单据应记录当时使用的模板版本。

不能因为当前模板改了，导致历史打印结果无法解释。

---

## 9. 临时数据与派生数据规则

### 9.1 临时数据

临时数据包括：

1. Excel 导入草稿。
2. 未确认解析结果。
3. 临时计算结果。
4. 未提交草稿附件。
5. 失败任务中间数据。

临时数据可以删除，但应明确其不是正式业务记录。

### 9.2 派生数据

派生数据包括：

1. 报表缓存。
2. 统计宽表。
3. 搜索索引。
4. 看板聚合结果。
5. 推荐结果。

派生数据可以清理和重建。

派生数据不能作为唯一业务事实来源。

真实来源应来自业务单据、库存流水、财务流水、审计记录等真源数据。

---

## 10. 审计数据规则

### 10.1 审计数据不允许随意删除

审计数据包括：

1. 操作日志。
2. 审批日志。
3. 状态变更日志。
4. 登录日志。
5. 权限变更日志。
6. 数据导入导出日志。
7. 打印日志。

审计数据用于追责、排查、审计和客户交付证据，不应被普通业务操作删除。

### 10.2 审计日志必须记录的操作

以下操作必须写入审计日志：

1. 新建。
2. 编辑。
3. 删除。
4. 启用。
5. 停用。
6. 冻结。
7. 解冻。
8. 归档。
9. 取消归档。
10. 提交。
11. 审核。
12. 驳回。
13. 取消。
14. 关闭。
15. 作废。
16. 冲销。
17. 更正。
18. 合并。
19. 替代。
20. 强制刷新快照。

### 10.3 审计日志字段

建议字段：

```text
id
tenant_id
object_type
object_id
action
before_status
after_status
before_data
after_data
reason
operator_id
operator_name
operated_at
request_id
ip
user_agent
```

## 11. 历史快照规则

### 11.1 为什么需要快照

业务单据不能只保存主数据 ID。

因为主数据未来可能会改名、改规格、改地址、改联系人、改单位、改税率。

如果历史单据只关联主数据 ID，那么主数据变化后，历史单据展示也会变化，导致历史记录不可信。

### 11.2 客户快照

销售订单、报价单、发货单、对账单中建议保存：

```text
customer_id
customer_code_snapshot
customer_name_snapshot
customer_contact_snapshot
customer_phone_snapshot
customer_address_snapshot
customer_tax_no_snapshot
customer_settlement_method_snapshot
```

### 11.3 产品快照

销售订单行、报价单行、生产单中建议保存：

```text
product_id
product_code_snapshot
product_name_snapshot
product_spec_snapshot
product_unit_snapshot
product_category_snapshot
product_version_snapshot
product_price_snapshot
```

### 11.4 材料快照

BOM、采购单、入库单、领料单中建议保存：

```text
material_id
material_code_snapshot
material_name_snapshot
material_spec_snapshot
material_unit_snapshot
material_color_snapshot
material_model_snapshot
supplier_name_snapshot
```

### 11.5 快照更新规则

| 场景          | 是否自动更新历史快照 |
| ----------- | ---------: |
| 主数据改名       |          否 |
| 主数据改规格      |          否 |
| 主数据改联系人     |          否 |
| 草稿单据重新选择主数据 |          是 |
| 已提交单据       |          否 |
| 已审核单据       |          否 |
| 已生效单据       |          否 |
| 管理员强制刷新     | 需要权限、原因和日志 |

## 13. API 设计规范

### 13.1 不推荐直接暴露通用删除接口

不推荐：

```http
DELETE /api/customers/{id}
DELETE /api/products/{id}
DELETE /api/materials/{id}
DELETE /api/sales-orders/{id}
DELETE /api/inventory-transactions/{id}
```

如必须保留删除接口，必须经过权限检查、引用检查、状态检查和审计记录。

### 13.2 推荐显式业务动作接口

推荐：

```http
POST /api/customers/{id}/disable
POST /api/customers/{id}/freeze
POST /api/customers/{id}/archive
POST /api/customers/{id}/merge

POST /api/products/{id}/stop-selling
POST /api/products/{id}/stop-production
POST /api/products/{id}/replace

POST /api/materials/{id}/stop-purchase
POST /api/materials/{id}/freeze
POST /api/materials/{id}/replace

POST /api/sales-orders/{id}/submit
POST /api/sales-orders/{id}/approve
POST /api/sales-orders/{id}/cancel
POST /api/sales-orders/{id}/close

POST /api/inbound-orders/{id}/void
POST /api/outbound-orders/{id}/void
POST /api/payments/{id}/reverse
```

动作接口比通用 delete 更适合 ERP，因为它表达的是业务语义。

## 14. 数据库约束规范

### 14.1 禁止危险级联删除

ERP 业务表之间不应使用会破坏业务链路的级联删除。

禁止：

```sql
ON DELETE CASCADE
```

尤其禁止：

```text
删除客户时级联删除订单
删除产品时级联删除 BOM
删除材料时级联删除库存流水
删除订单时级联删除出入库记录
```

推荐：

```sql
ON DELETE RESTRICT
```

或者：

```sql
ON DELETE NO ACTION
```

### 14.2 状态字段不能只靠 deleted_at

不推荐只设计：

```text
deleted_at
```

推荐：

```text
status
disabled_at
frozen_at
archived_at
deleted_at
merged_into_id
replaced_by_id
```

说明：

| 字段             | 用途            |
| -------------- | ------------- |
| status         | 表达业务生命周期      |
| disabled_at    | 停用时间          |
| frozen_at      | 冻结时间          |
| archived_at    | 归档时间          |
| deleted_at     | 仅用于未引用数据的受控删除 |
| merged_into_id | 合并关系          |
| replaced_by_id | 替代关系          |

## 15. 权限规范

不同动作应拆分权限，不应只有一个“管理”权限。

推荐权限点：

| 权限点                | 含义      |
| ------------------ | ------- |
| data:view          | 查看数据    |
| data:create        | 新建数据    |
| data:update        | 编辑数据    |
| data:delete_unused | 删除未引用数据 |
| data:disable       | 停用      |
| data:freeze        | 冻结      |
| data:archive       | 归档      |
| data:merge         | 合并      |
| data:replace       | 替代      |
| document:submit    | 提交单据    |
| document:approve   | 审核单据    |
| document:cancel    | 取消单据    |
| document:close     | 关闭单据    |
| fact:void          | 作废业务事实  |
| fact:reverse       | 冲销业务事实  |
| fact:correct       | 更正业务事实  |
| audit:view         | 查看审计日志  |

高风险操作必须单独授权：

1. 删除未引用主数据。
2. 冻结客户。
3. 供应商拉黑。
4. 客户合并。
5. 产品替代。
6. 材料替代。
7. 作废入库单。
8. 作废出库单。
9. 冲销收付款。
10. 强制刷新历史快照。

## 16. 多租户与客户差异

### 16.1 产品核心规则

以下规则应作为产品核心，不建议客户随意修改：

1. 已引用数据禁止物理删除。
2. 业务事实不允许删除。
3. 历史单据必须保留关键快照。
4. 删除前必须做引用检查。
5. 状态变更必须记录审计日志。
6. 数据库不得使用危险级联删除。
7. 新业务默认只选择可用状态数据。
8. 历史查询必须能显示停用、冻结、归档数据。

### 16.2 可配置规则

以下规则可以按客户配置：

1. 哪些角色可以停用客户。
2. 哪些角色可以冻结客户。
3. 客户冻结是否自动阻止发货。
4. 供应商冻结是否自动阻止采购。
5. 产品停产后是否允许售后。
6. 材料停采后是否允许继续领料。
7. 归档数据默认隐藏多久。
8. 是否允许普通管理员删除未引用数据。
9. 作废单据是否必须审批。
10. 冲销财务记录是否必须二次确认。

## 18. Codex 实现约束建议

可以将以下内容追加到 `AGENTS.md` 或开发规范中：

```text
ERP 项目中，业务页面不要默认实现删除和回收站心智。

客户、供应商、产品、材料、仓库、工序等主数据，应优先实现启用、停用、冻结、归档、合并、替代、查看引用、查看历史。

报价单、销售订单、采购订单、生产单等业务过程数据，草稿可以删除，提交后应通过撤回、取消、关闭、完成、归档等状态流转处理。

入库单、出库单、库存流水、收付款记录、对账记录等业务事实数据，不允许物理删除，应通过作废、冲销、更正、反向流水或调整单处理。

已被业务引用的数据禁止物理删除。

删除接口必须先调用引用检查，不允许直接删除。

业务单据必须保存关键主数据快照，不能只依赖主数据 ID 展示历史内容。

数据库外键不得使用 ON DELETE CASCADE 删除业务数据。

关键状态变更必须写入审计日志。

新业务选择数据时默认只显示可用状态；历史查询必须能显示停用、冻结、归档、已合并、已替代的数据。
```

## 19. 推荐落地顺序

### 19.1 第一层：统一删除保护

先完成：

1. 已引用数据禁止删除。
2. 删除前引用检查。
3. 删除操作写审计日志。
4. 禁止危险级联删除。

完成：

### 19.3 第三层：业务单据状态流转

2. 提交。
3. 审核。
4. 取消。
5. 关闭。
6. 完成。

### 19.4 第四层：业务事实作废更正

## 20. 最终规则

项目应统一采用以下规则：

```text
ERP 核心业务数据不以删除和回收站作为默认心智。

草稿可以删除。
临时数据可以删除。
派生数据可以清理。
未被引用的主数据可以受控删除。

主数据被引用后不能删除，应停用、冻结、归档、合并或替代。

业务过程数据提交后不能删除，应撤回、取消、关闭、完成或归档。

业务事实数据不能删除，应作废、冲销、更正或生成反向流水。

配置数据不应随意删除，应启停、版本化或归档。

审计数据不应删除，应长期保留或按策略归档。

历史业务必须保留快照。
关键操作必须记录审计日志。
删除前必须做引用检查。
数据库禁止危险级联删除。
```

一句话总结：

> ERP 系统不是 CMS，也不是普通后台。它的核心不是“删掉数据”，而是“保留业务发生过的证据，并用状态流转表达当前结果”。
