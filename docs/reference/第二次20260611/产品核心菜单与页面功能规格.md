# Plush Toy ERP 产品核心菜单与页面功能规格

> 适用阶段：ERP MVP / 客户试用候选
> 产品定位：玩具制造业私有化 ERP
> 目标：明确每个核心菜单页面要做什么、为什么做、页面字段、流程逻辑、状态流转和实现边界。
> 使用方式：可直接用于原型图设计、前端页面拆分、后端 API 拆分、数据库字段设计和 Codex 实现任务拆解。

---

# 1. 产品核心目标

当前系统要优先跑通一条制造业 ERP 主链路：

```text
客户
销售订单
SKU / 产品
BOM
采购订单
采购收货
质检
入库
生产 / 外协
出货
财务业务事实
```

所有页面设计都围绕这条链路展开。

# 2. 菜单设计原则

## 2.1 菜单不要按开发阶段命名

禁止出现在产品菜单中的命名：

```text
Phase 7
Phase 8
Phase 9
business_records
测试事实
内部闭环
```

产品菜单必须使用业务语言：

```text
销售订单
采购订单
库存流水
生产任务
外协任务
出货单
财务事实
```

---

## 2.2 菜单必须围绕真实业务事实

每个菜单页面都要回答三个问题：

```text
1. 这个页面对应哪个业务事实？
2. 谁会使用这个页面？
3. 这个页面的操作会改变哪些事实源？
```

例如：

```text
销售订单页面 -> sales_orders / sales_order_items
库存流水页面 -> inventory_txns
出货单页面 -> shipments / shipment_items / inventory_txns
```

---

## 2.3 列表页、详情页、动作页分开设计

每个核心模块建议拆成三类页面：

```text
列表页：查找、筛选、批量操作
详情页：查看完整业务对象
动作页/弹窗：执行状态变更或业务动作
```

例如销售订单：

```text
销售订单列表
销售订单详情
创建销售订单弹窗
审核销售订单动作
关闭销售订单动作
```

---

## 2.4 所有状态变化必须有审计

这些动作必须写审计日志：

```text
创建
编辑
删除 / 作废
审核
反审核
取消
关闭
入库
出库
发货
取消发货
质检通过
质检拒收
生成财务事实
权限变更
导入执行
```

# 4. 通用页面组件规范

## 4.1 列表页通用布局

每个列表页建议包含：

```text
顶部标题区
快捷操作按钮
筛选区
数据表格
批量操作区
分页
导出按钮
列设置
```

### 通用筛选字段

| 字段     | 为什么               |
| ------ | ----------------- |
| 关键词    | 快速搜索编号、名称、联系人、备注  |
| 状态     | ERP 业务高度依赖状态      |
| 日期范围   | 按创建、下单、收货、发货等日期筛选 |
| 客户/供应商 | 销售和采购核心维度         |
| 仓库     | 库存和出入库核心维度        |
| 创建人    | 追踪责任人             |
| 是否异常   | 快速定位问题数据          |

---

## 4.2 详情页通用布局

详情页建议使用 Tab：

```text
基本信息
明细
流程/任务
库存影响
财务影响
附件
评论
审计日志
```

不是每个页面都需要全部 Tab，但核心业务页面建议保留结构。

---

## 4.3 通用动作按钮

| 按钮   | 使用场景             |
| ---- | ---------------- |
| 新建   | 创建主数据或业务单据       |
| 编辑   | 修改草稿或允许修改的数据     |
| 删除   | 仅允许草稿或未使用数据      |
| 作废   | 已被引用的数据不能删除，只能作废 |
| 审核   | 业务确认             |
| 反审核  | 开发阶段可有，正式阶段需权限控制 |
| 关闭   | 业务完成但不删除         |
| 导出   | 列表导出             |
| 导入   | 批量创建数据           |
| 查看审计 | 追踪变更             |
| 复制   | BOM、订单、采购单常用     |
| 打印   | 采购单、出货单、BOM、质检单  |

# 5. 首页

---

## 5.1 ERP 总览

### 页面目标

让用户进入系统后立即看到：

```text
今天要处理什么
哪些订单有风险
哪些库存不足
哪些采购未到
哪些出货延期
```

### 为什么需要

ERP 用户不是为了看数据，而是为了处理异常。首页必须突出待办和风险。

### 页面模块

| 模块   | 内容              |
| ---- | --------------- |
| 今日待办 | 当前用户待处理任务       |
| 销售概览 | 待审核、生产中、待出货订单数  |
| 采购概览 | 待审批、待收货、逾期未到采购单 |
| 库存风险 | 缺料、低库存、负库存风险    |
| 质检风险 | 待检、拒收、返工        |
| 出货风险 | 待发货、逾期未发、部分发货   |
| 财务提醒 | 待确认应收、待确认应付     |
| 最近操作 | 最近审计日志          |

### 字段设计

| 字段   | 类型   | 为什么               |
| ---- | ---- | ----------------- |
| 日期范围 | 日期选择 | 首页数据必须支持看今日、本周、本月 |
| 业务角色 | 下拉   | 不同角色看不同重点         |
| 客户   | 下拉   | 管理层常按客户查看         |
| 订单状态 | 多选   | 快速筛风险订单           |
| 风险类型 | 多选   | 聚焦缺料、延期、质检失败等异常   |

### 核心功能

1. 点击 KPI 跳转对应列表页。
2. 点击异常项跳转到过滤后的业务页面。
3. 根据角色显示不同卡片。
4. 支持刷新。
5. 支持客户维度筛选。
6. 支持日期维度筛选。

### 流程逻辑

```text
用户登录
  -> 读取用户角色
  -> 读取权限范围
  -> 加载对应 KPI
  -> 加载待办
  -> 加载异常提醒
  -> 点击卡片跳转到业务列表
```

## 5.2 异常提醒

### 页面目标

集中处理所有跨模块异常。

### 为什么需要

ERP 实际使用中，用户最关心的不是“所有单据”，而是“哪里出问题了”。

### 异常类型

| 异常        | 来源        | 处理人   |
| --------- | --------- | ----- |
| 销售订单逾期未交付 | 销售订单 / 出货 | 销售    |
| 采购订单逾期未收货 | 采购订单      | 采购    |
| 质检未完成     | 质检任务      | 质检    |
| 质检拒收      | 质检记录      | 采购/仓库 |
| 库存不足      | 库存余额 / 预留 | 仓库/采购 |
| 出货缺库存     | 出货单 / 库存  | 仓库    |
| 生产未完工     | 生产任务      | 生产    |
| 外协未回货     | 外协任务      | 外协    |
| 应收未确认     | 财务事实      | 财务    |
| 应付未确认     | 财务事实      | 财务    |

### 字段设计

| 字段     | 为什么                                    |
| ------ | -------------------------------------- |
| 异常编号   | 追踪异常                                   |
| 异常类型   | 区分业务来源                                 |
| 严重程度   | 排优先级                                   |
| 关联单据   | 跳转业务对象                                 |
| 责任角色   | 谁处理                                    |
| 责任人    | 具体处理人                                  |
| 首次发生时间 | 判断持续多久                                 |
| 截止时间   | 判断是否超期                                 |
| 状态     | open / processing / resolved / ignored |
| 处理备注   | 记录处理说明                                 |

### 流程逻辑

```text
系统根据规则生成异常
  -> 异常进入 open
  -> 责任人处理
  -> 关联业务单据状态变化
  -> 异常自动 resolved 或人工关闭
```

# 6. 我的待办

---

## 6.1 我的任务

### 页面目标

显示当前用户需要处理的任务。

### 为什么需要

一线人员不应该从各个菜单里找工作，而应该直接看到“我现在要做什么”。

### 任务类型

```text
销售订单审核
采购订单审核
采购收货确认
质检任务
库存调整审批
生产上报
外协发料
外协回货
出货确认
财务确认
导入异常处理
```

### 列表字段

| 字段     | 为什么        |
| ------ | ---------- |
| 任务编号   | 追踪任务       |
| 任务类型   | 判断要做什么     |
| 标题     | 人可读        |
| 关联业务对象 | 跳转原始单据     |
| 优先级    | 先做重要任务     |
| 状态     | 待处理/处理中/完成 |
| 发起人    | 谁发起        |
| 当前处理人  | 谁负责        |
| 截止时间   | 是否超期       |
| 创建时间   | 排序         |
| 最近更新时间 | 判断是否卡住     |

### 操作按钮

| 按钮 | 说明      |
| -- | ------- |
| 查看 | 打开任务详情  |
| 处理 | 执行动作    |
| 转交 | 转给其他人   |
| 评论 | 留业务说明   |
| 完成 | 完成任务    |
| 驳回 | 审批类任务使用 |

### 流程逻辑

```text
业务单据创建/状态变化
  -> 生成 workflow task
  -> 分配给角色或用户
  -> 用户处理
  -> 任务完成
  -> 触发业务 usecase 状态变化
```

### 注意

Workflow 不直接写库存、财务、出货事实。
Workflow 只触发对应领域 usecase。

## 6.2 我发起的

### 字段

| 字段    | 为什么        |
| ----- | ---------- |

| 流程类型  | 区分审批/任务/异常 |

# 7. 主数据

---

## 7.1 客户管理

### 页面目标

维护销售客户信息，作为销售订单、出货、应收的基础。

### 为什么需要

客户是销售链路的起点。客户信息不稳定会影响订单、交期、收款和发货。

### 列表字段

| 字段     | 为什么            |
| ------ | -------------- |
| 客户编号   | 唯一识别           |
| 客户名称   | 主显示字段          |
| 客户简称   | 列表展示更方便        |
| 客户类型   | 区分品牌方、贸易商、终端客户 |
| 状态     | 是否可下单          |
| 联系人    | 快速联系           |
| 电话     | 业务联系           |
| 邮箱     | 发送文件           |
| 国家/地区  | 出货和贸易信息        |
| 地址     | 发货基础           |
| 创建时间   | 追踪             |
| 最近订单时间 | 判断活跃度          |

### 表单字段

| 字段          | 必填 | 为什么     |
| ----------- | -: | ------- |
| 客户编号        |  是 | 系统唯一识别  |
| 客户名称        |  是 | 业务主名称   |
| 客户简称        |  否 | 页面展示    |
| 客户类型        |  是 | 后续筛选和权限 |
| 统一社会信用代码/税号 |  否 | 财务和发票预留 |
| 默认币种        |  否 | 销售和财务   |
| 默认付款方式      |  否 | 应收管理    |
| 默认联系人       |  否 | 订单联系    |
| 默认收货地址      |  否 | 出货默认值   |
| 状态          |  是 | 控制是否可用  |
| 备注          |  否 | 补充信息    |

### 状态

```text
ACTIVE
INACTIVE
ARCHIVED
```

### 核心功能

1. 新建客户。
2. 编辑客户。
3. 禁用客户。
4. 查看客户订单。
5. 查看客户联系人。
6. 查看客户收货地址。
7. 查看客户应收事实。
8. 导入客户。

### 流程逻辑

```text
创建客户
  -> 状态 ACTIVE
  -> 可创建销售订单

禁用客户
  -> 不允许新建销售订单
  -> 历史订单保留

归档客户
  -> 只读
  -> 不出现在默认选择列表
```

### 页面目标

### 为什么需要

### 列表字段

| 字段     | 为什么           |
| ------ | ------------- |
| 供应商编号  | 唯一识别          |
| 供应商名称  | 主显示           |

## 16.1 导入任务

### 页面目标

管理客户历史数据导入任务。

### 为什么需要

客户试用和私有化部署通常需要导入历史 Excel、PDF、图片、旧系统数据。

### 列表字段

| 字段         | 为什么           |
| ---------- | ------------- |
| 导入任务号      | 唯一识别          |
| 客户         | 数据来源          |
| 导入类型       | 客户/产品/BOM/订单等 |
| 文件数量       | 范围            |
| 状态         | 流程            |
| dry-run 结果 | 是否可执行         |
| 成功数量       | 导入结果          |
| 失败数量       | 异常            |
| 创建人        | 责任            |
| 创建时间       | 追踪            |

### 状态

```text
DRAFT
DRY_RUNNING
DRY_RUN_PASSED
DRY_RUN_FAILED
READY_TO_APPLY
APPLYING
APPLIED
FAILED
CANCELLED
```

### 核心功能

1. 创建导入任务。
2. 上传或选择 source manifest。
3. dry-run。
4. 查看预检结果。
5. 查看未匹配队列。
6. 修复映射。
7. apply 导入。
8. 生成导入报告。
9. 回滚导入，可选。

---

## 16.2 导入预检

### 页面目标

展示导入 dry-run 的检查结果。

### 字段

| 字段         | 为什么            |
| ---------- | -------------- |
| 检查项        | 哪类检查           |
| 结果         | pass/warn/fail |
| 影响行数       | 范围             |
| 错误样例       | 方便修复           |
| 建议动作       | 指导用户           |
| 是否阻塞 apply | 控制导入           |

### 检查类型

```text
文件是否存在
sha256 是否匹配
字段是否缺失
客户是否匹配
供应商是否匹配
SKU 是否匹配
物料是否匹配
单位是否匹配
BOM 是否可生成
订单是否重复
```

### 流程逻辑

```text
创建导入任务
  -> 读取 manifest
  -> 检查文件
  -> 解析数据
  -> 匹配主数据
  -> 生成 dry-run report
  -> 如果无阻塞错误，允许 apply
```

| 置信度  | 自动匹配可信度       |

| 状态   | 流程            |

### 状态

```text
OPEN
RESOLVED
IGNORED
BLOCKED
```

### 操作

1. 匹配到已有主数据。
2. 新建主数据。
3. 忽略。
4. 标记阻塞。
5. 批量处理。

## 22.2 每个页面交给 Codex 的通用提示词

```text
请根据 PRODUCT_CORE_MENU_SPEC.md 实现页面：<页面名称>。

要求：
1. 只实现这个页面相关的前端、API、usecase、repo、schema/migration，不要顺手改无关模块。
2. 页面必须包含列表、筛选、详情、创建/编辑动作。
3. 字段按照文档中的字段设计。
4. 状态流转按照文档中的状态机。
5. 所有关键动作必须写 audit event。
6. 不允许通过 business_records 写正式业务事实。
7. 不允许新增 phase8 API。
8. data 层只做 repository，不做 JSON-RPC dispatch。
9. 增加必要测试。
10. 完成后输出修改文件、测试结果、未完成项和风险。
```

# 23. 总结

这个产品核心菜单规格的核心思想是：

```text
每个页面都有明确业务事实源。
每个动作都有明确状态流转。
每个字段都服务于流程、审计、追踪或后续自动化。
先实现制造业 ERP 主链路，再做报表、扫码、SaaS 和高级扩展。
```

最短 MVP 菜单链路是：

```text
客户
产品
SKU
物料
BOM
销售订单
采购订单
采购收货
质检
库存
生产
外协
出货
应收/应付
审计
```

只要这条链路跑通，项目就从“模块很多”变成“真正可试用的 ERP”。
