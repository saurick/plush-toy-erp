# Plush Toy ERP 自动化测试方案

> 当前状态：产品化内测候选
> 下一目标：可控客户试用候选
> 目标：建立一套可复现、可扩展、可进入 CI 的自动化测试体系，覆盖安全、后端领域逻辑、数据库迁移、JSON-RPC API、前端页面、端到端业务流程、客户导入、备份恢复和 release package。
> 使用方式：可直接用于拆 Codex 任务、补测试目录、配置 CI、制定客户试用前验收标准。

---

# 1. 自动化测试总体目标

本项目的自动化测试不是为了追求形式上的覆盖率，而是为了保证 ERP 核心事实不会错。

自动化测试必须优先保护这些风险点：

1. 不能泄漏 secret / token。
2. 生产危险配置必须启动失败。
3. SQL tracing / logging 不能记录 SQL 参数。
4. 运行时不能再出现 `phase8`。
5. `business_records` 不能重新成为正式事实写入源。
6. Workflow 不能直接写库存、出货、采购、财务事实。
7. 库存不能负数。
8. 库存预留不能超预留。
9. 出货不能重复扣库存。
10. 取消出货不能重复回滚库存。
11. 采购收货不能超收。
12. BOM 激活规则必须正确。
13. 销售订单到出货到应收的主链路必须可跑通。
14. 客户导入 dry-run 必须可复现。
15. release package 不能包含敏感信息。
16. 权限控制必须由后端强制执行，不只依赖前端隐藏菜单。

# 2. 测试分层模型

建议采用如下测试金字塔：

```text
              E2E / Playwright / Release Smoke
            ------------------------------------
          API Contract / JSON-RPC Integration
        ----------------------------------------
      Domain Integration / DB Transaction Tests
    --------------------------------------------
  Unit Tests / Pure Functions / Validators / Utils
--------------------------------------------------
Static Checks / Type / Lint / Secret Scan / Migration Check
```

每层目标不同：

| 测试层                      | 目标                   |            是否进 CI |   优先级 |
| ------------------------ | -------------------- | ----------------: | ----: |
| Static Checks            | 发现低成本问题              |                 是 |    P0 |
| Secret Scan              | 防止敏感信息入库             |                 是 |    P0 |
| Config Preflight Tests   | 防止生产危险配置             |                 是 |    P0 |
| Unit Tests               | 验证纯逻辑                |                 是 |    P0 |
| Domain Integration Tests | 验证领域事实和事务            |                 是 | P0/P1 |
| DB Migration Tests       | 验证 schema 可迁移        |                 是 |    P0 |
| JSON-RPC API Tests       | 验证接口行为和权限            |                 是 |    P1 |
| Frontend Component Tests | 验证页面组件行为             |                 是 | P1/P2 |
| E2E Tests                | 验证核心业务链路             |              是，分级 | P1/P2 |
| Import Tests             | 验证客户导入 dry-run/apply |                 是 | P0/P1 |
| Security Tests           | 验证认证、权限、安全头          |                 是 | P0/P1 |
| Backup/Restore Tests     | 验证私有化交付              | Nightly / Release |    P3 |
| Performance Smoke        | 验证基础性能退化             | Nightly / Release |    P3 |

# 6. 测试数据策略

## 6.1 不要在测试里散落手写数据

建议建立统一 test builder：

```text
server/test/testkit/builders.go
```

包含：

```go
NewCustomerBuilder()
NewSupplierBuilder()
NewProductBuilder()
NewSKUBuilder()
NewMaterialBuilder()
NewBOMBuilder()
NewSalesOrderBuilder()
NewPurchaseOrderBuilder()
NewInventoryBalanceBuilder()
NewShipmentBuilder()
```

---

## 6.2 测试数据必须可读

示例：

```text
customer: "Test Customer A"
supplier: "Test Supplier A"
product: "Plush Bear"
sku: "PLUSH-BEAR-PINK-S"
material: "Pink Fabric"
warehouse: "Main Warehouse"
```

不要使用无意义数据：

```text
abc
test1
foo
bar
```

---

## 6.3 所有测试必须确定性

禁止：

```text
依赖当前时间导致结果不稳定
依赖测试运行顺序
依赖外部网络
依赖真实 token
依赖真实客户原始文件
依赖本机路径
```

建议：

```text
统一 fake clock
统一 test IDs
统一 fixture path
统一 timezone
统一 random seed
```

---

## 6.4 时间测试要求

ERP 状态很依赖日期，测试中必须显式设定：

```text
order_date
expected_arrival_date
delivery_date
shipment_date
created_at
updated_at
```

不要依赖 `time.Now()` 直接决定断言。

## P1-T04：库存预留并发测试

### 目标

并发预留不能超过可用库存。

### 建议测试文件

```text
server/test/integration/inventory_reservation_concurrency_test.go
```

### 必须使用 PostgreSQL

该测试必须使用 PostgreSQL，因为 SQLite 无法可靠模拟生产锁行为。

### 测试数据

```text
SKU: PLUSH-BEAR-PINK-S
Warehouse: MAIN
Inventory available: 100
Concurrent requests: 10
Each request reserve: 20
Expected successful reservations: 5
Expected total reserved: 100
Expected failed reservations: 5
```

### 测试场景

| 场景                      | 预期                   |
| ----------------------- | -------------------- |
| 10 个请求并发各预留 20，总库存 100  | 只有 5 个成功             |
| 2 个请求并发各预留 80，总库存 100   | 只有 1 个成功             |
| 使用同一 idempotency key 重试 | 返回同一结果，不新增           |
| 取消预留后重新预留               | pass                 |
| 取消两次                    | no-op 或明确 fail，不释放两次 |

### Go 测试命令

```bash
cd server
go test ./test/integration -run TestInventoryReservationConcurrency -count=20
```

### Codex 提示词

```text
请实现 P1-T04：库存预留并发测试。

要求：
1. 使用 PostgreSQL 测试库。
2. 构造库存 available=100。
```

# 16. 覆盖率目标

覆盖率不要一开始追求虚高，但核心领域必须硬性要求。

## 16.1 初始目标

| 模块                  | 覆盖率目标 |
| ------------------- | ----: |
| config/preflight    |  90%+ |
| inventory usecase   |  85%+ |
| shipment usecase    |  85%+ |
| purchase usecase    |  80%+ |
| sales usecase       |  80%+ |
| bom usecase         |  80%+ |
| auth/rbac           |  85%+ |
| import scripts      |  80%+ |
| frontend components |  60%+ |

## 16.2 不建议只看总覆盖率

不推荐这种目标：

```text

```

原因：

```text
容易为了覆盖率测试无关 getter、DTO、简单组件，却漏掉库存并发、出货幂等等真正风险。
```

推荐：

```text
核心领域必须覆盖
风险动作必须覆盖
状态机必须覆盖
事务和并发必须覆盖
```

# 27. 人工 Review 清单

每个测试 PR 人工 Review 时检查：

```text
[ ] 测试是否真的断言了业务事实，而不是只断言无报错
[ ] 测试是否依赖真实 secret
[ ] 测试是否依赖本机路径
[ ] 测试是否依赖当前时间
[ ] 测试是否依赖运行顺序
[ ] 测试是否会污染其他测试
[ ] 并发测试是否使用 PostgreSQL
[ ] 幂等测试是否断言行数和库存数量
[ ] 权限测试是否走后端 API
[ ] 前端权限测试是否没有替代后端权限测试
[ ] E2E 是否太脆弱
[ ] 测试数据是否可读
[ ] 错误信息是否清晰
[ ] CI 是否会运行该测试
[ ] 如果测试暂不进 CI，是否说明原因
```

# 28. 最终目标

自动化测试最终要达到：

```text
每次提交都能保护安全和基础质量。
每次合并都能保护核心领域事实。
每次发布都能证明客户试用包可运行、可导入、可备份、可恢复、无 secret。
```

一句话原则：

```text
ERP 自动化测试的核心不是“页面能打开”，而是“业务事实不会错”。
```
