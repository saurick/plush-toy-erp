# 甲方配置、Seed、数据导入、测试与部署规范

> 适用项目：Plush Toy ERP
> 适用阶段：客户试用候选 / 私有化部署候选
> 目标：规范每个甲方的配置包、初始化 seed、历史数据导入、测试验证、部署发布、回滚和签收流程。
> 核心原则：每个甲方都是一个独立交付单元；配置、seed、导入、部署必须可审计、可复现、可回滚、可测试。

---

# 1. 为什么需要这份规范

ERP 私有化项目和普通 SaaS 不同。每个甲方通常都有自己的：

* 公司名称和品牌。
* 菜单开关。
* 角色和权限。
* 字段叫法。
* 单据编号规则。
* 默认仓库。
* 默认单位。
* 物料分类。
* 产品分类。
* 客户/供应商历史数据。
* SKU / BOM / 库存期初数据。
* 历史订单。
* 财务期初或业务应收应付。
* 导入文件格式。
* 部署环境。
* 账号体系。
* 数据保密要求。

如果没有统一规范，很容易出现：

* A 客户配置污染 B 客户。
* demo seed 进入生产。
* 导入数据重复。
* 历史文件丢失或无法复现。
* 期初库存和库存流水不一致。
* 客户字段含义映射错误。
* 生产部署无法回滚。
* 上线后无法证明导入过什么、跳过了什么、失败了什么。

所以必须把每个甲方作为一个可管理的交付包。

# 2. 核心原则

## 2.1 每个甲方一个独立配置包

每个甲方必须有独立目录：

```text
customers/
  <customer-code>/
    config/
    seed/
    import/
    deployment/
    evidence/
    reports/
```

示例：

```text
customers/
  yoyoosun/
    config/
    seed/
    import/
    deployment/
    evidence/
    reports/
```

注意：

* 真实客户原始文件不建议直接提交到产品源码仓库。
* 可以提交脱敏样例、manifest、字段映射、导入报告。
* 真实文件应放在受控存储，并通过 hash、manifest、签收记录关联。

---

## 2.2 配置、seed、导入必须分开

三者不能混在一起。

| 类型           | 作用                 |     是否可重复执行 |  是否可删除 |
| ------------ | ------------------ | ----------: | -----: |
| Config       | 控制系统行为、菜单、字段、品牌、开关 |           是 |    可替换 |
| Seed         | 初始化必要主数据、角色、权限、默认值 |        必须幂等 |  不随意删除 |
| Import       | 导入客户历史业务数据         |      需要批次控制 | 不可随意重跑 |
| Demo Seed    | 演示数据               | 只能 dev/demo |   生产禁止 |
| Test Fixture | 自动化测试数据            |      只能测试环境 |   生产禁止 |

---

## 2.3 生产环境禁止 demo/test 数据

生产或客户试用环境禁止：

```text
demo customer
demo supplier
test order
test SKU
test password
mock SMS code
debug seed
debug cleanup
sample finance fact
fake shipment
```

生产或客户试用环境只允许：

```text
system seed
customer config
customer-approved seed
customer import data
customer migration data
```

---

## 2.4 所有导入必须 dry-run 先行

任何客户数据导入都必须先：

```text
manifest check
schema check
mapping check
duplicate check
referential integrity check
business invariant check
dry-run report
customer confirmation
backup
apply
post-apply verification
```

不允许直接 apply。

---

## 2.5 所有客户数据导入必须可追溯

每次导入必须记录：

```text
import_job_id
customer_code
source_manifest_version
source_file_hash
mapping_version
operator
started_at
finished_at
dry_run_report
apply_report
unresolved_queue
created_count
updated_count
skipped_count
failed_count
rollback_plan
```

# 4. 客户配置规范

## 4.1 `customer.config.json`

### 作用

定义甲方基础信息和部署标识。

### 示例

```json
{
  "version": 1,
  "customerCode": "yoyoosun",
  "customerName": "YoyooSun",
  "deploymentType": "private",
  "timezone": "Asia/Shanghai",
  "locale": "zh-CN",
  "defaultCurrency": "CNY",
  "defaultLanguage": "zh-CN",
  "industry": "plush-toy-manufacturing",
  "enabled": true
}
```

### 字段说明

| 字段                | 必填 | 说明                     |
| ----------------- | -: | ---------------------- |
| `version`         |  是 | 配置版本                   |
| `customerCode`    |  是 | 稳定客户编码，不可随意变更          |
| `customerName`    |  是 | 客户显示名称                 |
| `deploymentType`  |  是 | private / trial / demo |
| `timezone`        |  是 | 时间解释规则                 |
| `locale`          |  是 | 语言和格式                  |
| `defaultCurrency` |  是 | 默认币种                   |
| `defaultLanguage` |  是 | 默认语言                   |
| `industry`        |  是 | 行业模板                   |
| `enabled`         |  是 | 是否启用                   |

### 注意事项

* `customerCode` 必须稳定，不能用中文名。
* 不要把客户真实 secret 放进该文件。
* 生产环境 secret 通过环境变量或 Secret Manager 注入。
* timezone 必须明确，否则历史日期导入会出错。

---

## 4.2 `menu.config.json`

### 作用

控制甲方启用哪些菜单。

### 示例

```json
{
  "version": 1,
  "menus": {
    "dashboard": true,
    "tasks": true,
    "masterData.customers": true,
    "masterData.suppliers": true,
    "masterData.products": true,
    "masterData.skus": true,
    "masterData.materials": true,
    "masterData.bom": true,
    "sales.orders": true,
    "purchase.orders": true,
    "purchase.receipts": true,
    "quality.inspections": true,
    "inventory.balances": true,
    "inventory.txns": true,
    "inventory.reservations": true,
    "production.tasks": true,
    "outsourcing.tasks": true,
    "shipments": true,
    "finance.facts": true,
    "imports": true,
    "reports": false,
    "system.audit": true,
    "legacyArchive": false
  }
}
```

### 注意事项

* 菜单隐藏不是权限控制。
* 后端 API 必须仍然检查权限。
* 客户试用阶段不成熟模块可以隐藏。
* 隐藏模块不能被路由直接访问绕过。

---

## 4.3 `field-alias.config.json`

### 作用

适配甲方自己的字段叫法。

### 示例

```json
{
  "version": 1,
  "aliases": {
    "customer": {
      "customerName": "客户名称",
      "customerOrderNo": "客户订单号"
    },
    "product": {
      "skuCode": "款号",
      "spec": "规格/颜色"
    },
    "material": {
      "materialCode": "物料编号",
      "materialName": "材料名称"
    }
  }
}
```

### 注意事项

* 字段别名只影响 UI 显示。
* 不应该改变数据库字段含义。
* 不允许同一页面两个字段显示成同一个名字。
* 导入映射应使用系统标准字段，不直接使用 UI alias。

---

## 4.4 `numbering.config.json`

### 作用

定义单据编号规则。

### 示例

```json
{
  "version": 1,
  "rules": {
    "salesOrder": "SO-{yyyyMMdd}-{seq4}",
    "purchaseOrder": "PO-{yyyyMMdd}-{seq4}",
    "purchaseReceipt": "PR-{yyyyMMdd}-{seq4}",
    "qualityInspection": "QC-{yyyyMMdd}-{seq4}",
    "inventoryTxn": "ITX-{yyyyMMdd}-{seq6}",
    "productionTask": "MO-{yyyyMMdd}-{seq4}",
    "outsourcingTask": "OS-{yyyyMMdd}-{seq4}",
    "shipment": "SH-{yyyyMMdd}-{seq4}",
    "financeFact": "FIN-{yyyyMMdd}-{seq4}"
  }
}
```

### 注意事项

* 编号规则必须保证唯一。
* 编号生成必须在数据库事务中防并发冲突。
* 导入历史单据时，要区分“客户原单号”和“系统单号”。
* 不要用客户原始 Excel 行号当系统单号。

---

### 作用

控制功能开关。

### 示例

```json
{
  "version": 1,
  "features": {
    "publicRegister": false,
    "smsMock": false,
    "debugSeed": false,
    "debugCleanup": false,
    "sqlArgsTracing": false,
    "legacyArchive": true,
    "financeLedger": false,
    "barcodeScan": false,
    "attachmentUpload": false,
    "pdaMode": false,
    "customerImportApply": false
  }
}
```

### 生产禁止开启

```text
publicRegister
smsMock
debugSeed
debugCleanup
sqlArgsTracing
```

### 注意事项

* feature flag 必须进入 production preflight。
* 客户试用前不要默认开启未完成能力。
* hidden feature 不能绕过权限和 API 校验。

## 4.6 `permissions.config.json`

### 作用

定义甲方角色和权限矩阵。

### 角色建议

```text
super_admin
admin
sales
purchase
warehouse
quality
production
outsourcing
finance
viewer
```

### 示例

```json
{
  "version": 1,
  "roles": [
    {
      "role": "sales",
      "permissions": [
        "customer.read",
        "salesOrder.read",
        "salesOrder.create",
        "salesOrder.updateDraft",
        "salesOrder.confirm"
      ]
    },
    {
      "role": "warehouse",
      "permissions": [
        "inventory.read",
        "inventory.reserve",
        "shipment.read",
        "shipment.ship"
      ]
    }
  ]
}
```

### 注意事项

* 权限配置必须由后端强制。
* 前端菜单只是展示控制。
* 初始 super admin 只能 bootstrap 一次。
* 不允许 seed 把普通用户自动提升为 super admin。

# 5. Seed 数据规范

## 5.1 Seed 分类

| Seed 类型         | 说明             |   是否生产允许 |
| --------------- | -------------- | -------: |
| System Seed     | 系统必需状态、枚举、默认配置 |        是 |
| Role Seed       | 角色             |        是 |
| Permission Seed | 权限             |        是 |
| Masterdata Seed | 客户确认过的基础主数据    |        是 |
| Unit Seed       | 单位和换算          |        是 |
| Warehouse Seed  | 默认仓库和库位        |        是 |
| Demo Seed       | 演示订单、演示客户、演示库存 |        否 |
| Test Fixture    | 自动化测试数据        |        否 |
| Import Seed     | 从客户文件导入的初始业务数据 | 是，但必须有报告 |

---

## 5.2 Seed 必须幂等

Seed 脚本必须支持重复执行。

重复执行时：

```text
不重复创建角色
不重复创建权限
不重复创建默认仓库
不重复创建单位
不重复创建 admin
不重复写库存期初
不重复创建业务单据
```

每条 seed 应该有稳定 key：

```text
seed_key
customer_code
entity_type
entity_code
version
```

---

## 5.3 `seed-manifest.json`

### 作用

记录 seed 文件、版本、hash、执行顺序。

### 示例

```json
{
  "version": 1,
  "customerCode": "yoyoosun",
  "seeds": [
    {
      "seedId": "roles-v1",
      "file": "roles-seed.json",
      "type": "roles",
      "sha256": "",
      "required": true,
      "order": 10
    },
    {
      "seedId": "permissions-v1",
      "file": "permissions-seed.json",
      "type": "permissions",
      "sha256": "",
      "required": true,
      "order": 20
    },
    {
      "seedId": "warehouse-v1",
      "file": "warehouse-seed.json",
      "type": "warehouse",
      "sha256": "",
      "required": true,
      "order": 30
    }
  ]
}
```

### 验收标准

* seed 执行顺序明确。
* 每个 seed 有 hash。
* required seed 缺失必须失败。
* seed 重复执行结果一致。
* seed 报告记录 created / updated / skipped。

---

## 5.4 Seed 执行报告

每次 seed 必须输出：

```json
{
  "customerCode": "yoyoosun",
  "seedBatchId": "seed-20260611-001",
  "status": "passed",
  "created": 10,
  "updated": 2,
  "skipped": 30,
  "failed": 0,
  "errors": [],
  "startedAt": "2026-06-11T10:00:00+08:00",
  "finishedAt": "2026-06-11T10:00:30+08:00"
}
```

---

## 5.5 Seed 注意事项

### 不能做

```text
不能在生产 seed 中创建 demo 订单
不能创建默认弱密码
不能创建 mock 手机验证码
不能写真实 secret
不能跳过权限检查创建 super admin
不能重复写期初库存
不能把 test fixture 放入客户试用环境
```

### 必须做

```text
seed 前备份数据库
seed 前检查 APP_ENV
seed 前执行 preflight
seed 后生成报告
seed 后做 smoke test
seed 后写 audit event
```

# 6. 客户数据导入规范

## 6.1 导入阶段

标准流程：

```text
1. Data Intake：接收客户文件
2. Data Classification：分类文件
3. Source Manifest：生成源文件清单
4. Mapping：字段映射
5. Validation：基础校验
6. Dry-run：预导入
7. Unresolved Queue：处理未匹配
8. Customer Confirmation：客户确认
9. Backup：导入前备份
10. Apply：正式导入
11. Post-apply Verification：导入后验证
12. Signoff：签收
```

---

## 6.2 原始文件接收注意事项

### 必须记录

```text
文件名
文件大小
sha256
文件类型
来源人
接收时间
是否原始文件
是否脱敏
是否客户确认
```

### 不建议

```text
把客户 raw files 直接提交到产品仓库
在聊天记录中传递长期保存的敏感文件
重命名后丢失 originalName
只靠中文文件名定位文件
不记录 hash
```

---

## 6.3 `source-manifest.json`

### 作用

让导入工具不依赖中文文件名、系统路径或人工记忆。

### 示例

```json
{
  "version": 1,
  "customerCode": "yoyoosun",
  "receivedAt": "2026-06-11T10:00:00+08:00",
  "sources": [
    {
      "sourceId": "customer-master-v1",
      "type": "customer_master",
      "originalName": "客户资料.xlsx",
      "storedName": "customer-master-v1.xlsx",
      "sha256": "",
      "required": true,
      "containsPII": true,
      "sanitized": false,
      "notes": ""
    },
    {
      "sourceId": "bom-26029-v1",
      "type": "bom",
      "originalName": "26029#夜樱烬色材料明细表2026-1-19.xlsx",
      "storedName": "bom-26029-v1.xlsx",
      "sha256": "",
      "required": true,
      "containsPII": false,
      "sanitized": false,
      "notes": ""
    }
  ]
}
```

### 必须校验

```text
sourceId 唯一
type 合法
storedName 存在
sha256 匹配
required 文件不可缺失
文件大小合理
文件扩展名符合 type
中文 originalName 不影响导入
```

---

## 6.4 字段映射规范

每种导入类型都要有 mapping 文件。

示例：

```text
import/mapping/customer.mapping.json
import/mapping/supplier.mapping.json
import/mapping/product.mapping.json
import/mapping/sku.mapping.json
import/mapping/material.mapping.json
import/mapping/bom.mapping.json
import/mapping/inventory-opening.mapping.json
import/mapping/sales-order.mapping.json
```

### mapping 示例

```json
{
  "version": 1,
  "sourceType": "customer_master",
  "targetEntity": "customers",
  "fields": [
    {
      "sourceColumn": "客户名称",
      "targetField": "customer_name",
      "required": true,
      "transform": "trim"
    },
    {
      "sourceColumn": "客户编号",
      "targetField": "customer_code",
      "required": false,
      "transform": "trim_upper"
    }
  ],
  "dedupe": {
    "keys": ["customer_code", "customer_name"],
    "strategy": "match_or_create"
  }
}
```

## 6.5 导入顺序

推荐顺序：

```text
1. 单位 units
2. 仓库 warehouses / locations
3. 客户 customers
4. 供应商 suppliers
5. 联系人 contacts
6. 产品 products
7. SKU product_skus
8. 物料 materials
9. BOM headers / items
10. 期初库存 inventory opening
11. 历史销售订单 sales orders
12. 历史采购订单 purchase orders
13. 历史收货 / 质检 / 库存流水
14. 历史出货 shipments
15. 期初应收 / 应付 finance facts
```

原因：

```text
后面的业务单据依赖前面的主数据。
BOM 依赖 SKU、物料、单位。
库存依赖 SKU/物料、仓库、批次。
销售订单依赖客户和 SKU。
采购订单依赖供应商和物料。
财务事实依赖销售/采购/出货/外协来源。
```

# 7. 关键导入注意事项

## 7.1 客户和供应商

### 注意

* 客户名称可能重复。
* 客户简称可能不是唯一。
* 客户编码可能缺失。
* 电话、联系人可能混在一个单元格。
* 客户和供应商可能同名。
* 旧系统里的客户状态可能不规范。

### 测试

```text
重复客户识别
缺失客户编号处理
同名不同客户处理
客户联系人拆分
禁用客户不允许新订单
```

---

## 7.2 产品和 SKU

### 注意

* 产品和 SKU 不要混为一谈。
* 客户款号不一定等于系统 SKU。
* 同一产品可能有颜色、尺寸、包装版本。
* SKU code 必须稳定唯一。
* 旧资料可能只有产品名，没有 SKU。

### 测试

```text
SKU code 唯一
客户 SKU 映射正确
同产品多个 SKU 正确关联
禁用 SKU 不可下单
SKU 可关联 BOM 和库存
```

---

## 7.3 物料

### 注意

* 物料名称可能不规范。
* 同一物料可能有不同颜色、规格。
* 单位可能不统一。
* 默认供应商可能不唯一。
* 安全库存可能缺失。

### 测试

```text
物料去重
单位映射
默认供应商映射
物料类别映射
物料用于 BOM
```

---

## 7.4 单位和换算

### 注意

单位是导入高风险点。

常见问题：

```text
个 / 只 / pcs 混用
箱和个未设置换算
米和码混用
kg 和 g 混用
客户 Excel 中单位为空
```


## 7.5 BOM

### 注意

BOM 是最容易导入错的核心数据。

必须确认：

```text

BOM 是否生效
物料是否都能匹配
用量单位是否正确
损耗率是否正确
替代料如何表达
同 SKU 是否已有 active BOM
```

### 测试

```text
BOM item qty > 0
loss_rate >= 0
所有物料必须匹配
所有单位必须匹配
同 SKU 只能有一个 ACTIVE BOM
ACTIVE BOM 不允许直接覆盖
导入新版本必须生成 DRAFT 或归档旧版本
```

## 7.6 期初库存

### 注意

期初库存不能随便写库存余额。

必须决定：

```text
是否生成 OPENING_BALANCE 库存流水
期初日期是什么
仓库/库位是什么
批次是否需要
质量状态是什么
是否待检
是否可用
```

### 推荐做法

```text
期初库存通过 inventory_txns 写入
txn_type = OPENING_BALANCE
source_type = CUSTOMER_IMPORT
source_id = import_job_id
```

### 禁止

```text
直接 update inventory_balances 而没有流水
重复导入导致库存翻倍
没有仓库的库存
没有单位的库存
待检/不良库存计入可用库存
```

### 测试

```text
期初库存导入后余额等于流水聚合
重复 apply 不重复增加库存
缺少仓库进入 unresolved queue
负库存进入错误
未知 SKU/物料进入 unresolved queue
```

## 7.9 历史出货

### 注意

历史出货会影响：


### 关键选择

如果期初库存已经是当前库存：

```text

```

如果导入从历史库存流水重放：

```text

```

### 测试

```text
历史出货不会重复扣当前库存

应收事实不重复
```

```text
未收金额计算正确

不会生成总账凭证
```

```text
5. 输出 import-apply-report.json。
6. 增加测试。
```

# 20. 最终原则

一句话：

```text
每个甲方都必须有独立、可验证、可回滚的交付包。
```

更具体地说：

```text
配置要可 lint。
seed 要幂等。
导入要 dry-run。
apply 要有批次。
库存要有流水。
财务要有边界。
部署要有 preflight。
上线要有 smoke。
失败要能回滚。
交付要有报告。
```
