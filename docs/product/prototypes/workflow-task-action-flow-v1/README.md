# Workflow 任务处理流程样板 / Workflow Task Action Flow

本目录保存任务中心、岗位任务端和业务页协同入口可共用的任务处理流程原型。资产状态为 `To Implement`；运行时共享抽屉已吸收核心三步交互，但本原型仍承担完整状态矩阵和后续一致性评审，不因此自动晋级为 Current。

## 文件

| 文件 | 用途 |
| --- | --- |
| `index.html` | 单文件可交互 HTML 原型，覆盖可点击三步导航、动作权限差异、原因校验、提交中、失败重试、成功回执、键盘、窄屏和暗色。 |
| `images/task-action-flow-redesign-reference.png` | 本轮生成的任务处理视觉方向图，只作层级比较；交互、校验和状态恢复以 HTML 原型为准。 |

## 设计结论

- 三步固定为 `核对任务 -> 选择处理 -> 确认与结果`。步骤 1、2 是直接可点击的导航按钮，不再要求用户先点击底部动作才能看到第二步。
- 步骤 3 是受控确认页：必须先选择当前任务允许的处理方式；若动作需要原因，还必须输入非空原因。未满足时步骤仍可获得焦点并解释缺失项，但不会越过校验。
- 催办只属于步骤 2 的处理方式，与完成、阻塞、退回、解除阻塞并列。催办不会变成“下一步”“进入表单”或“打开处理页”的代名词。
- 步骤切换不丢失已选动作与原因；返回核对页后可以再次直接进入步骤 2。
- 提交使用显式 `提交中` 状态并禁用重复提交。成功后停留在步骤 3，展示动作、时间和影响范围回执，由用户主动返回任务列表。
- 失败状态不假定任务已经改变，保留动作和原因供重试；失败示例首次提交失败，第二次重试成功，用于 review 恢复路径。
- 只读终态没有提交入口；仅催办示例只显示催办，不通过 disabled 假按钮暗示其他能力。
- 原型采用真实按钮、radio、textarea 和 `tabpanel` 语义。场景页签支持左右方向键、`Home`、`End`；`Ctrl/⌘ + Enter` 用于继续或提交。

## 运行时吸收边界

- 运行时 `WorkflowTaskActionDrawer` 已改为真正可点击的核对 / 选择处理 / 确认提交三步：步骤 1、2 可直接进入，步骤 3 由动作与必填原因解锁；催办只是处理方式之一。完整失败重试、结果回执和移动端状态仍按本原型继续评审。
- 真实动作集合、只读原因和提交权限必须继续来自现有任务 action explain、RBAC、owner / assignee、任务状态与领域命令合同；前端不得根据这个静态原型自行放宽。
- 真实实现必须使用当前共享 Workflow API 和提交前权限复核，不从样例任务、静态状态或页面按钮推导授权。
- 完成、阻塞、解除阻塞、退回和催办只处理当前 Workflow 任务 / 事件。库存、质检、出货、应收、应付、开票和付款仍由对应业务页面与领域 usecase 处理。
- `done`、`rejected` 等终态不可重复提交；`blocked` 只显示真实允许的解除阻塞 / 催办动作。
- “返回任务看板”只链接到同目录下的原型样板。正式运行时路径、菜单、客户显隐和 RBAC 不由本资产决定。
- 本原型继续保持 `To Implement`，不得因静态交互自测或未来局部 runtime 吸收而自动晋级 `Current`。

## Review 与验收关注点

- 步骤 1、2 可用鼠标、触控和键盘直接进入；步骤 3 在前置不满足时给出原因并把焦点带回动作或原因输入。
- 选择无原因动作可直接进入确认；选择阻塞、退回、解除阻塞或催办时，空原因必须阻断。
- 提交中不能重复提交或返回修改；成功后不自动关闭，失败后不清空输入。
- 切换常规、失败、只读和仅催办示例时，旧动作、旧原因、旧回执和失败次数全部清理，避免残值。
- radio 方向键、场景页签方向键、Tab 顺序、`Ctrl/⌘ + Enter`、焦点目标和可访问名称清楚。
- 360px 级窄屏采用单列步骤与动作，不出现页面级横向滚动；暗色主题保持状态、边框、焦点和错误对比度。
- 用户可见文案使用岗位语言，不暴露 raw ID、权限 key、API、schema、trace 或内部状态字段。
