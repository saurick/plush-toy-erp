# 后台工作台样板 / Admin Workbench Pattern

本目录保存后台首页 / 工作台的 To Implement 标准样板，用于重新确认“工作台到底做什么”和“哪些入口不应继续并列放大”。

## 文件

| 文件 | 用途 |
| --- | --- |
| `index.html` | 单文件可交互 HTML 原型，把工作台收敛为今日处理台：今日焦点、优先队列、当前处理卡、相关对象快捷入口和交接边界。 |

## 设计结论

- 工作台的意义是登录后第一屏判断：今天先处理什么、为什么重要、应该进入哪个业务对象。
- 工作台不应继续展示成“工作台 / 任务看板 / 业务看板 / 打印中心 / 异常闭环”五个并列视图；详细任务、经营看板、打印编辑和异常处理可以从工作台跳转，但不应在工作台里重复铺开。
- 工作台只保留能继续处理当前事项的相关对象入口，例如销售订单、入库通知/检验/入库、库存和单据模板；这些入口必须标注为常用入口 / 快捷入口，并随当前任务高亮，具体客户菜单仍由客户配置和 RBAC 决定。
- 任务完成只代表 Workflow 协同状态变化，不代表库存、出货、财务或发票事实过账。

## 边界

- 本原型自身不接后端、不接权限、不进入 ERP 正式菜单、不进入生产构建。
- 本原型不替代 `docs/当前真源与交接顺序.md`、Workflow / Fact 架构文档、schema、migration、API 或测试。
- 真实实现时应优先复用后台布局、业务列表共享组件、任务详情面板和打印模板组件，不把 HTML 直接复制进 `web/src`。
- 如果后续继续扩展任务看板、业务看板或工作台菜单，需要同步评审菜单配置、RBAC 权限码、seed、帮助文档和浏览器回归。

## 验收关注点

- 优先队列筛选可切换，当前处理卡随选择变化。
- 优先队列筛选需要有明确的 `tablist / tab / tabpanel` 状态语义，当前选中项应同步 `aria-selected`。
- 相关对象快捷入口数量克制，并随当前处理卡高亮；首屏不重复铺开完整菜单，且明确不是新的正式菜单结构。
- 桌面宽屏和移动窄屏不出现横向溢出。
- 文案避免把 Workflow 任务状态误写成事实层过账结果。
