# 岗位任务中心 v2 / Mobile Role Tasks v2

结论：这是岗位任务端当前 **Current** 选中任务流程，但不是整页替换：保留 v1 的待办 / 已办 / 提醒 / 我的列表、筛选和分页，选中任务后进入 v2“查看任务 → 处理任务 → 可信结果回执”，结束后恢复原列表状态。v1 列表基线与 v2 选中任务流程共同组成当前移动任务主路径。

## 单一任务

岗位人员在手机上完成一件事：从任务列表找到当前事项，核对信息；有多种处理方式时明确选择动作，只有一种动作时直接填写所需信息，确认后得到可追溯回执。

组合主路径固定为：

1. v1 任务列表：保留现有标签、筛选、分页 / 分批展开和任务卡片。
2. v2 全屏查看任务：核对任务、来源、权限和真实风险。
3. v2 全屏处理任务：只显示服务端允许的动作和对应输入。
4. v2 全屏结果回执：只展示可信提交结果，并可恢复原列表状态。

详情与处理步骤可按当前权限直接点击切换；结果步骤只有在已有已确认回执或“提交结果待确认”记录时才开放。没有回执时不能进入空结果页，用户需先在处理步骤完成一次有效提交。`催办` 只属于第 2 步里的处理动作，不再充当“打开处理页”的隐含入口。

## 重新设计的关键决策

| 问题                               | v2 决策                                                                                                                   |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| 点击催办才像是进入下一步           | 三个步骤使用真实按钮；详情、处理按权限直接导航，结果只在已有确认回执或待确认记录时开放。                                  |
| 详情、动作输入与结果挤在同一长页面 | 拆成三个全屏任务阶段，每屏只完成一个判断。                                                                                |
| 催办与流程导航混在一起             | 催办与完成、阻塞、退回、解除阻塞同属“处理动作”；没有额外导航权限。                                                        |
| 无权限时只能看见禁用按钮           | 仅催办任务显示非交互的“本次操作：催办”摘要并直接填写原因，不伪装成可选择按钮；只读任务显示原因和已有回执，不展示假动作。  |
| 选择与执行容易混为一谈             | 只有多种处理方式时使用原生单选框；卡片内标题和说明负责建立选择语境，底部“确认催办”等按钮才是真正执行命令。                |
| 提交后缺少闭环反馈                 | 提交后进入结果回执，明确成功、待确认、动作、当前状态和说明；历史处理线索只读保留，原型不补造处理人或时间。                |
| 列表与深层处理的职责容易混淆       | 保留 v1 列表标签、筛选和分页交互；`todo / risk / history` 仍是各自服务端查询视图，v2 只承接选中任务后的查看、处理与回执。 |
| 长文本、键盘、暗色和窄屏容易失控   | 文本允许自然换行；内容区独立滚动；底部动作区固定；适配 390 / 430、暗色和移动键盘。                                        |

视觉方向采用成熟企业移动工作台常见的克制模式：单列任务队列、低装饰层级、语义色只表达风险或结果、44px 以上触控目标、固定主动作和可见焦点；不复制任何外部产品的品牌或业务结构。

## 可评审状态

列表中四条样本分别承担不同的回归职责：

| 样本                 | 进入详情        | 第 2 步                                                            | 第 3 步                        |
| -------------------- | --------------- | ------------------------------------------------------------------ | ------------------------------ |
| 确认来料存放位置     | 可处理          | 完成 / 阻塞 / 退回 / 催办；覆盖必填校验和提交                      | 提交前不可进入 -> 提交成功回执 |
| 等待成品质检结论     | 仅可催办        | 显示非交互的“本次操作：催办”，直接填写催办原因，不允许代替品质完成 | 提交前不可进入                 |
| 查看上周成品入库结果 | 只读            | 解释只读原因，无提交按钮                                           | 已确认成功回执                 |
| 多批次混装长文本任务 | 可处理 + 长文本 | 长标题、长来源、长说明、长输入                                     | 结果待确认 -> 刷新确认恢复路径 |

处理校验保持当前项目语义：

- `完成`：完成反馈必填；处理页不再重复收集附件编号、链接或自由文本“证据”。
- `阻塞 / 退回 / 解除阻塞 / 催办`：对应原因必填；催办不开放附件上传。
- 照片、异常截图和处理凭证统一在任务详情的“任务附件”中按权限查看或管理；旧 `evidence_refs` 仅作为“历史处理线索”只读显示。
- 校验失败时定位并聚焦第一个错误字段；提交时禁用重复动作。
- `Ctrl/Cmd + Enter` 可提交；`Escape` 返回详情；触屏和浏览器 / 系统返回都不丢失当前草稿。
- 使用 `visualViewport` 收敛软键盘出现后的可视高度，输入框聚焦时滚动到可见区域。

## 业务与权限边界

- 原型只表达页面骨架、信息层级、导航、处理表单和反馈状态，不新增 API、RBAC、菜单、路由、schema、migration 或客户专属逻辑。
- 列表中的 `todo / risk / history` 只是三类服务端查询视图的交互表达；正式运行时继续使用各自游标、快照与分页结果，不在前端把三类数据重新拼成第二套任务真源。
- 可见不等于可处理。仅催办和只读样本用于评审 `canOperateTask` / `canUrgeTask` 的页面投影，不是新的权限真源。
- 催办权不等于附件写入权。附件上传继续要求 `workflow.task.update` 和服务端 owner / assignee 范围；无上传权时不显示假“选择附件”控件。
- Workflow task 的完成、阻塞、退回、解除阻塞或催办只更新协同任务，不等于库存、出货、质检、应收、应付、开票或付款事实过账。
- 关联来源只显示业务可读单号与摘要，不暴露内部主键、权限 key、API 名称或开发者字段。
- 详情中的处理提示是只读的原型规则示例，不是人工输入内容。正式运行时应由任务状态、当前账号可用操作和来源入口共同派生；来源入口还必须通过当前账号的页面权限投影，无法可靠派生时不应使用静态样例兜底。
- 原型数据全部是评审样本，不是 yoyoosun 真实数据或客户确认事实。

## 已吸收切片与待验收差异

本地工作树已把以下内容映射回当前组件、API、权限与回归测试：

1. `MobileRoleTasksPage` 的列表选择与三步 screen state。
2. `MobileTaskDetailScreen` 的全屏详情、直接步骤导航、权限解释，以及真实“任务附件”与历史处理线索的分离展示。
3. 独立全屏处理页的动作选择、原因 / 反馈草稿保留、字段校验、提交中与重复提交保护；多动作使用原生单选框，单一催办不显示伪选择控件或证据输入。
4. 独立结果回执页的 `confirmed / unknown / failed` 三态、重新确认与恢复入口；提交前不开放空回执页。
5. 390 / 430、暗色、长文本、焦点返回、软键盘和固定动作区浏览器回归。

以下现有能力是 v2 的非回退边界，不因页面结构调整而删除或用静态样例替代：

- 客户运行态 guard、岗位权限与 action explain 仍在进入列表和提交动作前生效；失败时保持 fail closed。
- `todo / risk / history` 各自保留服务端游标分页、快照一致性、空态、失败重试与过期请求保护。列表只显示一个“显示更多”入口：优先展开已加载内容，耗尽后按同一快照续取下一页；顶部刷新才重新获取最新快照。
- 任务附件继续使用真实附件组件和上传权限；无上传权时只显示查看能力。回执里的动作、状态、原因与历史处理线索只消费后端确认结果或受权限保护的事件投影，不由前端推断处理人或时间。
- 账号信息、切换正式允许的入口和退出登录仍需保留清晰入口；浏览器 / 系统返回、刷新恢复、焦点返回和软键盘可视区都属于正式回归范围。

上述代码吸收已经形成有意的 v1 列表 + v2 选中任务流程，不以“整页切成 v2”为完成条件。Current 只表示当前产品与本地运行态采用该流程；自动化、浏览器验收、发布、目标环境 smoke 和客户 UAT 仍必须分别举证，不能互相替代。

## 人工评审路径

1. 在 390px 和 430px 宽度分别打开 `index.html`，确认页面无横向滚动；再在真实岗位页确认 v1 列表标签、筛选和分页仍可使用。
2. 从列表任务进入详情，确认“任务附件”按权限显示“查看与补充附件”或“查看任务附件”，历史处理线索不与真实附件混称；没有回执的任务第 3 步应禁用，已有可信回执的任务才能进入第 3 步。
3. 在可处理任务选择“完成”，空提交应聚焦完成反馈；填写后提交应进入成功回执。
4. 在仅催办任务确认第 2 步不出现“选择处理方式”、可点击的催办选项、证据输入或附件上传，只显示“本次操作：催办”摘要、催办原因与底部“确认催办”；在只读任务确认没有提交按钮。
5. 打开长文本任务的第 3 步，先看到结果待确认，再点击刷新结果确认恢复态。
6. 切换暗色并用 Tab、Shift+Tab、Escape、Ctrl/Cmd+Enter 检查键盘路径与可见焦点，再用浏览器 / 系统返回核对详情和列表的筛选、已加载分页、滚动与焦点恢复。
