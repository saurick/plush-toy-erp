# 业务管理中心样板 / Business Management Center Pattern

本目录保存后台“业务管理”入口的 To Implement 标准样板，用于重新确认业务管理菜单的意义。

## 文件

| 文件 | 用途 |
| --- | --- |
| `index.html` | 单文件可交互 HTML 原型，把业务管理入口收敛为业务对象总控：按链路选择对象、查看当前风险、进入标准业务页或详情页。 |

## 设计结论

- 业务管理菜单的意义不是把销售、采购、库存、生产、出货和财务全部堆成一个大菜单；它应回答“当前要管理哪条业务链路、哪类对象、下一步进入哪个标准页”。
- 业务管理中心只做对象选择、链路摘要、风险聚合和跳转；真实字段、保存、导出、过账、冲正和状态机仍回到各业务模块标准页、详情页、表单页和动作浮层。
- 业务管理不能成为第二套 `business_records` 或第二套事实写入入口；它也不能因为页面上出现对象就承诺对应 API、schema 或 RBAC 已完成。
- 未实现或只恢复入口壳的领域必须明确标注“待接入”，避免菜单看起来比能力更成熟。

## 待实现吸收状态

截至 2026-06-13，本原型按 `待实现 / To Implement` 登记。当前运行时已有若干正式 V1 页面和正式入口壳；本样板只说明后续如果需要“业务管理”总入口，应怎样避免变成模块堆叠或第二套菜单。

| 原型区域 | 当前运行时参考路径 | 尚待人工确认点 | 边界 |
| --- | --- | --- | --- |
| 业务链路切换 | 现有业务菜单和正式入口壳 | 是否需要一个业务管理总入口，或继续保留分组菜单。 | 不替代正式菜单评审。 |
| 当前链路对象 | `BusinessModulePage` 标准页、详情页、表单页 | 是否按客户与订单、采购入库质检、库存出货、财务等链路组织。 | 不新增未评审 API。 |
| 风险摘要 | 工作台、业务看板、异常闭环 | 是否从只读投影展示风险。 | 只读 projection，不写 Fact。 |
| 进入标准页 | 各领域正式页面 | 是否统一跳转到标准列表页、详情页或动作浮层。 | 字段、动作和权限仍由目标页负责。 |

明确不直接吸收：原型里的固定数字、静态对象、mock 脚本、链路名称和模块顺序不进入正式运行时；真实实现必须重新核对当前菜单配置、RBAC、路由和领域能力成熟度。

## 验收关注点

- 链路切换可用，当前链路对象和详情说明同步变化。
- 页面明确区分“可进入的标准页”和“待接入的事实动作”。
- 桌面和移动视口不出现页面级横向溢出。
- 文案不能把业务管理中心写成第二套菜单、第二套 API 或第二套事实真源。
