# 任务看板中心样板 / Task Command Center Pattern

本目录保存后台“任务看板”的 `To Implement` 标准样板。它把任务入口重构为紧凑的职责处理台，不替代正式业务菜单，也不代表运行时已经实现。

## 文件

| 文件 | 用途 |
| --- | --- |
| `index.html` | 单文件可交互 HTML 原型，覆盖四类互斥任务队列、组合筛选、键盘选中、主从详情、相关单据入口和三步任务处理抽屉。 |
| `images/task-board-redesign-reference.png` | 本轮生成的桌面信息层级方向图，只作视觉比较，不替代 HTML 交互原型或运行真源。 |

共用任务处理流程的完整状态样板见 `../workflow-task-action-flow-v1/index.html`。任务中心只吸收该流程的入口和核心交互，不为自己维护第二套业务规则。

## 本轮重设计结论

- 首屏改为“分类指标 + 紧凑筛选 + 任务队列 / 当前详情”主从结构。删除大面积宣言卡、重复业务对象捷径和独立边界说明区，把空间留给任务判断与处理。
- 四个指标本身就是队列分类入口，不再额外复制一组页签；再次点击当前指标或点击“查看全部分类”回到总队列。
- 任务卡单击选中并同步右侧详情，桌面双击或按 `Enter` 打开处理；上下方向键、`Home`、`End` 可在当前结果中移动。
- 当前详情只展示“为什么现在要看、负责岗位、到期时间、业务来源、相关单据、处理提示、可用动作、最近协同记录”。不展示 raw ID、权限 key、schema 字段或开发说明。
- “处理提示”是只读的原型规则示例，不是用户填写的下一步计划。正式运行时应由任务状态、当前账号可用操作和来源入口共同派生；来源入口还必须通过当前账号的页面权限投影，无法可靠派生时不应显示静态兜底建议。
- 任务处理使用三步状态机：`核对任务 -> 选择处理 -> 确认与结果`。步骤 1、2 可直接点击切换；步骤 3 必须先选择有效动作，并在需要时填写原因。
- 催办是与完成、阻塞、退回、解除阻塞并列的处理选项，不再充当“进入第二步”的入口，也不会因为点击催办才显示处理表单。
- 完成动作不要求原因；阻塞、退回、解除阻塞和催办要求原因。提交中禁用重复提交，成功后停留在结果页并给出回执，再由用户返回队列。
- 原型内含常规可处理、阻塞、退回只读、已完成只读和仅催办任务，便于 review 权限与状态差异；静态任务均为中性样例。
- 保留暗色主题、窄屏单列、抽屉全屏、焦点返回、`Escape` 关闭、焦点循环与 `Ctrl/⌘ + Enter` 继续 / 提交语义。

## 页面单一主任务

任务看板只帮助有权限的桌面用户完成三件事：

1. 从当前可见范围中判断谁该处理、哪里阻塞、哪些到期。
2. 选中一条任务，核对当前上下文和可用操作。
3. 记录 Workflow 处理结果，或跳转到相关业务对象继续办理。

它不复制销售、采购、生产、库存、出货或财务菜单，不提供通用业务对象快捷入口，也不把任务完成写成业务事实完成。

## 与当前真源的边界

- 本原型继续按 `待实现 / To Implement` 登记；未获用户明确确认前不得晋级 `Current`。
- 运行时 `/erp/task-board` 本轮已吸收四类可点击指标、组合筛选、左侧任务队列 / 右侧当前任务、只读处理提示、单一“处理任务”入口和共享三步任务抽屉；原型还覆盖更完整的只读、仅催办、失败重试和回执状态，因此继续保持 To Implement，不自动晋级 Current。
- 真实实现必须继续连接现有 `get_task_board`、任务可见范围、动作 explain、RBAC、`owner_role_key`、assignee 和任务状态合同；原型固定数字和 mock 脚本不得进入运行时。
- `rejected` 与 `done` 在本样板中只读；仅催办示例表达动作权限受限，不代表新增权限码或状态。
- “查看相关单据”只表达跳转语义，并且仅在正式路由可解析且当前账号菜单权限允许时显示；不因任务中心新增菜单入口。
- Workflow 任务完成、阻塞、解除阻塞、退回或催办只更新当前协同任务 / 事件；库存、质检、出货、应收、应付、开票和付款仍由对应领域 usecase 处理。

## Review 与验收关注点

- 四个分类数字在当前组合筛选下与各自队列数量一致；全部结果等于四类互斥数量之和。
- 搜索、状态、负责岗位、业务来源和分类筛选可组合；结果变化后只在当前结果内恢复选择，不默认选中已被筛掉的任务。
- 单击、双击、上下方向键、`Home`、`End`、`Enter`、`Escape`、Tab 焦点循环和 `Ctrl/⌘ + Enter` 均有明确语义。
- 步骤 1、2 可任意切换；步骤 3 在未选动作或必填原因为空时保持受控，并把焦点带回需要补充的位置。
- 催办只出现在处理方式中；仅催办任务不显示完成、阻塞、退回或解除阻塞。
- 只读任务仍可核对详情与相关单据，但不能进入提交确认；终态不能重复提交。
- 提交中、成功回执、原因校验、窄屏和暗色状态不出现页面级横向溢出、遮挡或错误焦点。
- 文案始终区分 Workflow 与 Fact，不把静态原型、局部交互或回执写成真实运行态已交付。
