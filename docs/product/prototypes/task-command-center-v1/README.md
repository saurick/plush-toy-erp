# 任务中心样板 / Task Command Center Pattern

本目录保存后台“我的任务 / 任务看板”的 To Implement 标准样板，用于重新确认任务入口的意义。

## 文件

| 文件 | 用途 |
| --- | --- |
| `index.html` | 单文件可交互 HTML 原型，把任务入口收敛为职责处理台：待我处理、我发起的、阻塞交接、当前任务详情、任务处理抽屉和关联业务对象。 |

## 设计结论

- 任务菜单的意义不是复制业务菜单，也不是给每个流程节点新增入口；它只回答“哪些任务现在需要我处理、我发起的卡在哪里、阻塞该交给谁”。
- 任务中心只处理 Workflow 协同任务、原因、催办、改派和跳转；任务完成不等于库存、出货、财务、发票或收付款事实过账。
- 业务对象只作为任务上下文和跳转入口出现，例如销售订单、入库通知/检验/入库、出货放行、单据模板；它们不替代正式业务菜单。
- 岗位任务端仍按职责投影，不复制桌面菜单树；桌面任务中心用于跨角色管理、跟进和交接。
- 任务处理抽屉应承载“核对上下文 -> 选择动作 / 填写原因 -> 提交回队列”的短路径；步骤提示和 Workflow / Fact 边界合并为紧凑导引，不再拆成多张说明卡。
- 抽屉底部只放当前可执行动作和关联对象跳转；阻塞、退回、催办必须前置原因输入，完成动作必须明确只关闭 Workflow 任务。

## 验收关注点

- 筛选可切换，当前任务详情随选择变化。
- 当前任务的关联业务对象会高亮，但明确只是跳转上下文。
- 任务处理抽屉可打开 / 关闭，完成、阻塞、催办三类动作状态清晰；阻塞和催办展示原因输入，完成不要求原因。
- 桌面和移动视口不出现页面级横向溢出。
- 文案不能把任务完成、异常关闭或催办写成事实层过账结果。
