# 业务主链路数据流向与字段来源规则 / Business Data Flow And Field Source Policy

## 0. 文档目的 / Purpose

本文回答两个问题：

1. `plush-toy-erp` 是否要做从订单到产品、材料、仓库、出库、出运和财务的数据流向。
2. 新建 / 编辑表单、列表、导入、导出和打印中的字段，哪些应该选择、导入、带值、派生或只读，哪些才允许人工输入。

本文是产品治理和实施边界文档，不替代 `docs/当前真源与交接顺序.md`、Ent schema、Atlas migration、后端 usecase、API、RBAC、前端实现或测试结果。

## 1. 结论 / Decision

业务主链路数据流向必须做，而且应作为产品内核主线推进。

但数据流向不是新增一套万能流程引擎，也不是恢复旧 `business_records`、创建泛化 `change_records`，更不是让 Workflow 直接写库存、出货或财务事实。正确主路径是：

```text
MasterData 主数据
-> Source Document 源单据
-> Domain Usecase 业务动作
-> Fact 事实表
-> Derived Projection / Report 派生展示
```

对应 ERP 口径是：

```text
客户 / 供应商 / 产品 / SKU / 材料 / 工序 / 单位 / 仓库
-> 销售订单 / BOM / 采购订单 / 生产或委外单 / 出货单
-> 入库 / 质检 / 领料 / 成品入库 / 出库 / 发货 / 对账
-> inventory_txns / shipments / finance_facts 等事实
-> 订单进度、库存可用量、未出货、应收应付、导出报表
```

## 2. 基础边界 / Core Boundaries

| 层级 | 写入职责 | 不能承担 |
| --- | --- | --- |
| MasterData | 客户、供应商、联系人、产品、SKU、材料、工序、单位、仓库等稳定资料 | 不承接订单临时字段、客户样本残值或事实状态 |
| Source Document | 销售订单、采购订单、生产单、委外单等业务承诺 | 不直接写库存、出货、应收、应付、发票或收付款事实 |
| Workflow | 任务分派、审批、阻塞、退回、协同状态推进 | 不写库存流水、出货事实、财务事实 |
| Fact | 采购、库存、质检、生产、委外、出货、财务等真实业务动作 | 不从菜单、旧快照、workflow payload 或表单显示伪造事实 |
| Derived | 进度、余额、剩余数量、应收应付状态、报表和导出 | 可以缓存，但必须能从事实或明确来源重算 |
| UI / API | 展示、选择、导入、动作触发、错误提示 | 不在前端本地补造后端事实或权限判断 |

后续任何“从 A 带到 B”的需求，都必须先判断它属于这六类中的哪一类，再决定字段、表、API 和 UI。

`ProcessRuntime` 只是按客户激活流程编排这些既有领域动作的运行包装，不是第七类事实真源。包装动作若真实创建源单据、质检、库存或财务事实，血缘和测试必须按它最终调用的领域 usecase 归类；只有 `EffectNone` 的启动、资料确认、质量门禁或财务放行才归为纯流程编排，不能因为入口名带 `execute_*` 就把它误记成 Fact。

服务端已注册也不自动等于正式 Web 页面可达。当前销售订单提交已统一使用 `start_sales_order_acceptance_process + execute_sales_order_acceptance_submit`；`add_purchase_receipt_item` 以及物料供应 / 成品交付的 9 个 ProcessRuntime 动作虽有公开 JSON-RPC，但正式 Web UI 尚无调用入口，页面血缘必须标为 `partial / backend-only`。发票事实可过账、取消并生成对账事实，但不支持直接 `settle_finance_fact`。

### 2.1 模块单一职责 / Module Purity

加工链路里的 `查货 / 手工 / 车缝 / 包装` 应按模块单一职责拆开理解，不把一个页面做成跨模块处理台。

| 模块 | 可以维护 | 不在本模块维护 |
| --- | --- | --- |
| 加工环节 / `processes` | 工序编号、名称、类别、可委外、可内制、需质检标记；`查货 / 手工 / 车缝 / 包装` 可作为行业默认候选 | 委外合同、生产路线、质检判定、库存流水、应付或结算 |
| 委外订单 / 加工合同 | 加工合同源单、加工厂承诺、工序明细、数量、单价、打印快照和合同生命周期 | 查货结果、合格 / 不合格 / 让步 / 返工判定、发料、回货、库存、应付、发票或付款 |
| 质检 | 质检草稿、提交、合格 / 让步 / 不合格 / 取消和批次状态联动 | 加工合同承诺、委外报价、库存数量流水、应付结算 |
| 库存 | 入库、出库、调整、冲正、余额、批次和可用量 | 工序候选、加工合同审批、质检判定文本或财务结算 |
| 财务 | 应收、应付、发票、收付款和对账事实 | 加工工序、查货判定、库存数量事实 |

因此，“查货”能作为加工环节候选出现在工序档案和加工合同明细里；但查货结果、缺陷、退回、让步和返工应进入质检 / 异常 / 回货链路，不写回加工合同页当事实。

## 3. 表单字段来源分级 / Field Source Levels

新建、编辑、导入、打印、导出和列表字段默认按下面规则分类。

| 字段来源级别 | 适用字段 | UI 策略 | 保存策略 |
| --- | --- | --- | --- |
| 主数据选择 / MasterData Select | 客户、供应商、联系人、产品、SKU、材料、单位、仓库 | 业务对象选择控件为主；选项、已选摘要和校验文案只显示名称、编号、规格、状态等业务信息，可按权限提供“新建主数据”入口 | 内部保存 ID；名称、编号、规格等只作展示或快照，用户可见层不展示裸主键 |
| 来源单据带值 / Source Carry-over | 来源订单号、客户、币种、产品明细、来源行号、采购来源、出货来源 | 通过“选择来源 / 导入来源”带出；核心来源字段只读或受限编辑，用户看到来源单号、来源行号、状态和余额 | 内部保存来源 ID / 来源行 ID；必要展示快照必须可回溯，页面不要求用户手填或识别 `source_id` / `source_line_id` |
| 动作事实输入 / Fact Action Input | 本次入库数量、本次出库数量、实发日期、运费、异常原因、质检判定 | 人工输入，但必须展示来源余额、库存、状态和限制 | 由领域 usecase 校验、过账、幂等和冲正 |
| 派生展示 / Derived Display | 剩余可采购、剩余可出运、库存可用量、已采购、已出库、已结汇 | 只读展示；可作为筛选或导出列 | 不写回源单据；如缓存必须能重算 |
| 客户配置显示 / Customer Config Display | 字段别名、显示顺序、编号前缀、客户菜单开关 | 按客户配置展示 | 不改变 Product Core 事实语义 |
| 导入候选 / Import Candidate | 客户历史 Excel / PDF / 截图中可结构化字段 | dry-run、匹配、人工确认和 unresolved queue | 未批准前不写库；禁止自动生成库存、出货、财务事实 |
| 人工备注 / Manual Note | 备注、内部说明、特殊要求、异常补充说明 | 文本输入 | 不参与事实计算，除非后续单独评审结构化 |

原则：

- 能从主数据选择的字段，不应长期保留纯手填。
- 能从来源单据带出的字段，不应要求用户重复输入。
- 能从事实推导的字段，不应作为普通可编辑字段。
- 允许手填的字段，应是当前动作现场发生的信息，而不是上游已经存在的事实或承诺。

## 4. 主链路数据流向 / Main Business Flow

### 4.1 销售订单到产品 / BOM

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 客户 | MasterData | 销售订单必须选择 active customer，不手填客户名作为真源 |
| 默认付款条件 | MasterData Default | 客户档案的 `default_payment_method / default_payment_term_days` 只作为新建销售订单带出的建议值；订单修改后不自动回写客户默认值 |
| 成交付款条件 | Source Document Snapshot | 销售订单主表的 `payment_method / payment_term_days / price_condition_note` 记录本单谈成条件；付款条件变化必须人工核对明细单价，不自动按账期重算价格 |
| 产品 / SKU | MasterData | 订单行必须选择 active product；可选 SKU 必须 active 且属于该产品，选择后带出产品 / 单位 / 快照并保存 `product_sku_id`；不从颜色、尺寸或客户款号文本自动创建 SKU |
| 单位 | MasterData | 订单行选择单位，不手填自由文本单位 |
| 销售订单 | Source Document | 只表达客户订单承诺，不写出货、库存、应收、发票或付款事实 |
| BOM | MasterData / Engineering Data | 维护产品用料版本，不直接生成采购、生产、库存或成本事实 |

### 4.2 BOM 到采购 / 入库 / 质检 / 库存

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 材料 | MasterData | BOM 和采购行选择材料，不手填材料名称作为真源 |
| 辅料 / 包材 | Material Category | 辅料、包材、面料等都先进入 `materials` 主数据，可通过 `category` 区分；BOM 明细和采购订单行都引用 `material_id`，不各自维护一套材料真源 |
| 采购订单 | Source Document | 表达供应商采购承诺；不写库存、应付、发票或付款事实 |
| 采购入库 | Fact | 可关联采购订单行；入库数量由当前动作输入并受来源数量约束。从采购订单创建收货、按完整明细创建收货，以及向收货草稿追加采购来源行，必须在同一事务为每条收货行建立待检 IQC，并创建或绑定 `HOLD` 批次登记；该批次可由库存批次读模型查询，但草稿阶段不写库存流水或余额。逐行追加不允许脱离采购订单行伪造来源 |
| 来料质检 | Fact | 质检判定影响批次状态，不直接写库存数量流水；采购收货只有全部必要 IQC 合格或让步后才能过账 |
| 库存流水 | Fact | 入库、出库、调整、冲正统一写 `inventory_txns` |
| 库存余额 | Derived / Query Acceleration | 由库存流水维护或重算，不手填余额 |

采购入库、采购退货和采购调整都必须区分草稿退出与已过账冲正。`DRAFT -> CANCELLED` 只终止尚未发生的事实，不写 `inventory_txns`；`POSTED -> CANCELLED` 才按原交易写独立 `REVERSAL`，不能改写或删除原流水。采购入库草稿取消还会在同一事务锁定并结清关联的来料质检和预备批次：`DRAFT / SUBMITTED` IQC 转为取消，已判定或已取消 IQC 保留审计；仅由该草稿准备的批次必须确认余额精确为零后才停用，任一非零余额或来源形状异常都会整笔回滚。取消后来源余额是否恢复，由订单来源与仍然有效的事实重新计算，不由页面手工加回。

### 4.3 生产 / 委外到成品库存

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 加工环节 / 工序主数据 | MasterData / Engineering Data | 维护工序编号、名称、自由文本类别、可委外 / 可内制 / 需质检标记；默认可提供 `查货 / 手工 / 车缝 / 包装` 等毛绒玩具行业候选，但仍允许自由扩展；`查货` 只代表一个可选加工环节，不代表查货结果；不代替生产 / 委外源单、发料、回货、质检判定或库存事实 |
| 生产 / 委外源单据 | Source Document | 表达生产或委外承诺，不能代替领料、回货或入库事实；通用 `create_production_order` 是生产页拥有的新建动作，销售订单 / BOM 只作显式关联来源；生产订单发布时原子生成 PMC 排程任务，并为显式路线行冻结 WIP 路线 / 初始批次，但任务仍属于 Workflow |
| 生产排程 | Workflow | 只由生产订单 `DRAFT -> RELEASED` 生成；任务保存生产订单来源和发布快照，完成后订单才可关闭，不写领料 / 完工 / 库存事实 |
| 材料领用 / 委外发料 | Fact | 生产领料从冻结物料需求行办理；委外发料从已确认加工合同行办理，实际数量由动作确认 |
| 生产 WIP / 分段质检 | Fact processing + Quality Fact | 显式 `PLUSH_SEW_HAND_V1` 路线通过 WIP 动作办理拆批、内制 / 外发、完成 / 回货、移交、返工与包材确认；外发分配把已确认委外合同行持久化为 `production_wip_outsourcing_allocations.outsourcing_order_item_id`，回仓按该锚点重验来源，不从名称或前端快照猜合同；工序完成或委外回货按冻结关口生成 `PRODUCTION_WIP` 质检，当前只有 `PASSED + PASS` 放行，不能用任务状态或前端带值推进 |
| 成品入库 / 委外回货 | Fact | 委外回货从已确认合同行生成；生产完工入库另受包装批次 `ACCEPTED` 数量约束。草稿使用新批号时会先创建或复用可见批次登记，只有正式过账才写库存流水和余额；Workflow / WIP / 质检完成都不单独等于库存入账 |
| 返工 / 异常 | Workflow + Fact 分层 | REWORK 事实过账先按 Fact 规则写库存，再在同一事务生成生产异常任务；任务分派和处理结论走 Workflow，返工冲销、报废或库存调整仍走 Fact |

委外订单的 `source_order_no / product_order_no_snapshot` 只是可读的打印 / 追溯快照，不能证明销售订单外键。公开 schema / API / 页面已移除 `source_sales_order_id`，`20260718125909_migrate.sql` 通过新 Atlas revision 删除目标列，不保留别名、fallback 或兼容读取。

生产和委外事实也使用同一状态边界：来源坐标完整的领料、完工、返工、委外发料或回货草稿可以直接转为 `CANCELLED`，不写库存；已过账事实取消才形成库存反向流水。历史数据如缺少当前合同要求的正式来源坐标，业务记录页同时禁用草稿过账和作废，不发起必然失败的 RPC。草稿取消不会绕过下游依赖：委外回货仍有非取消质检或应付时必须先结清，已过账委外发料仍受 WIP 外发分配约束，已过账生产完工仍受未取消返工事实约束，已过账返工仍受来源异常任务终态约束。子事实进入允许终态后，父生产 / 委外订单的关闭或取消阻断才解除，系统不在父单动作中暗中取消子事实。

### 4.4 出货 / 出库 / 财务

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 出货放行 | Workflow / Business Status | 只由草稿出货单显式“提交出货放行”生成；任务完成写 `shipping_released`，只表示具备继续办理资格，不等于 shipped、扣库存或财务事实 |
| 出货单 | Fact | 通过服务端 `list_shipment_source_candidates` 按关键词 / 销售订单 / 分页查询可出货行，返回订单与来源行、客户、产品 / SKU、单位、订单量、已出货、剩余量、是否可选和不可选原因；来源导入和选定销售订单后的逐行选择都必须使用同一候选合同，页面不再全量拼接订单、明细和出货三张列表，也不能退回裸 `list_sales_order_items` 绕过资格校验 |
| 出货前成品检验 | Quality Fact side flow | 只从尚未提交放行的 `DRAFT` 出货单按产品 / SKU、仓库、批次送检粒度生成；同一送检粒度的非取消质检原子防重。没有发起质检时按可选检验策略进入放行；一旦发起，必须在提交放行前得到合格 / 让步接收，提交后不再改变检验集合 |
| 出库确认 | Fact | 候选余额只统计 `SHIPPED`，普通 `DRAFT` 不占用；只在对应 `shipment_release` 已完成后，确认事务重新锁定并校验销售来源、累计已出货量、预留和库存，再写 shipment fact 和 inventory_txns；重复确认必须幂等，来源查询和 Workflow 任务本身都不执行出库 |
| 应收 / 开票 | Finance Fact | 只从 `SHIPPED` 出货事实显式生成，不从放行任务或销售订单状态直接生成 |
| 收款 / 对账 / 核销 | Finance Fact / Derived | 已过账应收、应付或发票支持单笔显式对账；`PAYMENT`、银行收付和多单核销仍未实现，不靠手填状态表示已结清 |

出货草稿与已出货取消必须分开：`DRAFT -> CANCELLED` 只终止草稿，不写库存；`SHIPPED -> CANCELLED` 才形成库存反向流水，原出货及其数量快照保留审计。两条路径都会阻断 active 成品交付流程、未结质检、active 放行任务或未取消财务事实；草稿已提交放行时，须先将任务完成或退回。服务端候选只从仍有效的 `SHIPPED` 事实聚合已出货量，因此已出货取消成功后来源余额才恢复；普通草稿本来就不占用候选余额，页面清空和 Workflow 状态也不能直接释放或增加来源余额。

正式来源财务事实也允许草稿退出，但它不是库存冲正：出货应收 / 发票、采购或委外应付以及财务事实对账的 `DRAFT -> CANCELLED` 会保留取消时间、操作人和原因，`POSTED -> CANCELLED` 沿用同一审计终止合同；两者都不写库存。无正式来源的手工 / 历史草稿不能借此入口确认或作废，业务记录页会对两个动作同时 fail closed；非取消的对账子事实继续阻断上游财务事实取消；完全相同的 actor + reason 可精确重放，改 actor 或 reason 明确冲突。

### 4.5 来源任务合同 / Source Task Contract

`production_scheduling`、`production_exception`、`shipment_release` 是保留的三个确定性来源任务组，统一使用 `workflow.source-task/v1`。生产者在来源事务中写入确定性 task code、`source_type + source_id`、producer、intent hash、created event 和初始业务状态；同意图重放返回原任务，不同意图或已有同码但来源合同不一致时回滚来源动作。

公开 `workflow.create_task` 还会精确拒绝 19 个能驱动领域状态或产生下游任务的 transition task group；其中包含上述三个来源任务组，但不会禁止受信的领域 / ProcessRuntime producer 生成真实任务。seed、fixture 和人工验收模拟统一使用 `trial_*`。来源取消也不偷写任务终态：生产订单关闭要求排程任务 `done`，已发布订单取消和已过账 REWORK 冲销要求对应任务已 `done / rejected`，实际出货要求放行任务 `done`。如果未来需要独立“撤销任务”，必须正式增加任务状态、事件、migration、看板和测试。

### 4.6 来源读取与结算边界 / Source Read And Settlement

公开动作只要从已有单据 / 事实查询候选或派生新单据、事实、任务、流程，就必须同时通过“目标动作权限 + 精确来源读权限 + 来源与目标模块状态”；不能用下游 create / update 权限偷读上游，也不能在来源模块缺失、`disabled` 或只读时继续写下游。合同由服务端共享 registry 管理，在进入来源 repo / usecase 前 fail closed；条件来源按实际绑定项逐项要求，动态财务来源先用候选读权收窄可探测范围，再按服务端读回的 authoritative FactType 校验精确读权。registry 同时驱动 permission usage，测试逐项删除任一来源读权并断言写用例未被调用；AST handler guard 还会检查每个已登记公开动作的 handler 分支确实调用了来源读 guard，避免“登记完整但接线漏检”。

事实取消和父单结算的统一语义如下；表中的“零库存”是指不新增任何 `inventory_txns`，不是写一笔数量为零的假流水。

| 事实 / 动作 | `DRAFT -> CANCELLED` | 已发生事实的取消 | 活跃依赖与父单解除 |
| --- | --- | --- | --- |
| 采购入库 | 锁定收货、IQC、批次；结清可取消 IQC，校验预备批次余额精确为零后停用；零库存 | `POSTED` 取消写逐行 `REVERSAL` | 任一非取消退货 / 调整或应付阻断已过账收货取消；收货草稿取消后不再阻断采购订单关闭 / 取消 |
| 采购退货 / 入库调整 | 锁定子单与父收货，只终止草稿；零库存 | `POSTED` 取消按原交易写 `REVERSAL` | 草稿子修正取消后解除父收货取消依赖；已有应付继续按既定规则阻断已过账修正取消 |
| 生产事实 | 校验生产订单、来源行 / 物料需求或完工事实坐标后终止草稿；零库存 | `POSTED` 取消写库存反向流水 | 完工的 active REWORK、返工的 active 来源任务、生产订单 WIP / 来源任务继续阻断；子事实终态后父单才可按关闭 / 取消规则推进 |
| 委外事实 | 校验已确认委外订单行后终止草稿；零库存 | `POSTED` 取消写库存反向流水 | 回货的 active 质检 / 应付和发料的 WIP 分配继续阻断；子事实终态后父委外单才解除关闭 / 取消阻断 |
| 出货单 | 终止未出库草稿；零库存 | `SHIPPED` 取消写库存 `REVERSAL` | active 成品交付流程、未结质检、active 放行任务或非取消应收 / 发票继续阻断；草稿本来不占用可出货余额 |
| 财务事实 | 校验正式来源，记录取消时间、操作人、原因；零库存 | `POSTED` 取消保留同一审计终止合同，不写库存 | 任一非取消对账子事实阻断上游取消；精确 actor + reason 重放，变更意图冲突 |

以上状态动作都在事务内锁定同一事实行，并按各领域的既定父子锁顺序追加来源、父单、批次或下游依赖锁。并发 post / cancel 只能串行成两种合法结果：cancel-first 时后续 post 失败且没有库存流水；post-first 时先完成过账，再由 cancel 写完整反向流水或财务取消审计。任何路径都不能留下半笔库存变动、只改状态未冲正，或取消审计缺字段。

| 结算动作 | 同一事务内必须阻断的依赖 |
| --- | --- |
| 销售订单取消 | 未取消出货单、`ACTIVE` 库存预留、未取消生产订单、active `sales_order_acceptance` 流程 |
| 采购订单关闭 / 取消 | 关闭阻断 `DRAFT` 入库；取消阻断所有未取消入库；两者都阻断 active `material_supply` 流程 |
| 生产订单关闭 / 取消 | 关闭阻断非 `POSTED / CANCELLED` 子事实；取消阻断所有非 `CANCELLED` 子事实；既有 WIP / 来源任务结算约束继续生效 |
| 委外订单关闭 / 取消 | 关闭阻断非 `POSTED / CANCELLED` 委外事实；取消阻断所有非 `CANCELLED` 委外事实；WIP 委外分配约束继续生效 |
| 已过账采购入库取消 | 任一未 `CANCELLED` 采购退货 / 入库调整，以及既有应付依赖；不自动取消子事实 |

## 5. 新建 / 编辑表单规则 / Form Rules

业务表单默认分为四块：

1. 主体字段：单据身份、生命周期、客户 / 供应商、日期、负责人。
2. 来源字段：来源单据、来源行、来源数量、剩余数量、来源快照。
3. 明细字段：产品、SKU、材料、单位、数量、价格、仓库、批次等。
4. 动作字段：本次处理数量、判定、原因、费用、附件或备注。

通用规则：

- 来源一旦选择，客户、供应商、币种、来源编号、来源行号等关键字段默认只读。
- 切换来源时，必须清空或重算从旧来源带出的字段，避免残值。
- 清空来源时，必须清空来源 ID、来源行 ID、来源快照和依赖字段。
- 查看或编辑来源绑定记录后再新建，必须从空白草稿重新初始化，并清空路由来源、客户 / 供应商带值、来源候选和行选择缓存；控件的来源锁定态只能由当前草稿真值派生，不能继承上一张单。
- 表单依赖的主数据或来源候选尚未加载完成或加载失败时，新建、编辑、来源选择和保存必须 fail closed，并给出“正在加载 / 请刷新后重试”的可恢复提示。只有当前请求成功进入 ready 后的零条才能解释为真实空结果；不能把延迟、失败或被新请求取消的响应写成“无可选数据”。
- 编辑已过账、已出货、已结算或终态对象时，禁止直接改影响事实追溯的字段；应走取消、冲正、调整或反向事实。
- 表单保存前不能只信任前端带值；后端 usecase 必须按来源 ID 重新校验关键字段和剩余数量。
- `numeric(20,6)` 数量在 API、候选带值、表单状态和提交参数之间保持规范十进制字符串，并以定点整数 / `BigInt` 做比较和减法；服务端公共输入拒绝 JSON number、指数形式、超过 14 位整数或 6 位小数的值，不把精度与溢出留给 PostgreSQL 隐式舍入；前端禁止通过 JavaScript `Number`、浮点减法或 `toFixed` 中转，必须保留 `0.000001` 和字段上限附近的值。
- 列表、详情、打印、导出和搜索应复用同一字段派生或映射层，不在多个页面各写一套条件。

### 5.1 当前明细字段矩阵 / Current Line Item Field Matrix

本矩阵用于当前采购、入库、出货等业务页的“明细”区收口。运行时真源仍以 Ent schema、usecase、JSON-RPC 和当前测试为准；本文只记录稳定字段口径，避免后续把展示快照、来源行号或历史迁移残留误当事实字段。

| 字段组 | 当前页面 / 表 | 字段 | 真源层级 | 保存与展示口径 | 残值 / 缺值防护 |
| --- | --- | --- | --- | --- | --- |
| 行身份 | 采购订单行、采购入库行、出货行 | `id`、`line_no` | Source Document / Fact line identity | `purchase_order_items.line_no` 是采购承诺行号；入库和出货事实行以各自事实行 `id` 为准 | 下游只保存来源行 ID 或事实行 ID，不用手填行号反推事实 |
| 来源追溯 | 采购入库行 | `purchase_order_item_id`、`source_line_no` | Source Carry-over + Display Snapshot | `purchase_order_item_id` 是来源真源；`source_line_no` 只是来源行号展示快照 | 若有关联来源行，后端必须按采购订单行覆盖 `source_line_no`，前端旧值不得继续显示成真源 |
| 来源追溯 | 出货行 | `sales_order_item_id` | Source Carry-over | 指向销售订单行，用于证明该出货行来自哪一条销售承诺 | 选择 / 导入销售订单行时带出；清空来源时必须同步清空来源行 ID 和依赖带值 |
| 来源追溯 | 出货前成品检验 | `source_type=SHIPMENT`、`source_id=shipment.id`、`subject_id=product.id`、`warehouse_id`、`inventory_lot_id` | Fact Source Anchor | 证明质检来自哪张草稿出货单及哪个产品 / SKU、仓库、批次送检组合；SKU 由批次和匹配出货明细共同校验，不另造页面真源 | 后端锁定出货单并复核组合；同组合重复明细聚合，不把 `quality_inspections` 误写成逐出货行事实；切换来源不复用旧单号或旧组合 |
| 主对象 | 采购订单行、采购入库行 | `material_id` | MasterData Select | 材料 ID 是主路径；`material_code_snapshot`、`material_name_snapshot`、`color_snapshot` 只用于采购订单时点展示 | 选择材料时带出快照；后续材料改名不反向改历史订单；入库事实不再重复写材料名称快照 |
| 主对象 | 销售订单行、出货行 | `product_id`、`product_sku_id` | MasterData Select | `product_id` 是必填业务主体；`product_sku_id` 可空，但一旦提供就是精确库存事实维度，必须贯穿销售订单、出货 / 预留、lot / txn / balance、生产 / 委外且来源一致 | 切换 / 清空来源时同步替换 / 清空 SKU；历史 `NULL` 保持 `NULL`，不猜测、不回填、不 fallback 到其他 grain |
| 仓库 / 批次 / 单位 | 入库行、出货行、库存事实 | `warehouse_id`、`lot_id`、`lot_no`、`unit_id` | Fact Action Input + MasterData Select | 仓库和单位为必填主数据；`lot_id` 指向库存批次。采购入库逐行 IQC 创建或绑定 `HOLD` 批次，生产完工 / 委外回货草稿使用新批号时创建或复用批次；批次登记可见不等于库存数量已入账 | 同时有 `lot_id` 和 `lot_no` 时以 `lot_id` 为事实追溯；草稿批次不得派生库存数量，只有正式过账才写 `inventory_txns / inventory_balances` |
| 数量 | 采购订单行、入库行、出货行 | `purchased_quantity`、`quantity`、`ordered_quantity` | Source Commitment / Fact Action Input | 采购订单数量是承诺；入库 / 出货 `quantity` 是本次动作事实数量；跨层传递保持 `numeric(20,6)` 规范十进制字符串 | 下游不能把上游承诺数量直接当已发生事实；来源导入只能作为默认值，后端 usecase 仍校验状态和数量；前端不得用浮点数导致微量归零或大数失真 |
| 金额 | 采购订单行、入库行 | `unit_price`、`amount` | Source Commitment / Fact Action Input / Display | 采购订单金额按数量和单价派生；已过账入库可通过独立来源动作按入库、退货和调整净额生成应付 | 金额为空时可按既定 helper 派生展示；入库保存 / 过账本身不自动生成应付、发票或付款，调用方也不能覆盖后端派生金额 |
| 备注 | 采购订单行、入库行、出货行 | `note` | Manual Note | 只记录人工说明，不参与库存、出货、财务计算 | 不用备注补造来源、材料、产品、批次、数量或状态真源 |

当前明显重复字段的主路径结论：

- `purchase_order_item_id` 与 `source_line_no` 不同级：前者是来源真源，后者只是展示快照。有关联来源行时，后端按来源行覆盖展示行号，避免前端残值。
- `product_sku_id` 不是仅展示追溯；有值时是下游 exact fact grain。订单行带 SKU 时出货 / 预留 / 库存事实必须透传同一 SKU；历史 `NULL` 独立保留，不按名称、颜色或备注推断，也不作为任一 SKU 的兜底池。
- 采购入库行和出货行都出现 `warehouse_id / lot_id / unit_id / quantity / note`，但它们分别属于不同事实表和 usecase；共享的是前端参数映射和展示语义，不共享一张万能明细表。
- 采购订单打印、列表导出和详情回显应继续复用共享映射 helper；打印合同只能使用采购订单承诺字段，不把入库、质检、库存或财务结果混入采购合同模板。

## 6. 导入规则 / Import Rules

导入分三类，不能混用。

| 导入类型 | 允许目标 | 禁止目标 |
| --- | --- | --- |
| 主数据导入 | 客户、供应商、联系人、产品、SKU、材料、单位、仓库等，经 dry-run 和人工确认后进入主数据 | 不自动生成出货、库存、财务事实 |
| 源单据导入 | 销售订单、采购订单等承诺类单据，经字段匹配和人工确认后进入源单据 | 不把订单状态直接写成库存、出货、应收状态 |
| 事实导入 | 仅在已有正式 loader、备份、回滚、审计和客户批准后单独评审 | 不从历史 Excel 静默生成 `inventory_txns`、`shipments`、`finance_facts` |

真实导入前必须满足：

1. source inventory 已完成。
2. field classification 已完成。
3. unresolved queue 已清空或有明确处置。
4. dry-run package 可复现。
5. 客户或项目负责人批准。
6. 有备份、回滚、审计和 post-import reconciliation。

当前没有批准和可执行数据时，只能做 dry-run、模拟数据和 evidence，不执行真实导入。

## 7. trade-erp 经验吸收 / trade-erp Lessons

`trade-erp` 可作为字段联动和来源锁定经验输入，但不能作为 `plush-toy-erp` 的架构真源。

吸收 trade 经验时必须先回答三件事：

1. 这个字段或弹窗模式解决的是通用 ERP 问题，还是 trade 的外贸 / 磁材行业问题。
2. plush 当前是否已有 Ent schema、usecase、API、RBAC、页面和测试真源承接。
3. 借鉴后是否会让前端变成隐藏事实源，或让 Workflow / Source Document 越界写 Fact。

### 7.1 可借鉴矩阵 / Borrowing Matrix

| trade 模块或模式 | 可借鉴内容 | plush 处理口径 | 原因 |
| --- | --- | --- | --- |
| 客户 / 供应商主数据 | 付款方式、付款周期、业务员、联系人明细、启停状态 | 可借鉴到 MasterData；付款文案优先使用“付款方式 / 付款周期(天)”；联系人保存必须走客户 / 供应商聚合保存路径 | 这些是通用 ERP 主数据能力，且能减少销售订单重复输入；但客户类型、国家地区和外贸字段不自动进入核心 |
| 产品资料 | 生命周期、品类、规格、描述、产品单重、附件证据 | 产品单重已作为可选 Product Core 主数据，按产品默认单位以 g 保存；SKU 可选单重只在同时维护显式 SKU 默认单位时成立，未维护 SKU 单重时才回退到产品单重；生命周期和附件证据可继续借鉴；HS Code、中英文外贸描述、磁材类别暂不进 Product Core | 单重是真实通用主数据；SKU 覆盖只改变后续出货确认时的取值优先级，不回写产品主档，也不改写历史出货快照；外贸报关和磁材字段仍不进入核心 |
| 报价 / 外销 | 客户默认带值、价格条款、交期、产品导入、金额派生、采购 / 出运进度只读摘要 | 当前只借字段联动和只读进度摘要；独立报价和外销字段待真实业务确认 | 销售订单是 Source Document，不自动生成采购、出货、财务事实；报价能力未闭环时不能提前铺模块 |
| 采购合同 | 来源导入、来源摘要、剩余数量、本次数量、交期、送货地址、包装 / 唛头、附件 | 高优先级借鉴来源导入和剩余量约束；包装 / 唛头只在采购或打印样本确认后加入 | 采购订单到采购入库是 plush 当前主链路；来源约束能直接防残值、缺值和超量 |
| 入库 / 质检 | 采购来源、待入库数量、本次数量、仓库、货位、质检状态、质检附件 | 借 UI 模式，不合并事实层；采购入库、质检、库存仍分别由对应 usecase 负责 | trade 的合并页面适合作为交互参考，但 plush 已明确 Purchase / Quality / Inventory Fact 边界 |
| 库存 | 可用数量、锁定数量、仓库 / 货位提示 | 借只读说明和字段帮助；库存数量必须来自库存流水 / 余额 read model | 可用 / 已预留解释有助于页面理解，但不能前端本地计算或手填余额 |
| 出运 / 出库 | 来源导入、来源锁定、剩余可出货、本次出货、收货地址、运输方式、件数、重量、体积、运费、附件 | 来源导入和剩余量继续沿用当前主路径；`DRAFT -> SHIPPED` 由后端事务按“SKU 单重 + SKU 显式默认单位优先，否则产品单重 + 产品默认单位”解析并冻结行单重快照。全部有效行可解析时按 decimal 计算最终总净重；任一行不可解析时不做部分合计，表头仍允许可空的人工实际总净重，第一版不因净重缺失阻断出货；报关、提单、到付账号、欠款提醒仍不进核心 | 总净重现已归 Shipment Fact 后端主路径，不由前端临时计算；历史出货不回填，取消出货保留原净重快照和总净重；毛重、单位换算、装箱 / 物流 / 打印仍需后续专项 |
| 结汇 / 水单 | 发票号、发货日、付款周期、到期日、已收 / 未收、多单认领、银行扣费 | 暂不借运行时字段；只记录未来 Finance Fact 评审候选 | plush 当前 finance facts 仍是收窄事实链，不是完整 AR/AP、结汇、总账或收款核销模块 |
| 来源选择 / record import | 分页、搜索、已选摘要、清空已选、不可导入原因、导入后回父弹窗 | 高优先级借鉴到来源选择 / 导入控件和业务弹窗标准 | 这是页面交互模式，不改变业务真源；能显式暴露来源、资格和残值风险，不能暴露裸 ID 或要求用户理解技术选择器 |
| 明细 items | 统一横向滚动、添加条目、数量 / 金额汇总、单位后缀、只读派生金额 | 高优先级继续收口到共享组件和 L1 断言 | 可降低多业务弹窗差异，且已符合 plush 业务表单标准 |
| 通用删除 / 回收站 | trade 正式模块也不走通用删除策略 | 继续不借通用删除；草稿明细行删除只表示表单内移除 | 与 plush 生命周期治理一致，避免前端假删除和回收站误导 |

### 7.2 借鉴落地优先级 / Adoption Priority

| 顺序 | 借鉴目标 | 先落地到哪里 | 验收重点 |
| --- | --- | --- | --- |
| 1 | 来源导入弹窗增强 | 销售订单转出货、采购订单转入库、入库转质检、委外来源选择 | 已选摘要、不可导入原因、清空已选、切换来源清值都可见 |
| 2 | 剩余数量约束 | 采购入库、出货、出库和后续领料 / 回货 | 展示来源数量、已处理数量、本次数量；后端 usecase 阻断超量和重复提交 |
| 3 | 字段联动 QA | 客户付款条件、产品 / SKU / 单位、材料 / 单位、来源行快照 | 新值覆盖旧值；切换来源清空旧值；缺少来源时不伪造 |
| 4 | 明细行共享体验 | 销售、采购、委外、BOM、出货、质检等带 items 弹窗 | 横向滚动、单位后缀、汇总和错误提示不挤压相邻区域 |
| 5 | 出货物流证据字段 | Shipment Fact V1 | 行单重快照和总净重已进后端事务主路径；收货地址、运输方式、件数、毛重、体积、运费和物流附件仍需后端字段或正式评审 |
| 6 | 财务收款 / 对账字段 | Finance Fact 来源动作与后续专项 | 已过账应收、应付或发票的单笔对账已接；收付款账户、银行流水、`PAYMENT` 和多单核销只在对应 usecase 成立后实现 |

不能照搬的部分：

- 不把前端模块配置当正式事实主路径。
- 不让导出 helper 反向决定 schema 或 usecase。
- 不把 `trade-erp` 的客户字段、编码、磁材行业类别、外贸报关字段、结汇 / 水单流程或旧流程写进毛绒玩具 Product Core。
- 不为“页面看起来完整”提前新增 schema、菜单、权限码、Workflow 规则或 Finance Fact。

`plush-toy-erp` 的正式主路径应落在 Ent schema、Atlas migration、repo/usecase、RBAC、API 和测试；前端负责选择、导入、展示和动作触发。

暂缓引入的候选字段：

- 质检数量链路：`本次送检数量 / 已检数量 / 待检数量 / 不合格数量回写` 需要 QualityUsecase 和 schema 评审，不能只做前端字段。
- 出货草稿占用：当前强阻断发生在 `SHIPPED` 确认事务，草稿不预占销售订单剩余量；若后续要让草稿占用，必须通过明确预留事实实现，不能把普通草稿计数写成库存预留。
- 出货重量扩展：毛重、单位换算、装箱数 / 箱号、体积、物流追踪和出货打印仍需对应 Shipment / Packing / Print 合同；不用已落地的净重快照倒推这些未实现能力。
- 委外回货、领料、出库的剩余数量字段需要对应 Source Document 和 Fact usecase 成立后再吸收。
- 报关、提单、结汇、水单、磁材编码、外贸到付账号等字段仍只作为 trade 背景，不进入 plush 当前 Product Core。

## 8. 残值与缺值防护 / Stale And Missing Value Guards

涉及来源带值、导入或派生字段时，必须同时防残值和缺值。

| 场景 | 必须处理 |
| --- | --- |
| 新建默认带值 | 来源选择后带出必要展示字段，并保存来源 ID / 行 ID |
| 编辑覆盖 | 切换来源后旧来源字段必须清空或重算 |
| 清空来源 | 来源 ID、来源行 ID、来源快照和依赖字段必须同步清空 |
| 删除或停用主数据 | 已被引用的主数据不能直接破坏历史追溯；按启停用、引用保护或快照显示处理 |
| 快照缺失 | 能从来源 ID 安全回补时按规则回补；无法证明来源时保持空值或待确认，不伪造 |
| 真源缺失 | 不用备注、历史文案或客户样本猜测事实 |
| 列表 / 详情 / 打印 / 导出 | 复用共享映射或派生 helper，确保同字段口径一致 |

## 9. 后续实施顺序 / Implementation Order

建议按可验证闭环推进，而不是一次性改完整 ERP。

| 顺序 | 目标 | 验收重点 |
| --- | --- | --- |
| 1 | 建立字段来源分类和表单规则 | 每个字段标注选择、来源、派生、动作输入或导入候选 |
| 2 | 收口销售订单行产品 / SKU / 单位选择 | 不手填主数据名；后端重新校验 active master data |
| 3 | 收口采购订单到采购入库来源带值 | 入库行可追溯采购订单行；剩余数量可校验 |
| 4 | 收口出货单到销售订单 / 出货来源带值 | 出货行可追溯来源行；出库不超过可出货 / 可用库存 |
| 5 | 收口财务事实来源 | 应收、开票、收付款从已出货或已采购 / 委外事实进入，不从 workflow 直接生成 |
| 6 | 统一列表 / 导出 / 打印派生层 | 同一字段在列表、详情、打印、导出中口径一致 |

每一轮都必须说明允许修改路径、禁止修改路径、验收命令和剩余风险；触达 schema、migration、usecase、API、RBAC 或 UI 时按 `docs/product/模块实施治理.md` 拆任务。

## 10. 测试要求 / Test Requirements

字段来源和数据流向改动至少覆盖：

- 新值覆盖旧值。
- 切换来源后旧值清空或替换。
- 清空来源后依赖字段清空。
- 真源缺失时不伪造值。
- 主数据 / 来源候选延迟、失败、刷新重试和旧请求晚到时，页面保持 fail closed，不把暂时零条当作业务空结果。
- 快照缺失时按既定口径回补或保持待确认。
- 列表、详情、打印、导出和搜索口径一致。
- 后端 usecase 重新校验来源 ID、来源行、数量余额、状态和权限。
- Workflow 完成不写库存、出货或财务事实。
- 已过账或已出货对象通过取消、冲正或调整修正，不直接改事实字段。

跨单据 / 跨事实链路必须分层验收，不能用一个层级替代另一个层级：

| 层级 | 必须证明 | 不能证明 |
| --- | --- | --- |
| Registry / 静态合同 | 每个公开来源动作唯一登记、permission usage 同步、handler 分支调用来源读 guard、页面 typed lineage 无漏项 | repository 事务和真实状态变更 |
| Service / RBAC / module 合同 | 缺任一目标权限、精确来源读权限或必要模块状态时，在读取来源或调用写 usecase 前 fail closed | 行锁、库存原子性和 PostgreSQL 隔离行为 |
| Data / usecase 集成 | 正常、边界、非法来源、幂等、active downstream、父单解除；草稿零库存、已过账完整冲正、财务取消审计 | PostgreSQL 真正的并发锁顺序 |
| PostgreSQL 事务 / 并发 | post-vs-cancel、child-create-vs-parent-cancel、依赖创建 / 结算竞争只有合法赢家，且没有半笔库存或悬空 active child | 页面按钮、岗位文案和用户操作可达性 |
| Web 组件 / mock 浏览器 | 真实 RPC 方法、参数、状态回显、DRAFT 与 POSTED 的不同确认文案、错误恢复和分页边界 | 后端已持久化，也不能证明目标环境身份 |
| 真实后端浏览器 + 数据库读回 | 当前 commit / image / migration / customer config 下，岗位账号完成来源选择、生成、过账、取消并读回事实与流水 | 客户业务验收或生产发布本身 |
| 目标 smoke / UAT | 目标环境 health / ready、关键业务 readback、角色验收和客户签收 | 不能由本地 `full / strict` 自动推导 |

数量 / 分页边界至少包含 0、1、服务端默认页、50、200、超过一页，以及 `numeric(20,6)` 的最小精度和字段上限；状态矩阵至少包含 `DRAFT / POSTED / CANCELLED`，有额外业务状态的链路还要覆盖 `SHIPPED / SETTLED` 等不可直接取消状态。PostgreSQL 用例如被环境变量跳过，必须明确报告 `skipped`，不能把普通 `go test ./...` 绿色写成并发已验收。

如果本轮只覆盖部分路径，最终回复和 `progress.md` 必须明确未覆盖路径、原因和剩余风险。
