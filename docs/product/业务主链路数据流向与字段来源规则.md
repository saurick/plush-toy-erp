# 业务主链路数据流向与字段来源规则 / Business Data Flow And Field Source Policy

## 0. 文档目的 / Purpose

本文回答两个问题：

1. `plush-toy-erp` 是否要做从订单到产品、材料、仓库、出库、出运和财务的数据流向。
2. 新建 / 编辑表单、列表、导入、导出和打印中的字段，哪些应该选择、导入、带值、派生或只读，哪些才允许人工输入。

本文是产品治理和实施边界文档，不替代 `docs/当前真源与交接顺序.md`、Ent schema、Atlas migration、后端 usecase、API、RBAC、前端实现或测试结果。

## 1. 结论 / Decision

业务主链路数据流向必须做，而且应作为产品内核主线推进。

但数据流向不是新增一套万能流程引擎，也不是恢复旧 `business_records`、创建泛化 `change_records`，更不是让 Workflow 直接写库存、出货或财务事实。正确主路径是：

```text
MasterData 主数据
-> Source Document 源单据
-> Domain Usecase 业务动作
-> Fact 事实表
-> Derived Projection / Report 派生展示
```

对应 ERP 口径是：

```text
客户 / 供应商 / 产品 / SKU / 材料 / 工序 / 单位 / 仓库
-> 销售订单 / BOM / 采购订单 / 生产或委外单 / 出货单
-> 入库 / 质检 / 领料 / 成品入库 / 出库 / 发货 / 对账
-> inventory_txns / shipments / finance_facts 等事实
-> 订单进度、库存可用量、未出货、应收应付、导出报表
```

## 2. 基础边界 / Core Boundaries

| 层级 | 写入职责 | 不能承担 |
| --- | --- | --- |
| MasterData | 客户、供应商、联系人、产品、SKU、材料、工序、单位、仓库等稳定资料 | 不承接订单临时字段、客户样本残值或事实状态 |
| Source Document | 销售订单、采购订单、生产单、委外单等业务承诺 | 不直接写库存、出货、应收、应付、发票或收付款事实 |
| Workflow | 任务分派、审批、阻塞、退回、协同状态推进 | 不写库存流水、出货事实、财务事实 |
| Fact | 采购、库存、质检、生产、委外、出货、财务等真实业务动作 | 不从菜单、旧快照、workflow payload 或表单显示伪造事实 |
| Derived | 进度、余额、剩余数量、应收应付状态、报表和导出 | 可以缓存，但必须能从事实或明确来源重算 |
| UI / API | 展示、选择、导入、动作触发、错误提示 | 不在前端本地补造后端事实或权限判断 |

后续任何“从 A 带到 B”的需求，都必须先判断它属于这六类中的哪一类，再决定字段、表、API 和 UI。

### 2.1 模块单一职责 / Module Purity

加工链路里的 `查货 / 手工 / 车缝 / 包装` 应按模块单一职责拆开理解，不把一个页面做成跨模块处理台。

| 模块 | 可以维护 | 不在本模块维护 |
| --- | --- | --- |
| 加工环节 / `processes` | 工序编号、名称、类别、可委外、可内制、需质检标记；`查货 / 手工 / 车缝 / 包装` 可作为行业默认候选 | 委外合同、生产路线、质检判定、库存流水、应付或结算 |
| 委外订单 / 加工合同 | 加工合同源单、加工厂承诺、工序明细、数量、单价、打印快照和合同生命周期 | 查货结果、合格 / 不合格 / 让步 / 返工判定、发料、回货、库存、应付、发票或付款 |
| 质检 | 质检草稿、提交、合格 / 让步 / 不合格 / 取消和批次状态联动 | 加工合同承诺、委外报价、库存数量流水、应付结算 |
| 库存 | 入库、出库、调整、冲正、余额、批次和可用量 | 工序候选、加工合同审批、质检判定文本或财务结算 |
| 财务 | 应收、应付、发票、收付款和对账事实 | 加工工序、查货判定、库存数量事实 |

因此，“查货”能作为加工环节候选出现在工序档案和加工合同明细里；但查货结果、缺陷、退回、让步和返工应进入质检 / 异常 / 回货链路，不写回加工合同页当事实。

## 3. 表单字段来源分级 / Field Source Levels

新建、编辑、导入、打印、导出和列表字段默认按下面规则分类。

| 字段来源级别 | 适用字段 | UI 策略 | 保存策略 |
| --- | --- | --- | --- |
| 主数据选择 / MasterData Select | 客户、供应商、联系人、产品、SKU、材料、单位、仓库 | 业务对象选择控件为主；选项、已选摘要和校验文案只显示名称、编号、规格、状态等业务信息，可按权限提供“新建主数据”入口 | 内部保存 ID；名称、编号、规格等只作展示或快照，用户可见层不展示裸主键 |
| 来源单据带值 / Source Carry-over | 来源订单号、客户、币种、产品明细、来源行号、采购来源、出货来源 | 通过“选择来源 / 导入来源”带出；核心来源字段只读或受限编辑，用户看到来源单号、来源行号、状态和余额 | 内部保存来源 ID / 来源行 ID；必要展示快照必须可回溯，页面不要求用户手填或识别 `source_id` / `source_line_id` |
| 动作事实输入 / Fact Action Input | 本次入库数量、本次出库数量、实发日期、运费、异常原因、质检判定 | 人工输入，但必须展示来源余额、库存、状态和限制 | 由领域 usecase 校验、过账、幂等和冲正 |
| 派生展示 / Derived Display | 剩余可采购、剩余可出运、库存可用量、已采购、已出库、已结汇 | 只读展示；可作为筛选或导出列 | 不写回源单据；如缓存必须能重算 |
| 客户配置显示 / Customer Config Display | 字段别名、显示顺序、编号前缀、客户菜单开关 | 按客户配置展示 | 不改变 Product Core 事实语义 |
| 导入候选 / Import Candidate | 客户历史 Excel / PDF / 截图中可结构化字段 | dry-run、匹配、人工确认和 unresolved queue | 未批准前不写库；禁止自动生成库存、出货、财务事实 |
| 人工备注 / Manual Note | 备注、内部说明、特殊要求、异常补充说明 | 文本输入 | 不参与事实计算，除非后续单独评审结构化 |

原则：

- 能从主数据选择的字段，不应长期保留纯手填。
- 能从来源单据带出的字段，不应要求用户重复输入。
- 能从事实推导的字段，不应作为普通可编辑字段。
- 允许手填的字段，应是当前动作现场发生的信息，而不是上游已经存在的事实或承诺。

## 4. 主链路数据流向 / Main Business Flow

### 4.1 销售订单到产品 / BOM

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 客户 | MasterData | 销售订单必须选择 active customer，不手填客户名作为真源 |
| 默认付款条件 | MasterData Default | 客户档案的 `default_payment_method / default_payment_term_days` 只作为新建销售订单带出的建议值；订单修改后不自动回写客户默认值 |
| 成交付款条件 | Source Document Snapshot | 销售订单主表的 `payment_method / payment_term_days / price_condition_note` 记录本单谈成条件；付款条件变化必须人工核对明细单价，不自动按账期重算价格 |
| 产品 / SKU | MasterData | 订单行必须选择 active product；可选 SKU 必须 active 且属于该产品，选择后带出产品 / 单位 / 快照并保存 `product_sku_id`；不从颜色、尺寸或客户款号文本自动创建 SKU |
| 单位 | MasterData | 订单行选择单位，不手填自由文本单位 |
| 销售订单 | Source Document | 只表达客户订单承诺，不写出货、库存、应收、发票或付款事实 |
| BOM | MasterData / Engineering Data | 维护产品用料版本，不直接生成采购、生产、库存或成本事实 |

### 4.2 BOM 到采购 / 入库 / 质检 / 库存

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 材料 | MasterData | BOM 和采购行选择材料，不手填材料名称作为真源 |
| 辅料 / 包材 | Material Category | 辅料、包材、面料等都先进入 `materials` 主数据，可通过 `category` 区分；BOM 明细和采购订单行都引用 `material_id`，不各自维护一套材料真源 |
| 采购订单 | Source Document | 表达供应商采购承诺；不写库存、应付、发票或付款事实 |
| 采购入库 | Fact | 可关联采购订单行；入库数量由当前动作输入并受来源数量约束 |
| 来料质检 | Fact | 质检判定影响批次状态，不直接写库存数量流水 |
| 库存流水 | Fact | 入库、出库、调整、冲正统一写 `inventory_txns` |
| 库存余额 | Derived / Query Acceleration | 由库存流水维护或重算，不手填余额 |

### 4.3 生产 / 委外到成品库存

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 加工环节 / 工序主数据 | MasterData / Engineering Data | 维护工序编号、名称、自由文本类别、可委外 / 可内制 / 需质检标记；默认可提供 `查货 / 手工 / 车缝 / 包装` 等毛绒玩具行业候选，但仍允许自由扩展；`查货` 只代表一个可选加工环节，不代表查货结果；不代替生产 / 委外源单、发料、回货、质检判定或库存事实 |
| 生产 / 委外源单据 | Source Document | 表达生产或委外承诺，不能代替领料、回货或入库事实 |
| 材料领用 / 委外发料 | Fact | 根据产品 / BOM / 来源单据生成候选，实际数量由动作确认 |
| 成品入库 / 委外回货 | Fact | 写库存事实；Workflow 完成只代表协同任务完成 |
| 返工 / 异常 | Workflow + Fact 分层 | 任务分派走 Workflow；真实库存或财务影响走 Fact |

### 4.4 出货 / 出库 / 财务

| 环节 | 当前定位 | 字段来源要求 |
| --- | --- | --- |
| 出货放行 | Workflow / Business Status | 只表示可发货 / 待出库，不等于 shipped |
| 出货单 | Fact | 选择销售订单或可出货来源，带出客户、产品、来源行和剩余可出货数量 |
| 出库确认 | Fact | 写 shipment fact 和 inventory_txns；重复确认必须幂等 |
| 应收 / 开票 | Finance Fact | 从已出货事实进入财务评审，不从放行任务直接生成 |
| 收款 / 对账 / 核销 | Finance Fact / Derived | 关联发票、应收或出货，不靠手填状态表示已结清 |

## 5. 新建 / 编辑表单规则 / Form Rules

业务表单默认分为四块：

1. 主体字段：单据身份、生命周期、客户 / 供应商、日期、负责人。
2. 来源字段：来源单据、来源行、来源数量、剩余数量、来源快照。
3. 明细字段：产品、SKU、材料、单位、数量、价格、仓库、批次等。
4. 动作字段：本次处理数量、判定、原因、费用、附件或备注。

通用规则：

- 来源一旦选择，客户、供应商、币种、来源编号、来源行号等关键字段默认只读。
- 切换来源时，必须清空或重算从旧来源带出的字段，避免残值。
- 清空来源时，必须清空来源 ID、来源行 ID、来源快照和依赖字段。
- 编辑已过账、已出货、已结算或终态对象时，禁止直接改影响事实追溯的字段；应走取消、冲正、调整或反向事实。
- 表单保存前不能只信任前端带值；后端 usecase 必须按来源 ID 重新校验关键字段和剩余数量。
- 列表、详情、打印、导出和搜索应复用同一字段派生或映射层，不在多个页面各写一套条件。

### 5.1 当前明细字段矩阵 / Current Line Item Field Matrix

本矩阵用于当前采购、入库、出货等业务页的“明细”区收口。运行时真源仍以 Ent schema、usecase、JSON-RPC 和当前测试为准；本文只记录稳定字段口径，避免后续把展示快照、来源行号或历史迁移残留误当事实字段。

| 字段组 | 当前页面 / 表 | 字段 | 真源层级 | 保存与展示口径 | 残值 / 缺值防护 |
| --- | --- | --- | --- | --- | --- |
| 行身份 | 采购订单行、采购入库行、出货行 | `id`、`line_no` | Source Document / Fact line identity | `purchase_order_items.line_no` 是采购承诺行号；入库和出货事实行以各自事实行 `id` 为准 | 下游只保存来源行 ID 或事实行 ID，不用手填行号反推事实 |
| 来源追溯 | 采购入库行 | `purchase_order_item_id`、`source_line_no` | Source Carry-over + Display Snapshot | `purchase_order_item_id` 是来源真源；`source_line_no` 只是来源行号展示快照 | 若有关联来源行，后端必须按采购订单行覆盖 `source_line_no`，前端旧值不得继续显示成真源 |
| 来源追溯 | 出货行 | `sales_order_item_id` | Source Carry-over | 指向销售订单行，用于证明该出货行来自哪一条销售承诺 | 选择 / 导入销售订单行时带出；清空来源时必须同步清空来源行 ID 和依赖带值 |
| 主对象 | 采购订单行、采购入库行 | `material_id` | MasterData Select | 材料 ID 是主路径；`material_code_snapshot`、`material_name_snapshot`、`color_snapshot` 只用于采购订单时点展示 | 选择材料时带出快照；后续材料改名不反向改历史订单；入库事实不再重复写材料名称快照 |
| 主对象 | 销售订单行、出货行 | `product_id`、`product_sku_id` | MasterData Select | `product_id` 是必填业务主体；`product_sku_id` 可空，但一旦提供就是精确库存事实维度，必须贯穿销售订单、出货 / 预留、lot / txn / balance、生产 / 委外且来源一致 | 切换 / 清空来源时同步替换 / 清空 SKU；历史 `NULL` 保持 `NULL`，不猜测、不回填、不 fallback 到其他 grain |
| 仓库 / 批次 / 单位 | 入库行、出货行、库存事实 | `warehouse_id`、`lot_id`、`lot_no`、`unit_id` | Fact Action Input + MasterData Select | 仓库和单位为必填主数据；`lot_id` 指向库存批次；入库允许先填 `lot_no`，过账时创建或绑定批次 | 同时有 `lot_id` 和 `lot_no` 时以 `lot_id` 为事实追溯；只有 `lot_no` 时仅由入库过账主路径绑定批次 |
| 数量 | 采购订单行、入库行、出货行 | `purchased_quantity`、`quantity`、`ordered_quantity` | Source Commitment / Fact Action Input | 采购订单数量是承诺；入库 / 出货 `quantity` 是本次动作事实数量 | 下游不能把上游承诺数量直接当已发生事实；来源导入只能作为默认值，后端 usecase 仍校验状态和数量 |
| 金额 | 采购订单行、入库行 | `unit_price`、`amount` | Source Commitment / Fact Action Input / Display | 采购订单金额按数量和单价派生；入库金额是入库行金额展示或后续财务候选，不生成应付事实 | 金额为空时可按既定 helper 派生展示；不能从入库金额自动生成应付、发票或付款 |
| 备注 | 采购订单行、入库行、出货行 | `note` | Manual Note | 只记录人工说明，不参与库存、出货、财务计算 | 不用备注补造来源、材料、产品、批次、数量或状态真源 |

当前明显重复字段的主路径结论：

- `purchase_order_item_id` 与 `source_line_no` 不同级：前者是来源真源，后者只是展示快照。有关联来源行时，后端按来源行覆盖展示行号，避免前端残值。
- `product_sku_id` 不是仅展示追溯；有值时是下游 exact fact grain。订单行带 SKU 时出货 / 预留 / 库存事实必须透传同一 SKU；历史 `NULL` 独立保留，不按名称、颜色或备注推断，也不作为任一 SKU 的兜底池。
- 采购入库行和出货行都出现 `warehouse_id / lot_id / unit_id / quantity / note`，但它们分别属于不同事实表和 usecase；共享的是前端参数映射和展示语义，不共享一张万能明细表。
- 采购订单打印、列表导出和详情回显应继续复用共享映射 helper；打印合同只能使用采购订单承诺字段，不把入库、质检、库存或财务结果混入采购合同模板。

## 6. 导入规则 / Import Rules

导入分三类，不能混用。

| 导入类型 | 允许目标 | 禁止目标 |
| --- | --- | --- |
| 主数据导入 | 客户、供应商、联系人、产品、SKU、材料、单位、仓库等，经 dry-run 和人工确认后进入主数据 | 不自动生成出货、库存、财务事实 |
| 源单据导入 | 销售订单、采购订单等承诺类单据，经字段匹配和人工确认后进入源单据 | 不把订单状态直接写成库存、出货、应收状态 |
| 事实导入 | 仅在已有正式 loader、备份、回滚、审计和客户批准后单独评审 | 不从历史 Excel 静默生成 `inventory_txns`、`shipments`、`finance_facts` |

真实导入前必须满足：

1. source inventory 已完成。
2. field classification 已完成。
3. unresolved queue 已清空或有明确处置。
4. dry-run package 可复现。
5. 客户或项目负责人批准。
6. 有备份、回滚、审计和 post-import reconciliation。

当前没有批准和可执行数据时，只能做 dry-run、模拟数据和 evidence，不执行真实导入。

## 7. trade-erp 经验吸收 / trade-erp Lessons

`trade-erp` 可作为字段联动和来源锁定经验输入，但不能作为 `plush-toy-erp` 的架构真源。

吸收 trade 经验时必须先回答三件事：

1. 这个字段或弹窗模式解决的是通用 ERP 问题，还是 trade 的外贸 / 磁材行业问题。
2. plush 当前是否已有 Ent schema、usecase、API、RBAC、页面和测试真源承接。
3. 借鉴后是否会让前端变成隐藏事实源，或让 Workflow / Source Document 越界写 Fact。

### 7.1 可借鉴矩阵 / Borrowing Matrix

| trade 模块或模式 | 可借鉴内容 | plush 处理口径 | 原因 |
| --- | --- | --- | --- |
| 客户 / 供应商主数据 | 付款方式、付款周期、业务员、联系人明细、启停状态 | 可借鉴到 MasterData；付款文案优先使用“付款方式 / 付款周期(天)”；联系人保存必须走客户 / 供应商聚合保存路径 | 这些是通用 ERP 主数据能力，且能减少销售订单重复输入；但客户类型、国家地区和外贸字段不自动进入核心 |
| 产品资料 | 生命周期、品类、规格、描述、附件证据 | 可借鉴生命周期和附件证据；HS Code、中英文外贸描述、磁材类别暂不进 Product Core | plush 当前是毛绒玩具产品 / SKU / 材料主路径，不能把外贸报关或磁材字段写成通用事实 |
| 报价 / 外销 | 客户默认带值、价格条款、交期、产品导入、金额派生、采购 / 出运进度只读摘要 | 当前只借字段联动和只读进度摘要；独立报价和外销字段待真实业务确认 | 销售订单是 Source Document，不自动生成采购、出货、财务事实；报价能力未闭环时不能提前铺模块 |
| 采购合同 | 来源导入、来源摘要、剩余数量、本次数量、交期、送货地址、包装 / 唛头、附件 | 高优先级借鉴来源导入和剩余量约束；包装 / 唛头只在采购或打印样本确认后加入 | 采购订单到采购入库是 plush 当前主链路；来源约束能直接防残值、缺值和超量 |
| 入库 / 质检 | 采购来源、待入库数量、本次数量、仓库、货位、质检状态、质检附件 | 借 UI 模式，不合并事实层；采购入库、质检、库存仍分别由对应 usecase 负责 | trade 的合并页面适合作为交互参考，但 plush 已明确 Purchase / Quality / Inventory Fact 边界 |
| 库存 | 可用数量、锁定数量、仓库 / 货位提示 | 借只读说明和字段帮助；库存数量必须来自库存流水 / 余额 read model | 可用 / 已预留解释有助于页面理解，但不能前端本地计算或手填余额 |
| 出运 / 出库 | 来源导入、来源锁定、剩余可出货、本次出货、收货地址、运输方式、件数、重量、体积、运费、附件 | 优先借来源导入、剩余量和物流证据字段；报关、提单、到付账号、欠款提醒先不进核心 | plush 出货单已是 Shipment Fact V1，适合吸收物流证据；外贸专属字段需要真实出口样本 |
| 结汇 / 水单 | 发票号、发货日、付款周期、到期日、已收 / 未收、多单认领、银行扣费 | 暂不借运行时字段；只记录未来 Finance Fact 评审候选 | plush 当前 finance facts 仍是收窄事实链，不是完整 AR/AP、结汇、总账或收款核销模块 |
| 来源选择 / record import | 分页、搜索、已选摘要、清空已选、不可导入原因、导入后回父弹窗 | 高优先级借鉴到来源选择 / 导入控件和业务弹窗标准 | 这是页面交互模式，不改变业务真源；能显式暴露来源、资格和残值风险，不能暴露裸 ID 或要求用户理解技术选择器 |
| 明细 items | 统一横向滚动、添加条目、数量 / 金额汇总、单位后缀、只读派生金额 | 高优先级继续收口到共享组件和 L1 断言 | 可降低多业务弹窗差异，且已符合 plush 业务表单标准 |
| 通用删除 / 回收站 | trade 正式模块也不走通用删除策略 | 继续不借通用删除；草稿明细行删除只表示表单内移除 | 与 plush 生命周期治理一致，避免前端假删除和回收站误导 |

### 7.2 借鉴落地优先级 / Adoption Priority

| 顺序 | 借鉴目标 | 先落地到哪里 | 验收重点 |
| --- | --- | --- | --- |
| 1 | 来源导入弹窗增强 | 销售订单转出货、采购订单转入库、入库转质检、委外来源选择 | 已选摘要、不可导入原因、清空已选、切换来源清值都可见 |
| 2 | 剩余数量约束 | 采购入库、出货、出库和后续领料 / 回货 | 展示来源数量、已处理数量、本次数量；后端 usecase 阻断超量和重复提交 |
| 3 | 字段联动 QA | 客户付款条件、产品 / SKU / 单位、材料 / 单位、来源行快照 | 新值覆盖旧值；切换来源清空旧值；缺少来源时不伪造 |
| 4 | 明细行共享体验 | 销售、采购、委外、BOM、出货、质检等带 items 弹窗 | 横向滚动、单位后缀、汇总和错误提示不挤压相邻区域 |
| 5 | 出货物流证据字段 | Shipment Fact V1 | 收货地址、运输方式、件数、重量、体积、运费和附件有后端字段或正式评审 |
| 6 | 财务收款 / 对账字段 | Finance Fact 后续专项 | 只在应收、发票、付款和核销 usecase 成立后实现 |

不能照搬的部分：

- 不把前端模块配置当正式事实主路径。
- 不让导出 helper 反向决定 schema 或 usecase。
- 不把 `trade-erp` 的客户字段、编码、磁材行业类别、外贸报关字段、结汇 / 水单流程或旧流程写进毛绒玩具 Product Core。
- 不为“页面看起来完整”提前新增 schema、菜单、权限码、Workflow 规则或 Finance Fact。

`plush-toy-erp` 的正式主路径应落在 Ent schema、Atlas migration、repo/usecase、RBAC、API 和测试；前端负责选择、导入、展示和动作触发。

### 7.3 当前已吸收字段 / Adopted Runtime Fields

下表只记录本轮已经进入运行态页面的字段和交互，不把尚无后端合同的候选字段写成已完成能力。

| 页面 | 表单 / 弹窗 | 已加入字段或交互 | 加入原因 | 仍需后端合同的边界 |
| --- | --- | --- | --- | --- |
| 采购订单 | `生成采购入库草稿` 弹窗 | 来源采购单摘要、来源行号、材料、采购数量、已生成入库、本次入库、剩余可生成、不可生成原因 | 让用户在生成入库草稿前看清来源、余额和不可生成原因，减少重复生成和来源残值 | 当前弹窗只做可见预检；最终超量、状态、重复提交仍必须由采购入库 usecase 校验 |
| 出货单 | `从销售订单行导入出货明细` 来源选择 / 导入控件 + 出货新建 / 编辑弹窗 | 来源销售订单摘要、来源行号、客户、产品 / SKU、订单数量、已生成出货、剩余可出货、不可导入原因、已选来源摘要、清空已选、导入后默认本次数量 | 出货必须追溯到销售承诺行；展示已生成和剩余数量用于可见预检 | 确认 `SHIPPED` 时后端在事务内校验 active 销售订单、客户、来源行、产品 / SKU、单位和累计已出货量，并发超量会阻断；草稿不占用剩余量，因此前端的“已生成”仍不是后端预留事实 |
| 质检记录 | `生成质检草稿` 表单 | 来源采购入库摘要、来源入库行、供应商、状态、到货数量、材料、仓库、批次提示 | 质检应从采购入库事实行进入，用户需要确认本次质检对应哪一行、哪批货 | 当前质检 schema/API 没有独立“本次送检数量 / 已检数量 / 待检数量”字段，本轮不在前端伪造 |
| 采购入库 | 入库明细详情 | 采购订单行从裸 `purchase_order_item_id` 改为“来源行 N / 已关联采购来源行” | 业务页面不展示裸数据库 ID；有来源时保留可读追溯，没有行号时保持关联语义 | 若后续要按来源行搜索或导出，需要 API / 映射层统一支持 |
| 出货 / 库存 / 生产 / 财务事实 | 表单、列表、筛选、导出 | 幂等键、内部主键、内部引用、trace / request ID、裸数据库 ID、`#123` 兜底、ID 选择控件和手填 `source_id` / `source_line_id` 不进入用户可见层；需要防重或追溯时隐藏携带，页面只展示单号、来源、来源行、状态、数量和“已关联”反馈 | 防止业务人员把工程字段当成业务字段沟通或录入，同时保留后端幂等和审计边界 | `web/src/erp/utils/userVisibleTechnicalFields.test.mjs` 已作为静态守卫；若未来确需展示内部字段，只能放在 dev-only / audit 专项并先评审 |

暂缓引入的候选字段：

- 质检数量链路：`本次送检数量 / 已检数量 / 待检数量 / 不合格数量回写` 需要 QualityUsecase 和 schema 评审，不能只做前端字段。
- 出货草稿占用：当前强阻断发生在 `SHIPPED` 确认事务，草稿不预占销售订单剩余量；若后续要让草稿占用，必须通过明确预留事实实现，不能把普通草稿计数写成库存预留。
- 委外回货、领料、出库的剩余数量字段需要对应 Source Document 和 Fact usecase 成立后再吸收。
- 报关、提单、结汇、水单、磁材编码、外贸到付账号等字段仍只作为 trade 背景，不进入 plush 当前 Product Core。

## 8. 残值与缺值防护 / Stale And Missing Value Guards

涉及来源带值、导入或派生字段时，必须同时防残值和缺值。

| 场景 | 必须处理 |
| --- | --- |
| 新建默认带值 | 来源选择后带出必要展示字段，并保存来源 ID / 行 ID |
| 编辑覆盖 | 切换来源后旧来源字段必须清空或重算 |
| 清空来源 | 来源 ID、来源行 ID、来源快照和依赖字段必须同步清空 |
| 删除或停用主数据 | 已被引用的主数据不能直接破坏历史追溯；按启停用、引用保护或快照显示处理 |
| 快照缺失 | 能从来源 ID 安全回补时按规则回补；无法证明来源时保持空值或待确认，不伪造 |
| 真源缺失 | 不用备注、历史文案或客户样本猜测事实 |
| 列表 / 详情 / 打印 / 导出 | 复用共享映射或派生 helper，确保同字段口径一致 |

## 9. 后续实施顺序 / Implementation Order

建议按可验证闭环推进，而不是一次性改完整 ERP。

| 顺序 | 目标 | 验收重点 |
| --- | --- | --- |
| 1 | 建立字段来源分类和表单规则 | 每个字段标注选择、来源、派生、动作输入或导入候选 |
| 2 | 收口销售订单行产品 / SKU / 单位选择 | 不手填主数据名；后端重新校验 active master data |
| 3 | 收口采购订单到采购入库来源带值 | 入库行可追溯采购订单行；剩余数量可校验 |
| 4 | 收口出货单到销售订单 / 出货来源带值 | 出货行可追溯来源行；出库不超过可出货 / 可用库存 |
| 5 | 收口财务事实来源 | 应收、开票、收付款从已出货或已采购 / 委外事实进入，不从 workflow 直接生成 |
| 6 | 统一列表 / 导出 / 打印派生层 | 同一字段在列表、详情、打印、导出中口径一致 |

每一轮都必须说明允许修改路径、禁止修改路径、验收命令和剩余风险；触达 schema、migration、usecase、API、RBAC 或 UI 时按 `docs/product/模块实施治理.md` 拆任务。

## 10. 测试要求 / Test Requirements

字段来源和数据流向改动至少覆盖：

- 新值覆盖旧值。
- 切换来源后旧值清空或替换。
- 清空来源后依赖字段清空。
- 真源缺失时不伪造值。
- 快照缺失时按既定口径回补或保持待确认。
- 列表、详情、打印、导出和搜索口径一致。
- 后端 usecase 重新校验来源 ID、来源行、数量余额、状态和权限。
- Workflow 完成不写库存、出货或财务事实。
- 已过账或已出货对象通过取消、冲正或调整修正，不直接改事实字段。

如果本轮只覆盖部分路径，最终回复和 `progress.md` 必须明确未覆盖路径、原因和剩余风险。
