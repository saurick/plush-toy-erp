# 页面来源生成入口规则 / Source-Generated Page Entry Policy

本文定义 ERP 业务页面的“新建 / 生成 / 登记”入口口径，避免来源生成单据、事实页和只读台账继续出现无来源的页面级新建按钮。

## 当前结论

由其他页面、来源单据、事实行或后续 usecase 生成的表单，不建议保留无来源的页面级“新建”按钮。页面可以提供来源选择 / 来源导入控件或“从 X 生成 / 生成 X 草稿”动作，但必须让用户看到来源、余额、状态和不可生成原因；如果真实后端路径还没完成，前端不放占位按钮。

| 页面类型 | 页面级入口口径 | 允许条件 | 禁止口径 |
| --- | --- | --- | --- |
| MasterData / 主数据 | 可以“新建” | 当前页面就是对象 owner，后端已完成 create / update / RBAC / 引用约束 | 不能用新建绕过启停、引用检查或客户配置 |
| Source Document / 源单据 | 可以“新建”或“从来源导入明细” | 当前页面拥有草稿生命周期；来源导入只填明细，不直接写事实 | 不能把销售订单、采购订单、委外订单直接说成库存、质检或财务事实 |
| 来源生成单据 | 用“生成 X 草稿 / 从 X 生成” | 来源对象、来源行、剩余量、状态和后端校验都可见 | 不使用无来源“新建 X” |
| Fact Action / 事实动作 | 用“过账 / 取消 / 冲正 / 判定 / 释放 / 消耗”等动作 | 后端 usecase、RBAC、幂等和状态机已完成 | 不用页面级“登记事实”代替来源生成 |
| Read Model / Ledger | 不显示创建入口 | 只读查询、筛选、导出或列配置 | 不放“新建 / 登记 / 生成”占位按钮 |
| Future / 未闭环能力 | 不显示按钮或降级为只读说明 | 后端合同、来源和测试未完成时先隐藏 | 不用前端本地表单伪造后端事实 |

## 23 个正式业务页面血缘

`web/src/erp/config/businessPageLineage.mjs` 是页面血缘的可执行注册表；每个 `formal-v1` 页面必须且只能登记一次。`producerActions` 只登记当前真实 RPC / 聚合保存动作，直接外键关系登记为上游页面，不能把“相关”误写成“已自动生成”。同一注册表用 `catalog_fill / source_link / source_query / domain_generate / unscoped_mutation / external_import / post_fact / fact_reversal / lifecycle_transition / process_orchestration / read_projection / source_navigation / print_snapshot` 区分目录带值、来源关联、服务端来源候选、领域生成、尚未收口的无来源写入、外部导入、事实过账、库存事实冲正、对象生命周期、只推进流程节点的编排、只读投影、来源导航和打印快照；“能跳转”“有外键”“查询到候选”“生成了下游事实”不能混写成同一种流。

血缘门禁同时从当前 JSON-RPC service dispatcher 和页面 API 调用反向提取真实 action，包含 `customer_config` 暴露的 ProcessRuntime start / execute wrapper：来源查询、跨页生成、事实过账、库存冲正、生命周期和流程编排动作必须在 typed flow 中登记；本页普通 CRUD / 只读查询只能进入精确 allowlist，allowlist 中不存在的旧 action 也会失败。ProcessRuntime wrapper 按真实领域副作用登记；仅创建流程、聚合读质量门禁或记录财务放行节点的动作才是 `process_orchestration`。这样门禁既验证“登记项存在”，也阻断“代码新增了跨页动作却漏登血缘”。

登记为血缘候选不等于正式页面可达。当前 `add_purchase_receipt_item` 以及 `material_supply` 4 个、`finished_goods_delivery` 5 个 ProcessRuntime 公开动作均已有服务端 JSON-RPC，但正式 Web UI 无调用入口，统一记为 `partial / backend-only`，不得计入页面可达的 implemented 链路。销售订单正式提交只走 `start_sales_order_acceptance_process + execute_sales_order_acceptance_submit`，Web API 不再保留直连 `submit_sales_order` 的第二条路径；发票页登记过账、取消和从发票生成对账，不登记 `settle_finance_fact`。

每个 `source_query / source_link / domain_generate / post_fact / fact_reversal / lifecycle_transition / process_orchestration` 公开动作还必须同时校验目标动作权限、精确来源读权限和来源 / 目标模块状态。服务端在查询来源 repo 前拒绝无读权请求，条件来源只在请求真实绑定该来源时加权；不能因为页面能创建下游对象，就允许猜 ID、穿透读取上游单据 / 事实，或在来源模块 `disabled / read_only / 缺失` 时继续写下游。共享 registry 同时驱动 permission usage；逐项缺权测试断言写 usecase 未被调用，AST handler guard 断言每个已登记 action 的 handler 分支确实调用来源读 guard，避免注册表和真实接线脱节。

主数据批量带值统一使用草稿级 `catalog_fill` 合同：调用页面必须显式声明 `append / replace` 和 `allow / skip / reject` 重复策略，选择器不替业务域全局去重。同一 SKU 或材料可因交期、价格、批次或部位拆成多行；带值只复制主档字段，不自动建立来源关系或过账事实。真实 Excel / 跨系统导入只有在后端具备校验、幂等、审计和错误报告后才能登记为 `external_import`；当前注册表保持为空。

## 单据流、事实流与任务流

1. 销售主链：客户 / 产品 → 销售订单；生产订单可显式关联销售订单，库存预留可从销售订单行创建，出货草稿可从服务端可出货候选导入来源行，但三者都不是“销售订单自动生成事实”。草稿出货单可按送检批次发起出货前成品检验，已有检验必须合格或让步接收。检验集合确定后从出货单提交放行，仓库完成 `shipment_release` 只形成 `shipping_released` 协同状态；随后仍须回出货单确认出货，才写库存流水，再按已出货来源生成应收 / 发票 → 对账。
2. 采购主链：供应商 / 材料 → 采购订单 → 采购入库草稿与逐行 IQC → 全部通过后入库确认 → 库存；采购入库还可继续办理退货 / 调整和应付。入库、退货、调整草稿作废都是零库存终止；入库草稿还要结清 IQC / 预备批次，已过账取消才写反向流水。子修正取消后父收货依赖解除，收货草稿取消后采购订单关闭 / 取消依赖解除。
3. 委外主链：供应商 / 产品 / 材料 / 加工环节 → 委外订单 → 由已确认合同行发起发料 / 回货草稿 → 在同页“委外记录”过账 / 作废 / 取消 → 已过账回货再进入质检、库存与应付。草稿作废零库存，已过账取消冲正；active 质检 / 应付或 WIP 分配仍先阻断。取消事实和关闭 / 取消源单分别走各自生命周期，不用删除、导航或父单动作代替冲正。
4. 生产主链：产品 / 显式销售订单关联 / BOM → 生产订单发布 → PMC 排程任务与显式路线行 WIP → 领料 / 工序执行 / 分段质检 / 完工 / 返工事实 → 库存；REWORK 过账同时生成生产异常任务。生产事实草稿作废零库存，已过账取消冲正并服从 active 下游；生产事实写入库存时，库存来源必须使用事实自身的 `PRODUCTION_FACT + fact.id`，不能回退成生产订单 ID；WIP、质检、排程或异常任务完成也不能代替事实过账。
5. 三类确定性来源任务统一保存 `workflow.source-task/v1 + source_task_producer + source_type + source_id + intent_hash`，使用确定性 task code；同意图重放返回原任务，不同意图冲突并回滚。公开 `workflow.create_task` 拒绝全部 19 个能驱动领域 transition 的 task group，其中包含 `production_scheduling / production_exception / shipment_release`；受信的内部 producer 仍可生成真实任务。试用任务只能使用 `trial_*` 组，普通协同任务不能猜测业务 ID。
6. 客户配置 ProcessRuntime 只是同一领域 usecase 的另一条受控入口：销售提交、采购订单生成入库和逐行 IQC、质检判定、入库过账、确认出货和应收草稿必须分别归回源单生命周期、领域生成、事实过账或库存冲正；只创建 / 推进流程节点且 `EffectState=None` 的动作才登记为流程编排，不能用 wrapper 名称掩盖真实副作用。当前只有销售提交的 start + execute 已接入正式 Web UI；物料供应和成品交付的 9 个 wrapper 均是 `partial / backend-only`。
7. 公开服务端流程启动保留三条来源绑定链：`DRAFT` 销售订单 → `sales_order_acceptance`、`APPROVED` 采购订单 → `material_supply`、`DRAFT` 出货单 → `finished_goods_delivery`。服务端在同一事务锁定真实来源，派生 canonical 单号并复核状态；只允许完全相同的已创建流程重放。其中只有销售链已页面可达，采购与出货流程启动仍属 backend-only；旧入库单起点 `start_material_supply_process` 已退役。

## 当前未闭环与停止线

- 三类来源任务已接入正式业务事件；来源取消不会把活动任务伪造成 `done / rejected`。生产订单关闭 / 取消、已过账 REWORK 冲销和实际出货均按各自终态规则 fail closed；若未来需要无条件撤销任务，应正式增加 `cancelled` 状态、migration、事件和看板语义，不能偷用现有终态。
- 入库页面与公开后端均已退出无来源新建：`create_purchase_receipt_draft / create_purchase_receipt_with_items` 返回 unknown method；正式新建只走 `create_purchase_receipt_from_purchase_order`，追加行必须绑定同一已审核采购订单行。内部 usecase 只供领域事务和测试准备复用，不能注册为公开无来源入口。
- 服务端具备不等于正式 Web 可达：`add_purchase_receipt_item` 和物料供应 / 成品交付 9 个 ProcessRuntime 动作在注册表中保持 `partial / backend-only`；页面只有真实 API import / 调用与对应交互合同后才能提升为 implemented。销售提交已统一走 ProcessRuntime start + execute，不再把服务端 `submit_sales_order` 误记为第二条页面链路；发票不支持直接 settle。
- 委外订单不存在可证明的销售订单外键链；公开 schema / API / 页面均不再接受或返回 `source_sales_order_id`，新 Atlas migration 只删除该伪来源列。可读的 `source_order_no / product_order_no_snapshot` 仍是打印 / 追溯快照，不提升为可用于导入、授权或结算的业务关联。
- 自动生成的初始逐行 IQC 与已过账入库后的追加检验必须分开：初始 IQC 的 `REJECTED` 只阻断入库，不形成库存退货；现有质检来源“退供应商”动作要求来源采购入库已 `POSTED`，不能把两者描述成同一步。
- 委外页面的 post / 草稿作废 / 已过账取消已经通过 canonical RPC 和写后重读接线；当前两个 mock 浏览器场景证明委外发料创建、过账、另一草稿作废和委外回货移动端创建可达，不等于真实后端库存反向流水或 PostgreSQL 并发已由浏览器证明。已过账取消按钮与状态矩阵由页面 / 组件合同覆盖，仍需真实后端岗位浏览器和数据库读回才能作为目标运行证据。
- 出货页已把草稿零库存退出与已出货库存冲正分成“作废草稿”和“撤销已出货”两种用户可见动作，并在取消 RPC 后重新加载列表。Node 24.14.0 下冻结快照与实际当前树的来源链 mock Chromium 均 `34 / 34` 通过，已覆盖草稿 / 已出货取消和新建恢复态清值；该层仍不证明真实后端库存变化，也不替代数据库读回。
- `SHIPMENT / FINISHED_GOODS` 是出货前检验侧链，不是“生产完工 → 成品质检 → 成品入库”主链。当前来源粒度是同一出货单内的产品 / SKU、仓库和批次组合；同组合的重复出货明细聚合为一个送检批次，不声称逐出货行留痕。若未来必须逐行审计，需要新增正式来源行锚点和 migration，不能用备注或前端映射伪造。
- `OUTSOURCING_FACT`、`FINANCE_FACT`、`PURCHASE_RETURN`、`PRODUCTION_ORDER_MATERIAL_REQUIREMENT` 等间接来源，要精确跳回最初单据时仍需后端返回可解析的来源投影；页面不得猜 ID 或用同名列表替代精确追溯。
- 没有真实 producer、状态规则、RBAC、幂等、审计和测试时，停止在只读 / deferred 口径；不能用前端表单补造业务来源。

## 新增按钮检查清单

新增或恢复业务页主按钮前，先回答：

1. 当前页面是不是该对象的唯一 owner。
2. 如果不是 owner，来源对象、来源行、剩余量、状态和不可生成原因是否可见。
3. 后端 usecase、JSON-RPC、RBAC、审计、幂等和状态机是否已经完成。
4. 按钮文案是否准确区分“新建”“生成草稿”“过账 / 冲正 / 取消 / 判定”“发起协同”。
5. 表单、表格、筛选、导出列是否只展示业务字段，不展示幂等键、内部主键、内部引用、trace / request ID、裸数据库 ID、`#123` 兜底、ID 选择控件或手填 `source_id` / `source_line_id`；这些字段如需提交，只能隐藏携带或由后端生成。
6. L1 页面回归或静态测试是否断言了按钮存在 / 不存在，以及技术字段不会重新出现在用户可见层。
