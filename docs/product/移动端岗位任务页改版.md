# 移动端岗位任务页改版 / Mobile Role Tasks Redesign

## 结论

岗位任务端采用有意组合的当前主路径：v1 列表继续负责“找到当前事项”，v2 负责选中任务后的“查看任务 → 处理任务 → 可信结果回执”，结束后恢复原列表。两份 HTML 都登记为 Current，分别描述列表基线和选中任务流程；v2 不替换 v1 列表。

移动端只更新 Workflow 协同任务。完成、阻塞、退回、解除阻塞或催办均不代表库存、质检、出货、应收、应付、开票或收付款 Fact 已过账。

## 当前工作流

1. 登录后从服务端刷新管理员、权限和客户 effective session。明确岗位深链优先；否则直接进入当前账号第一个可用移动岗位，不在登录入口重复选择岗位。
2. v1 列表首屏继续保留待办 / 已办 / 提醒 / 我的、`全部 / 风险 / 超时 / 我负责` 主筛选、服务端游标分页 / 分批展开和任务卡片。`todo / risk / history` 继续使用各自服务端游标、快照和分页，不在前端合并成第二套任务真源。
3. 点选任务进入全屏详情。详情展示完整关键信息、完成条件、全部关联业务、当前结果和真实任务附件；长标题、长来源和历史处理线索允许换行。旧动作留下的文本证据引用只作为“历史处理线索”只读展示。
4. 有操作权限时进入独立全屏处理页；仅催办任务只提供催办，只读任务不展示假动作。完成必须填写可核验反馈，阻塞、退回、解除阻塞和催办必须填写原因。处理页不再重复提供自由文本证据框或附件上传，附件统一回到详情页按权限查看或管理。
5. 提交后进入独立结果回执。回执区分 `confirmed / unknown / failed`，显示服务端确认的任务快照和本次反馈或原因；只有旧任务确有历史证据引用时才显示“历史处理线索”。没有可信来源时不补造处理人或处理时间。

浏览器或系统返回键按“回执 → 列表”或“处理 → 详情 → 列表”恢复，并保留列表筛选、已加载分页、滚动位置和焦点；处理草稿在处理路径内恢复。移动软键盘出现时使用 `visualViewport` 收敛可视高度；校验失败聚焦第一个缺失字段，`Ctrl/Cmd + Enter` 提交，`Escape` 返回详情。主要触控目标不小于 44px，底部主动作不随正文滚走。

## 权限、可见性与性能边界

- 列表只消费服务端已经授权的 `list_role_tasks` 结果。前端不得再按当前岗位二次删除跨岗位但明确指派给当前用户的任务。
- 可见不等于可处理。处理前继续消费后端 action explain，并在提交前复核权限、任务版本和幂等意图；确认失败时 fail closed。
- 初次客户运行态同步失败时不挂载任务页；后台刷新瞬时失败时保留最后一次有效投影并提供重试，不因临时网络抖动伪造撤权结果。
- 列表只提供一个“显示更多”入口：先展开已加载内容，已加载内容耗尽且服务端仍有下一页时，继续读取同一 `server_time` 快照并直接显示新批次。顶部刷新才重新获取最新快照；过期请求保护、失败保留上次数据和刷新反馈继续沿用现有实现。
- 当前不增加移动搜索。岗位人员的第一入口是优先事项、风险和超时，而不是一组重复指标或桌面菜单树。

本改版不新增 schema、migration、RBAC 权限码、菜单、seed 或业务 Fact 写路径。原型与运行时不一致时，优先按正式文档、服务端合同、当前代码和测试收敛。
