# 自动化测试策略 / Test Strategy

本文只负责测试选择、证据层级和最终报告口径。当前能力状态看产品台账，具体命令入口看 `scripts/qa/README.md`、`scripts/README.md` 和各子系统 README。

## 原则

- 按影响面升级验证，不按“只改一行/一个文件”降级。
- 单元测试证明局部规则，合同测试证明跨层接口，浏览器回归证明真实页面，目标环境 smoke 证明部署可用；这些证据不能互相冒充。
- 自动化通过是必要条件，不代表业务事实、客户验收或生产发布已经完成。
- warning、skip、未运行和环境缺失必须单独说明，不能写成“全部通过”。
- 测试不得依赖调用者偶然安装的更高 Node/Go 版本；先用项目锁定工具链。
- 以前 AI 草稿、本地实验和未发布实现不构成兼容基线。非目标 key、字段、别名或入口应由精确集合测试证明已从 code/schema/API/UI/test 全链删除；一次性 migration 清理样本必须与常驻运行时合同隔离。

## 证据维度

| 维度 | 例子 | 回答什么 |
| --- | --- | --- |
| 验证层级 | T0-T8 | 本轮需要测到多深 |
| 测试形态 | unit、integration、contract、UI regression、E2E、smoke | 测试证明哪类行为 |
| 证据环境 | static、local mock、local DB、browser、test env、target env | 结果发生在哪里 |

最终回复至少写清：验证层级、执行命令/测试形态、环境、通过/失败/跳过数量、剩余盲区。

## 验证层级 T0-T8

| 层级 | 适用改动 | 最小验证 |
| --- | --- | --- |
| T0 现场与静态 | 所有改动 | `git status --short`、`git diff --stat`、`git diff --check`、未跟踪文件检查 |
| T1 文档 | Markdown、目录职责、正式口径 | T0 + `node --test scripts/qa/docs-inventory.test.mjs`；新增/删除/重命名同步目录 README 和文档清单 |
| T2 Schema/Migration | Ent schema、constraint、index、generated code | `cd server && make data` 后证明仓库零漂移，运行 schema/migration 定向测试和 `go test ./internal/data/...`；静态 `db-guard` 不能替代 Atlas/Ent diff、fresh/upgrade 或目标库 apply 证据 |
| T3 Core/Usecase/Repo | 状态机、金额、事务、领域规则 | core/workflow-fact 边界测试 + `go test -count=1 ./internal/core/... ./internal/biz ./internal/data` |
| T4 API/RBAC | JSON-RPC、auth、权限码、客户 key | T3 + `go test -count=1 ./internal/service ./internal/server`；覆盖未登录、disabled、无权限、有权限、多角色、super admin |
| T5 Frontend/UI | 页面、菜单、表单、字段、样式、错误提示 | 前端 unit/contract + `pnpm --dir web lint`、`pnpm --dir web css`、`pnpm --dir web test`；按影响面追加 `pnpm --dir web style:l1` 和真实浏览器回归 |
| T6 Config/Seed/Import | 客户配置、客户包、fixture、dry-run | customer package/runtime manifest、test-data-isolation、import manifest/extract/freeze/dry-run 测试；不连接真实客户库 |
| T7 Business Integration/E2E | 库存、采购、质检、出货、Workflow/Fact 真链 | `full/strict` 必跑本地隔离 PostgreSQL 关键事务门禁，再按范围补真实 JSON-RPC / 浏览器 E2E；验证写后读、幂等、取消/冲正和审计，禁止并发测试静默 skip |
| T8 Release | Compose、migration、备份恢复、客户部署 | `full`/`strict`、production preflight、固定镜像、migration 整段锁、目标 health/smoke、备份恢复和回滚 evidence |

选择最高受影响层级后，低层关键测试仍需保留。例如改 JSON-RPC 同时要跑 T3 领域测试；改页面字段联动同时要跑相关纯函数/合同测试。

## 按领域的必测合同

### Source Document

销售订单、采购订单、加工合同至少覆盖：

- 只有 `draft` 可修改头和明细；提交后拒绝无痕覆盖。
- 新建、保存、提交、批准/确认、取消、关闭的合法和非法状态。
- 数量必须为正；金额由后端 `数量 × 单价` 计算，客户端不一致金额被拒绝。
- 来源切换后旧快照清空/替换；列表、详情、打印、导出一致。
- 重复提交幂等；事务中任一明细失败整体回滚。
- 已有草稿更新必须携带正整数 `expected_version`；PostgreSQL 并发使用同一旧版本保存时只能一个成功，失败方不得改写单头或明细，成功响应必须返回递增后的版本。
- 保存结果遇到网络中断、超时、5xx 或结构不完整的成功响应时，页面必须保留弹窗和已编辑字段；结果确认前不得上传待传附件、刷新列表或关闭表单。

### Inventory/Purchase/Quality/Shipment/Finance Facts

至少覆盖 happy path、非法状态、重复提交、取消/冲正、负数/越界、事务失败、已过账不可编辑、事实表与余额/read model 一致。

采购入库必须防止超过采购剩余量；从采购订单生成收货草稿和向草稿追加明细的公开入口必须使用服务端规范化 payload hash 与稳定幂等键，覆盖并发同 intent 单一事实集、同 key 改 intent 冲突、commit 结果未知回查、replay 早于当前引用校验，以及收货行 / lot / 来料质检结果边界损坏时 fail closed。出货必须验证销售来源、库存可用量和预留归属；财务事实必须有稳定来源对象和审计。

BOM 生命周期的精确集合只能是 `DRAFT / ACTIVE / ARCHIVED`；普通创建、激活、归档、重新激活和非法转换测试必须共同阻断多余 key、旧别名和无正式命令的状态。

### Workflow/ProcessRuntime

至少覆盖：

- 公开 `create_task` 只能接受正式创建字段；config/process/node 锚点、`customer_key`、receipt / CAS 和未知字段必须 fail closed，且不能把该拒绝误写成 create replay / 幂等实现。
- owner role、assignee、责任池、capability、customer scope 和 RBAC 同时成立。
- Workflow task 的精确集合只能是 `ready / blocked / done / rejected`：公共创建固定为 `ready`，`ready → blocked / done / rejected`，`blocked → ready`，`done / rejected` 终态不可重开；同状态、未登记状态和动态恢复目标必须 fail closed，公开 `create_task` 不冒充 receipt replay。
- 后端常量、Schema CHECK、JSON-RPC 参数与返回、看板筛选、前端映射和 fixture 必须对上述精确集合做一致性断言；不得保留旧状态别名、聚合筛选别名、动态恢复字段或退出分支。一次性 migration 清理可以使用隔离样本，但不能让运行时测试继续承认非目标 key。
- 普通任务 action 必须覆盖 `version / expected_version / idempotency_key`、服务端 intent hash、payload patch 保留、前次阻塞 / 退回原因清理，以及 stale version 零副作用。receipt 必须使用稳定 `workflow.task-mutation-result/v1`；损坏、缺字段、错误 contract、padded key / command / status、双方一致但非法的状态和非法业务状态一律 fail closed。
- PostgreSQL 并发必须覆盖 blocked/blocked、blocked/done、urge/urge、urge/terminal、同 key 同 intent、同 key 改 intent 和 stale version；事件 receipt、任务快照、业务投影、派生任务及成功 break-glass 审计必须与单一赢家一致。精确重放不得重复副作用，新 key 不得绕过终态。
- migration 测试必须把 `20260711063237 / 20260711075355` 当作已执行不可变 revision，并验证 `20260711104729` 的 portable receipt bundle CHECK 以及 `20260711204000` 将两张 Workflow 投影表的迁移前提醒值规范为 `shipment_pending`；不得把 `shipment_release_pending` 升格为第二个业务状态真源。本项目迁移前且无法证明准确版本的事件必须保持 `task_version=NULL`，不得按行号或当前任务版本伪造 backfill。此类一次性规范化和证据保全不建立运行时兼容入口。每轮 fresh / upgrade 验证都必须记录当时实际的 latest revision、执行数量和 pending 数，不在长期策略里固化会漂移的 migration 总数；本地 migration evidence 仍不替代目标环境 evidence。
- 前端必须覆盖安全 UUID 生成、一次用户 intent 冻结参数、相同 intent 保留 key、语义变化换 key、HTTP 408 / 网络中断 / 5xx / 结构不合法 success response 保留同一冻结 key、未知结果不刷新 / 不关闭输入，以及 retained attempt 跳过重复 explain 但仍由后端重新校验当前权限和 receipt。
- 同一 task 的完成 / 阻塞 / 退回 / 催办必须共用同步 in-flight guard：首个动作在任何 await 前取得 lease，跨动作双击不得发出第二个请求，且只能在 `finally` 用同一 lease 释放。共享 `scripts/qa/workflow-task-mutation-intent-v1.vectors.json` 必须由 Go / JS 同时消费，覆盖全字符串 evidence 规范化、非字符串 evidence 类型与顺序、空格 key、mobile 精确重复 key 和 changed-intent；两端均通过前不得宣称跨语言 golden 闭环。
- linked task 的 done/rejected 与流程节点 outcome 一致；拒绝后按 manifest 回退或阻塞，不遗留 active 节点。
- domain command 只调用白名单 usecase，带稳定业务引用和幂等键。
- Process Instance 的精确集合只能是 `active / completed / blocked`，Process Node 的精确集合只能是 `waiting / active / completed / blocked`；常量、Schema、repo、API 和 fixture 必须拒绝未评审状态，未来取消、跳过或失败方向不得通过空壳命令或旧样本提前进入当前合同。
- `StartProcessInstance` 重试必须覆盖 active human/approval 缺 task、completed end 缺实例结算、durable domain result 待结算、active domain/wait 不重复执行，以及 task immutable intent 不一致时 fail closed。
- 节点阻塞和 ProcessInstance 阻塞必须在同一 PostgreSQL 事务内完成；实例 CAS 失败时节点更新必须回滚。
- Workflow 不直接写库存、质检、出货或财务事实。

### Customer Config/RBAC

至少覆盖：

- validate → publish → activate → rollback revision 合同。
- 模块 `enabled/read_only/disabled` 的页面、动作和后端写门禁。
- role profile、entitlement、work pool member 无悬空引用；disabled role 不投影。
- effective actions = backend RBAC ∩ active entitlement ∩ enabled module。
- 普通账号页面取 RBAC 与 active pages 交集；无投影不退回 RBAC-only。
- super admin 的前端看全不扩大后端领域写权限。
- 部署固定 `ERP_CUSTOMER_KEY`；业务请求不能切换客户配置。

### Import/Seed/Fixture

当前只测试 manifest、只读 extract、freeze 和 dry-run：

- `canExecuteRealImport=false`、`noRealImport=true`、`executesImport=false`。
- 不连接后端/数据库，不写正式表，不生成 migration，不创建 Fact。
- demo/fixture 明确 simulated，Product Core demo 不包含 yoyoosun 名称/PII。
- 原始客户资料不进入公开前端配置或构建产物。

仓库没有真实客户导入执行器，因此不保留 execution-loader 测试。未来真实导入应以通用批次 usecase 的 T3/T4/T7 测试进入，而不是给 dry-run 增加执行参数。

## 前端与浏览器回归

### 功能合同

- 菜单、路由、客户 page catalog 和后端 effective session 使用同一 page key；路由 path 只在前端导航 registry 维护。
- 写按钮同时服从 RBAC、effective actions、模块状态和 Source Document 状态。
- 用户错误使用中文场景 fallback，不透传 `error.message`；业务界面不显示 raw id、内部 key、Workflow/Fact 等开发术语。
- 请求切换和卸载使用 abort/stale guard，旧响应不能覆盖新状态。

### 视觉回归

页面或样式改动至少验证默认态、交互态、恢复态、错误/空/无权限态和相邻区域。检查：

- `getBoundingClientRect()`、`offset/client/scrollWidth/Height`。
- flex/grid 收缩、overflow、sticky/fixed、z-index 和响应式断点。
- 长文本、连续字符串、大数字、多标签和异常图片比例。
- 键盘焦点、关闭/返回后恢复、移动端关键动作。

局部单页样式至少 L1；共享组件、页面容器或同区域多状态至少 L2；全局样式、基础布局或多页面链路至少 L3。截图是视觉证据，不替代 DOM/盒模型断言。

## 推荐命令

```bash
# 查看当前改动需要哪些验证；默认不执行
bash scripts/qa/affected.sh --plan

# 执行受影响测试；仍需完成输出中的 required follow-up
bash scripts/qa/affected.sh --run

# 固定高频守卫
bash scripts/qa/fast.sh

# 提交前
bash scripts/qa/full.sh

# 发版前
bash scripts/qa/strict.sh
```

`affected.sh` 优先复用同名测试，并按前端、后端领域/API、业务事实 PostgreSQL、客户配置/导入、schema 和发布路径逐级升级。它不会自动执行可能改写生成文件的 `make data`，也不能自动猜测页面对应的唯一 L1 场景；这些步骤会在计划中作为 required follow-up 明示。未知路径、全局入口、部署路径和无独立测试的 QA 脚本保守升级为 `full.sh`，不会静默判成无需测试。

按需验证只服务开发期快速反馈。`pre-push` 仍固定执行 `full.sh`；发布前仍需 `strict.sh`、目标环境 migration、health/smoke、备份恢复和回滚 evidence。禁止用 affected 通过替代全量收口或目标环境证据。

`full.sh` 默认包含真实 Chromium browser-smoke：它拒绝继承的 `STYLE_L1_BASE_URL`，在动态空闲端口自启当前 worktree Vite，并只清理自身进程。server 全量 Go 测试还会显式启用真实 Chromium PDF 安全集成，验证安全 HTML 生成 `%PDF` 且被阻断的子资源不发出网络请求；本机自动发现 Chrome/Chromium，CI 通过 `ERP_PDF_CHROME_PATH` 绑定本轮 Playwright 下载的精确可执行路径。Ubuntu runner 还必须把同一发行包的 `chrome_sandbox` 安装为 root 所有、`4755` 的独立 helper，并通过 `CHROME_DEVEL_SANDBOX` 显式绑定；helper 缺失或所有者、权限不正确时 CI 直接失败。产品运行参数继续禁止 `--no-sandbox` 与 `--disable-setuid-sandbox`，不会为适配 CI 降低 PDF 渲染安全边界。该测试未执行将以 Go skip 阻断 full。`strict.sh` 直接复用 full 后再追加 shell、YAML、零 warning 与严格漏洞检查，必须始终是 full 的真实超集；资源排队只改变执行顺序，不能把 browser 移出 full 或静默 skip。fast / full 的固定 Node 与 Go 测试必须同时证明可解析结果、实际执行数大于 0、失败数与跳过数均为 0；缺 summary、零执行或 skip 不能只凭退出码 0 通过。

直接 full / strict 遇到 `SKIP_*`、`STRICT_SKIP_*` 或调用者提供的旧 coverage 变量必须以 `incomplete` 失败。full 始终真实执行 secrets 与 govulncheck，不接受调用者自签静态 JSON 代替前序成功。pre-push 必须先按每个真实 push ref 完成完整历史 secrets，随后调用普通 full；新 remote ref 的 full 聚合 diff 使用 `empty-tree..HEAD`，避免单 SHA 在干净工作树下产生空 diff。只有全部真实子命令依次成功，外层才能报告完整结果。

仓库的 `.github/workflows/ci.yml` 在 pull request、main push 和手工触发时使用单一 strict job：先证明 Ent / Atlas 生成零漂移，再直接运行 `strict.sh` 的完整本地门禁，最后检查 committed source archive。CI 不复制 quality、backend、PostgreSQL、frontend/browser 等第二套命令清单。workflow 文件只证明仓库已定义 CI；GitHub branch protection / required check 是否启用仍需要远端设置证据，本地 hook、full / strict 绿色或 CI YAML 存在都不能替代该证据。

# 客户配置与数据隔离

### 试用账号与浏览器脚本的证据边界

`fast/full/strict` 中的试用账号检查是无凭据守卫：它只证明本地 RBAC 检查形状、输入模板 / preflight no-write 边界、静态角色投影和入口脚本语法未损坏；不执行真实登录、不调用本地后端。真实账号和浏览器 smoke 同样包含上述无凭据单测和脚本语法守卫，仍需在本地后端与演示账号密码可用时单独运行。

`--print-input-template` 只打印所需环境变量、账号清单、可选报告路径、effective session 脱敏诊断读取计划和真实核对命令；不读密码、不登录、不调用后端、不启动浏览器、不启动 Vite、不读取客户配置脚本、不写报告、不写数据库。

```bash
node scripts/qa/trial-account-rbac.mjs \
  --preflight-report output/trial-account-rbac/preflight.json
node web/scripts/trialDemoAccountBrowserSmoke.mjs --preflight-report output/trial-demo-account-browser-smoke/preflight.json
node web/scripts/trialDemoAccountBrowserSmoke.mjs \
  --report output/trial-demo-account-browser-smoke/report.json
node web/scripts/mobileWorkflowRuntimeBrowserSmoke.mjs \
  --preflight-report output/mobile-workflow-runtime-browser-smoke/preflight.json
node web/scripts/mobileWorkflowRuntimeBrowserSmoke.mjs \
  --report output/mobile-workflow-runtime-browser-smoke/report.json
```

RBAC preflight 报告必须保留 `preflightOnly=true`；不调用 `admin_login / me`，不证明真实 RBAC、真实账号或有效客户配置已经可用。浏览器 preflight 不读密码、不登录、不调用 JSON-RPC、不启动浏览器、不启动 Vite、不读取客户配置脚本、不创建任务、不写数据库。真实执行生成的 real-smoke report 也不证明目标环境发布、真实客户导入或 release evidence 已完成。

岗位任务端真实回归只写本地脱敏报告，包含模拟任务计划 coverage 摘要和未证明项，不保存密码、token、Authorization header、raw customer package 或 action 列表，也不进入 release evidence；覆盖范围应包含品质完成和仓库入库完成等岗位动作。显式外部地址未命中 yoyoosun 配置和资产时，preflight 必须以 `external-base-url-not-yoyoosun-entry` 阻断，避免把其他客户或 Product Core 端口误当验证目标。

## 发布证据边界

本地 full/strict 通过只证明仓库质量门禁。目标发布还必须证明：

1. Git commit、镜像 tag/digest 和客户配置 revision 可追溯。
2. runtime env 通过 preflight，Secret 非占位，`ERP_CUSTOMER_KEY` 稳定。
3. migration 在同一锁内完成 status → dry-run → apply。
4. 备份可恢复，恢复后的 migration/数据状态和 smoke 通过。
