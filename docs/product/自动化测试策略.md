# 自动化测试策略 / Test Strategy

本文用于把自动化测试和验收命令按改动影响面归入验证层级，帮助每个实施任务或日常改动选择合适的验证范围。

## 1. 定位

- 本文是测试选择和验收分层文档，不替代 `docs/当前真源与交接顺序.md`、代码、Ent schema、Atlas migration、`scripts/README.md` 或本轮具体任务说明。
- 具体任务的允许修改文件、禁止修改文件和验收命令，仍以当前用户要求、正式设计文档或任务说明为准。
- 文档里的“未来 / 后续”测试项只表示后续能力成熟后应补的方向，不代表当前仓库已经有对应 runner、脚本、API 或 E2E 环境。
- 自动化测试通过是必要条件，不等于业务事实、权限边界、部署状态和客户交付已经完成。涉及当前实现状态时，必须回到代码、测试和当前真源文档交叉确认。

## 3. 术语边界

测试治理同时使用几类口径。它们是互补关系，不应互相替代。

| 口径 | 回答的问题 | 典型值 | 用法 |
| --- | --- | --- | --- |
| 架构层级 | 系统责任在哪里 | MasterData、Workflow、Fact、RBAC、API/UI、Help/Debug/QA、Productization/Delivery | 判断要读哪些代码和正式文档，以及是否破坏 Workflow / Fact、RBAC、菜单、部署或客户差异边界 |
| 验证层级 T0-T8 | 本轮验证需要测到哪里 | T0 静态现场、T2 Schema / Migration、T5 Frontend UI、T8 发布 / 回滚 | 按改动影响面选择验证范围，避免 docs-only 机械全跑，也避免核心逻辑只跑最低层 |
| 测试形态 | 某个测试证明了什么 | unit、integration、contract、UI regression、smoke、E2E、deployment | 在最终回复或验收记录里说明测试证据的能力边界 |
| 证据环境 | 测试在哪里发生、可信度到哪 | static scan、local mock、local real DB、browser runtime、dev/test env、target deployment env | 区分静态检查、本地真实库、浏览器运行时、目标环境 smoke 和部署后验证 |

示例：一次页面表单改动可能选择 `T5 Frontend UI`，测试形态是 `UI regression + contract`，证据环境是 `browser runtime + local mock`；如果没有触达真实写入，就不能声称覆盖 `T7 业务事实 / E2E`。

## 5. 静态边界检查

T0 / T1 至少按本轮影响面选择下列检查，不要无脑扫超大目录后忽略结果。

```bash
git status --short
git diff --stat
git diff --check
git ls-files --others --exclude-standard
```

常用边界 grep：

```bash
grep -R "tenant_id" docs/customers docs/product docs/architecture docs/reference config deployments server web || true
grep -R "shipping_released" docs/customers docs/product docs/architecture docs/reference web server || true
grep -R "shipped_quantity\|shipment_id\|inventory_txn_id\|invoice_id\|payment_id\|ar_id\|ap_id\|product_sku_id" server web docs || true
grep -R "markAsShipped\|shipSalesOrder\|reserveStock\|deductInventory\|generateInvoice\|generateReceivable\|receivePayment" server web docs || true
grep -R "ChangeUsecase\|change_records" server web docs || true
```

这些 grep 不是“出现即失败”。如果命中的是禁止说明、边界说明、未来 deferred 说明或历史 imported note，必须在验收记录里说明命中语义；如果命中 runtime、schema、migration、seedData、API 或用户可见误导文案，应视为风险。

```bash
  --unit-id 1 \
  --warehouse-id 1 \
  --out output/customers/yoyoosun/mvp-closure
```

该入口默认不连接数据库、不调用后端、不写正式表，也不执行真实客户导入；真正写入模拟数据仍必须进入具体模拟脚本的 `--apply` 路径。

采购入库第一条真实写入链路验收：

```bash
cd /Users/simon/projects/plush-toy-erp
node scripts/qa/purchase-receipt-real-write-e2e.mjs --print-input-template
node scripts/qa/purchase-receipt-real-write-e2e.mjs \
  --preflight-report output/qa/purchase-receipt-real-write-e2e/preflight.json
node scripts/qa/purchase-receipt-real-write-e2e.mjs \
  --out output/qa/purchase-receipt-real-write-e2e
```

`--print-input-template` 只输出服务层真实写入 e2e 的 `--out`、`--with-postgres`、本地测试库边界和后续真实命令，不运行 Go test、不调用 `make`、不连接 PostgreSQL、不写报告、不写数据库。

`--preflight-report` 只写本地脱敏 JSON 前置报告，确认 Go 命令、服务层测试锚点、PostgreSQL guard 脚本和 Make target 是否存在；请求 `--with-postgres` 时还会用 `pg_isready` 对脱敏后的本地 host / port 做无凭据可达性检查。本模式不运行 Go test、不调用 `make`、不使用凭据连接 PostgreSQL、不保存 `PURCHASE_RECEIPT_PG_DB_URL` 值、不写业务报告、不写数据库，也不证明采购入库事实链已通过。

如需追加本地 PostgreSQL 防呆测试：

```bash
cd /Users/simon/projects/plush-toy-erp
node scripts/qa/purchase-receipt-real-write-e2e.mjs \
  --with-postgres \
  --out output/qa/purchase-receipt-real-write-e2e
```

该入口覆盖采购入库草稿创建、加行、非法明细失败、过账、`get/list` 回显、看板 projection、库存流水 / 批次 / 余额、取消冲正、重复操作幂等和权限拒绝；不执行真实客户导入，不把 Workflow task done 当成采购入库过账。

## 8. 任务验收记录 / Task Verification Notes

非平凡任务的最终回复、`progress.md` 或用户明确要求的正式验收文档应包含：

- 本轮选择的验证层级 T0-T8。
- 本轮未选择的验证层级及原因。
- 本轮测试形态、证据环境和跳过原因。
- 实际运行命令和结果。
- 失败命令、关键日志和是否与本轮相关。
- 边界 grep 或禁止路径检查结果。
- Git 状态、未跟踪文件和并行现场说明。
- 若跳过 Go / pnpm / browser / migration / E2E，必须写明原因和剩余风险。

普通文档整理不自动生成额外审查报告；如只改正文且未触达 runtime、schema、前端配置、菜单或路由，应在最终回复中说明验证范围。

## 9. 后续 CI 分层建议

当前是个人开发主路径，暂不配置 GitHub CI。下面只作为后续协作人数增加、分支回归频率升高或需要远端门禁时的可选方向，不代表当前仓库已经配置 CI。

| 层级 | 建议内容 |

CI 落地前仍以本地命令、Git hooks、部署文档和当前任务说明为准。
