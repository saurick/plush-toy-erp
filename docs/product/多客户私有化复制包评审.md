# 多客户私有化复制包评审 / Private Deployment Package Review

- 文档类型：产品化交付评审 / Productization Delivery Review
- 状态：最小模板候选 / Minimal Template Candidate
- 作用域：新增客户的 docs、config 与部署参数边界
- 不代表：真实客户导入、生产部署、release evidence、客户签收、SaaS、多租户或新运行时平台

## 结论

继续使用模块化单体和一客户一私有化实例。新增客户复用同一个 Product Core、后端、前端、schema/migration 和 `server/deploy/compose/prod`；客户差异只进入已有声明式配置和受控投影。

`reference-customer` 是中性工程参考，帮助评审差异和验证已有纵向链路，不是第二个真实客户，也不是可直接复制其业务结论的模板。新客户骨架应从最小模板开始，再逐项评审差异。

## 当前五类内容

| 内容 | 当前职责 | 边界 |
| --- | --- | --- |
| `demo` | 最小 lint/compile smoke fixture | 不承担完整交付示例 |
| `reference-customer` | 工程参考配置和三份人本化文档 | draft/preview，未部署、未签收 |
| `yoyoosun` | 当前真实客户资料、配置和部署包 | 客户事实不能升格为 Product Core 默认 |
| `private-deployment-template` | 三份文档、三份配置、最小部署参数的边界 | 不复制部署架构，不生成 evidence |
| `industry-templates/plush` | 行业候选输入 | 不是 runtime 默认 |

## 最小交付结构

| 层级 | 必须内容 | 目的 |
| --- | --- | --- |
| `docs/customers/<customer-key>/` | `README.md`、`差异与边界.md`、`实施测试部署验收.md` | 让产品、开发、测试、运维和业务共享同一口径 |
| `config/customers/<customer-key>/` | `README.md`、`customerPackage.mjs`、`customer-config.example.js` | 只保存当前消费者需要的声明式配置 |
| 部署资料 | 客户 README 与受控 env；可选薄 override | 复用生产 Compose，不建立第二套主路径 |

真实客户可以在 `deployments/<customer-key>/` 保存经脱敏的 runbook 和 evidence 索引，但 reference 工程样例不创建 `deployments/reference-customer/`。

## 必须守住的质量底线

- `ERP_CUSTOMER_KEY` 由环境固定，请求不能切换实例身份。
- 每客户独立数据库/账号、文件、日志、备份、恢复权限和 secrets；不以 `tenant_id` 做同库多租户。
- customer-config revision 发布后不可变；同 revision 同 hash 幂等，不同内容 conflict。
- publish/activate/rollback、compiled rows 与 audit 的事务和唯一 active 语义由 PostgreSQL 测试证明。
- 客户配置只能收窄 Product Core/RBAC，前端隐藏不是权限边界。
- Workflow task done 不等于库存、质检、出货或财务 Fact posted。
- 真实客户原件进入客户专属私有仓库或受控对象存储；Product Core 不使用 Git 子模块，也不依赖私有资料仓库。
- 目标机只加载已构建制品；发布前保留备份和回滚点，发布后形成 health/ready/smoke 证据。

## 当前不建设

- SaaS、多租户、微服务、license、billing 或动态插件。
- 客户专属 schema、migration、RBAC、usecase 或前端应用。
- acceptance matrix 平台、跨文档 validator、通用脚手架语言或大型部署 shell 框架。
- reference 专属完整部署包、虚假 release evidence 或第二个临时客户全栈闭包。
- 没有真实消费者的编号、字段编辑/必填/公式或导入 apply 平台。

只有第三个真实客户开始、出现明确重复维护成本，或能力本身成为 Product Core 需求时，才单独评审这些抽象。

## 模板边界检查

```bash
node scripts/qa/private-deployment-boundaries.mjs
node --test scripts/qa/private-deployment-package-closure.test.mjs
node scripts/qa/private-deployment-package-closure.mjs --out output/private-deployment-template-boundary
```

报告字段 `boundariesSatisfied=true` 只表示禁止项和最小清单成立；`deliveryCompleted`、`releaseEvidencePresent` 和 `customerAccepted` 必须保持 `false`。真实目标机部署、备份恢复、浏览器 smoke 和客户签收只能由对应环境证据证明。
