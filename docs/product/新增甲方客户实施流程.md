# 新增甲方客户实施流程 / New Customer Implementation Flow

本文定义新增私有化甲方从资料接收、差异评审到部署与验收的产品级顺序。它服务开发、测试、运维和业务验收，不替代具体客户目录、代码、migration、目标环境 release evidence 或客户签收。

## 固定原则 / Fixed Principles

- 每个真实客户先确定稳定的小写 `customer key`；部署环境固定 `ERP_CUSTOMER_KEY`，请求不能切换实例身份。
- 继续复用同一个 Product Core、后端、前端、schema / migration 和 `server/deploy/compose/prod`；一客户一私有化实例，不新增 `tenant_id`、客户分支或第二套 Compose 真源。
- 每客户独立数据库与账号、对象存储或文件边界、日志、secrets、备份和恢复权限。目标机只加载已构建制品，不承担源码构建。
- 真实 Excel、PDF、图片、合同和私密 manifest 进入客户专属私有仓库或受控对象存储；Product Core 不使用 submodule / subtree，也不依赖客户原件完成普通构建和 CI。
- Workflow task done 不等于库存、质检、出货或财务 Fact posted；客户配置不能提升后端 RBAC、绕过领域状态机或改变事实真源。
- `reference-customer` 只用于工程参考，不创建 `deployments/reference-customer/`，也不形成真实发布、恢复、浏览器验收或签收证据。

## 参与者与责任 / Readers And Responsibilities

| 参与者 | 主要责任 | 不能替代 |
| --- | --- | --- |
| 开发 | 评审差异归属，复用现有 catalog / schema / compiler / consumer，实现受控客户配置或通用 Product Core 能力 | 业务确认、目标环境证据 |
| 测试 | 按风险验证配置、权限、Workflow / Fact、导入 dry-run、迁移和用户可见行为 | 客户人工签收 |
| 运维 | 准备隔离资源，执行 preflight、备份、migration、部署、升级、回滚和恢复演练 | 代码测试或业务口径确认 |
| 业务验收 | 按真实岗位、真实账号和已确认差异检查页面、动作、打印与业务链路 | 自动化测试、模板报告 |

## 实施顺序 / Delivery Sequence

### 1. 建立客户身份与资料授权

1. 确认稳定 `customer key`、客户负责人、实施范围、目标环境和验收负责人。
2. 记录资料来源、授权范围、敏感等级、hash / 版本和保留策略；未获授权的原件不得进入导入或模板实现。
3. 在 `docs/customers/<customer-key>/` 只保存可进入产品仓的脱敏说明；私密 manifest 与原件保持客户级隔离。

停止条件：customer key、资料授权或敏感性边界未确认时，只能做中性样例评审，不进入真实导入、部署或签收。

### 2. 归类客户差异

逐项归入 Product Core、Industry Template、Customer Config、Customer Extension、Data Import Adapter、Print Template 或 Reporting，并记录实现位置、消费者、验证位置和延期项。

- 只有通用领域事实或已经出现真实复用需求的能力，才进入 Product Core 评审。
- 品牌、菜单投影、低风险字段显示、角色责任池、打印默认值等优先进入声明式客户配置。
- 涉及 schema、migration、权限码、事实写入、动态代码或跨领域事务的需求必须单独评审，不能用前端条件分支代替。

参考：[客户差异策略](客户差异策略.md)、[多客户私有化复制包评审](多客户私有化复制包评审.md)和[标准样例客户差异与边界](../customers/reference-customer/差异与边界.md)。

### 3. 准备配置与导入

1. 在 `config/customers/<customer-key>/` 只声明当前 catalog、schema、compiler 和正式消费者已经支持的内容；raw package 在正式评审前保持 draft / preview-only。
2. 构建期品牌资源只进入该客户公开 overlay，不作为后端授权或业务事实真源。
3. 导入先做只读提取、字段映射、dry-run、unresolved queue 和对账计划；没有备份、审批、幂等与 forward-fix 方案时不得 apply。
4. 客户配置 publish / activate / rollback 必须继续走后端权限、不可变 revision、审计和固定 customer key 门禁。
5. 任何会改变正式 manifest 内容的客户包变更都必须提升 package/version 并生成新 revision；同 revision 同 hash 只允许幂等重放，同 revision 不同 hash 必须拒绝，已发布 revision 不得覆盖或改写。

### 4. 开发与测试验证

开发期先按受影响范围运行定向验证；全部 tracked draft 包使用 preview manifest，不伪装成 release-ready：

```bash
node scripts/qa/customer-package-lint.mjs --all
node scripts/qa/customer-config-runtime-manifest.mjs --all --mode preview
node scripts/qa/private-deployment-boundaries.mjs
bash scripts/qa/affected.sh --plan
```

最低覆盖包括：客户索引与 key 一致性、package/schema/catalog 禁止项、manifest 确定性、RBAC 上限、字段与责任池消费者、旧 revision、导入隔离、Workflow / Fact 禁区和部署模板边界。涉及 PostgreSQL 唯一约束、事务或并发时必须补 PostgreSQL 集成测试；涉及用户可见页面、打印或角色动作时补真实浏览器回归。

`fast` 用于高频静态和合同反馈；前端生产构建、浏览器 smoke、PostgreSQL 关键事务与服务端 build 进入 `full` / `strict`。本地绿色仍不等于目标环境发布或客户验收。

### 5. 首次部署

1. 从 `config/private-deployment-template/` 准备受控 env；真实客户在 `deployments/<customer-key>/` 保存脱敏 runbook 和 evidence 索引。
2. 固定产品 commit、镜像 digest、customer-config revision、migration 前后版本和回滚点。
3. 核对独立数据库/账号、对象存储或文件目录、日志、secrets、备份目标和恢复权限；先完成 preflight 与可恢复备份。
4. 目标机加载已构建制品，串行执行受控 migration，再使用唯一生产 Compose 启动。
5. 检查 health、ready、桌面入口、登录、服务端 RBAC、active revision 读回和选定业务纵向链路。

### 6. 升级

升级前重新核对旧/新产品版本、镜像 digest、customer-config revision、migration 影响和兼容回滚点。先备份并验证恢复入口，再串行 migration、启动和 post-deploy smoke；不得用历史发布报告或模板检查替代本次证据。

### 7. 应用回滚与 forward-fix

- 应用回滚只使用兼容旧制品和已验证配置 revision；配置 rollback 不等于业务数据或数据库回滚。
- 已执行 migration 默认不编造逆向 SQL；按正式评审选择兼容旧应用或 forward-fix。
- 回滚或 forward-fix 后重新执行 health、ready、登录、RBAC、active revision 和关键业务 smoke，并绑定同一发布批次。

### 8. 备份恢复演练

在隔离环境恢复数据库及客户文件/对象，核对备份 hash、migration 状态、关键数量与金额、附件或对象完整性和只读 smoke。模板、空报告、未回读对象或仅执行 `pg_dump` 都不能证明可恢复。

### 9. 业务验收与收口

业务验收使用目标环境真实岗位账号，检查品牌与菜单、字段显示、服务端拒权、Workflow 责任与状态、Source Document / Fact 边界、打印真源、默认/交互/恢复/错误态和相邻页面。

最终证据至少绑定：产品 commit / image、customer key、配置 revision、migration、备份恢复、health / ready、真实账号 smoke、差异逐项结果、未完成项、回滚点和客户负责人签收。任一外部步骤未执行时必须明确写为未执行。

## 完成口径 / Completion Boundary

以下各层必须分别陈述，不能合并为“已完成”：

| 层级 | 最低证据 |
| --- | --- |
| 工程实现 | 代码、配置、migration 和定向自动化 |
| 本地收口 | affected / fast / full / strict 中与风险匹配的结果 |
| 目标环境发布 | 固定制品、migration、health / ready、smoke 和回滚点 |
| 数据与恢复 | 导入对账、备份、隔离恢复和完整性核对 |
| 业务验收 | 真实岗位账号、差异逐项验收和客户负责人签收 |

reference 工程样例、preview manifest、Compose `config`、本地浏览器或模板边界报告都只能证明各自局部合同，不能替代真实交付证据。
