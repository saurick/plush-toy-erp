# 多甲方角色能力流程编排优先级 / Multi-client Role Workflow Priority

本文把 `docs/reference/**` 的多批 GPT 参考材料收敛为当前仓库可执行的优先级口径。日期越新的参考材料权重越高，其中 `docs/reference/第四次20260627/` 是本主题的最新输入；所有参考材料都只作为输入，真正能进入实现的内容必须继续回到 `AGENTS.md`、`docs/当前真源与交接顺序.md`、正式产品 / 架构文档、当前代码和测试复核。

## 结论

当前最值得优先做的不是“大平台”或通用流程引擎，而是先把 Workflow / RBAC / 角色投影的边界补稳。执行顺序如下：

机器可读执行队列以 `node scripts/qa/multi-client-role-workflow-priority-audit.mjs --json` 的 `implementationOrder` 为准；该字段按 P0-P5 输出阶段目标、当前本地 / 目标环境证据状态、允许路径、禁止路径、明确不做内容、验证命令和下一条 release closeout 动作。P5 阶段的 `nextAction.inputChecklist` 会列出第一条 blocked action 的已解析输入、缺失输入、字段级 operator checklist、report-only 报告路径和真实执行确认边界；`releaseEvidenceProgress.closeoutInputChecklist` 会把整条 closeout 队列的缺失输入按 env / file / manual 聚合，列出每个输入关联的 action、env 模板、secret 标记和 report-only 边界，并通过 `collectionPlan` 按 release closeout action 顺序列出每组证据要先收集的输入，方便一次性收集真实 release batch 输入再逐组执行。全局 `missingInputs` 和每个 `collectionPlan[*].missingInputs` 会直接带出 `source/sourceHint/evidenceTarget/validation/status`，可作为外部 release batch 收集表的字段来源；`collectionGroups` 会再按镜像 / 版本、preflight、备份恢复、目标 smoke、客户配置读回、回滚处置和签收等收集责任面聚合 action 与缺失输入，不需要再把缺失输入与 operator checklist 二次拼接。只需要收集 P5 输入时，可运行 `node scripts/qa/multi-client-role-workflow-priority-audit.mjs --input-checklist-json` 输出轻量 JSON；如果外部表格只接受平铺行，可运行 `node scripts/qa/multi-client-role-workflow-priority-audit.mjs --input-checklist-csv` 输出带 `evidence_dir / runtime_env_file / release_ready / completion_state / blocking_category` 和 P5 `phase_*` 上下文的 group / action / input / source / evidence / validation 维度 CSV。轻量 JSON / Markdown 会带脱敏 `implementationBreakdown` / `Implementation Breakdown`，列出 P0-P5 阶段目标、本地状态、目标证据状态、下一条只读 closeout action 和 `executionContract`（allowed paths / forbidden paths / not doing / validation commands）；轻量模式仍只包含 read-only 状态、目标 evidence 路径、阶段拆解、执行合同、全队列输入清单和不由本审计证明的范围，不输出完整 checks / reference coverage，也不写 release evidence。当前没有正式生产环境时，P5 只能停在 `target-evidence-required`，不能把本地 ready、输入清单、collection plan、阶段拆解、执行合同或 report-only 命令写成目标 release evidence 已完成。

如果需要人工阅读或贴进外部 release batch 收集表，可运行 `node scripts/qa/multi-client-role-workflow-priority-audit.mjs --input-checklist-markdown`。Markdown 模式同样只读、不写 release evidence、不输出 execute 确认入口，并额外提供 `Implementation Breakdown`、`Collection Groups` 与 `Collection Input Details` 表；`Implementation Breakdown` 给出 P0-P5 目标拆解、当前阻塞阶段和阶段执行合同，`Collection Groups` 按收集责任面分组，`Collection Input Details` 把每个 closeout action 的输入、source、evidence target 和 validation 展开成逐 action 明细，方便分别交给镜像构建、preflight、备份恢复、目标 smoke、回滚演练和签收负责人收集。CSV 模式也只用于外部表格导入，不包含 report command、execute command、真实 DSN、token 或任何 secret 值。

## 本会话先落地的范围

本会话优先处理前三类后端边界和 engineering 最小闭环里能直接由当前代码验证的部分：

1. `actor_role_key` 由服务端推导，不再采信客户端参数；完成 / 阻塞 / 退回动作新增受控 action 合同，前端任务动作入口不再提交 raw `task_status_key`。
2. Workflow 任务列表按当前管理员角色和指派人收敛，保留 super admin 的全局可见性。
3. engineering 角色补入内置角色、移动端访问权限和 RBAC 同步测试。
4. 成品返工完成后的业务状态投影迁入后端 WorkflowUsecase，移动端不再本地 upsert 成品返工状态。
5. 委外返工完成后的业务状态投影迁入后端 WorkflowUsecase，移动端不再本地 upsert 委外返工状态。
6. 委外回货跟踪完成后的 QC 任务派生和 `qc_pending` 业务状态投影迁入后端 WorkflowUsecase，移动端不再本地创建回货检验任务。
7. 委外入库完成后的应付登记任务派生和 `inbound_done` 财务待处理投影迁入后端 WorkflowUsecase，移动端不再本地创建委外应付任务。
8. 应收登记完成后的开票登记任务派生、开票登记完成后的 `reconciling` 协同状态投影迁入后端 WorkflowUsecase，移动端不再本地创建开票任务或写入应收 / 开票完成状态。
9. 应付登记完成后的采购 / 委外对账任务派生、对账完成后的 `settled` 协同状态投影迁入后端 WorkflowUsecase，移动端不再本地创建对账任务或写入应付 / 对账完成状态。
10. 财务应收 / 开票 / 应付 / 对账 blocked / rejected 预警投影迁入后端 WorkflowUsecase，移动端不再本地 `upsertWorkflowBusinessState`。
11. `NormalizeAdminRoleKeys` 不再按 `BuiltinRoles()` 过滤；账号创建 / 更新角色时按当前 `roles` 表的有效角色校验，自定义角色不会被静默丢弃，不存在或已停用角色会拒绝写入。
12. 客户配置包 lint 明确 `validate / compile / preview` 三类安全模式；raw 客户包脚本入口仍拒绝 `activate / rollback`，不把 preview-only 包伪装成可发布 runtime。
13. Workflow / Fact 边界进入 QA 守卫：受控 `complete_task_action`、`block_task_action` 和 `reject_task_action` 链路不得直接引用 Operational Fact、库存、出货或财务事实写入口；旧 `update_task_status` 已退出运行时，真实领域命令仍从对应领域 usecase 显式执行。
14. Workflow UI action 边界进入 QA 守卫：正式 `web/src/erp` 运行时代码不得直接调用 `createWorkflowTask`、`updateWorkflowTaskStatus`、`upsertWorkflowBusinessState` 或对应 raw JSON-RPC 方法；`updateWorkflowTaskStatus` wrapper 和 raw `update_task_status` mock 已退出，style:l1 场景和测试代码只保留验收用途。
15. `fast.sh` / `strict.sh` 接入客户包 validate + compile、Workflow / Fact 边界守卫、Workflow UI action 边界守卫、正式前端客户配置投影边界守卫、`multi-client-role-workflow-priority-audit` 和 release evidence status 检查；导入、部署、备份和回滚仍以既有 dry-run / release evidence / production preflight 文档和脚本为准，不伪造目标环境通过。
16. 客户配置版本运行时接入 Ent / Atlas：新增 `customer_config_revisions`、`deployment_module_states`、`role_profiles`、`access_entitlements`、`work_pools` 和 `work_pool_memberships`，只保存受控编译后的配置 revision、模块状态、角色画像、授权和责任池，不新增多租户隔离字段，也不新增业务事实表。
17. 后端新增 `customer_config` JSON-RPC 域：`validate_customer_config / publish_customer_config / activate_customer_config / rollback_customer_config / get_effective_session` 进入 usecase / repo / RBAC 主路径；active revision 不允许被覆盖，发布 payload 禁止 secret、token、password、SQL、Go、JS 和 raw records。
18. 从已跟踪客户包生成后端可验证的 runtime manifest，锁住模块状态、角色画像、授权、责任池、页面投影、字段策略和 `purchase -> purchasing` 等后端角色 key 映射；客户包可选 `moduleStates` 只能声明 catalog 内模块的 `enabled / read_only / disabled`，非 `enabled` 必须写 reason，并只编译为后端 `module_states` 受控输入，不提供运行时安装 / 卸载模块，也不证明完整模块关闭已 ready；该 manifest 可作为 `validate_customer_config / publish_customer_config` 的受控输入，但不上传文件、不调用后端、不执行激活或真实导入。页面投影必须来自已登记正式菜单 key，manifest 编译器和后端 validate / publish 都会拒绝缺失、空列表或未知 page key 的 `compiled_snapshot.pages`。字段策略只发布当前前端已消费的 `customers.default`、`suppliers.default` 和 `sales_orders.default`，不把 `sales_order_items` 的产品 / 款式 / 颜色尺码草案候选编译成 active revision 字段策略；后端 validate / publish 也拒绝未登记字段策略，effective session 输出会过滤旧 revision 中的非法字段策略。
19. 正式前端登录后读取 `get_effective_session`，把页面、动作、字段策略和责任池投影到当前 admin profile；正式普通账号的后端页面投影只在 RBAC 菜单和 active revision 页面清单之间取交集，菜单可见性和直接打开当前 URL 都按同一交集强收窄，隐藏 URL 会重定向到可见入口或进入空入口提示；本地开发态只保留 RBAC 已允许且已登记直达 URL 的诊断入口，不放开普通账号侧边菜单；super admin 是产品核心 / 客户系统全功能审阅和配置账号，在前端不被 active revision pages、actions 或 field policy 收窄，用于查看当前系统所有已登记业务能力做到哪里；active revision 缺失页面清单或页面清单无有效 key 时，后端不回退 RBAC 全量页面，正式普通账号前端也不得退回 RBAC-only 放开入口；动作投影通过共享权限 helper 只在普通账号 RBAC 已允许的基础上继续收窄，字段策略先接入客户 / 供应商主数据和销售订单列表 / 导出列可见性；销售订单明细字段草案仍只作导入 / 客户评审线索，未接入运行时页面前不得发布为 `sales_order_items.default`；若 active revision 返回空页面清单，或 effective session 同步失败且没有可复用缓存投影，正式普通账号必须阻止业务 Outlet 渲染并提示复核当前客户有效配置；本地开发态诊断路径只影响前端排障可见性，super admin 产品核心看全只影响导航、按钮和字段可见性，不扩大后端 RBAC、动作权限或 Workflow / Fact usecase；`formal-frontend-customer-config-boundary.test.mjs` 锁住源码边界，`adminProfileSync.test.mjs` 锁住正式普通账号交集收窄、隐藏 URL 跳转、local-dev 直达 URL 诊断不放开普通账号菜单、active revision 空页面清单不回退 RBAC-only、super admin 产品核心看全，以及模块 disabled 后端投影隐藏业务页时正式普通账号需要跳转；`style:l1` 的 `erp-effective-session-super-admin-product-core`、`erp-effective-session-direct-url-local-dev-diagnostic`、`erp-effective-session-sync-failure-local-dev-diagnostic`、`erp-effective-session-empty-pages-local-dev-diagnostic`、`erp-no-visible-menu-blocks-outlet` 和 `erp-effective-session-action-projection-business-pages` 锁住本地 DEV 诊断、super admin 产品核心业务页可见、无可见菜单空态，以及普通账号菜单按 active pages 投影、页面可见但 `effective_session.actions=[]` 时写按钮禁用；这些前端场景不能扩大后端 RBAC 权限，也不能代替领域 usecase、JSON-RPC 授权或后端 action permission。

22. 客户配置 publish / activate / rollback 新增控制面审计：同一事务写入脱敏 `runtime_audit_events`，记录 actor、customer / revision、config hash、计数、状态和回滚目标；审计日志可读为“客户配置发布 / 激活 / 回滚”，不保存 compiled snapshot、raw 客户配置或 secret。
23. 客户前端配置 overlay 补专门测试：`apply-customer-web-config` 已由 Dockerfile 引用和 `.dockerignore` 放行，并新增脚本测试锁住 customer-config.js 与 assets 复制行为，避免客户私有化构建时只靠字符串检查。
24. Workflow 新增后端只读解释合同：`explain_action_access` 返回完成 / 阻塞 / 退回 / 催办的权限、归属、终态原因、owner role、当前 action 可见 owner role 集合、candidate owner role keys、责任池命中状态和 work pool entitlement scope 命中状态，`explain_task_assignment` 返回当前管理员与任务 owner / assignee / 责任池 / PMC / 老板 / super admin 边界、各动作 required permission、各动作 candidate owner role keys 以及各动作 work pool scope 命中映射；桌面任务看板、Workflow V1 页面、业务页协同 Drawer 和岗位任务端提交前预检已开始消费后端 explain，仍不写 Workflow 事件或 Fact，也不暴露 entitlement ID 或全局候选人清单。

26. Workflow 任务可见范围接入 active customer config 的责任池成员关系：非 super admin 的 `list_tasks`、任务动作和只读 explain 会把当前账号角色、assignee 和当前 active revision 中命中的 `work_pool_memberships.role_key` 合并成服务端可见 owner role 集合；由责任池扩出的 owner role 必须在同一 active revision 下由同一条 entitlement 同时命中当前任务场景所需 workflow task capability 和当前 customer scope，列表按同 scope 的 `workflow.task.read`，完成 / 阻塞 / 退回 / 催办按同 scope 的对应 action permission 过滤，不能用一个角色的动作权限、另一个责任池成员关系或另一个 customer scope 的 entitlement 拼接处理任务；请求参数不能越权扩大可见范围；`workflow_tasks.owner_pool_key / required_capability_key / config_revision` 已作为可空 runtime 解释锚点接入，新任务默认写责任池、动作能力和配置版本线索，旧任务允许为空，当前授权过滤仍以 owner role / assignee / active revision entitlement scope 为主。
27. engineering 最小可用入口闭环补齐：工程角色已有后端 RBAC、BOM / 工序 / 产品资料权限和 boss 审批后派生工程资料任务；本轮补齐 `/m/engineering/tasks` 岗位入口、`mobile.engineering.access` 前端权限映射、`demo_engineering` 试用账号 seed / RBAC 核对、行业模板默认角色、customer package catalog / runtime manifest 中的 engineering 责任池、角色画像和岗位端 access entitlement，以及 JSON-RPC 层 engineering 账号列出并完成派生任务的合同测试。

## 领域命令进入条件

补充口径：ProcessInstance / ProcessNodeInstance 最小运行时当前负责持久化流程版本、配置版本、业务引用、节点类型、责任池、所需能力、状态和幂等键；WorkflowTask 当前新增可空 `process_instance_id / process_node_instance_id` 追踪锚点。ProcessRuntime 可显式 `StartProcessInstance` 启动 active 流程实例的首个 `waiting` 节点，启动后只按该节点类型复用现有处理边界：`human_task / approval` 创建 linked WorkflowTask，`domain_command / wait_event` 进入 active 等待显式执行 / 唤醒，`end` 完成流程实例。ProcessRuntime 可显式从已 `active` 且 `expected_version` 匹配的 `human_task / approval` 节点创建 linked WorkflowTask，未显式传 `owner_role_key` 时只接受 active customer config 解析出的唯一候选 owner role，相同 `task_code` 的同一节点重试会返回已有任务，并在 linked WorkflowTask 到达 `done` 后完成当前节点；受控 `complete_task_action` 成功完成 linked WorkflowTask 后会触发这条节点完成路径，旧 `update_task_status` 已退出运行时，不再作为流程节点完成触发路径。当前最小运行时还会在完成当前节点后，按路由策略激活下一组 `waiting` 节点：默认走同一流程实例内节点创建顺序；如果当前节点声明了 `policy_snapshot.branch_policy_key`，则必须由已注册的命名 policy handler 返回下一节点 key；如果声明了 `policy_snapshot.fan_out_node_keys`，则只激活这些明确 node key；如果声明了 `policy_snapshot.join_node_key / join_policy / join_source_node_keys`，则按 join-all / join-any 判断来源节点是否已完成；如果声明了 `policy_snapshot.return_to_node_key / return_outcomes / return_max_attempts`，则只在 outcome 命中时创建并激活目标 node key 的下一 attempt。所有路由都只作用于同一 ProcessInstance；未注册 policy、空目标、目标不存在、目标重复、目标不是允许状态、returnTo 目标最新 attempt 未 settled 或超过 attempt 上限都会拒绝或 no-op，且不会解析自由表达式、客户 JS / SQL 或任意脚本。如果刚激活的目标节点是 `human_task` 或 `approval`，会创建 linked WorkflowTask；如果刚激活的是 `end`，会完成 end 节点并把 ProcessInstance 标记为 completed，显式 end 节点才会完成 ProcessInstance。`domain_command` 只开放 usecase 层显式 `ExecuteDomainCommandNode`：节点必须 active、版本匹配、`policy_snapshot.command_key` 已声明，调用方必须提供幂等键且 handler 已注册，handler 成功后才完成节点并复用当前路由推进；未注册 handler、command key 不匹配或缺幂等键都不完成节点。`wait_event` 只开放 usecase 层显式 `WakeProcessWaitEventNode`：节点必须 active、版本匹配、`policy_snapshot.event_key` 已声明，调用方必须提供事件 key 和幂等键，事件 key 匹配后才完成节点并复用当前路由推进。`blocked / due_at` 只开放 usecase 层显式阻塞：active 节点可通过 `BlockProcessNodeInstance` 阻塞，带 `due_at` 的 active 节点可在到期后通过 `EscalateDueProcessNode` 阻塞，二者都只把节点和流程置为 blocked，不推进后续节点、不创建任务、不写 completed_at。这不把流程创建等同于启动，不提供后台 scheduler，不扫描流程定义、不自动扫描 overdue、不自动跳过或 settle 非选中分支、不为 `domain_command / wait_event / end` 创建 WorkflowTask、不让 task action 自动调用 handler 或唤醒事件，也不改变 Workflow / Fact 边界。后续让事件订阅器、提醒升级通知或具体 `domain_command` 调用领域 usecase 时，必须另拆阶段并补 JSON-RPC/RBAC/幂等/审计测试。

第 7 项不能用“看到任务完成按钮”直接推进。当前这些任务仍明确是 Workflow-only，不得在本轮改成自动过账：

| 任务 / 场景 | 当前允许动作 | 当前禁止动作 |
| --- | --- | --- |
| `shipment_release` | 写 `shipping_released` 协同状态和后续财务提醒 | 自动调用出货、库存扣减、应收或开票事实 |
| `warehouse_inbound` / 采购入库协同 | 推进入库协同状态或后续质检 / 财务提醒 | 自动调用 `post_purchase_receipt` 或写库存流水 |
| `quality` / IQC / 成品质检协同 | 推进质检协同和异常 / 返工任务 | 自动判定 `quality_inspections` 或生成采购退货事实 |
| `engineering_data` | 补齐产品资料、工序和 BOM 协同任务 | 自动生成采购需求、生产任务、成本或库存事实 |
| 财务登记 / 对账提醒任务 | 推进 Workflow 财务提醒和下一步协同任务 | 自动写应收、应付、发票、付款或总账事实 |

后续如果要把某个任务升级为领域命令入口，必须先在对应业务域评审 source-of-truth 字段、状态机、幂等键、RBAC、审计、重复提交、取消 / 冲正和测试范围，再由该领域 usecase 写事实；不能在 `WorkflowUsecase` 或前端任务端直接写事实。

## 暂不做

- 把 yoyoosun 客户资料写成 Product Core 规则。
- 让前端本地拼装业务事实、库存、财务、出货或质检落账。
