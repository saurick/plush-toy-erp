# 正式菜单运行时实施拆分清单 / Formal Menu Runtime Implementation Plan v1

- 文档类型：产品实现拆分计划
- 状态：已部分实施；当前 runtime 真源仍以代码、测试和 `docs/当前真源与交接顺序.md` 为准
- 对应计划：`docs/product/正式产品入口与菜单配置计划.md`
- 对应评审表：`docs/product/菜单映射评审表.md`
- 对应原型：`docs/product/prototypes/formal-menu-candidate-v1/index.html`

## 0. 文档目的

本文把正式菜单候选、当前运行时菜单和后端 RBAC 菜单拆成可执行批次，避免把 12 个候选主入口或 51 个内部覆盖细项一次性写进左侧导航。

2026-06-13 已执行一轮运行时吸收：前端默认菜单、yoyoosun 客户菜单、毛绒行业候选模板、后端内置菜单、菜单权限预设、路由和正式页面壳已同步。该轮不新增权限码、API、schema、migration、WorkflowUsecase 或 Fact usecase，不恢复旧 `business_records`。

## 1. 当前运行时真源

| 面 | 当前文件 | 作用 | 实施注意 |
| --- | --- | --- | --- |
| 前端默认导航 | `web/src/erp/config/seedData.mjs` | 定义工作台、任务看板、业务看板、打印中心、异常闭环、权限管理等默认入口，并组合业务模块导航 | 改 key / label / path / section 都会影响前端菜单、权限项、测试和截图 |
| 前端业务模块 | `web/src/erp/config/businessModules.mjs` | 定义业务页分组、模块标题、路由、摘要和页面口径 | 改 title / route 会影响侧边栏、权限项、业务页标题和文档 |
| 客户菜单配置 | `web/src/erp/config/customerMenuConfig.mjs`、`config/customers/yoyoosun/menuConfig.mjs` | 控制客户侧显示、隐藏、排序和文案 | 只影响入口可见性，不是 RBAC 或事实层安全边界 |
| 前端菜单权限项 | `web/src/erp/config/menuPermissions.mjs` | 从当前导航生成菜单权限分组和角色预设 | 改菜单会影响 `ERP_MENU_PERMISSION_GROUPS`、默认角色预设和旧路径归一化 |
| 后端内置菜单 | `server/internal/biz/rbac.go` 的 `BuiltinAdminMenus()` | 登录返回 menus、RBAC 菜单可见性和 menu_options | 必须和前端正式入口保持 path / label / required permissions 一致 |
| 后端菜单测试 | `server/internal/biz/rbac_test.go`、`server/internal/service/jsonrpc_*` | 锁定管理员可见菜单、旧入口退出和 JSON-RPC 返回结构 | 前端菜单变更若影响后端内置菜单，必须同步这些测试 |

当前实现已经不是“只有前端菜单”。如果只改 `seedData.mjs` 或客户配置而不改后端 `BuiltinAdminMenus()`，登录返回菜单、权限管理选项和前端导航会漂移。

当前仍是正式入口壳的入口只剩：`生产排程`、`生产异常`、`出货放行`。这些入口只恢复页面结构和交互，不读取或写入旧 `business_records`，也不代表对应领域写入 API 已完成。`产品档案`、`BOM 管理`、`采购订单`、`入库管理`、`来料质检`、`库存台账`、`出货单`、`委外订单`、`生产进度`、`出库管理`、`对账管理`、`应付管理`、`应收管理` 和 `发票管理` 已有对应 V1 或收窄 V1 页面，仍需按各自边界区分主数据、工程资料、采购承诺、采购入库事实、质检事实、含 ACTIVE 预留可用量 read model 的库存只读台账、出货事实、委外加工合同源单、生产 operational facts、库存预留和财务 facts。

## 3. 实施批次

| 批次 | 目标 | 允许修改 | 禁止修改 | 验收重点 |
| --- | --- | --- | --- | --- |
| R0 前后端菜单真源一致性 | 只把后端内置菜单补齐到当前前端默认导航，修正明显标签漂移，不改页面结构 | `server/internal/biz/rbac.go`、`server/internal/biz/rbac_test.go`、必要 `server/internal/service/jsonrpc_*` 测试、本文和 `progress.md` | 不改前端导航、客户菜单、路由、权限码、schema、migration、Workflow / Fact | `BuiltinAdminMenus()` 包含前端默认正式入口；旧业务记录入口仍不存在；无权限 / disabled / super_admin 边界不退化 |
| R1 低风险命名清理 | 只做当前已实现入口的稳定命名，例如 `产品` -> `产品档案`、`材料 BOM` -> `BOM 管理`；不新增入口 | `businessModules.mjs`、`seedData.mjs`、`rbac.go`、前后端菜单测试、正式文档 | 不改 route，不把 51 个细项铺成菜单，不新增 API | 前端导航、后端 menus、权限标签、yoyoosun 菜单和文档口径一致 |
| R2 客户显隐决策 | 决定 yoyoosun 是否恢复 `任务看板`、`异常 / 阻塞闭环` 或继续隐藏 | `config/customers/yoyoosun/menuConfig.mjs`、客户培训 / 验收清单、客户菜单测试 | 不把客户显隐写成 Product Core，不改变 RBAC 安全边界 | 隐藏后仍有任务处理替代路径；恢复后角色权限和培训说明一致 |
| R3 页面内 tab / 动作吸收 | 把库存流水、批次、质检记录、财务明细等内部细项放进页面内 tab、筛选或动作 | 对应页面、共享组件、前端测试、必要文档 | 不新增事实语义，不用前端本地派生后端事实 | 默认态、交互态、恢复态、暗色、移动视口和相邻区域回归 |
| R4 新能力入口评审 | 报表、数据导入、审计日志、客户配置 UI、备份恢复 UI 等新入口 | 先补正式设计、API / RBAC / schema / migration 评审、测试策略 | 不直接加左侧菜单，不抢跑 SaaS、多租户、license、客户工单 | 能力边界、权限、数据真源、运维风险和回滚路径明确 |

R0 是建议的第一批真实代码任务。它不改变当前用户看到的前端默认导航，只修后端菜单真源与当前前端正式入口的明显不一致。

## 4. R0 建议实现边界

R0 只做后端内置菜单与当前前端默认导航对齐：

1. 把后端 `global-dashboard` label 从 `任务看板` 改为 `工作台`。
2. 在后端 `BuiltinAdminMenus()` 增加 `task-board`，path 为 `/erp/task-board`，label 为 `任务看板`，权限先沿用 `PermissionERPDashboardRead`。
3. 把后端 `business-dashboard` label 从 `业务总览` 改为 `业务看板`。
4. 在后端 `BuiltinAdminMenus()` 增加 `exception-flow`，path 为 `/erp/operations/exceptions`，label 为 `异常 / 阻塞闭环`，权限先沿用 `PermissionERPDashboardRead` 或当前异常闭环实际所需权限；实施前需在代码中确认现有页面动作权限。
5. 把后端 `receivables` label 对齐为 `应收/开票登记`。
6. 把后端 `invoices` label 对齐为 `发票登记`。

R0 不做：

- 不改 `web/src/erp/config/seedData.mjs`、`businessModules.mjs` 或 yoyoosun 客户菜单。
- 不新增路由、页面、权限码、API、schema、migration。
- 不把任务看板或异常闭环写成 Fact 入口。
- 不改变客户菜单显隐；yoyoosun 仍可隐藏任务看板和异常闭环。

## 6. 停止条件

遇到以下情况必须停止并重新评审，不继续顺手扩范围：

- 需要新增或修改权限码。
- 需要改路由、页面路径或旧路径兼容策略。
- 需要新增 API、schema、migration 或 seed。
- 需要改变 yoyoosun 菜单显隐。
- 需要把报表、数据导入、审计日志、客户配置 UI、备份恢复 UI 等候选能力变成正式入口。
- 发现 active workflow task 只能通过被隐藏或将删除的入口处理。

## 7. 剩余风险

- 本文是实施拆分清单，不是自动化守卫。实际代码仍可能在前后端菜单之间产生漂移，后续可考虑补前后端菜单快照比对测试。
- R0 中 `exception-flow` 使用哪个后端 required permission 需要在实现时回到当前页面动作和 RBAC 代码确认，不能只按文档猜。
- R1 之后的命名清理会影响用户培训和截图证据，应和客户试用说明同步，不应在同轮混进 R0。
