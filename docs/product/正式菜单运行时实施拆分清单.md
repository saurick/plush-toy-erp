## 1. 当前运行时真源

| 面 | 当前文件 | 作用 | 实施注意 |
| --- | --- | --- | --- |
| 前端默认导航 | `web/src/erp/config/seedData.mjs` | 定义工作台、任务看板、业务看板、打印中心、异常闭环、权限管理等默认入口，并组合业务模块导航 | 改 key / label / path / section 都会影响前端菜单、权限项、测试和截图 |
| 前端业务模块 | `web/src/erp/config/businessModules.mjs` | 定义业务页分组、模块标题、路由、摘要和页面口径 | 改 title / route 会影响侧边栏、权限项、业务页标题和文档 |
| 客户菜单配置 | `web/src/erp/config/customerMenuConfig.mjs`、`web/public/customer-config.js`、`config/customers/yoyoosun/customer-config.example.js`、`config/customers/yoyoosun/menuConfig.mjs` | 默认产品包只带中性占位；客户部署注入 `window.__PLUSH_ERP_CUSTOMER_CONFIG__` 控制客户侧显示、隐藏、排序和文案 | 只影响入口可见性，不是 RBAC 或事实层安全边界；不把具体客户包静态打进默认前端产物 |
| 前端菜单权限项 | `web/src/erp/config/menuPermissions.mjs` | 从当前导航生成菜单权限分组和角色预设 | 改菜单会影响 `ERP_MENU_PERMISSION_GROUPS`、默认角色预设和旧路径归一化 |
| 后端内置菜单 | `server/internal/biz/rbac.go` 的 `BuiltinAdminMenus()` | 登录返回 menus、RBAC 菜单可见性和 menu_options | 必须和前端正式入口保持 path / label / required permissions 一致 |
| 后端菜单测试 | `server/internal/biz/rbac_test.go`、`server/internal/service/jsonrpc_*` | 锁定管理员可见菜单、旧入口退出和 JSON-RPC 返回结构 | 前端菜单变更若影响后端内置菜单，必须同步这些测试 |

当前实现已经不是“只有前端菜单”。如果只改 `seedData.mjs` 或客户配置而不改后端 `BuiltinAdminMenus()`，登录返回菜单、权限管理选项和前端导航会漂移。

运行时已不再保留正式入口壳。`生产排程`、`生产异常`、`出货放行` 由 Workflow 协同页承接任务读取和受控处理，不代表对应领域事实写入已完成；其余正式入口已有对应 V1 或收窄 V1 页面，页面文案和动作边界按主数据、工程资料、采购承诺、采购入库、质检、库存、出货、加工合同、生产和财务事实分清。

## 3. 行为接入门禁

页面治理已经完成当前可见入口、动作文案和权限只读态收口；后续只有在新增真实行为或正式入口时才进入单页闭环：

1. 新能力入口：报表、数据导入、审计日志、客户配置 UI、系统配置 UI、备份恢复 UI 等不得直接进入左侧菜单；进入前必须先有设计、API / RBAC / schema / migration 影响评审、测试策略、数据真源和回滚路径。
2. 行为接入升级：库存写入、完整财务核销、税控、总账、Workflow 协同页升级为领域事实写入、打印、删除或业务数据导出，都必须按具体页面单独闭环；不能用页面存在、菜单可见或 Workflow 协同任务完成来替代事实能力验收。

## 4. 已完成检查项的边界

前后端菜单真源一致性检查当前只做后端内置菜单与当前前端默认导航对齐：

1. 后端 `global-dashboard` label 为 `工作台`，path 为 `/erp/dashboard`。
2. 后端 `task-board` path 为 `/erp/task-board`，label 为 `任务看板`，required permission 为 `workflow.task.read`。
3. 后端 `business-dashboard` label 为 `业务看板`，path 为 `/erp/business-dashboard`。
4. 后端 `exception-flow` path 为 `/erp/operations/exceptions`，label 为 `异常 / 阻塞闭环`，required permission 为 `workflow.task.read`。
5. 后端 `receivables` label 为 `应收管理`。
6. 后端 `invoices` label 为 `发票管理`。

已完成检查项不做：

- 不用 seed 或客户菜单新增未评审业务入口；`businessModules.mjs` 只同步当前已实现的来源动作说明。
- 不新增路由、页面、权限码、API、schema、migration。
- 不把任务看板或异常闭环写成 Fact 入口。
- 不改变客户菜单显隐；当前 yoyoosun 客户菜单显隐源仍以 `config/customers/yoyoosun/menuConfig.mjs` 为准，部署运行时通过 `customer-config.js` 注入。

页面文案和动作边界已完成本轮收口：

1. 库存台账：余额、批次、流水 tab 保持只读台账语义；页面动作只允许查看关联，不提供库存写入、盘点调整、出库确认、批次状态变更或预留自动消耗。
2. 财务事实页：应收、应付、发票、单笔核对入口只表达当前收窄的 `finance_facts`；创建动作位于已出货出货单、已过账采购入库、质量合格的委外回货或已过账财务事实来源页，财务列表不提供无来源通用新增，`PAYMENT` 保持只读，也不承诺完整核销、税控、查验、纳税申报、总账或多账簿能力。
3. Workflow V1 协同页：`生产排程`、`生产异常`、`出货放行` 只读取、创建、完成、阻塞和催办 Workflow 协同任务；页面明确不提供删除、打印、业务数据导出或生产 / 出货 / 库存 / 财务领域事实写入。`shipping_released != shipped`，`出货放行` 不等于 `SHIPPED`。

## 6. 停止条件

遇到以下情况必须停止并重新评审，不继续顺手扩范围：

- 需要新增或修改权限码。
- 需要改路由、页面路径或旧路径退出策略。
- 需要新增 API、schema、migration 或 seed。
- 需要改变 yoyoosun 菜单显隐。
- 需要把报表、数据导入、审计日志、客户配置 UI、备份恢复 UI 等候选能力变成正式入口。
- 发现 active workflow task 只能通过被隐藏或将删除的入口处理。
