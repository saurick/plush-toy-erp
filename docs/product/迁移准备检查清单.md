# 迁移准备检查清单 / Migration Readiness Checklist

## 目的

本清单用于真正落 Ent schema 前拦截重复设计、客户耦合、Workflow / Fact 混淆和迁移风险。任何 V1 / V2 schema 任务必须先逐项过表。

如果无法证明不重复已有真源，则停止。
如果影响库存 / 出货 / 财务事实但没有架构评审，则停止。
如果 schema 草案包含多租户运行时字段，则停止。
如果只是 永绅 yoyoosun 客户样本字段但被写成 Product Core，则停止。

## 检查清单 / Checklist

| Check | Question | Required Evidence | Stop If |
|---|---|---|---|
| Existing schema | 当前是否已有等价 schema 对象？ | `server/internal/data/model/schema` 文件和 `docs/当前真源与交接顺序.md` 对照 | 与 `products/materials/units/warehouses/inventory/bom/purchase/quality` 重复 |
| Existing migration | 是否已有对应 Atlas migration？ | `server/internal/data/model/migrate` 和 `make migrate_status` 计划 | migration 已存在但草案另建同义表 |
| Existing usecase | 是否已有 usecase / repo 真源？ | `server/internal/biz`、`server/internal/data` 搜索 | 新对象绕开已有 usecase |
| business_records impact | 是否影响 `business_records` 兼容层？ | 迁移来源、保留快照、双写退出计划 | 两套页面继续同时写同一业务真源 |
| Data migration | 是否需要从旧快照迁移？ | source table、字段映射、失败样本和回滚方案 | 无法说明迁移来源或无法保留历史 |
| Backfill | 是否需要历史数据回补？ | backfill SQL / Go job 设计、幂等键、dry-run 计划 | 缺值会导致历史数据错误显示 |
| Unique index | 是否需要唯一索引？ | code / no / source_line_no / idempotency_key 规则 | 业务唯一性只能靠前端校验 |
| State machine | 是否需要状态机？ | 状态 key、允许转移、终态保护 | 一个 `status` 同时承接 workflow、fact、derived |
| Idempotency | 是否需要幂等键？ | 幂等 key 生成规则和重复提交测试 | 重复提交可能重复扣库存、开票、生成事实 |
| Inventory impact | 是否影响库存？ | InventoryUsecase 评审和 `inventory_txns` 关系 | 没有库存架构评审却写库存事实 |
| Shipment impact | 是否影响出货？ | ShipmentUsecase 评审和 `shipping_released != shipped` 证明 | 从 workflow done 或 order status 伪造 shipped |
| Finance impact | 是否影响财务？ | Finance 评审，AR/AP/invoice/payment 生成时机 | 从 purchase_order 或 shipping_released 直接生成财务事实 |
| RBAC | 是否需要权限码？ | `rbac.go` 设计、动作权限、数据范围 | 只靠菜单隐藏或前端按钮控制 |
| API | 是否需要 API？ | JSON-RPC / service DTO、错误码、权限守卫计划 | UI 直接拼事实或绕过后端 usecase |
| UI | 是否需要 UI？ | 页面入口、状态文案、默认/交互/恢复态回归计划 | UI 文案和 canonical status 混用 |
| Seed / demo / import | 是否需要 seed、demo 或导入？ | seed 范围、debug 标记、导入 dry-run | seed 被当真实客户事实或 Product Core |
| Customer config | 是否需要客户配置？ | config/customer draft、字段显示/必填/编号规则 | 永绅 yoyoosun 样本字段写死进 core schema |
| Tests | 需要哪些测试？ | biz/data/API/UI 对应命令和用例清单 | 只有文档或截图，没有可运行验证 |
| yoyoosun material classification | yoyoosun 资料分类是否明确？ | Product Core / Industry Template / Customer Config / Customer Material 分类表 | 只在永绅 yoyoosun 样本出现却进入 V1 必填 |
| current-source-of-truth | 是否需要更新真源索引？ | `docs/当前真源与交接顺序.md` diff | 实现状态变化但真源索引仍说 draft |
| domain-model-v1 | 是否需要更新领域模型？ | `docs/product/领域模型第一版.md` diff | 表状态、分类或影响面变化未同步 |
| release gates | 是否需要更新 release gates？ | `docs/product/发布门禁.md` diff | 新能力缺少 DoD / stop conditions |
| Rollback | migration 回滚计划是什么？ | rollback / forward-fix / backup 方案 | 无法安全回滚或前滚修复 |

## 合格标准

1. 每张表都有明确 category：MasterData / Source Document / Fact / Derived / Workflow / Config。
2. 每个状态字段都能说明属于单据生命周期、事实动作状态、派生状态还是系统横切状态。
3. 每个会影响库存、出货或财务的对象都有对应架构评审。
4. 每个 source snapshot 到正式表的迁移都有缺值、残值、重复主体和历史回显方案。
5. 每个 usecase 都明确按顺序校验：

```text
Feature Flag
RBAC
Data Scope
State Machine
Business Rule
Idempotency
Audit Log
```

6. 菜单权限不等于动作权限，前端隐藏按钮不是安全边界。
7. `business_records` 的兼容期、退出路径和只读 / 迁移策略有文档记录。

## 停止条件 / Stop Conditions

- 无法证明不重复已有真源。
- 影响库存 / 出货 / 财务事实但没有对应架构评审。
- schema 草案包含当前阶段禁止的多租户运行时字段。
- 永绅 yoyoosun 客户样本字段被写成 Product Core 必填。
- `shipping_released` 被当作 `shipped`。
- `purchase_order` 被当作 `purchase_receipt`。
- `shipped_quantity`、`received_quantity`、AR/AP 等派生值没有事实来源。
- 只改 schema，没有 migration、repo、usecase、权限、测试和文档计划。
