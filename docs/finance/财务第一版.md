# 财务 v1 / Finance v1

## 当前定位

财务 v1 使用 `finance_facts` 保存来源可追溯的业务财务事实，不建设总账、会计凭证、银行支付、税控或多单据核销平台。Workflow 只负责协同、导航和关联引用；任务完成不能替代应收、应付、发票或核对事实。

财务列表页负责查询、详情、附件、审计和允许的状态动作。正式创建入口位于业务源单页面，不提供无来源通用新增，也不让用户填写 `source_type`、内部 ID 或幂等键。

| 类型 | 正式来源与入口 | 当前边界 |
| --- | --- | --- |
| 应收 `RECEIVABLE` | 已出货的出货单 | 客户、币种和金额由出货快照派生；收款分类固定为应收款，账期冻结销售订单的精确天数；同一出货单只能有一笔有效应收 |
| 应付 `PAYABLE` | 已过账采购入库；质量已合格或让步接收的已过账委外回货 | 供应商和金额由来源派生；采购退货、入库调整影响采购应付金额；委外不合格或质检未完成时拒绝生成 |
| 发票 `INVOICE` | 已出货的出货单 | 创建时必须明确选择发票类别；仅表示业务开票记录，不表示税控开票完成；同一出货单只能有一笔有效发票 |
| 单笔核对 `RECONCILIATION` | 已过账的应收、应付或发票 | 往来方、币种和金额与来源一致；不是多单合并对账或收付款核销 |
| 收付款 `PAYMENT` | 无正式来源合同 | 仅允许具有报表读取权限的用户查看历史记录，不开放创建、过账、结清或取消 |

## 来源、金额与唯一性

- 出货来源必须是 `SHIPPED`，财务放行或 Workflow task done 不能生成应收或发票。
- 应收只读取销售订单 `payment_term_days`，不把“银行转账”等自由文本付款方式猜成账期。0 / 30 / 45 天保留标准码和精确天数，其他非负天数只保留精确天数并显示为自定义账期；销售订单未填写账期时拒绝新建应收。
- 采购应付来源必须是 `POSTED` 采购入库，并具有稳定 `supplier_id`；历史记录无法从唯一采购订单确定供应商时不猜测回填，也不能自动生成应付。
- 委外应付来源必须是 `POSTED` 回货事实，并且恰有一张非取消质检达到 `PASSED` 且结果为 `PASS` 或 `CONCESSION`；缺质检、质检中、不合格或重复有效质检均 fail closed。
- 单笔核对来源必须是已过账、未取消的应收、应付或发票；不允许从 `PAYMENT` 或另一条核对记录继续派生。
- `fact_type + source_type + source_id` 的有效记录唯一约束与事务内复核共同阻止调用者更换幂等键重复生成。
- 来源存在有效财务或质检下游依赖时，源出货、采购入库、委外回货或源财务事实不能直接取消。

## 状态与权限

财务事实状态为 `DRAFT / POSTED / SETTLED / CANCELLED`。创建、过账、结清和取消均由后端按事实类型执行精确权限；前端隐藏按钮不是安全边界。

- 应收、应付、发票和单笔核对使用各自的 read / confirm 权限。
- `PAYMENT` 不继承这些确认权限，只通过报表读取范围只读展示。
- 发票页面不提供“结清”；`SETTLED` 不能解释为税控、银行到账或总账已记账。
- 取消需要原因并保留操作者、时间和来源审计，不删除原事实。

## 页面显示

- 应收：客户、出货来源、金额、币种、收款分类、精确账期和状态；不显示发票类别。
- 应付：供应商、采购入库或委外回货来源、金额、币种和状态；当前没有账期或发票类别真源，不显示这些列。
- 发票：客户、出货来源、金额、发票类别、附件和状态。
- 单笔核对：往来方、被核对记录、金额、币种、差异说明和状态。

适用字段在历史记录中为空时显示“历史未记录”；未知枚举显示“待核对”。非标准账期显示“自定义账期 / N 天”。非取消记录的取消记录列显示 `-`；已取消记录必须完整显示取消时间、操作者和原因。

页面展示业务编号与名称，不显示数据库 ID、权限 key、原始 source 字段、API 方法或幂等键。

## 字段测试覆盖门槛

这里的百分比是业务合同和验收数据覆盖率，不是用单一代码行覆盖率代替业务正确性。

| 层级 | 必须达到的门槛 |
| --- | --- |
| Biz / Repo | 出货应收的直接来源和流程来源 2 / 2；缺账期、0 / 30 / 45 天、自定义天数、发票类别必填与越界字段拒绝场景 100% |
| JSON-RPC | 应收禁止发票类别、发票类别必填、合法类别透传、缺来源账期错误文案等接口合同场景 100% |
| 页面投影 | 四类财务页面的适用列矩阵 4 / 4；已知值、历史空值、未知值和自定义账期渲染场景 100% |
| 试用数据与 readiness | 本批全部财务记录逐行执行适用字段与取消审计合同，必须 100%，任何无法解释的空值直接失败 |
| 浏览器 | 四页表头矩阵和绑定代表行 4 / 4；非取消行 `-` 与取消行完整审计均须有证据 |
| 目标环境 / UAT | 绑定版本、数据库和报告后执行清单；必测项 100%完成，阻塞项不得记为通过 |

代码覆盖率只作辅助趋势：本切片的变更逻辑建议行覆盖不低于 90%、分支覆盖不低于 85%；即使达到该数字，只要上述业务场景缺一项仍不能收口。

## 当前不做

- 银行账户、收付款申请、银行流水、支付凭证和核销行。
- 多条应收 / 应付合并对账、差异明细、自动核销。
- 总账、会计科目、凭证、多账簿和复杂成本会计。
- 税控开票、查验、红冲、纳税申报和自动生成真实发票文件。
- 在财务列表恢复通用新增表单或动态表单引擎。

如需开放真实收付款或多单据对账，必须先完成独立 header / line 数据模型、银行或支付来源合同、权限、审计、迁移和客户业务口径评审，不能复用自由填写的 `PAYMENT` 记录冒充。
