# 角色权限矩阵 v1 / Role Permission Matrix v1

## 当前口径

当前项目权限模型是标准 RBAC：用户通过 `admin_user_roles` 绑定角色，角色通过 `role_permissions` 获得权限码，后端接口、桌面菜单和岗位任务端入口统一消费 permission code。`admin_users.level`、`admin_users.menu_permissions`、`admin_users.mobile_role_permissions` 不再作为权限来源。

权限码和内置角色真源在 `/Users/simon/projects/plush-toy-erp/server/internal/biz/rbac.go`。登录和 `auth.me` 返回当前管理员、角色、权限码和后端推导出的菜单；前端只负责按这些结果展示入口，不能把菜单隐藏当成安全边界。

菜单注册表显式区分 `RequiredAny` 和 `RequiredAll`：`RequiredAny` 中具备任意一项即可满足共享页面入口，`RequiredAll` 必须全部具备；两组同时存在时必须同时满足。当前多岗位共享页面和产品 / SKU 聚合维护页都按任一相关读取能力开放入口，页面内部 API 和动作继续各自鉴权；当前没有必须组合权限才能显示的正式菜单，但 `RequiredAll` 合同和测试已经保留，后续新增组合页面不得退回模糊数组。

岗位任务端角色入口只认 `mobile.<role>.access` 权限码，`auth.me.roles` 用于身份展示、任务归属匹配和审计语义，不能替代入口权限。

`is_super_admin=true` 的账号拥有全部权限，用于初始化和紧急管理；普通管理员必须通过角色获得权限。普通管理员不能随意修改 super admin。

普通管理员的权限只来自未禁用角色；禁用角色不再授予权限。当前 `permissions` 表不提供独立 disabled 字段，权限停用方式是从角色权限中移除，或从 `rbac.go` 真源移除后由后端归一化丢弃，不能再用旧字段或历史权限码推导菜单、岗位任务端入口或接口权限。

workflow 任务处理有两层校验：RBAC 只判断“能不能做这类动作”，业务归属继续由 `owner_role_key`、`assignee_id`、`task_status_key` 判断“这条任务是不是你该处理”。受控动作 API 中，老板审批完成要求 `workflow.task.approve`，其他完成要求 `workflow.task.complete`，退回要求 `workflow.task.reject`，阻塞和催办要求 `workflow.task.update`。`workflow.task.update / complete / approve / reject` 都不能绕过任务归属；boss / pmc 的查看、关注、催办能力也不等于可以替销售、采购、仓库、品质、财务完成业务事实。

跟单角色如果甲方没有，不新增独立角色；业务跟进由 `sales` 或 `pmc` 承担。

## 权限分层与当前完成度

权限中心不建立独立页面授权，也不把所有业务字段做成角色勾选项。页面入口从客户模块状态和角色功能权限推导；字段权限只用于后端能够统一保护的敏感字段组。

| 层级 | 解决的问题 | 当前状态 |
| --- | --- | --- |
| 客户能力配置 | 当前客户是否启用某个业务模块 | 已由 customer config revision 发布、激活和回滚控制 |
| 功能权限 | 当前角色能查看、创建、修改或审批什么 | 已由 RBAC 权限码控制 |
| 数据范围 | 当前角色能操作哪些客户、订单、仓库或财务数据 | Workflow 任务已有责任岗位 / 指定处理人约束；普通业务资源尚未形成统一可配置策略 |
| 敏感字段 | 成本、结算账户、联系方式等是否可见、可修改、可导出或可打印 | 尚未建立角色级后端策略；现有客户字段显示只控制少量列表 / CSV 列，不是安全权限 |
| 状态与归属 | 当前状态是否允许操作、任务是否属于当前处理人 | 已由领域 usecase、状态和 Workflow 归属共同控制 |
| 页面与按钮 | 给用户展示哪些入口和动作 | 只作上述能力的前端投影，不是授权真源 |

权限中心角色详情按“功能权限 / 数据范围 / 敏感字段 / 权限地图”展示。只有功能权限当前可编辑；数据范围和敏感字段在后端合同闭环前只展示真实完成度，不提供仅隐藏前端内容的开关。权限地图只读解释“功能影响哪些页面、哪类控件以及还受什么条件限制”，由当前权限定义与菜单注册表投影，不反向授予页面或接口权限。

后续数据范围采用按业务资源定义的小型固定策略，例如本人负责、指定仓库或全部；不能把页面默认筛选宣传为数据隔离。后续敏感字段采用稳定领域字段组，并要求查询、修改、导出和打印共同执行；前端隐藏列不能代替后端保护。

| --- | --- | --- |
| 系统 | `system.user.read/create/update/disable/revoke`、`system.user.role.assign` | 管理员账号读取、创建、更新、临时启停、正式注销和独立角色分配 |
| 系统 | `system.role.read`、`system.permission.read`、`system.role.permission.manage` | 查看角色与权限定义、维护允许编辑的业务角色功能权限 |

| 销售订单 | `sales_order.*`、`sales_order_item.*` | 销售订单和订单行 V1 动作权限 |
| 工作流 | `workflow.task.read/create/update/assign/approve/reject/complete` | 任务查询、创建、更新、指派、审批、驳回和完成 |
| 采购入库 / 退货 | `purchase.receipt.read/create`、`purchase.return.read/create` | 当前为后端 usecase/repo 权限码预留；外部 JSON-RPC/API 接入时必须加对应守卫 |
| 岗位任务端 | `mobile.<role>.access` | 岗位任务端角色入口，例如 `mobile.sales.access` |
| 调试 | `debug.seed`、`debug.cleanup`、`debug.business.clear`、`debug.business_chain.run` | 开发 / 测试调试能力 |

## 预设角色

| 角色 | 定位 | 默认权限范围 | 任务处理边界 |
| --- | --- | --- | --- |
| `boss` | 管理层、审批和风险查看 | 全局 read、审批、报表和看板；不默认给 `debug.business.clear` | 可处理老板角色池或指派给自己的任务；关注高风险不等于替其他角色完成事实 |
| `sales` | 销售 / 业务跟进 | 销售链路、客户订单、出货协同、基础 workflow 权限、销售岗位任务端 | 只能处理 `owner_role_key=sales` 或 `assignee_id=自己` 的任务 |
| `purchase` | 采购 | 采购单、采购收货、采购退货、采购异常、采购岗位任务端 | 只能处理 `owner_role_key=purchase` 或 `assignee_id=自己` 的任务 |
| `warehouse` | 仓库 | 库存、入库、出库、盘点、仓库岗位任务端 | 只能处理 `owner_role_key=warehouse` 或 `assignee_id=自己` 的任务 |
| `quality` | 品质 | IQC、成品抽检、异常处理、返工复检、品质岗位任务端 | 只能处理 `owner_role_key=quality` 或 `assignee_id=自己` 的任务 |
| `finance` | 财务 | 应收、应付、收付款、对账、财务报表、财务岗位任务端 | 只能处理 `owner_role_key=finance` 或 `assignee_id=自己` 的任务 |
| `pmc` | 计划和风险跟进 | 计划、进度、风险查看和处理、PMC 岗位任务端 | 可看风险和卡点；不能替生产、仓库、品质、财务完成业务事实 |
| `production` | 生产执行 | 生产计划、生产进度、返工、生产岗位任务端 | 只能处理 `owner_role_key=production` 或 `assignee_id=自己` 的任务 |
| `admin` | 系统管理 | 用户、角色、权限、基础配置 | 不天然拥有业务审批、品质放行、仓库入出库或财务结算事实权 |
| `debug_operator` | 本地开发调试 | `debug.seed`、`debug.cleanup`、`debug.business.clear`、`debug.business_chain.run` | 只允许在 local / dev 环境由超级管理员分配，test / production 都不开放 |

## 操作边界

- 管理员账号状态由 `disabled` 与 `revoked_at` 派生为 `active / suspended / revoked`。临时禁用可恢复；正式注销保留账号和历史身份，不物理删除，也不能通过普通启用恢复。
- 临时禁用、正式注销和重置密码都在同一事务递增 `auth_version` 并注销该账号全部 active `admin_sessions`。审计只记录注销会话数量和稳定原因，不保存 session key；重复设置相同禁用状态保持幂等，不重复递增版本、注销会话或写审计。正式注销还会在同一事务将该账号未完成的个人待办退回原岗位任务池；超级管理员和当前操作人不能被该路径注销。
- `admin` 与 `debug_operator` 是 `role_type=system` 的产品系统角色，权限组合只读；是否可分配仍以服务端针对当前管理员返回的 `assignable_by_current_admin` 为准，不能在前端把 system role 一律判为不可分配。`business_default` / `custom` 同样消费该 metadata。非超级管理员不能改手机号、角色、状态或密码来接管持有系统角色的账号。
- 持有 `system.user.create/update` 与 `system.user.role.assign` 的账号属于高信任凭据管理员（credential administrator）：可以为非系统账号建立或恢复登录凭据并分配业务角色，相关动作必须进入控制面审计。“不天然拥有业务事实权”只描述当前账号自己的 RBAC，不等于绝对无法代管其他账号。若后续要求管理员绝对不能代持业务账号凭据，应单独评审邀请 / 自助激活、一次性恢复和手机号变更验证合同，不能用局部 guard 假装完成隔离。
- 角色记录以 `role_type` 区分 `system / business_default / custom`，以正整数 `version` 承担角色权限 CAS。相关 schema 变更必须通过正式 Ent / Atlas migration 进入 fresh 与 upgrade 链，不能只依赖 dirty worktree 的 schema 或 generated code。

- 旧通用业务记录删除入口已退出；领域事实单据的取消、冲正、删除限制和审计要求由对应 usecase 与表结构定义，不能用旧 `business.record.*` 权限推导。
- 权限中心只展示 `server/internal/biz/rbac.go` 当前内置权限；旧内置权限码从该真源移除后，后端 seed 会清理旧 `permissions` 行和对应 `role_permissions` 绑定，列表接口也不会继续返回历史残留。
- 当前权限中心维护稳定预设角色的功能组合，不开放没有客户配置上限合同的自定义角色创建，也不提供角色物理删除。`system.role.create/update/delete` 已退出正式权限真源，不能用无实现的权限码暗示可用能力。
- 启动 seed 会把 `admin` / `debug_operator` 两个系统角色的代码定义视为产品真源：仅当权限集合真实变化时原子同步并递增版本。`business_default` / `custom` 的人工选择不会被重启覆盖；后续调整业务角色默认权限必须走显式、可审查的数据迁移。
- `admin` JSON-RPC 对每个方法使用精确参数合同，未知字段和未知权限码会整体拒绝，不能静默忽略后返回“保存成功”。
- 可审批必须同时满足权限码、业务归属和状态条件，不能只靠 `admin` 角色推导。
- 可催办必须写入 `workflow_task_events`，不能只是页面红点或备注。
- 可打印和可导出只表示入口权限，导出字段和数据范围仍要按后续数据权限规则收口。
- 移动端只处理当前角色任务和规则允许的风险任务，不展示低代码配置、状态字典或系统权限页面。
