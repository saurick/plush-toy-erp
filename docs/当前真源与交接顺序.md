## 当前产品结构

产品目标是一套 Product Core 服务多个毛绒/轻工制造客户的独立私有化部署。当前不实现 SaaS 多租户，不新增 `tenant_id`、license server、套餐计费或客户工单系统。

客户差异通过声明式客户配置表达：品牌公开资源、模块状态、页面/字段投影、编号、打印、角色责任池和批准的流程 manifest。客户配置不能改 schema、提升后端 RBAC、绕过领域状态机或改变库存/质检/出货/财务事实底线。

当前稳定客户 key 是 `yoyoosun`。永绅是种子客户和验证来源，不是 Product Core 规则。新增客户复用同一代码和 migration，使用独立数据库、配置包和部署包，不创建客户分支。

`reference-customer` 只是一套中性 draft/preview 工程参考，用于验证已有配置纵向链路和新增客户边界；它不是当前客户、生产实例、真实资料来源、release evidence 或客户签收。仓库不创建 `deployments/reference-customer/`，其部署参数示例复用 `config/private-deployment-template/` 与唯一生产 Compose。

### 业务对象退出与纠错

当前业务模块不提供通用回收站。主数据默认走启用/停用，源单据走取消、关闭或归档，事实/台账走取消、冲正、调整或只读。只有允许删除的草稿 / 误建记录才能进入对象专项评审；后端必须检查状态和引用关系，被引用或已生效时拒绝删除并返回可读原因。

“不再活跃”用筛选、状态过滤或归档视图表达；已归档记录仍然是有效历史，归档记录不进入回收站，不使用“回收站 / 还原删除”语义表达取消归档。必须清理脏数据时走管理员级数据修复 / 合并 / 迁移专项，保留审计和可重放脚本。

## Workflow 与 Fact

Workflow 只负责任务、责任池、流程实例/节点、提醒、阻塞、退回和必要的协同派生。ProcessRuntime 的节点类型收窄为人工任务、审批、白名单领域命令、等待事件和结束节点；它不是通用 BPMN。

生产排程和生产异常各有唯一业务生产者：生产订单从 `DRAFT` 下达到 `RELEASED` 时，在同一领域事务生成 PMC 排程任务；来源完工事实创建并过账返工事实时，在同一领域事务生成生产异常任务。新的出货财务放行不再由 `operational_fact.submit_shipment_release` 生成仓库任务，而由 `finished_goods_delivery` active revision 在质量关口后创建 `workflow.task.approve` 任务；审批通过后，绑定领域命令原子写入 Shipment 财务放行门禁。历史 `shipment_release` 只保留读取和既有协同语义，公开 producer 已退出。

三类任务完成、阻塞或退回只收口协同结果：排程完成投影为生产执行中，生产异常完成投影为已安排处理，出货放行完成投影为 `shipping_released`；均不创建、重过账、冲正或取消 Production / Quality / Inventory / Shipment / Finance Fact。来源对象随后发生真实终态时，领域事务会把同一来源投影继续推进：生产订单关闭 / 取消为 `closed / cancelled`，返工事实取消为 `cancelled`，出货单真实出货为 `shipped`，草稿取消或已出货冲正为 `cancelled`。该来源投影不修改既有 task status，不能把来源终态伪装成任务曾经 `done` 或 `rejected`。

出货草稿取消是源单生命周期，不是库存冲正：没有放行任务时可直接取消且不造任务；存在 `ready / blocked` 放行任务时 fail closed；任务已 `done / rejected` 时允许取消，但保留任务结论和退回原因。只有曾经 `SHIPPED` 的出货单取消才写库存 `REVERSAL`。`ship_shipment / cancel_shipment` 无论当前来源状态都统一要求 `shipments / inventory / workflow_tasks` 三模块，避免服务层预读状态与事务内真实状态之间形成门禁竞态。

公开 `create_task` 使用精确顶层 allowlist，只允许当前正式创建字段；`config_revision / process_instance_id / process_node_instance_id`、`customer_key`、receipt / CAS 字段和任意未知字段都会被拒绝，流程锚点只能由 ProcessRuntime 内部生成。该收紧只阻止静默忽略，不给 `create_task` 增加幂等或网络 replay。五个正式 Workflow 写入口都会在模块状态查询前按各自精确合同拒绝 `customer_key` 和未知字段；`workflow_tasks` 模块门禁只使用服务端部署上下文解析客户，不采信请求参数。正式前端任务动作必须走受控 action API，后端同时校验 RBAC、owner/assignee、责任池、客户 scope、模块状态和任务终态。PMC 与生产岗位具备退回各自来源任务的能力，但仍受相同 owner / assignee、责任池和客户 scope 约束；未指派任务的 owner 资格必须由同一角色在 active revision 中同时满足启用 profile、当前动作 entitlement 和当前客户 scope，不能把一个角色的账号级动作权限与另一个角色的 owner 身份拼接使用。

存量来源任务修复只允许使用 `server/cmd/backfill-workflow-source-tasks`：先执行默认事务 dry-run，再在核对数据库名与统计结果后显式 `--apply --confirm-database=<精确数据库名>`。它只补当前仍为 `RELEASED` 的生产订单和当前仍为 `POSTED REWORK` 的返工事实，遇到确定性编号被占用或 task / created event / business state 包不完整时整批失败；新建任务保持 `ready`，不推断历史 `done / rejected`。`DRAFT` 出货单不回补、不猜测曾经提交，仍必须由业务人员在出货页显式提交放行。

普通 `workflow_tasks` 以 `version` 作为乐观锁真源。正式完成、阻塞、退回和催办必须携带页面读到的 `expected_version` 与顶层 `idempotency_key`；repo 在同一事务内按任务 ID、旧状态和版本执行 CAS，并把新 task version、服务端计算的 intent hash、命令 key 与首次任务返回快照写入 `workflow_task_events`。receipt 使用稳定的 `workflow.task-mutation-result/v1` 合同；writer 在落库前校验 V1 DTO，reader 要求存储的 key、command、任务状态为 canonical，并验证任务 / 非空业务状态属于正式字典，内部一致但非法或带空格的损坏快照也会 fail closed。相同 task、key、actor、命令和业务 intent 的网络重放会在终态 / 版本校验前返回首次快照，不重复事件、催办计数、break-glass 成功审计、业务投影或派生任务。同 key 改 intent 返回 `40920`，新 key 操作终态任务仍按 settled 拒绝；公开 `create_task` 不在该 replay 合同内。

任务归属调整使用独立的 `get_task_assignment_options / reassign_task` 合同。`workflow.task.assign` 当前默认只授予 `boss`；`super_admin` 通过全权限兜底可以执行转交，但不会因此自动成为任务负责岗位、接收候选人或业务办理人，`pmc` 的跨岗监督仍保持只读，普通 `admin` 默认没有该权限。转交只接受 `ready / blocked`，必须提交非空原因、当前 version、幂等键，以及明确的接收人 ID 或 `assignee_id=null` 退回原岗位待办池。接收人必须是 active 账号，直接持有任务 `owner_role_key`，并在任务冻结 revision 中同时具备读取、更新和完成 / 审批能力；当前处理人、停用账号、仅有 super admin 身份但未持有负责岗位的账号都不会进入候选列表。repo 在事务内再次锁定并核对账号、岗位关联、岗位权限、任务 version、状态和原处理人，只更新 `assignee_id / updated_by / version`，同时写 `reassigned / unassigned` 事件 receipt 与运行审计；任务状态、负责岗位、责任池、payload、ProcessRuntime 锚点和任何 Fact 都保持不变。

桌面任务看板使用 `get_task_board` 只读投影，不再以 `list_tasks(limit: 200)` 的当前页长度推断总数。后端在同一可见性与筛选口径下返回全量 `total`、互斥的常规待办 / 阻塞与退回 / 到期提醒 / 已结束计数和每栏有界分页；四项之和必须等于 `total`。`approval_only=true` 只按 `required_capability_key=workflow.task.approve` 从同一完整可见集合筛选 `ready / blocked` 审批任务，不再识别老板岗位、销售来源或历史 task group。所有 ProcessRuntime approval 节点必须声明该 capability；老板和财务分别通过 active revision 的责任池、entitlement、owner / assignee 与任务 version 取得处理资格。`rejected` 仍是不可操作、不计 due 的终态，但可在看板中只读投影到阻塞 / 退回泳道承接交接。

前端把一次用户 intent 的业务参数、`expected_version` 和 `idempotency_key` 冻结为同一个 attempt；HTTP 408、网络中断、5xx 或结构不合法的 success response 都按“结果未知”处理，保留同一 key 继续读取 / 重放 receipt，不刷新列表或关闭输入。五类正式任务入口已接 task 级同步 in-flight lease，同一 task 的跨动作重复点击在请求发出前互斥，且只由持有该 lease 的调用在 `finally` 释放。移动岗位任务的完成、阻塞、退回和催办按钮以后端 `explain_action_access` 投影为本地执行真源，提交前仍会再做一次新鲜的单动作校验；退回使用 `workflow.task.reject`，不再由任务类型启发式隐藏。Go 与 JS 已共同消费 `scripts/qa/workflow-task-mutation-intent-v1.vectors.json`，mixed evidence 类型 / 顺序、raw whitespace key、transport-only equality 与 changed-intent relations 均由同一 golden 锁定；定向证据不代表 final full/strict/L1 已完成。

`20260711063237` 与 `20260711075355` 已在本地隔离 migration chain 执行并固定为不可变 revision；`20260711104729` 新增 PostgreSQL / SQLite 均可生成的 receipt bundle 完整性 CHECK；`20260711204000` 只把本项目迁移前可能落入两张投影表的提醒值 `shipment_release_pending` 规范为正式业务状态 `shipment_pending`，不修改 payload 中同名 notification / alert 类型。对本项目在 receipt migration 前已经存在、无法证明准确版本的事件，`task_version` 保持 `NULL`，不按行号或当前任务版本伪造回填；这不是旧项目、旧客户端或旧 API 兼容路径。历史 PostgreSQL fresh / upgrade 的执行数量只保留在 `progress.md` 过程证据，不作为本文常驻总数；latest revision、执行数量和 pending 必须按当前代码与 Atlas status 重查。133 较早固定 V5 的目标 migration 已到 `20260722000505`，但当前 HEAD 后续 migration（当前 latest 为 `20260723155358_reconcile_permission_assignment_boundaries.sql`）尚未整体发布；不得把本地 latest 或旧目标数量写成当前已发布事实。

`task done`、流程节点完成和业务状态投影都不等于 Fact posted。库存、质检、出货和财务必须调用对应领域 usecase，并保留幂等、状态、事务和审计证据。

显式 SKU 已贯通销售订单行、库存批次 / 流水 / 余额、生产 / 委外事实、出货与预留。`product_sku_id=NULL` 仍是合法且独立的未分规格产品库存，不能自动回填、猜测归类或作为 SKU 库存不足时的兜底。BOM SKU 粒度、客户导入自动建 SKU 和旧库存重分类工具仍未接入。

ProcessRuntime 的 domain command 先做 handler 只读预检，再以单条条件 `UPDATE ... RETURNING` 绑定不可变 SHA-256 intent fingerprint。写领域命令在各自领域事务内同时写 side effect 与 durable result / effect ref；销售审批后的 `SUBMITTED → ACTIVE`、采购审批后的 `SUBMITTED → APPROVED` 和出货财务放行均由 approval 节点完成后的受控自动命令执行，公开 JSON-RPC 不再暴露销售提交 / 激活、采购直批或财务放行旁路。销售、采购的生命周期更新与 ProcessRuntime durable result 在同一事务提交；出货单的财务放行以 `finance_release_status / version / released_at / released_by / process anchors` 作为 Shipment 领域门禁并与 durable result 同事务提交，`ship_shipment` 和流程出货命令都必须读到 `APPROVED`。Workflow task 更新与随后 ProcessRuntime 对账仍是可重试的两个持久步骤，任务成功但命令结算失败时保持待恢复证据，不伪造领域完成。`quality_inspection.aggregate_gate` 继续只评估正式 Quality Fact。重试先校验并重放持久结果，不再次执行领域副作用。

重复 `StartProcessInstance` 只做基于已持久化状态的有界恢复：补建或核对 active human / approval 节点的 linked task，补齐 completed end 的流程结算，并继续 durable domain result 的节点结算；不会自动重复 active domain command，也不会唤醒 wait event。显式阻塞在同一 PostgreSQL 事务内同时更新节点和 ProcessInstance，任一更新失败整体回滚，不留下节点已阻塞而流程仍 active 的分裂状态。

销售订单取消、采购入库取消冲正、已出货取消冲正和财务事实取消会在各自领域事务内把对应 effect 标记为 compensated；正式认证 JSON-RPC 主路径会把当前管理员写入 `domain_command_compensated_by`。财务事实取消同时永久保留原 `posted_at`，记录 `cancelled_at / cancelled_by / cancel_reason`；本项目迁移前已取消但无法还原审计的行只读显示“历史记录，取消审计信息缺失”，不猜造数据。active 节点读到补偿结果时会阻塞流程，已完成节点在补偿后重放会返回 `40921` 且不再推进下游，不把已撤销事实冒充成功。若补偿发生前下游已激活，当前只保留补偿证据并失败关闭，不自动回滚既有下游，后续必须走明确恢复决策。migration `20260710150000` 只给升级前 active 空 fingerprint 写 fail-closed sentinel；合并 migration `20260710150001` 才增加 protocol / result / hash / effect-ref / compensation schema，并把本项目迁移前已有 fingerprint 标为 protocol 0。本项目迁移前的 protocol 0、已提交销售订单但缺 exact result，以及省略判定时间且无法精确证明的迁移前成品质检 result-missing 仍返回 `40921` 人工核对。这两条 schema migration 已处于较早固定 V5 的目标 apply 范围；原子回滚、恢复与并发仍以本地 PostgreSQL 测试为证，当前 HEAD 的后续能力、目标重发和客户 UAT 另验。

流程命令 payload 只保留真实生效字段：入库过账、财务放行、显式出货都只接收对应业务主引用；成品质检执行人取认证 actor；应收和发票的 customer 必须与已出货 shipment 的 customer 真源一致，不能由 caller 自报或从 Workflow payload 派生。

## 角色、页面与客户配置

后台权限真源是用户、角色、权限码和 `is_super_admin`。用户可以持有多个角色；客户岗位差异优先用多角色和责任池表达，不能让流程节点写死员工或公司岗位名。

普通账号的有效页面取后端 RBAC 菜单、active revision 全局页面和角色页面投影的交集；有效动作取后端 RBAC、active revision entitlement、角色 revoke 和模块状态的交集。`role_profiles` 只保留角色启停、能力包标记和撤权；字段按已登记 surface 投影。客户配置只能收窄，不能增加账号本来没有的后端权限。部署固定了非 demo `ERP_CUSTOMER_KEY` 时，缺少 active revision 必须 fail closed，普通账号和 super admin 都不能回退到 RBAC-only 执行业务动作；系统修复和客户配置控制面仍按独立权限处理。super admin 用于产品核心/客户系统审阅和配置，但后端业务写动作仍受领域 usecase 约束。

永绅“财务根据工程表下采购合同”采用同一用户 `finance + purchase` 多角色；不把采购权限加入所有客户的财务核心角色。角色、菜单、字段、状态和流程完整表见 `docs/product/多甲方角色能力与流程编排.md`。

## 永绅当前客户线索

- 加工合同包括外发车缝、布料加工和手工加工：复用 `outsourcing_orders`，明细主体为产品或材料二选一，工序类别表达差异。
- 甲方已确认高层生产主线为 `布料加工 → 裁片检验 → 车缝 → 皮套检验 → 手工 → 成品检验 → 针检 → 抽检 →（订单要求时）客户验货 → 包装 → 成品入库`，其中车缝必须先于手工，两道工序分别由生产经理决定本厂或外发。当前 Product Core 本地源码已以固定 `PLUSH_SEW_HAND_V1` v1、工序主档标准位置绑定、订单工序快照、WIP 批次 / 事件、分段 Quality Fact、包材业务确认、JSON-RPC / RBAC 和生产订单页工序办理入口承接；完整迁移链已在一次性 PostgreSQL 18 隔离库和登记的本地验收库验证。migration 不会自动创建工序标准位置绑定，固定路线使用前仍须通过工序页或受控 seed 显式绑定。133 较早固定 V5 已有该路线的 migration、配置激活和岗位 / 页面技术检查记录；当前 HEAD 后续异常切片未整体重发，客户 UAT / 签收未完成。详细口径见 `docs/architecture/生产工艺路线与在制品边界评审.md`、`docs/customers/yoyoosun/需求线索.md`与 `docs/customers/yoyoosun/流程编排闭环矩阵.md`；只有 `processes.production_route_operation_code` 绑定固定路线位置，名称、类别、普通编码和 `sort_order` 都不是路线真源。
- 采购合同包括主料和其他材料：复用采购订单，需求来源应追溯到工程/BOM或明确人工批准来源。
- 仓库包括主料仓、成品仓、其他仓；质检待检仓可独立保留。它们是仓库主数据类型/名称，不是客户 schema。
- 受控客户来源中的汇报材料和表单只作为业务线索；存在疏漏时必须用领域边界补齐，不能照搬成运行时规则。

## 客户资料、公开资源与导入

客户原始 Excel/PDF/图片、工程图和员工信息保留在受控客户资料边界，不进入 Product Core 或公开前端产物。前端只可发布 `config/customers/<key>/public-assets/` 和经过审查的品牌/菜单配置。

长期边界是“一客户一专属 Private 资料仓库或经确认的等价受控存储”。永绅使用独立 Private Git 仓库保存原件、manifest、来源评审和私有验证入口；Product Core 只保存通用工具、合成 fixture 和脱敏类别映射。依赖方向由客户仓库固定产品版本并执行兄弟目录或 CI multi-checkout 校验，Product Core 不把客户仓库作为 submodule，也不在普通构建 / CI 中读取私有资料。

Product Core 当前提交树与索引已移除旧原件目录、真实 manifest 和客户工程指导图片；客户原件数量、hash / size、结构化提取和远端回读结果只以永绅专属 Private 仓库的 manifest、README 和正式验证记录为准，本仓库不硬编码易漂移的客户仓提交号。可变产品版本锁定的唯一真源是客户私有仓的 `product.lock.json`；每次 Product Core `HEAD` 推进后，必须在两边均已提交且工作树清洁时更新该锁，并以 `FORMAL_PRODUCT_PIN=1` 运行私有仓 `scripts/validate.sh` 后才能声明当前版本已固定。

当前已完成的是 Product Core 当前树的客户原件隔离；仍未完成的是既有 Git 历史中旧副本的单独治理、后续 Product Core 新提交的重新锁定与远端回读，以及真实导入批准和客户签收。`config/customers/yoyoosun/` 与 `deployments/yoyoosun/` 仍保留非原件配置与部署资料，本轮不做运行时客户配置或部署体系迁移。

当前没有经客户确认、可执行的永绅真实数据。`scripts/import/` 只提供 manifest、只读 extract、freeze 和 dry-run；它们不连接后端/数据库，且 `canExecuteRealImport=false`。仓库没有真实客户导入执行器，模拟 seed/fixture 不能写成真实导入结果。

未来真实导入必须另行实现通用批次 usecase、RBAC、审计、幂等、失败恢复和对账，不能为某客户恢复隐藏 `--execute` 脚本。

## 打印边界

采购合同、加工合同和工程资料打印复用正式模板工作台。客户配置只提供买方抬头、模板选择和公开资源；供应商、产品/材料、数量、单价、金额和业务行必须来自 Source Document/正式映射，不由模板默认值伪造。

打印预览/生成 PDF 不等于签章、合同批准、入库、结算或事实过账。源单提交后打印使用冻结快照；客户原始 Excel/PDF/图片只作模板输入证据。

## 前端入口

- 正式 ERP 菜单只显示可用或明确受控的业务页；开发、治理、文档和原型入口只在 `/__dev/**`。
- 岗位任务端使用 `/m/<role>/tasks`；当前业务角色为 boss、sales、purchase、production、warehouse、quality、finance、pmc、engineering。
- 岗位使用帮助使用 `/erp/help-center` 单一登录态入口，按当前有效岗位显示常用入口、正常案例、完成标准、异常处理和退回对象；多岗位账号必须显式切换当前办理岗位。永绅客户菜单采用 `role_guided` 视觉分层：已授权看板统一放在最前的“看板中心”，岗位帮助固定进入“更多功能”；“常用工作”默认由系统按岗位推荐，普通岗位约 3 个，财务为应付、应收、发票、对账 4 个。管理员可在权限管理为业务岗位切换“系统推荐 / 自定义常用”，自定义从当前最终可进入页面中选择并排序 1–5 个。岗位布局随角色版本持久化并写审计，但不授予 RBAC、客户页面投影、页面内操作或直接路由权限。权限中心“可用功能”按后端菜单合同和客户有效投影展示未保存草稿的页面结果、缺少入口及常用 / 更多位置；权限分组名称由后端注册表统一返回，BOM、客户退货、生产执行、敏感字段等不再落入多个同名“其他功能”，未知分类只合并为一个“未分类功能”。桌面使用吸顶分类直达和已选 / 总数，手机使用“跳到功能分类”下拉，只保留“只看已选”复核，不提供低频全文搜索；分类导航只定位页面，不改变权限。主办理页面明确且入口唯一时只补齐该入口，不连带开启其他页面。草稿预览不落库、不写审计，也不替代保存回读和目标账号实登。“页面与导航”继续承接完整页面解释、布局编辑和导航预览。旧 docs/qa 路由、formal-shell 假页面、开发验收页和旧权限别名仍不恢复。
- 用户可见文案使用业务语言，不展示 Workflow/Fact、raw id、内部 key 或底层英文异常。

## 测试与发布

工具链版本以根 Node 版本文件、`web/package.json#packageManager` 和 `server/go.mod#toolchain` 为准。使用项目指定 Node/pnpm/Go；QA 不应依赖调用者偶然安装的更高版本。

- `scripts/qa/fast.sh`：开发期高频合同和边界检查。
- `scripts/qa/full.sh`：提交前完整门禁。
- `scripts/qa/strict.sh`：发版前严格检查，warning 视为失败。
- 页面/样式改动按影响面运行浏览器回归并保存截图证据。

生产只使用 `server/deploy/compose/prod`。镜像在本地/CI 构建并使用固定 tag/digest；低配服务器只负责加载/拉取、migration、启动、健康检查和 smoke。`ERP_CUSTOMER_KEY` 由部署环境固定，业务请求不能切换到其他客户配置。migration 的 status → dry-run → apply 必须持有同一把部署锁。

本地测试通过不等于目标环境已发布。真实发布仍需固定版本、preflight、backup/restore、migration、smoke、回滚/前向修复和签收 evidence。
