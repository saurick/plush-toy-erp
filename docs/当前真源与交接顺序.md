# 当前真源与交接顺序 / Current Source Of Truth And Handoff Order

本文档只解决一件事：当前这份仓库到底应该先读哪里，才能避免把历史占位、现场猜测或过期文档误当成真源。

## 本文职责边界

本文是当前真源索引和交接入口，不是完整能力台账、roadmap、测试报告或实现流水。它只保留会影响阅读顺序、当前判断和禁区边界的摘要；具体能力成熟度、客户可见性、证据和代码行为必须继续回到对应真源交叉确认。

| 你要判断什么 | 先看哪里 | 本文怎么用 |
| --- | --- | --- |
| 运行时到底怎么做 | 当前代码、Ent schema、Atlas migration、自动化测试 | 本文只指路和提醒禁区，不替代代码和测试 |
| 某个能力做到哪一层、能不能试用 | `docs/product/产品能力进度台账.md` 和 `docs/product/产品能力证据详情.md` | 本文只保留最容易误判的状态摘要 |
| 客户能否看到、试用或验收 | `docs/customers/<customer-key>/客户交付矩阵.md` 和 `docs/customers/<customer-key>/客户差异台账.md` | 本文只说明客户资料和 Product Core 的边界 |
| 菜单、入口、入口壳和客户菜单差异 | `docs/product/正式产品入口与菜单配置计划.md`、当前前端 / 后端菜单代码 | 本文只记录正式入口和入口壳的交接口径 |
| 新模块进入实现前怎么拆 | `docs/product/模块实施治理.md` | 本文只作为阅读顺序的一环 |
| 历史记录或外部建议能不能当真源 | `docs/archive/**`、`docs/reference/**` 只能作线索 | 本文提醒必须回到当前代码、测试和正式文档复核 |

如果本文与代码、测试、产品能力台账或客户交付文档出现冲突，先按当前代码和测试定位真实行为，再按影响范围修正文档；不要让本文中的旧摘要继续覆盖当前 runtime truth。

这里说的信息差，主要指读者只看到本文中的一句状态摘要，就误以为对应能力已经完成所有 schema、migration、repo/usecase、API、RBAC、UI、客户可见性和验收证据。处理方式不是把所有细节复制到本文，而是把本文保持为索引，并在对应台账、专题文档、代码和测试里保留可核对证据。

判断模块是否可交付时，必须同时看 `docs/product/产品能力进度台账.md` 的“模块交付门禁表”和对应客户交付矩阵；页面存在、菜单出现或 API 可调用都不能单独证明已经交付。

## 真源原则

- 运行时行为的最终真源始终是代码。

- 历史 changes 文档不再作为阅读入口；当前状态必须回到本文档、正式能力账本（`docs/product/产品能力进度台账.md`）、当前代码和当前测试交叉确认。
