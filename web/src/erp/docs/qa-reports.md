# 专项报告

> 目的：把当前值得长期关注的验收专项放在一个聚合入口里，避免左侧菜单继续膨胀，也避免把还没稳定的专项伪装成已落地功能。

## 1. 当前结论

- 当前专项报告是聚合说明页；字段联动覆盖已经单独升级为读取 latest JSON 的状态页。
- 当前不把字段联动、打印口径、权限边界、通知预警、移动端 smoke 等拆成更多左侧菜单。
- 每个专项只有在有稳定测试、明确真源和可读摘要后，才升级成页面卡片。

## 3. 链路覆盖矩阵专项报告口径

| 分类              | 报告口径                                                                                                                                           | 统计规则                                           |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| v1 主干闭环       | 订单审批到工程资料、采购 IQC 入库、委外回货入库、成品入库出货、出货应收开票、采购/委外应付对账                                                     | 允许显示为“已接入 v1”，但必须同时显示当前盲区      |
| deferred 扩展链路 | BOM 材料需求、订单变更、排产分派、欠料补料、发料领料、库存盘点、返工重检、售后退货、收付款、发票异常、成本毛利、供应商绩效、权限审计、完整通知中心 | 只能显示为 deferred 或 partial，不能纳入已完成统计 |
| out_of_scope      | 固定资产 / 低值易耗品、总账 / 凭证 / 纳税申报、PDA / 条码枪 / 图片识别、复杂低代码表单设计器、任意 SQL 控制台                                      | 只作为边界提醒，不进入本轮通过率                   |

链路覆盖矩阵的结论必须避免“全量业务链路已覆盖”这类表述。当前页面和报告只说明 ERP v1 主干闭环已经接入，扩展链路继续按 deferred / partial 管理。

## 4. 当前专项验收口径

| 专项                        | 至少要验什么                                                                                                                   |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| 字段联动                    | 新值覆盖旧值、来源切换清值、真源缺失不伪造、快照缺失按口径回补                                                                 |
| 打印快照                    | 编辑区和纸面同步、分页稳定、PDF 预览可打开、历史快照不被静默改                                                                 |
| workflow 状态字典           | 前后端状态一致，任务创建、更新和业务状态 upsert 不拒绝合法 key                                                                 |
| 角色权限矩阵                | 普通管理员只看到授权菜单，旧路径别名能归属到当前权限                                                                           |
| 通知 / 预警 / 催办          | due soon、overdue、critical、blocked、finance_pending、urged、escalated 等预警能按角色可见，并能在 `workflow_task_events` 追溯 |
| 协同任务可见性 / 角色池诊断 | `owner_role_key`、`task_group`、终态、PMC / boss / production 扩展可见性和任务加载策略可解释，页面不提供修复按钮               |
| 错误码同步                  | 服务端真源、前端生成码表、消费层和调用方保持一致                                                                               |
| 业务闭环覆盖                | 6 条 v1 主干闭环能从业务记录推进到任务、状态、事件和移动端可见性，同时保留扩展链路 deferred 边界                               |
| 移动端角色 smoke            | 角色入口、登录态、主路由和任务加载主路径可用                                                                                   |
| 后端 schema / migration     | 改 schema 才生成迁移；未改 schema 不伪造 migration 结果                                                                        |
| workflow usecase review     | 明确当前前后端边界、迁入后端候选优先级、迁移风险、回滚策略和第一条试迁建议                                                     |
| industry schema review      | 明确通用表继续使用边界、专表候选、P1 评审对象、数据一致性策略和不生成 migration 的原因                                         |

## 5. 当前不新增的菜单

| 不新增菜单   | 原因                                         |
| ------------ | -------------------------------------------- |
| 数量一致性   | 当前还没有独立结构化报告，先归到专项报告     |
| 打印口径覆盖 | 当前由打印测试和浏览器回归承接，不单独占菜单 |
| 权限边界     | 当前由权限中心和测试承接，不单独占菜单       |
| 通知预警     | 当前归到专项报告和通知预警文档，不额外拆菜单 |
| 并发一致性   | 更适合留在后端测试或 CI，不适合做日常菜单    |

## 6. 后续升级条件

专项升级为自动报告前，必须具备：

- 稳定 runner。
- 结构化摘要。
- 明确失败线索。
- 明确未覆盖范围。
- 可重复执行的验证命令。
