# 专项报告

> 目的：把当前值得长期关注的验收专项放在一个聚合入口里，避免左侧菜单继续膨胀，也避免把还没稳定的专项伪装成已落地功能。

## 1. 当前结论

- 当前专项报告是聚合说明页；字段联动覆盖已经单独升级为读取 latest JSON 的状态页。
- 当前不把数量一致性、打印口径、权限边界、字段联动等拆成更多菜单。
- 每个专项只有在有稳定测试、明确真源和可读摘要后，才升级成页面卡片。

## 2. 当前专项池

| 专项                   | 当前状态                                    | 当前真源                                                            |
| ---------------------- | ------------------------------------------- | ------------------------------------------------------------------- |
| 字段联动 / 残值 / 缺值 | 已有统一 catalog、runner 和 latest 覆盖报告 | `fieldLinkageCatalog.mjs`、`erp-field-linkage-coverage.latest.json` |
| 打印 / 合同 / 快照     | 已有采购合同、加工合同工作台和浏览器回归    | 打印模板测试、合同 smoke、打印快照文档                              |
| workflow 状态          | 已有任务、业务状态和事件表                  | Ent schema、JSON-RPC、任务状态测试                                  |
| 权限边界               | 已有菜单权限剪裁和权限管理页                | 前端权限配置、后端权限 option、权限测试                             |
| 错误码同步             | 已有统一错误码真源和同步脚本                | `server/internal/errcode/catalog.go`、前端生成码表                  |

## 3. 当前专项验收口径

| 专项          | 至少要验什么                                                   |
| ------------- | -------------------------------------------------------------- |
| 字段联动      | 新值覆盖旧值、来源切换清值、真源缺失不伪造、快照缺失按口径回补 |
| 打印快照      | 编辑区和纸面同步、分页稳定、PDF 预览可打开、历史快照不被静默改 |
| workflow 状态 | 状态流转不清空业务字段、阻塞原因可回显、任务状态事件可追踪     |
| 权限边界      | 普通管理员只看到授权菜单，旧路径别名能归属到当前权限           |
| 错误码同步    | 服务端真源、前端生成码表、消费层和调用方保持一致               |

## 4. 当前不新增的菜单

| 不新增菜单   | 原因                                         |
| ------------ | -------------------------------------------- |
| 数量一致性   | 当前还没有独立结构化报告，先归到专项报告     |
| 打印口径覆盖 | 当前由打印测试和浏览器回归承接，不单独占菜单 |
| 权限边界     | 当前由权限中心和测试承接，不单独占菜单       |
| 并发一致性   | 更适合留在后端测试或 CI，不适合做日常菜单    |

## 5. 后续升级条件

专项升级为自动报告前，必须具备：

- 稳定 runner。
- 结构化摘要。
- 明确失败线索。
- 明确未覆盖范围。
- 可重复执行的验证命令。
