# 产品化与交付

> 目的：跟踪这套 ERP 如何从当前甲方项目演进成可卖给同行的通用产品。本文档只记录产品化边界、客户差异隔离和交付能力路线，不实现真正 SaaS 多租户，不新增 `tenant_id`，不改 Ent schema。

## 1. 页面定位

- 本页只属于“开发与验收”模块。
- 本页不进入普通帮助中心主入口，不作为业务用户操作教程。
- 本页不实现真正 SaaS 多租户。
- 本页不实现 `tenant_id`。
- 本页不实现 license server。
- 本页不实现套餐计费。
- 本页不实现客户工单系统。

## 2. 产品定位

- 本系统长期目标是面向毛绒玩具及相近轻工制造企业的 ERP 产品。
- 当前甲方是第一个标杆客户 / 种子客户。
- 后续可能作为 SaaS / 私有化部署产品卖给同行。
- 后续按年收维护费。
- 当前阶段优先做单客户私有化版本，不提前实现复杂 SaaS 多租户。

## 3. 产品化原则

| 原则              | 口径                                                                           |
| ----------------- | ------------------------------------------------------------------------------ |
| 一个通用产品内核  | 核心 usecase、schema、权限、事实规则和帮助中心通用口径尽量沉淀为行业产品能力。 |
| 客户差异配置化    | 公司名、logo、菜单、字段显示、编号规则、打印模板和初始化数据优先用配置表达。   |
| 不写死当前甲方    | 不把当前甲方名称、logo、特殊字段、特殊流程或特殊报表写进核心 usecase。         |
| 极端定制隔离      | 真正客户专属逻辑进入 customer-specific / extension 边界，并记录原因。          |
| 核心事实规则统一  | 库存、出货、财务、状态机、迁移路径和权限安全边界不能按客户随意分叉。           |
| 先私有化，后 SaaS | 当前先稳定单客户私有化，再评审多客户私有化，最后才评审 SaaS 多租户。           |

## 4. 租户 / 客户分层预留

本轮只做文档与约束，不实现真正多租户。

### Product Core 通用产品内核

Product Core 是 `server / web / docs` 中的通用 ERP 能力，包括：

- MasterData
- Workflow
- Inventory
- Quality
- Purchase
- Shipment
- Finance
- RBAC
- API / UI
- Help / QA

Product Core 不允许写死当前甲方名称、logo、特殊流程、特殊字段或特殊报表。

### Industry Template 行业模板

Industry Template 面向毛绒玩具 / 相近轻工制造企业，沉淀默认角色、菜单、流程、字段、帮助文档和 seed template。

- 当前甲方可以作为第一个行业模板样本。
- 当前甲方样本不能直接等同通用产品真源。
- 通用口径必须沉淀到正式架构文档、schema、usecase、测试和帮助中心中。

### Tenant Config 客户 / 租户配置

Tenant Config 用于承接每个客户的差异项：

- 公司名
- logo
- 主题色
- 菜单开关
- 模块开关
- 字段显示 / 必填
- 编号规则
- 打印模板
- 角色模板
- 权限模板
- 初始化数据
- 默认仓库
- 默认单位

后续建议目录可以预留为：

```text
config/tenants/<tenant-key>/
deployments/<tenant-key>/
docs/customers/<tenant-key>/
```

本轮只在文档中说明，不创建复杂目录树。当前甲方资料目录隔离后续应单独做评审，不在本轮移动、重命名或大规模重构。

### Customer Extension 客户扩展层

只有极端客户专属逻辑才进入 customer-specific / extension 边界。

- 客户扩展不得修改核心库存、出货、财务事实规则。
- 客户扩展不得让核心 schema 为单个客户长期分叉。
- 客户扩展必须在产品化与交付文档中记录原因、范围、退出条件和维护责任。

### Runtime Tenant 多租户运行时

当前阶段不实现 Runtime Tenant：

- 不新增 `tenant_id`。
- 不改 Ent schema。
- 不改 RBAC 为多租户模型。
- 不做租户隔离中间件。
- 不做套餐计费。
- 不做 license server。
- 不做客户工单系统。

后续只有当多客户 SaaS 明确启动时，才单独评审 `tenant_id`、数据库隔离、对象存储隔离、用户隔离、权限隔离和升级策略。

## 5. 推荐部署形态

### 第一阶段：单客户私有化部署

- 单仓库
- 单租户
- 单客户私有化部署
- 通过 env / config / seed template 区分客户

当前阶段优先完成这一阶段，不提前实现第三阶段能力。

### 第二阶段：多客户私有化部署

- 多客户私有化部署
- 每个客户一套数据库 / 对象存储 / 配置
- 代码版本统一

### 第三阶段：SaaS 多租户

- `tenant_id`
- 租户隔离
- 套餐计费
- License / 授权
- 统一升级

当前不要提前实现第三阶段。

## 6. 通用产品能力

后续产品内核应逐步沉淀这些通用能力：

- MasterData
- Workflow
- Inventory
- Quality
- Purchase
- Production
- Shipment
- Finance
- RBAC
- Help / QA / Debug

这些能力应优先服务毛绒玩具及相近轻工制造行业，而不是只服务当前甲方的一套固定习惯。

## 7. 客户差异配置

### 建议优先配置化

- 公司名
- logo
- 主题色
- 菜单开关
- 模块开关
- 字段显示 / 必填
- 编号规则
- 打印模板
- 角色模板
- 权限模板
- 初始化数据
- 默认仓库
- 默认单位
- 帮助中心客户版说明

### 不建议客户级随意配置

- 库存流水语义
- 出库 / 入库 / 冲正规则
- 财务事实规则
- 核心状态含义
- 数据库结构
- 迁移路径

客户差异优先通过 tenant config、feature flag、初始化模板、打印模板、字段显示配置、菜单开关和客户扩展层隔离，避免污染核心 usecase。

## 8. 仓库与部署建议

### 推荐

- 一个通用产品主仓库
- 多个客户部署配置
- 必要时少量客户扩展模块
- 极端定制才考虑独立仓库

### 不推荐

- 每个客户复制一份完整代码仓库长期维护
- 每个客户一套不同 schema
- 每个客户分叉核心库存 / 财务规则

当前仓库内已经存在与当前甲方深度耦合的资料、样本、截图、字段口径或业务说明时，本轮只记录“后续需要隔离”，不做目录迁移。后续隔离应单独评审文件清单、引用关系、docs registry、测试断言和回滚风险。

## 9. 维护费交付能力

后续需要逐步补：

- 版本号
- 升级日志
- migration 执行说明
- 备份恢复脚本
- 巡检清单
- 客户问题记录
- 补丁记录
- 部署文档
- 培训文档
- 帮助中心业务版说明

这些交付能力应进入产品化 / 私有化交付边界，不应散落在业务 usecase 中。
